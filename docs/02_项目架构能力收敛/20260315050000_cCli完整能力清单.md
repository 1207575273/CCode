## 一、产品定位

**CCode** 是一个开源的、支持多模型的 AI 编程 CLI 助手。对标 Claude Code 的交互体验，但不绑定单一 LLM 提供商。

**目标用户**：需要在终端环境中使用 AI 辅助编码、调试、重构的开发者。

**核心价值**：自由选择 Claude / GPT / Gemini / GLM / DeepSeek / 本地 Ollama 等任意模型，获得一致的 AI 辅助编码体验，同时拥有完全的可控性和可扩展性。

## 二、三种运行模式

CCode 通过启动参数自动判断运行模式：

### 2.1 交互模式（REPL）— 默认模式

```bash
ccode              # 进入交互式对话
ccode --web        # 交互模式 + Web Dashboard
ccode --resume     # 恢复上一次会话
```

- **触发条件**：无 prompt 参数
- **特性**：Ink/React 终端 UI、流式输出、工具交互确认、多轮对话
- **支持选项**：`--model`、`--provider`、`--resume [sessionId]`、`--web`

### 2.2 管道模式（Pipe）— 非交互

```bash
ccode "这段代码有什么问题"                    # 单次问答
cat error.log | ccode "分析这个错误日志"       # stdin 管道输入
ccode -p "生成 API 文档" --json               # JSON 结构化输出
ccode "跑测试" --yes                          # 自动批准工具执行（CI 场景）
```

- **触发条件**：有 prompt 参数（位置参数 / `-p` 标志 / stdin）
- **特性**：纯文本流式输出到 stdout，执行完自动退出
- **支持选项**：
  - `-p / --prompt` — 指定问题
  - `-m / --model` — 指定模型
  - `--provider` — 指定供应商
  - `--yes / -y` — 自动批准所有工具调用（适用于 CI/CD）
  - `--no-tools` — 禁用工具调用，纯对话
  - `--json` — JSON 格式结构化输出（含 response + usage + cost）
  - `--verbose / -v` — stderr 输出工具执行进度
- **权限链**：项目级白名单 → `--yes` 全放行 → 拒绝
- **实现**：`src/core/pipe-runner.ts`

### 2.3 Web 桥接模式（Bridge + Dashboard）

```bash
ccode --web        # 启动 CLI + Bridge Server + Web Dashboard
```

- **触发条件**：`--web` 标志
- **工作流**：
  1. 第一个 CLI 实例：启动 Bridge Server（默认端口 9800）+ Vite 开发服务器
  2. 后续 CLI 实例：连接已有 Bridge Server（作为 WebSocket 客户端）
  3. 所有 CLI 实例平等连接，按 sessionId 隔离消息路由
- **端口**：默认 9800，冲突时自动递增（9801 → 9802），最多重试 3 次
- **Web 访问**：浏览器打开 `http://localhost:9800`

详见「第九节 Web 能力全景」。

| 传输 | 说明 |

- `/mcp` 指令查看连接状态

## 九、Web 能力全景

### 9.1 架构

```
Browser (React SPA)
  ↕ WebSocket
Bridge Server (port 9800) — 纯消息路由器
  ↕ WebSocket             ↕ WebSocket
CLI-1 (session-aaa)     CLI-2 (session-bbb)
```

## 十、Skills 系统

### 10.1 Claude Code Skill 生态兼容

CCode 的 Skills 系统**有限度地兼容** Claude Code 的 Skill 格式：

| 特性 | Claude Code | CCode | 兼容 |
|------|------------|-------|------|
| SKILL.md 格式 | YAML frontmatter + Markdown body | 相同 | ✅ |
| `name` / `description` 字段 | 必填 | 必填 | ✅ |
| `allowed-tools` 字段 | 数组 / 逗号分隔 | 两者均支持 | ✅ |
| `user-invocable` 字段 | 布尔值 | 布尔值 | ✅ |
| 驼峰写法 (`allowedTools` / `userInvocable`) | 不支持 | 同时支持 | ✅ 超集 |
| 发现路径 `~/.claude/skills/` | 默认 | 不扫描 | ❌ 需复制到 `~/.ccode/skills/` |
| superpowers 等社区 Skill | 原生支持 | SKILL.md 格式兼容可直接使用 | ✅ |
| Skill 支撑文件（同目录非 SKILL.md 文件） | 支持 | 支持 | ✅ |

**迁移方式**：将 Claude Code 的 `~/.claude/skills/` 下的 SKILL.md 目录复制到 `~/.ccode/skills/` 即可使用。社区 Skill（如 [skills.sh](https://skills.sh/) 275+ Skill）只要遵循 SKILL.md 格式均可直接兼容。

> 限度说明：CCode 不扫描 `~/.claude/` 目录，需要手动复制迁移；部分 Claude Code 专属的 Skill 内部引用（如特定内置工具名差异）可能需要微调。

### 10.2 发现路径（优先级从低到高，同名覆盖）

1. 源码 `src/skills/builtin/` — 内置
2. 插件包内 `skills/` — 插件级
3. `~/.ccode/skills/` — 用户级
4. `<cwd>/.ccode/skills/` — 项目级（最高优先级）

### 10.3 Skill 元数据

```yaml
---
name: my-skill
description: 触发条件描述（LLM 决策用）
allowed-tools: [bash, read_file]    # 可选：工具白名单（兼容 allowedTools 驼峰写法）
user-invocable: true                # 可选：支持 /skills <name> 手动触发
---
```

### 10.4 触发方式

- LLM 自动触发（基于 description 匹配用户意图）
- `/skills <name>` 手动触发（需 user-invocable: true）
- Skills 段落自动注入 system prompt

## 十二、Hooks 事件钩子

### 12.1 三层级配置（优先级从高到低）

1. `<cwd>/.ccode/hooks.json` — 项目级
2. `~/.ccode/hooks.json` — 用户级
3. `<plugin-dir>/hooks/hooks.json` — 插件级

### 12.2 事件类型

| 事件 | 触发时机 | 返回值 |
|------|---------|--------|
| SessionStart | 会话启动（startup / resume / compact） | additionalContext / userMessage — 注入对话上下文 |
| PreToolUse | 工具调用前 | allow / block / modify（修改参数） |
| PostToolUse | 工具执行后 | additionalContext / userMessage |

### 12.3 配置示例

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "bash",
      "hooks": [{ "type": "command", "command": "echo $CCODE_TOOL_INPUT", "timeout": 5000 }]
    }]
  }
}
```

### Bootstrap 并行编排

```
链 A：Skills → Instructions → Hooks → SessionStartHooks
链 B：文件索引扫描（磁盘 IO）
链 C：Runtime Plugin 发现与激活
链 D：MemoryManager 初始化（扫描 + BM25 索引）
链 E：Shell 快照创建
       ↓
      屏障（等 A + D 完成）
       ↓
buildSystemPrompt(hookContext, memoryContext)
       ↓
后台异步：MemoryManager.embedPending() + MCP 连接
```

### Session 生命周期

```
CLI 启动 → useChat mount → ensureSession() → sessionId 可用
  → Bridge Client register → 路由生效
  → submit() → AgentLoop → 事件流 → SessionLogger / TokenMeter / EventBus
  → CLI 退出 → session_end 汇总 → 打印 resume 命令
  → ccode --resume <id> → 恢复完整历史
```

## 二十、支持的模型

CCode 支持两大协议族：**Anthropic 原生协议** 和 **OpenAI Chat Completion 兼容协议**。只要你的模型服务实现了其中任一协议，即可接入 CCode。

> 本项目全程在 **智谱 GLM** 模型下进行开发与测试。

| Provider | 协议 | 模型示例 | 备注 |
|----------|------|---------|------|
| 智谱 GLM | OpenAI 兼容 | GLM-5 / GLM-4.7 | **项目主力测试模型** |
| Anthropic | Anthropic 原生 | Claude Opus/Sonnet/Haiku 4.x | 官方 SDK 直连 |
| DeepSeek | OpenAI 兼容 | deepseek-chat / deepseek-reasoner | |
| OpenAI | OpenAI 兼容 | GPT-4o / GPT-4o-mini | |
| Google Gemini | OpenAI 兼容 | gemini-2.5-pro / gemini-2.5-flash | |
| Ollama | OpenAI 兼容 | 任意本地模型 | 本地部署 |
| **任意服务** | OpenAI 兼容 | — | 配置 `baseURL` + `apiKey` 即可接入 |

**接入原则**：只要模型服务提供 OpenAI Chat Completion API（`/v1/chat/completions`）或 Anthropic Messages API，配置 `baseURL` 和 `apiKey` 即可使用，无需任何代码改动。
