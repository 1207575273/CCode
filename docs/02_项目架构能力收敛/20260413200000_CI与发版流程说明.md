# CI 与发版流程说明

> **创建日期**：2026-04-13
> **CI 配置文件**：`.github/workflows/ci.yml`
> **适用版本**：v0.10.0+

---

## 一、整体架构

```
                    ┌──────────────┐
                    │  开发者 Push  │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        release/**    PR → main     v* tag
         分支 push      合并请求      版本标签
              │            │            │
              ▼            ▼            ▼
        ┌─────────────────────────────────┐
        │  Job 1: TypeCheck & Test        │
        │  Node 20 + 22 矩阵并行          │
        │  ├── pnpm install               │
        │  ├── tsc --noEmit               │
        │  └── vitest run                 │
        └──────────────┬──────────────────┘
                       │ 通过
                       ▼
        ┌─────────────────────────────────┐
        │  Job 2: Build Verify            │
        │  ├── CLI 构建 (tsup)             │
        │  │   └── 校验 dist/bin/ccli.js  │
        │  ├── Web 构建 (vite)             │
        │  │   └── 校验 dist/web/index.html│
        │  └── 上传 artifact (release/tag) │
        └──────────────┬──────────────────┘
                       │ 通过
                       ▼
        ┌─────────────────────────────────┐
        │  Job 3: Release Notes           │
        │  (仅 release/* 和 v* tag)        │
        │  ├── 提取 CHANGELOG 段落         │
        │  ├── 生成 git commit 列表        │
        │  ├── 统计变更文件数              │
        │  └── 创建 GitHub Release         │
        │      ├── release/* → Draft 预览  │
        │      └── v* tag   → 正式发布     │
        └─────────────────────────────────┘
```

## 二、触发条件

| 触发事件 | 执行 Job | 说明 |
|----------|---------|------|
| `release/**` 分支 push | test + build + release-notes(draft) | 发版准备阶段，每次提交都验证 |
| `v*` tag push | test + build + release-notes(正式) | 打版本标签时，生成正式 Release |
| PR → main | test + build | 合并前门控，不生成 Release Notes |

### 并发控制

```yaml
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true
```

同一分支新 push 会自动取消正在运行的旧 CI，节省 GitHub Actions 分钟数。

## 三、各 Job 详解

### Job 1: TypeCheck & Test

**目的**：确保代码类型安全 + 测试通过

| 项目 | 说明 |
|------|------|
| 运行环境 | ubuntu-latest |
| Node 版本 | 20 + 22 矩阵（覆盖 `engines: >=20`） |
| 包管理器 | pnpm 9，使用 `--frozen-lockfile` 确保可复现 |
| 类型检查 | `pnpm run typecheck` → `tsc --noEmit` |
| 单元测试 | `pnpm test` → `vitest run`（当前 80 个测试文件，673 个用例） |

**注意事项**：
- 测试中不要硬编码分支名（CI 可能在 `release/*` 分支上）
- mock 路径匹配要用精确前缀（`startsWith`），不能用 `includes(homedir())`，因为 CI 上 cwd 在 homedir 子目录下

### Job 2: Build Verify

**目的**：确保 CLI 和 Web Dashboard 都能构建成功

| 步骤 | 命令 | 校验 |
|------|------|------|
| CLI 构建 | `pnpm run build` (tsup) | `dist/bin/ccli.js` 存在 |
| Web 依赖 | `cd web && pnpm install` | web/ 有独立的 pnpm-lock.yaml |
| Web 构建 | `cd web && pnpm build` (vite) | `dist/web/index.html` 存在 |

**关键路径说明**：

```
cCli/
├── dist/
│   ├── bin/ccli.js          ← tsup 输出（CLI 入口）
│   └── web/index.html       ← vite 输出（outDir: '../dist/web'）
└── web/
    ├── src/                 ← Web 源码
    ├── vite.config.ts       ← 配置 outDir 到 ../dist/web
    └── pnpm-lock.yaml       ← 独立 lockfile
```

> Vite 的 `outDir` 配置为 `../dist/web`，构建产物输出到 `cCli/dist/web/` 而非 `cCli/web/dist/`。
> 这是为了和 CLI（`dist/bin/`）统一在 `dist/` 下，npm publish 时一起打包。

**Artifact 上传**：release 分支和 tag push 时上传 `dist/bin/` + `dist/web/` + `package.json`，保留 30 天。

### Job 3: Release Notes

**目的**：自动生成版本说明，减少手动编写

| 触发 | 行为 | GitHub Release 状态 |
|------|------|-------------------|
| `release/*` 分支 push | 生成预览 | **Draft**（可编辑/删除） |
| `v*` tag push | 正式发布 | **Published** |

**生成内容包括**：

1. **CHANGELOG 段落** — 从 `CHANGELOG.md` 提取 `## [x.y.z]` 对应版本的内容
2. **Commit 列表** — `git log <上一个tag>..HEAD --pretty=format:"- %s (%h)"`
3. **变更统计** — 文件数、增删行数（折叠显示）
4. **npm 安装命令** — `npm install -g ccode-cli@x.y.z`

Draft 预览会在顶部标注：

> **Draft** — 此为 `release/0.11.0` 分支的预览版本，正式发布前可能继续变更。

## 四、完整发版流程

### 4.1 日常开发

```bash
# 在 main 或 feature 分支上开发
git checkout main
# ... 写代码 ...
git add && git commit
```

### 4.2 准备发版

```bash
# 1. 创建 release 分支
git checkout -b release/0.12.0

# 2. 更新版本号（两处同改）
#    - cCli/package.json → "version": "0.12.0"
#    - cCli/src/version.ts → APP_VERSION = '0.12.0'

# 3. 更新 CHANGELOG.md（新增 ## [0.12.0] 段落）

# 4. 提交并推送
git add -A && git commit -m "chore: bump version to 0.12.0"
git push origin release/0.12.0
```

此时 CI 自动运行：
- test ✓ → build ✓ → 生成 **Draft Release Notes**（GitHub Releases 页面可查看）

### 4.3 修复问题（可选）

```bash
# 在 release 分支上继续修复
git commit -m "fix: ..."
git push origin release/0.12.0
# CI 重新运行，Draft Release Notes 自动更新
```

### 4.4 合并到 main

```bash
# 合并 release 分支
git checkout main
git merge release/0.12.0 --no-ff -m "merge: release/0.12.0 → main"
git push origin main
```

### 4.5 打 tag + 正式发布

```bash
# 打版本标签
git tag v0.12.0
git push origin v0.12.0
```

CI 自动运行：
- test ✓ → build ✓ → 生成**正式 Release Notes**（非 Draft）

### 4.6 npm 发包（手动）

```bash
cd cCli
pnpm run build            # CLI 构建
cd web && pnpm build      # Web 构建
cd ..
npm publish               # 发包到 npm
```

> **为什么 npm publish 不集成到 CI？**
>
> | 风险 | 说明 |
> |------|------|
> | 误发无法撤回 | npm unpublish 只能在 72h 内操作 |
> | tag 即发包 | 误打 tag = 发错版本，没有确认环节 |
> | 需要目视检查 | 版本号、构建产物、CHANGELOG 都需要人工确认 |
> | Secret 维护 | NPM_TOKEN 需要定期轮换 |

## 五、CHANGELOG 维护规范

文件路径：项目根目录 `CHANGELOG.md`

### 格式

遵循 [Keep a Changelog](https://keepachangelog.com/) 规范：

```markdown
## [0.12.0] - 2026-05-01

### Added
- 新增功能描述

### Fixed
- 修复问题描述

### Changed
- 变更说明
```

### 分类标签

| 标签 | 含义 |
|------|------|
| `Added` | 新增功能 |
| `Fixed` | Bug 修复 |
| `Changed` | 已有功能的变更 |
| `Deprecated` | 即将移除的功能 |
| `Removed` | 已移除的功能 |
| `Security` | 安全修复 |

### 编写时机

- 每个 release 分支创建时，**先写 CHANGELOG 再写代码**
- CI 的 Release Notes 会自动提取对应版本段落，写好了就能自动生成

## 六、踩坑记录

### 6.1 Web 构建产物路径

**问题**：CI 校验 `web/dist/index.html` 报 Not Found。

**原因**：Vite 配置 `outDir: '../dist/web'`，产物输出到 `cCli/dist/web/` 而非 `cCli/web/dist/`。

**修复**：CI 校验路径改为 `dist/web/index.html`。

### 6.2 测试硬编码分支名

**问题**：`git-tool.test.ts` 断言 `output.toContain('main')`，CI 在 `release/*` 分支上失败。

**原因**：`git branch` 只列出本地分支，CI checkout 的是 release 分支。

**修复**：改为检查输出非空，不硬编码分支名。

### 6.3 Mock 路径匹配在 CI 上失败

**问题**：`skill-store-plugins.test.ts` 中 user/project 路径区分用 `p.includes(homedir())`，CI 上 project 路径也命中了 user 条件。

**原因**：CI 的 `cwd` = `/home/runner/work/CCode/CCode`，位于 `homedir()` = `/home/runner` 的子目录下，`includes` 无法区分。

**修复**：改为 `p.startsWith(join(homedir(), '.ccode', 'plugins'))` 精确前缀匹配。

### 6.4 新增 import 导致测试 mock 不完整

**问题**：`instructions-loader.ts` 新增 `import { dbg } from '../debug.js'`，`debug.ts` 模块顶层调用 `mkdirSync`，但测试 mock 的 `node:fs` 没有导出 `mkdirSync`。

**修复**：测试 mock 补全 `mkdirSync` 和 `appendFileSync`。
