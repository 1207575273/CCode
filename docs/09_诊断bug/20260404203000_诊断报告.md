## 四、解决方案

### 问题 1：SubAgent 重复调用 write_file

- **现状**：GLM-5.1 在 SubAgent 场景下反复写同一文件，19 次中有 17 次是无意义的重复
- **根因**：
  - 模型判断力弱，收到 success 后仍不确定是否完成
  - SubAgent 系统没有去重/防循环机制
  - tool_result 的格式对弱模型来说"确认感"不够强
- **方案**（三层防御）：

  **A. agent-loop 层 — 添加重复调用检测**
  ```typescript
  // 在 agent-loop.ts 的工具执行循环中
  // 记录最近 N 次工具调用的 hash（toolName + args hash）
  // 如果连续 3 次相同 → 注入一条 system message："你已经成功完成了这个操作，请继续下一步"
  // 如果连续 5 次相同 → 强制跳过并返回最后一次结果
  ```

  **B. write_file 工具层 — 增强反馈信息**
  ```typescript
  // 在 write-file.ts 的 execute 返回中
  // 改为：`✅ 文件写入成功。路径: xxx，大小: N 字符。此步骤已完成，请继续执行下一个任务。`
  // 加入"请继续执行下一个任务"这种显式引导
  ```

  **C. SubAgent systemPrompt — 加入防重复指令**
  ```
  在 built-in.ts 的 general agent systemPrompt 中加入：
  "If a tool call succeeds, move on to the next step immediately. 
   Do NOT call the same tool with the same arguments again."
  ```

- **验证**：用同样的 prompt 重新跑一次 SubAgent 任务，检查 write_file 是否还重复

### 问题 2：token usage 为 undefined

- **现状**：所有 `llm_call_end` 事件中 `usage.inputTokens` 等字段均为 undefined
- **根因**：wrapper.ts 在从 LangChain stream response 中提取 usage 时，字段路径可能不对（LangChain 不同版本的 usage 字段位置不同）
- **方案**：
  ```typescript
  // 在 wrapper.ts 中，检查 response_metadata.usage 或 usage_metadata
  // LangChain 的 usage 可能在以下位置：
  // 1. chunk.usage_metadata（新版）
  // 2. chunk.response_metadata?.tokenUsage（老版）
  // 3. chunk.additional_kwargs?.usage（某些模型）
  // 需要逐个检查并 fallback
  ```
- **验证**：跑一次 session 后检查 JSONL 中的 usage 字段

### 问题 3：重复调用检测机制

- **现状**：agent-loop 没有任何防循环机制
- **根因**：设计时假设 LLM 不会重复调用，但弱模型（GLM）打破了这个假设
- **方案**：
  ```typescript
  // agent-loop.ts 中添加 RepetitionDetector
  interface CallRecord { toolName: string; argsHash: string; timestamp: number }
  
  // 维护滑动窗口（最近 5 次调用）
  // 检测规则：
  // 1. 连续 3 次相同 toolName+argsHash → 注入引导消息
  // 2. 连续 5 次相同 → 强制跳过 + 日志告警
  // 3. 单次 run 中同一文件被 write_file 超过 3 次 → 注入引导
  ```
- **验证**：编写单元测试模拟重复调用场景
