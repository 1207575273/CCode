## 四、解决方案

### 问题 1：agent-loop 层面的重复调用检测（硬性防御）

- **现状**：提示词 + tool_result 反馈都无法阻止 GLM-5.1 的循环行为
- **根因**：弱模型的指令遵从度不足以仅靠"软约束"解决问题
- **方案**：在 `agent-loop.ts` 中实现 RepetitionDetector

```typescript
// 在 agent-loop.ts 中添加
interface CallRecord {
  toolName: string
  argsHash: string  // 用 JSON.stringify(args) 的简单 hash
}

// 维护最近 N 次工具调用记录
const recentCalls: CallRecord[] = []

// 每次工具调用后检查：
// 1. 连续 3 次相同 toolName + argsHash → 
//    替换 tool_result 为："⚠️ 你已经用相同参数调用了 {toolName} 3 次，每次都成功了。
//    这个操作已经完成。请立即执行下一个不同的步骤。"
// 2. 连续 5 次相同 → 强制跳过工具调用，直接返回：
//    "❌ 检测到循环调用（{toolName} × 5）。此操作已被系统跳过。请执行其他操作。"
//    并 yield 一个 post_tool_feedback 事件
```

**关键**：第 2 级（5 次后强制跳过）是必须的——如果模型连 3 次的警告都无视（GLM 确实会这样），就不能再给它机会浪费 token 了。

- **验证**：用同样的 prompt 再跑一次，检查 frontend SubAgent 是否还循环

### 问题 2：确认 streamUsage 生效

- **方案**：确认 cCli 的运行方式
  - 如果用 `npm run dev`（tsx）→ 改动应该已生效，需检查 GLM API 是否支持 `stream_options.include_usage`
  - 如果用 `dist/`（编译后）→ 需要先 `npm run build`
  - 也可能是 GLM-5.1 的 API 不支持 `stream_options`，需要加 fallback：在 stream 结束后用 `response_metadata.usage` 提取
- **验证**：开 debug 模式跑一次，grep 日志中 `usage_metadata`
