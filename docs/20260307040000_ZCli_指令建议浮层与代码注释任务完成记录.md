## 二、功能说明

### 2.1 指令建议浮层

**交互流程：**

```
输入 "/"         -> 显示所有指令（全量）
输入 "/cl"       -> 过滤为 [/clear]（前缀匹配，大小写不敏感）
按 Up / Down     -> 在建议列表中移动高亮
按 Enter         -> 直接执行高亮指令（清空输入，关闭浮层）
按 Esc           -> 清空输入，关闭浮层
输入非 / 字符    -> 不显示浮层
```

**外观：**

```
  /clear    Clear current conversation
> /help     Show available commands       <- 高亮项（绿色 > + 白色文字）
  /model    Switch AI model
```

- 左侧 `>` 标记高亮行，其余行两空格缩进
- command name 绿色，description dimColor
- 无边框，轻量样式
- 渲染于 InputBar 正上方

## 三、架构变更

### 3.1 InputBar 改为受控组件

```typescript
// Before
function InputBar({ onSubmit, ... })  // 内部管理 inputValue state

// After
function InputBar({ value, onChange, onSubmit, ... })  // state 提升至 App.tsx
```

`App.tsx` 新增 `const [inputValue, setInputValue] = useState('')`，通过 `onChange={setInputValue}` 与 InputBar 同步，使父组件能实时观测输入内容以计算建议列表。

### 3.2 新增 CommandSuggestion 组件

纯显示组件（无 `useInput`，无内部 state），接收两个 props：

- `items: SuggestionItem[]` — 已过滤的建议列表
- `selectedIndex: number` — 当前高亮行（由 App.tsx 管理）

### 3.3 App.tsx 导航状态

```typescript
const [suggestionIndex, setSuggestionIndex] = useState(0)
const suggestionIndexRef = useRef(0)       // 双轨：供 useInput 闭包读取最新值
const suggestionConsumedRef = useRef(false) // 防 Enter 双触发
```

**双轨 ref 模式**（项目内一贯约定）：
- `setSuggestionIndex` 驱动 UI 重渲染
- `suggestionIndexRef.current` 在 `useInput` 异步闭包中读取最新值，避免陈旧 state

**Enter 防双触发**：
- `useInput` 处理 Enter -> 设 `suggestionConsumedRef.current = true` -> 调 `handleSubmit`
- `handleSubmit` 检测到 ref 为 true -> 重置 ref -> 提前返回（跳过 TextInput.onSubmit 的重复触发）

## 六、关键设计决策

| 决策 | 原因 |
|------|------|
| CommandSuggestion 纯显示，不含 `useInput` | 避免多个 `useInput` 实例的键盘冲突，状态集中在 App.tsx 更易调试 |
| `useInput({ isActive: suggestions.length > 0 })` | 浮层不可见时不拦截任何按键，防止与 TextInput 的正常 Enter 冲突 |
| `suggestionIndexRef` 双轨 | Ink `useInput` 依赖 `useEffect` 重订阅，闭包捕获 state 理论上安全，但显式 ref 消除对 Ink 内部行为的隐式依赖 |
| `suggestionConsumedRef` | TextInput.onSubmit 与 useInput.return 在同一渲染周期内均会触发，ref 是最低成本的防重复方案 |
| Enter 直接执行（不填充后确认） | 减少交互步骤，符合 CLI 用户习惯 |

## 七、不做的事（YAGNI）

- Tab 自动补全（Enter 即执行，无需 Tab）
- 模糊匹配（前缀匹配足够，降低复杂度）
- 建议项分组（当前指令数量少，分组无实际收益）
- 为注释写测试
