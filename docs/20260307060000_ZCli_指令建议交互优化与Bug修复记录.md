# ZCli 指令建议交互优化与 Bug 修复记录

> 完成日期：2026-03-07 | 分支：master | 测试：78/78

---

## 一、任务背景

在上一轮迭代（CommandSuggestion 浮层 + 全量注释）完成后，用户反馈了三类问题，本次迭代逐一修复，并对建议浮层的交互模式做了整体重设计：

1. 建议浮层显示位置在输入框**上方**，不符合用户预期
2. Tab 键补全缺失；Enter 键直接执行指令，而非先填充到输入框
3. ModelPicker 的 Esc 退出在多种终端环境下失效

---

## 二、功能变更

### 2.1 建议浮层位置调整（InputBar 下方）

**前：** `<CommandSuggestion>` 渲染在 InputBar 块的上方。

**后：** JSX 结构调整，InputBar/PermissionDialog/ModelPicker 三选一块在上，`<CommandSuggestion>` 紧跟其后，终端 flex-column 布局下自然显示在下方。

### 2.2 Tab/Enter 改为补全模式

**前：** Enter 直接执行高亮指令（`handleSubmit('/' + cmd.name)`）。

**后：**
- Tab → 填充输入框为 `/cmdname `（尾部空格，方便继续输入参数），不执行
- Enter（浮层可见时）→ 与 Tab 相同，填充输入框，不执行
- 填充后用户需再按 Enter 才真正执行指令

**交互流程：**
```
输入 "/cl"  -> 浮层出现：/clear
按 Tab 或 Enter -> 输入框变为 "/clear "，浮层消失
再按 Enter  -> 执行 /clear
```

## 三、Bug 修复

### 3.1 Tab 补全后光标停在字符串中间

**现象：** 按 Tab 补全 `/clear ` 后，光标停在原输入 `/cl` 的末尾（位置 3），而非 `/clear ` 的末尾（位置 7）。

**根因：** `ink-text-input` 的 `cursorOffset` 只在 `prevCursorOffset > newValue.length - 1` 时自动归位到末尾。外部 `setInputValue` 将 value 变长时，旧 cursor 位置合法，组件不会移动光标。

**修复：** 新增 `inputResetKey` state。每次 Tab/Enter 补全时 `setInputResetKey(k => k + 1)`，通过改变 InputBar 的 `key` prop 强制重挂载，挂载时 `cursorOffset` 初始化为 `value.length`（末尾）。

### 3.2 Enter 补全后输入框被清空（最关键 Bug）

**现象：** 按 Enter 选择建议后，输入框出现一瞬 `/model ` 随即变为空。

**根因：** React 批量更新的"最后写入者胜"问题：
1. `useInput` 处理 Enter → `setInputValue('/model ')`
2. `TextInput.onSubmit` 在同 tick 触发 → `handleSubmit` 第一行调用 `setInputValue('')`
3. React 批量合并，`setInputValue('')` 在后 → 最终值为空

**修复：** 将 `handleSubmit` 中的 `setInputValue('')` 从函数第一行移到 `suggestionConsumedRef` 检查之后，确保补全路径提前 return，不执行清空操作。

```typescript
const handleSubmit = useCallback((input: string) => {
  if (suggestionConsumedRef.current) {
    suggestionConsumedRef.current = false
    return  // 补全路径：不清空输入，直接返回
  }
  setInputValue('')  // 普通提交路径：清空输入
  // ...
}, [...])
```

### 3.3 Tab 补全导致 suggestionConsumedRef 残留

**现象：** Tab 补全后，`suggestionConsumedRef` 被置为 true，但 Tab 不触发 `TextInput.onSubmit`，`handleSubmit` 不执行，ref 永远不重置。下次用户按 Enter 正常提交时被静默拦截。

**修复：** 拆分 `key.tab || key.return` 为两个独立分支，仅 Enter 分支设置 ref，Tab 分支只填充输入框。

### 3.4 Esc 不重置 suggestionConsumedRef

**现象：** 若用户按 Enter 补全（ref 置 true）后按 Esc 取消，ref 残留为 true，下次 Enter 提交被丢弃。

**修复：** 在 Esc 分支中显式重置 `suggestionConsumedRef.current = false`。

### 3.5 ModelPicker Esc 失效（标准终端）

**根因：** `onCancel` 是 App.tsx JSX 中的 inline lambda，每次渲染生成新引用。ModelPicker 内 `useInput` 回调捕获旧引用，Ink 在 deregister/re-register 的间隙中丢失按键事件。

**修复：** 在 ModelPicker 中应用双轨 ref 模式：
- `onCancelRef` / `onSelectRef` / `itemsRef` / `selectedRef` 通过 `useEffect` 实时同步最新值
- `stableHandler` 用 `useCallback(fn, [])` 确保引用永远稳定，Ink 不重新注册，消除竞态窗口

同时在 App.tsx 层面也用 `useCallback(fn, [])` 注册一个 ModelPicker 专属的 Esc 保险处理器。

### 3.6 ModelPicker Esc 失效（IDE 终端）

**现象：** 即使修复了竞态问题，在 IntelliJ IDEA / VSCode 内置终端中按 Esc 仍无法退出 ModelPicker。

**根因：** IDE 在框架层拦截 Esc（`\x1b`），作为"关闭对话框/取消操作"的全局快捷键消费掉，PTY/stdin 根本收不到该字节。这是操作系统/IDE 层面的限制，任何 Node.js / Ink 代码都无法绕过。

**修复：** 新增 `q` 键作为可靠的备用退出键（普通字符在所有 IDE 终端中均可透传），同步更新提示文字为 `↑↓ 选择   Enter 确认   Esc/q 取消`。

## 六、关键技术决策

| 决策 | 原因 |
|------|------|
| Tab/Enter 均为补全而非执行 | 用户习惯：先看清补全结果再确认，减少误操作 |
| `inputResetKey` 强制重挂载 | `ink-text-input` 无受控 cursor API，重挂载是最低成本的归位方案 |
| `setInputValue('')` 移到 ref 检查之后 | React 批量更新"最后写入者胜"，顺序决定结果 |
| Tab/Enter 分支拆开，只有 Enter 设 ref | Tab 不触发 onSubmit，设 ref 无意义且有残留风险 |
| `useCallback(fn, [])` for all Ink handlers | 彻底消除 handler 重注册的竞态窗口，是 Ink 项目的标准防御模式 |
| `q` 作为 Esc 备用键 | IDE 框架层拦截 `\x1b` 是不可改变的平台行为，字符键是唯一可靠方案 |
