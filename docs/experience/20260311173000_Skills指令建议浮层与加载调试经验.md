# Skills 指令建议浮层与加载调试经验

> 日期: 2026-03-11 | 模块: UI (App.tsx) + Skills (store.ts)

## 背景

F6 Skills 注入与卸载一阶段完成后，在 UI 层将 Skills 融入斜杠指令建议浮层时，遇到了一系列 Ink useInput + TextInput 交互的时序问题。核心场景：用户输入 `/` 后弹出建议浮层，方向键选择 skill，Enter 提交。

---

## 问题一：Enter 被建议浮层吃掉，第一次按 Enter 无效

### 现象
用户输入 `/hello-world` 或方向键选到某个建议后按 Enter，什么都不执行。必须再按一次 Enter 才能提交。

### 根因
建议浮层的 `useInput` Enter 处理设置了 `suggestionConsumedRef = true`，导致同 tick 触发的 `handleSubmit` 看到 consumed 后直接 return，第一次 Enter 永远被吃掉。

```typescript
// ❌ 旧方案：useInput 拦截 Enter → 设 ref → handleSubmit 读 ref 返回
if (key.return) {
  suggestionConsumedRef.current = true
  setInputValue('/' + cmd.name + ' ')  // 补全到输入框
}
// handleSubmit 里：
if (suggestionConsumedRef.current) { return }  // 第一次 Enter 被跳过
```

### 修复
删除 useInput 中的 Enter 拦截，改为 handleSubmit 直接通过 `suggestionsRef` 读取当前选中项。

```typescript
// ✅ 新方案：handleSubmit 直接读 ref，一步到位
const activeSuggestions = suggestionsRef.current
if (activeSuggestions.length > 0 && trimmed.startsWith('/')) {
  const selected = activeSuggestions[suggestionIndexRef.current]
  if (selected) trimmed = '/' + selected.name
}
```

---

## 问题二：overrideSubmitRef 方案仍然晚一拍

### 现象
尝试用 `overrideSubmitRef` 在 useInput 的 Enter handler 里设值，让 handleSubmit 读取。结果第一次 Enter 仍无效，第二次才行。

### 根因
**Ink 的 handler 执行顺序**：TextInput 的内部 `onSubmit` 先于外部 `useInput` 触发。因此 handleSubmit 执行时 ref 还没被设值，读到 null。

```
Enter 按下 → TextInput.onSubmit → handleSubmit (ref=null) → useInput (设 ref)
                                    ↑ 此时 ref 还没设       ↑ 此时才设，太晚了
```

### 修复
不再依赖跨 handler 的 ref 传递。改用 `suggestionsRef`（每次渲染同步赋值）+ `suggestionIndexRef`，handleSubmit 直接读取最新建议列表和选中索引。

**关键教训：Ink 中多个 useInput handler 和 TextInput.onSubmit 的执行顺序不可靠，不要用 ref 在它们之间传值。**

---

## 问题三：纯 `/` 输入被发送到 LLM

### 现象
用户方向键选中建议后按 Enter，input 值仍是 `/`（因为用户只输入了 `/` 就开始用方向键）。handleSubmit 中 `trimmed === '/'` 的 guard 在建议解析之前就 return 了。

### 根因
guard 顺序错误：先判断 `trimmed === '/'` 返回，后面的建议解析代码永远执行不到。

```typescript
// ❌ 错误顺序
if (!trimmed || trimmed === '/') return  // 先退了
const selected = suggestions[index]       // 永远走不到
```

### 修复
建议解析移到 guard 之前：

```typescript
// ✅ 正确顺序
// 先解析建议 → trimmed 可能变成 /hello-world
const selected = activeSuggestions[index]
if (selected) trimmed = '/' + selected.name
// 再 guard → /hello-world 不等于 /，放行
if (!trimmed || trimmed === '/') return
```

## 问题四：suggestionsRef 在 suggestions 定义前赋值

### 现象
启动报错 `Cannot access 'suggestions' before initialization`。

### 根因
`suggestionsRef.current = suggestions` 写在了 `const suggestions = useMemo(...)` 之前（JavaScript TDZ）。

### 修复
把 `suggestionsRef.current = suggestions` 移到 `useMemo` 之后。

---

## 问题五：Skills 三源加载串行导致启动偏慢

### 现象
builtin / user / project 三个目录串行 `await`，每个目录内的文件也串行读取。

### 修复
两层并行化：
1. 三个目录 `Promise.all` 同时扫描
2. 每个目录内所有 SKILL.md `Promise.all` 同时读取解析
3. 按 `[...builtin, ...user, ...project]` 顺序合并，保证高优先级覆盖

---

## 总结：Ink useInput 交互的核心教训

1. **不要在 useInput 的 Enter handler 里设 ref 给 handleSubmit 读** — 执行顺序不可靠
2. **用 suggestionsRef（渲染时同步赋值）替代跨 handler 的 ref 传值** — 永远是最新的
3. **guard 语句要放在数据转换之后** — 先解析建议项，再判断是否过滤
4. **Tab = 补全，Enter = 提交** — 符合终端/IDE 惯例，避免两步操作
5. **React 变量声明顺序即执行顺序** — ref 赋值必须在 useMemo 之后
