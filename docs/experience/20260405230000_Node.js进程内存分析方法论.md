## 一、Node.js 内存模型基础

### 1.1 process.memoryUsage() 五个指标

```typescript
const mem = process.memoryUsage()
// {
//   rss:          200 * 1024 * 1024,  // 常驻集大小（Resident Set Size）
//   heapTotal:     80 * 1024 * 1024,  // V8 堆总量（已申请）
//   heapUsed:      60 * 1024 * 1024,  // V8 堆实际使用
//   external:      10 * 1024 * 1024,  // C++ 对象绑定的内存（Buffer、TypedArray）
//   arrayBuffers:   5 * 1024 * 1024,  // ArrayBuffer + SharedArrayBuffer
// }
```

**它们的关系**：

```
RSS（操作系统视角的进程总内存）
 ├── V8 Heap（JS 对象）
 │    ├── heapUsed（活跃对象）
 │    └── heapTotal - heapUsed（已申请但空闲的堆空间）
 ├── External（C++ 绑定对象，如 Buffer）
 ├── Code（编译后的 JS 代码）
 └── Stack + Other（线程栈、libuv、DNS 缓存等）
```

| 指标 | 含义 | 关注场景 |
|------|------|---------|
| **RSS** | 操作系统分配给进程的总物理内存 | 最终用户看到的"占了多少内存" |
| **heapTotal** | V8 已向 OS 申请的堆空间 | 堆增长趋势（是否持续膨胀 = 泄漏） |
| **heapUsed** | V8 堆中实际使用的空间 | 代码中创建了多少 JS 对象 |
| **external** | V8 管理但不在堆中的 C++ 对象 | Buffer 大量使用时关注 |
| **arrayBuffers** | ArrayBuffer 占用（external 的子集） | 二进制数据处理时关注 |

### 1.2 为什么 RSS > heapTotal + external？

差值来自：
- **V8 编译缓存**：JIT 编译后的机器码占内存但不算在 heap 里
- **线程栈**：每个 libuv worker thread 默认 4MB 栈
- **系统库映射**：动态链接库（.dll/.so）的内存映射
- **Node.js 内部**：DNS 缓存、TLS 会话缓存、stream buffer 等

## 二、排查方法：从粗到细

### 2.1 第一步：全局快照（5 分钟定位方向）

```bash
# 查看进程 RSS
node -e "console.log(JSON.stringify(process.memoryUsage(), null, 2))"

# 启动项目后查看
node --expose-gc -e "
  global.gc();  // 强制 GC 排除垃圾影响
  const m = process.memoryUsage();
  console.log('RSS:', (m.rss/1024/1024).toFixed(1), 'MB');
  console.log('Heap Used:', (m.heapUsed/1024/1024).toFixed(1), 'MB');
  console.log('External:', (m.external/1024/1024).toFixed(1), 'MB');
"
```

**判断方向**：
- heapUsed 占 RSS 大头 → JS 对象多（大数组、缓存、依赖树）
- external 大 → Buffer / native 模块多
- RSS 远大于 heapTotal + external → 编译缓存 / 线程 / 系统库

### 2.2 第二步：依赖热力图（定位哪个 npm 包最重）

**核心方法：逐个 import 测量内存增量。**

```typescript
// measure-imports.mjs
async function measure(name, importFn) {
  // 强制 GC 排除干扰（需要 --expose-gc 启动）
  if (global.gc) global.gc()
  const before = process.memoryUsage().rss
  const t0 = performance.now()
  await importFn()
  if (global.gc) global.gc()
  const after = process.memoryUsage().rss
  const ms = (performance.now() - t0).toFixed(0)
  const mb = ((after - before) / 1024 / 1024).toFixed(1)
  console.log(`${name.padEnd(35)} ${ms.padStart(5)}ms ${mb.padStart(7)} MB`)
}

// 按你项目的 import 列表逐个测
await measure('@langchain/openai',     () => import('@langchain/openai'))
await measure('@anthropic-ai/sdk',     () => import('@anthropic-ai/sdk'))
await measure('ink',                   () => import('ink'))
await measure('hono',                  () => import('hono'))
// ...
```

```bash
node --expose-gc measure-imports.mjs
```

**原理**：每次 import 前后测 RSS 差值。由于 Node.js 的模块缓存，同一模块第二次 import 是 0 开销，所以能精确归因。

**实战结论（cCli 项目）**：

| 依赖 | 内存开销 | 加载耗时 |
|------|---------|---------|
| @langchain/openai | ~26MB | ~450ms |
| ink + react | ~30MB | ~450ms |
| @anthropic-ai/sdk | ~5MB | ~50ms |
| hono + node-server | ~1MB | ~15ms |
| libsql | ~0.5MB | ~10ms |

### 2.3 第三步：V8 堆快照（精确到对象级别）

**适用场景**：怀疑内存泄漏，或者 heapUsed 异常大时。

#### 方法 A：Chrome DevTools 远程调试

```bash
# 以 inspect 模式启动
node --inspect your-app.js

# 打开 Chrome 访问
chrome://inspect
```

在 DevTools → Memory 面板中：
1. **Heap Snapshot**：拍摄当前堆快照，看所有对象的类型、大小、引用链
2. **Allocation Timeline**：实时记录对象分配，找出"谁在不停创建对象"
3. **Allocation Sampling**：采样模式，低开销，适合生产环境

#### 方法 B：代码内生成 .heapsnapshot 文件

```typescript
import v8 from 'node:v8'
import fs from 'node:fs'

// 在关键时刻触发快照
function dumpHeap(label: string) {
  const filename = `heap-${label}-${Date.now()}.heapsnapshot`
  const snapshotStream = v8.writeHeapSnapshot(filename)
  console.log(`Heap snapshot written to ${snapshotStream}`)
}

// 示例：启动后 10 秒拍一次
setTimeout(() => dumpHeap('after-boot'), 10000)
// 运行 5 分钟后再拍一次，对比
setTimeout(() => dumpHeap('after-5min'), 300000)
```

然后在 Chrome DevTools → Memory → Load 加载 .heapsnapshot 文件分析。

#### 方法 C：命令行工具 clinic.js

```bash
# 安装
npm install -g clinic

# 生成内存报告
clinic heapprofiler -- node your-app.js

# 生成整体诊断（CPU + 内存 + 事件循环）
clinic doctor -- node your-app.js
```

### 2.4 第四步：泄漏检测（生产问题核心）

**内存泄漏的特征**：RSS / heapUsed 随时间**单调递增**，GC 后不回落。

#### 定时采样法

```typescript
// 每 30 秒记录一次内存，观察趋势
setInterval(() => {
  const m = process.memoryUsage()
  console.log(JSON.stringify({
    time: new Date().toISOString(),
    rss: (m.rss / 1024 / 1024).toFixed(1),
    heapUsed: (m.heapUsed / 1024 / 1024).toFixed(1),
    external: (m.external / 1024 / 1024).toFixed(1),
  }))
}, 30000)
```

**判断标准**：
- 5 分钟内 heapUsed 增长 < 5MB → 正常（JIT 预热 + 缓存填充）
- 持续 30 分钟单调增长 → 高度怀疑泄漏
- GC 后回落到稳定基线 → 正常的锯齿形曲线

#### 常见泄漏模式

| 模式 | 症状 | 典型原因 |
|------|------|---------|
| EventEmitter 泄漏 | listener 数量警告 | 注册了 listener 但没 removeListener |
| 闭包引用 | heapUsed 持续增长 | setInterval/setTimeout 回调闭包持有大对象引用 |
| 全局缓存无限增长 | Map/Set 越来越大 | 缓存没有淘汰策略（LRU/TTL） |
| Stream 未关闭 | external 持续增长 | 打开文件/网络流后没 destroy |
| Buffer 累积 | arrayBuffers 增长 | 日志 Buffer、响应 Buffer 没释放 |

### 2.5 第五步：依赖树分析（瘦身方向）

```bash
# 查看依赖树深度和大小
npx depcheck                    # 找出未使用的依赖
npm ls --all | wc -l            # 依赖树总节点数
du -sh node_modules/*/ | sort -rh | head -20  # 目录级大小排名
```

```bash
# 精确分析某个包拉入了多少依赖
npm ls @langchain/openai --all  # 看它的完整依赖子树
```

## 三、不同场景的诊断路径

### 3.1 启动内存偏高（本次场景）

```
诊断路径：
  RSS 194MB
    ↓ process.memoryUsage() 看分布
  heapUsed 60MB, external 5MB, 差值 ~130MB（编译缓存+线程+系统库）
    ↓ 逐个 import 测量依赖开销
  @langchain/openai 26MB, ink 30MB（两大头）
    ↓ 分析是否可替换
  LangChain 只用了 ChatOpenAI → 可换成 openai 官方 SDK 省 ~23MB
```

### 3.2 运行中内存持续增长（泄漏）

```
诊断路径：
  RSS 从 200MB 持续涨到 500MB+
    ↓ 定时采样确认趋势
  heapUsed 单调递增，GC 后不回落
    ↓ 两次 Heap Snapshot 对比
  发现某个 Map 从 1000 条涨到 100000 条
    ↓ Retainers 面板追溯引用链
  找到根因：全局缓存没有淘汰策略
```

### 3.3 请求处理期间内存飙升（峰值问题）

```
诊断路径：
  处理大文件时 RSS 飙到 1GB
    ↓ 检查 external / arrayBuffers
  Buffer 占 800MB
    ↓ 追踪 Buffer 分配点
  发现一次性 readFileSync 整个大文件到内存
    ↓ 改为 createReadStream 流式处理
```

## 四、生产环境实用技巧

### 4.1 V8 内存参数调优

```bash
# 限制堆上限（默认 ~1.5GB，可调小节省 RSS）
node --max-old-space-size=512 app.js   # 限制老生代堆为 512MB

# 限制新生代大小（减少 GC 频率但增加单次 GC 耗时）
node --max-semi-space-size=16 app.js   # 默认 16MB

# 打印 GC 日志（排查 GC 停顿）
node --trace-gc app.js
```

### 4.2 进程级内存监控

```typescript
// 超过阈值时报警
const MEMORY_LIMIT_MB = 500

setInterval(() => {
  const rss = process.memoryUsage().rss / 1024 / 1024
  if (rss > MEMORY_LIMIT_MB) {
    console.error(`[MEMORY ALERT] RSS ${rss.toFixed(0)}MB exceeds ${MEMORY_LIMIT_MB}MB`)
    // 可选：写 heapsnapshot 后优雅退出
    v8.writeHeapSnapshot()
    process.exit(1)
  }
}, 10000)
```

### 4.3 常用命令速查

```bash
# 查看 Node 进程内存（Linux/Mac）
ps aux | grep node | awk '{print $6/1024 "MB", $11}'

# 查看 Node 进程内存（Windows）
tasklist | findstr node

# V8 堆统计
node -e "console.log(require('v8').getHeapStatistics())"

# 强制 GC 后看真实占用
node --expose-gc -e "global.gc(); console.log(process.memoryUsage())"

# 生成堆快照
node -e "require('v8').writeHeapSnapshot()"
```

## 五、cCli 项目深度分析：194MB 到底花在哪

### 5.1 内存构成实测

通过逐模块 `import` + RSS 差值测量得到的精确数据：

| 消耗来源 | 大小 | 占比 | 能否优化 |
|---------|------|------|---------|
| Node.js 运行时 + V8 编译缓存 | ~100MB | 52% | ❌ 不可控 |
| ink + react（终端 UI 框架） | ~30MB | 15% | ❌ 核心框架无法替换 |
| @langchain/openai（LLM SDK） | ~26MB | 13% | ✅ **可优化** |
| bootstrap 其他（MCP/memory/skills） | ~36MB | 19% | ⚠ 部分可懒加载 |
| FileIndex（paths[] + Fzf） | ~1.5MB | **0.8%** | ❌ 已经很小 |

### 5.2 走过的弯路和纠偏

#### 弯路 1：以为文件索引是内存大头

直觉：194MB → 文件被加载到内存 → 优化文件索引

事实：FileIndex 只存路径字符串数组（不存文件内容），万级文件也就 ~1.5MB。已有 80 条 ignore 规则 + .gitignore + .ccodeignore 过滤。**这块没有优化空间。**

#### 弯路 2：以为需要持久化索引（JSON/SQLite）

调研了 VS Code、JetBrains、Cursor、ripgrep 的索引方案后发现：**VS Code 和 ripgrep 根本不索引**。靠现代 SSD + SIMD 暴力扫描就够了。cCli 的 Glob/Grep 工具底层也是 ripgrep，场景一致。持久化索引（无论 JSON 还是 SQLite）都是过度设计。

#### 弯路 3：想用 tree-shaking 优化 FileIndex 的文件列表

Tree-shaking 的方向是对的，但目标搞错了——不是 shake 文件索引中的文件条目，而是 shake **npm 依赖的 import 树**。

### 5.3 真正的洞察：Node.js 没有运行时 tree-shaking

**核心问题**：

```typescript
// 这行代码看起来是按需导入
import { ChatOpenAI } from '@langchain/openai'

// 但 V8 实际执行时：加载整个 @langchain/openai 的 index.js
// 及其所有依赖（embeddings、vector stores、parsers...）
// 即使你只用了 ChatOpenAI 一个类 → 26MB 全部驻留
```

webpack/esbuild 的 tree-shaking 是**构建时**静态分析 + 死代码消除。Node.js 运行时没有这个能力——`import` 就是 `require` 整个模块，所有副作用代码都会执行。

**cCli 全项目只有 2 处 import @langchain**：

```
providers/message-converter.ts → @langchain/core/messages  （4 个消息类）
providers/openai-compat.ts     → @langchain/openai         （1 个 ChatOpenAI 类）
```

但这 2 行 import 在启动时就被 V8 执行，26MB 立即驻留，无论用户当前选的是 Anthropic 还是 OpenAI 兼容模型。

### 5.4 行业文件索引方案调研（供参考）

| 工具 | 索引策略 | 持久化 | 适用规模 |
|------|---------|--------|---------|
| **VS Code** | 不持久化，实时遍历 + ripgrep 暴力扫描 | 无 | 万级文件 |
| **ripgrep** | 不索引，SIMD 加速正则 + 并行 | 无 | 10GB 以下 |
| **JetBrains** | 重度持久化（Stub + 倒排 + 三元组） | 自定义二进制 mmap | 百万级文件 |
| **Cursor** | VS Code + 向量嵌入（AI 语义检索） | 本地 HNSW 数据库 | 语义场景专用 |
| **ctags** | 排序文本 + 二分查找 | 纯文本 tags 文件 | 符号跳转专用 |
| **Zoekt** | 三元组倒排索引 | 自定义二进制 .zoekt | 代码搜索服务 |

**结论**：只有 JetBrains 做重度持久化索引（100-500MB 索引目录）。VS Code/ripgrep 证明了对万级文件规模，暴力扫描比维护索引更划算。

### 5.5 可执行的优化方向

| 优先级 | 方向 | 预期收益 | 原理 | 复杂度 |
|--------|------|---------|------|--------|
| **P1** | **LangChain dynamic import** | **-26MB**（Anthropic 用户） | 用到 OpenAI 兼容模型时才 `await import('@langchain/openai')`，Anthropic 用户完全不加载 | 低 |
| P2 | MCP 按需初始化 | -5~10MB | 没配 MCP server 时不加载 MCP 模块 | 低 |
| P3 | FileIndex 懒加载 | 省 100-300ms 启动耗时 | 延迟到首次 `@mention` 时扫描（内存不变，只是延迟加载） | 低 |
| ~~P?~~ | ~~文件索引持久化~~ | ~~无~~ | ~~已否决：VS Code 不索引，1.5MB 不值得优化~~ | — |
| ~~P?~~ | ~~去掉 LangChain~~ | ~~无法完全移除~~ | ~~ChatOpenAI 封装了 stream/tool_calls/重试，不只是一个类~~ | — |

#### P1 实现方案

```typescript
// providers/openai-compat.ts
// 现在：启动时加载（26MB 立即驻留）
import { ChatOpenAI } from '@langchain/openai'

// 优化后：首次调 OpenAI 兼容模型时才加载（单例缓存）
let _ChatOpenAI: typeof import('@langchain/openai').ChatOpenAI | null = null
async function getChatOpenAI() {
  if (!_ChatOpenAI) {
    const { ChatOpenAI } = await import('@langchain/openai')
    _ChatOpenAI = ChatOpenAI
  }
  return _ChatOpenAI
}

// chat() 方法中
async *chat(request: ChatRequest) {
  const ChatOpenAI = await getChatOpenAI()
  const model = new ChatOpenAI({ ... })
  // ...
}
```

用户只用 Anthropic → 26MB 完全不加载。用 OpenAI 兼容 → 首次调用时加载一次，后续缓存。

### 5.6 已否决的方案：切换 Bun 运行时

实测对比（同一台机器、同一批依赖）：

| 场景 | Node.js v20 | Bun v1.3 | 结论 |
|------|------------|----------|------|
| 空进程 | 30MB | 101MB | Bun 高 3.4× |
| 加载 LangChain+Ink+Anthropic | 92MB | 309MB | Bun 高 3.4× |

**Bun 内存比 Node.js 高 3 倍以上。** JavaScriptCore（Safari 引擎）的 JIT 编译器（DFG + FTL）预编译更多代码路径，空间换时间。Bun 的优势是启动速度和原生 TS 支持，不是内存。此方案否决。

### 5.7 最终结论

**194MB 对 CLI Agent 是正常水平**（Claude Code 自身 200-300MB）。P1 优化在 Anthropic 用户场景下可降到 ~168MB。

短期不需要做任何内存优化。如果未来需要，优先做 dynamic import 懒加载（最低成本）。去 LangChain 用 openai 原生 SDK（省 ~20MB）是个大重构，ROI 不高。
