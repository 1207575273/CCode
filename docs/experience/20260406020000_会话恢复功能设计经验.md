# 会话恢复功能设计经验

> 创建时间：2026-04-06 02:00
> 关联功能：Web 端历史会话恢复、CLI /resume 命令

---

## 一、需求本质

用户在 Web 端浏览历史会话时，想要**一键回到那个对话现场继续聊天**，而不是只能看回放。等同于 CLI 端的 `/resume` 命令。

## 二、架构选择：HTTP API vs WebSocket

### 为什么选 HTTP

| 方案 | 描述 | 优劣 |
|------|------|------|
| **WS 消息** | Web 通过 WebSocket 发 `{ type:'resume', sessionId }` | ConversationsPage 需要建 WS 连接（它本来只用 HTTP 取数据） |
| **HTTP API** ✅ | Web POST `/api/session/resume` | 复用现有 HTTP 调用，无需新建 WS；API 通过 eventBus 通知 CLI |

**结论**：ConversationsPage 是纯展示页，不需要实时通信。HTTP POST 触发 → eventBus 通知 CLI → CLI 执行 loadSession → Bridge 推 session_init → ChatPage 自动渲染。链路已有，只需加一个 API 入口。

## 三、恢复时不切换模型

### 踩过的坑

原始实现中 `loadSession` 会执行：
```typescript
if (snapshot.provider) setCurrentProvider(snapshot.provider)
if (snapshot.model) setCurrentModel(snapshot.model)
```

这导致恢复一个用 glm-5 聊的历史会话后，当前模型被切回 glm-5——但用户可能已经切到了 claude-opus-4-6。

### 正确做法

**恢复只加载消息历史和 LLM 上下文，provider/model 保持当前配置不变。**

历史消息中每条 assistant 消息携带的 `model` / `provider` 字段仅用于展示（"这条回复是 glm-5 生成的"），不影响后续请求使用哪个模型。

### 类比

就像打开一个别人写的文档继续编辑——你用你自己的编辑器设置，不会因为文档是用 Vim 写的就自动切到 Vim。

## 四、用户感知反馈

### 问题

恢复会话后 CLI 端和 Web 端都没有明显变化，用户不确定"恢复成功了没有"。

### 解决

两端都追加一条 system 消息：
```
已恢复会话 019d5abc...，12 条历史消息已加载，当前模型: GLM-4.6V-Flash
```

**原则**：涉及状态切换的操作，必须给用户明确的文字反馈。不能只靠 UI 状态变化（如消息列表刷新）——用户可能不确定是恢复了还是页面刷新了。

## 五、完整数据流

```
Web 列表页/详情页
  → 点击 [恢复] / [恢复对话]
  → POST /api/session/resume { sessionId }
  → Dashboard API: eventBus.emit('resume_session')
  → CLI useChat: loadSession(sessionId)
      → sessionStore.loadMessages() — 恢复消息快照
      → contextManager.restoreHistory() — 恢复 LLM 上下文
      → sessionStore.append(session_resume) — 标记恢复点
      → updateBridgeSession() — 通知 Bridge
      → appendSystemMessage() — CLI 端提示
  → Bridge: 推 session_init 给 Web
  → ChatPage: 渲染历史消息 + system 提示
  → 用户继续输入提问
```
