# AgentLoop 事件与 UI 状态同步

> 创建时间：2026-04-06 02:01
> 关联问题：用户以为 Agent 空闲但实际还在工作，误发消息导致中断

---

## 一、问题现象

用户反馈：**"CCode 有时候还在干活但 loading 态不出，让我以为它没干活了，结果再发消息就变成打断了。"**

CLI 端没有 loading 提示，Web 端也看不到任何活动迹象。但实际上 AgentLoop 正在执行多轮工具调用。

## 二、根因分析

### 2.1 AgentLoop 的多轮执行模型

```
用户提问
  → LLM 第 1 轮：回复工具调用（无文字）
  → 执行工具 A（tool_start → tool_done）
  → 执行工具 B（tool_start → tool_done）
  → LLM 第 2 轮：回复工具调用（无文字）     ← ⚠ 这个间隙 UI 什么都没有
  → 执行工具 C
  → LLM 第 3 轮：回复文字"完成了"
  → done
```

### 2.2 UI 状态的空白窗口

| 时间点 | streamingMessage | toolEvents | UI 表现 |
|--------|-----------------|------------|---------|
| LLM 第 1 轮开始 | 空 | 空 | **空白** ← 问题 |
| 工具 A 执行中 | 空 | [A: running] | 有工具指示器 |
| 工具 A 完成 | 空 | 空 | **空白** ← 问题 |
| LLM 第 2 轮开始 | 空 | 空 | **空白** ← 问题 |
| LLM 第 3 轮回文字 | "完成了" | 空 | 有流式文字 ✅ |

**"空白"时刻**就是用户以为空闲的原因。`isStreaming` 虽然一直是 `true`（InputBar 变灰），但没有任何可见的活动内容。

### 2.3 关键发现：AgentLoop 已有 `llm_start` 事件，但 UI 没处理

```typescript
// agent-loop.ts 第 249 行
yield { type: 'llm_start', provider, model, messageCount: messages.length }
```

这个事件在**每轮 LLM 调用开始时**都会触发，包括多轮工具调用中的每一次重新调 LLM。但 useChat 和 ChatPage 都没有处理它。

## 三、AgentLoop 事件全景

| 事件 | 触发时机 | UI 应该做什么 |
|------|---------|-------------|
| `llm_start` | 每轮 LLM 调用开始 | **显示"思考中..."** ← 本次修复 |
| `text` | LLM 流出文字 | 流式展示文字（覆盖"思考中"） |
| `thinking` | Extended Thinking 内容 | 展示思考过程 |
| `tool_start` | 工具开始执行 | 显示工具执行指示器 |
| `tool_done` | 工具执行完成 | 更新指示器为完成态 |
| `llm_done` | 一轮 LLM 调用结束 | 更新 token 统计 |
| `permission_request` | 需要用户授权 | 弹出权限确认对话框 |
| `user_question_request` | AskUser 工具 | 弹出问卷表单 |
| `subagent_progress` | SubAgent 进度 | 更新 SubAgent 面板 |
| `error` | 发生错误 | 显示错误信息 |
| `done` | AgentLoop 全部结束 | 关闭 streaming 态 |

**关键洞察**：`llm_start` 是唯一能填补"工具完成 → LLM 重新调用"间隙的事件。不处理它，UI 在多轮工具调用场景下必然有"假空闲"。

## 四、修复方案

### CLI 端（useChat.ts）

```typescript
} else if (event.type === 'llm_start') {
  // 每轮 LLM 调用开始时显示思考提示
  accumulated = ''
  setStreamingMessage('⏳ 思考中...')
}
```

### Web 端（ChatPage.tsx）

```typescript
case 'llm_start':
  setStreaming('⏳ 思考中...')
  break
case 'text':
  // 首次收到文字时覆盖"思考中"占位
  if (turnTextRef.current === '') setStreaming('')
  turnTextRef.current += event.text
  setStreaming(prev => prev === '⏳ 思考中...' ? event.text : prev + event.text)
  break
```

### 修复后的时间线

| 时间点 | UI 表现 |
|--------|---------|
| LLM 第 1 轮开始 | **⏳ 思考中...** ✅ |
| 工具 A 执行中 | 工具指示器 ✅ |
| 工具 A 完成 | 工具完成标记 |
| LLM 第 2 轮开始 | **⏳ 思考中...** ✅ |
| LLM 第 3 轮回文字 | "完成了"（流式） ✅ |

全程无空白窗口。

## 五、设计原则总结

### 原则 1：AgentLoop 事件是 UI 的唯一真相来源

AgentLoop 通过 AsyncGenerator yield 出所有状态变化事件。UI 层不应该"猜测"当前状态，而应该**对每个事件做出反应**。漏处理一个事件 = UI 状态与实际不同步。

### 原则 2：任何"等待"状态都必须有可见指示

| 等待什么 | 指示方式 |
|---------|---------|
| LLM 正在生成 | `⏳ 思考中...` → 流式文字 |
| 工具正在执行 | 工具名 + spinner |
| 等待用户授权 | 权限对话框 |
| 等待用户填表 | 问卷表单 |
| 等待网络响应 | 至少有 spinner |

**绝对不能有"什么都没显示但实际在工作"的状态。**

### 原则 3：事件驱动 > 轮询 > 猜测

```
✅ eventBus.on('llm_start')  → 立即显示"思考中"
⚠  setInterval 轮询 isStreaming → 延迟 + 浪费
❌ 不做任何处理              → 用户困惑
```

## 六、AgentLoop 事件利用率审计

| 事件 | SessionLogger | TokenMeter | CLI UI | Web UI | 是否充分利用 |
|------|:---:|:---:|:---:|:---:|:---:|
| `llm_start` | ✅ | - | ✅ (本次) | ✅ (本次) | ✅ |
| `text` | - | - | ✅ | ✅ | ✅ |
| `thinking` | - | - | ✅ | ✅ | ✅ |
| `tool_start` | ✅ | - | ✅ | ✅ | ✅ |
| `tool_done` | ✅ | - | ✅ | ✅ | ✅ |
| `llm_done` | ✅ | ✅ | ✅ | - | ⚠ Web 端未用 llm_done 更新统计 |
| `permission_request` | - | - | ✅ | ✅ | ✅ |
| `user_question_request` | - | - | ✅ | ✅ | ✅ |
| `subagent_progress` | - | - | ✅ | ✅ | ✅ |
| `subagent_done` | - | - | ✅ | ✅ | ✅ |
| `error` | ✅ | - | ✅ | ✅ | ✅ |
| `done` | ✅ | - | ✅ | ✅ | ✅ |
| `llm_error` | ✅ | - | - | - | ⚠ UI 未显示 LLM 调用失败细节 |
| `tool_fallback` | ✅ | - | - | - | ⚠ UI 未显示工具 fallback 信息 |

**待优化**：
- `llm_done`：Web 端可用来更新本轮 token 消耗展示
- `llm_error`：UI 可显示"LLM 调用失败，正在重试..."
- `tool_fallback`：UI 可显示"工具 X 不可用，已降级为 Y"
