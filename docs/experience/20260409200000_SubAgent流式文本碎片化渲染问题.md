# SubAgent 流式文本碎片化渲染问题

> 日期：2026-04-09
> 分类：流式渲染 / 事件架构
> 影响范围：CLI SubAgentPanel + Web SubAgentDrawer

---

## 一、现象

SubAgent 的文本输出在 CLI 和 Web 端都呈现为"逐词换行"：

```
📝 项目
📝 已经
📝 初始化
📝 过了
📝 。
📝 接下来
📝 添加
📝 依赖
📝 并
📝 编写
📝 代码
📝 。
```

预期应该是一整段话：

```
📝 项目已经初始化过了。接下来添加依赖并编写代码。
```

---

## 二、根因分析

### 流式输出的本质

LLM 的流式响应是**逐 token 返回**的，每个 chunk 通常只有 2-4 个中文字符。这是所有流式 LLM API 的标准行为，不是 bug。

主 Agent 的文本输出不受影响，因为主 Agent 的 text chunk 在 UI 层做了流式拼接渲染（ChatView/MessageBubble 持有一个累积的字符串，每个 chunk 追加上去）。

### 但 SubAgent 走了不同的路径

SubAgent 的事件链路是：

```
AgentLoop yield { type: 'text', text: '项目' }     // 每个 chunk 2-4 字
    ↓
dispatch-agent.ts: 每个 chunk 都发一个独立事件
    eventBus.emit({ type: 'subagent_event', detail: { kind: 'text', text: '项目' } })
    ↓
store.ts: 每个事件都 append 为独立条目
    state.events.push({ type: 'text', text: '项目' })
    ↓
UI 层: 每个条目渲染为独立一行
    📝 项目
```

**核心问题**：主 Agent 的 text 是"流式拼接后渲染"，但 SubAgent 的 text 是"先存储为事件列表再渲染"——事件列表的设计天然把每个 chunk 当成独立事件。

### 架构层面的洞察

这是**事件驱动架构中"事件粒度"与"渲染粒度"不匹配**的经典问题：

| 层级 | 粒度 | 适合做什么 |
|------|------|-----------|
| LLM 流式 chunk | 2-4 字符/token | 逐字渲染动画 |
| AgentEvent | 与 chunk 1:1 | 事件日志、观测 |
| UI 渲染单元 | 一段完整文本 | 人类阅读 |

SubAgent 的 text 事件直接从"LLM chunk 粒度"映射到"UI 渲染单元"，中间缺少了**聚合层**。

---

## 三、修复方案

在事件存储层做聚合——连续的 text 事件合并到同一条，而非 append 新条目。

### 判断逻辑

```
收到 text 事件时：
  如果 events 数组的最后一条也是 text → 追加到它的 text 字段
  如果最后一条是其他类型（tool_start/tool_done/error）→ 创建新的 text 条目
```

这样 tool 调用之间的文本各自聚合为一段，tool 调用不会被文本合并打断。

### 修改点

**CLI 端 — `src/tools/agent/store.ts`**：

```typescript
case 'text': {
  const state = store.get(agentId)
  const lastEvent = state?.events[state.events.length - 1]
  if (lastEvent?.type === 'text') {
    // 合并到上一条
    lastEvent.text = (lastEvent.text ?? '') + event.text
  } else {
    // 新建一条
    appendSubAgentEvent(agentId, { type: 'text', timestamp: now, text: event.text })
  }
  break
}
```

**Web 端 — `web/src/pages/ChatPage.tsx`**：

```typescript
if (d.kind === 'text') {
  const lastEvent = existing.events[existing.events.length - 1]
  if (lastEvent?.type === 'text') {
    // 不可变更新：复制数组，合并最后一条
    const merged = [...existing.events]
    merged[merged.length - 1] = { ...lastEvent, text: (lastEvent.text ?? '') + (d.text ?? '') }
    next.set(event.agentId, { ...existing, events: merged })
  } else {
    next.set(event.agentId, {
      ...existing,
      events: [...existing.events, { type: 'text', text: d.text }],
    })
  }
}
```

注意 Web 端必须用**不可变更新**（React state 要求），不能直接 mutate。

---

## 四、为什么主 Agent 没有这个问题

主 Agent 的文本渲染走的是完全不同的路径：

```
AgentLoop yield { type: 'text', text: '项目' }
    ↓
useChat.ts: accumulatedText += event.text    // 累积到一个字符串
    ↓
ChatView: 渲染 accumulatedText              // 一个完整的气泡
```

主 Agent 用的是**累积字符串 + 单气泡渲染**模式，天然不存在碎片化问题。

SubAgent 用的是**事件列表 + 逐条渲染**模式，因为 SubAgent 需要在一个面板里混合展示 text / tool_start / tool_done / error 等不同类型的事件。这个设计是合理的，只是在 text 聚合这一环缺了处理。

## 五、通用教训

### 流式系统中的"聚合层"设计模式

当系统架构是 `流式源 → 事件存储 → UI 渲染` 时，需要在存储层或渲染层加入聚合逻辑：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **存储层聚合**（本次采用） | 一次聚合，CLI/Web 都受益 | 丢失了原始 chunk 粒度 | 不需要逐字动画的场景 |
| 渲染层聚合 | 保留原始事件，灵活控制 | 每个 UI 端都要实现一遍 | 需要逐字动画 + 最终合并的场景 |
| 防抖聚合（debounce） | 时间窗口内的 chunk 合并 | 引入延迟 | 高频事件降频展示 |

### 检查清单

以后遇到"流式数据 → 列表存储 → 逐条渲染"的模式时，问自己：

1. **同类型事件是否需要合并？** — text chunk 几乎总是需要
2. **合并的边界是什么？** — 本例中是"遇到非 text 事件就断开"
3. **合并在哪一层做？** — 尽量在存储层，避免多端重复实现
4. **可变 vs 不可变？** — 内部 store 可以 mutate，React state 必须不可变更新
