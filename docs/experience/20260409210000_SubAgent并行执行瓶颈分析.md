# SubAgent 并行执行瓶颈分析

> 日期：2026-04-09
> 分类：Agent 架构 / 并行执行
> 来源：真实用户反馈——"创建了多个 agent 但没有真正并行干活"

---

## 一、现象

用户在使用 SubAgent 功能时发现：虽然 LLM 派发了多个子 Agent，但它们是**逐个串行执行**的——第一个跑完才开始第二个，没有发挥出并行的能力。

---

## 二、根因分析

瓶颈有两层，一层是工程硬限制，一层是 LLM 行为倾向。

### 2.1 工程层：StreamableTool 被强制归入串行路径

CCode 的工具执行分为两组：

```
LLM 返回 N 个 tool_calls
  → classifyToolCalls() 分组
      → safe（普通工具）→ Promise.allSettled 并行执行
      → dangerous（危险工具 + StreamableTool）→ for 循环逐个执行
```

**关键代码**（`parallel-executor.ts:67`）：

```typescript
if (!tool || registry.isDangerous(tc.toolName) || isStreamableTool(tool)) {
  dangerous.push(tc)  // dispatch_agent 是 StreamableTool → 归入 dangerous
}
```

**串行执行**（`agent-loop.ts:389-392`）：

```typescript
for (const tc of dangerous) {
  yield* this.#executeOneTool(tc, history)  // 逐个 await，A 跑完才跑 B
}
```

即使 LLM 在一轮响应中返回 3 个 `dispatch_agent` tool_call，也会串行执行。

### 2.2 为什么 StreamableTool 不能并行？

`dispatch_agent` 实现了 `StreamableTool` 接口，它的 `stream()` 方法是一个 AsyncGenerator，通过 `yield` 实时透传子 Agent 的中间事件（进度、工具调用、文本等）。

AgentLoop 通过 `yield*` 消费这些事件：

```typescript
result = yield* (tool.stream(args, ctx) as AsyncGenerator<AgentEvent, ToolResult>)
```

`yield*` 是**同步委托**——在当前 generator 消费完所有事件之前，控制权不会回来。这意味着：

- 同一个 AsyncGenerator 中不能同时 `yield*` 两个子 generator
- JavaScript 的 AsyncGenerator 没有"多路复用"能力
- 要并行就必须放弃 `yield*`，改用事件合并机制

这是 **JavaScript AsyncGenerator 的语言层面限制**，不是简单的代码 bug。

### 2.3 LLM 行为层：模型倾向于串行派发

即使工程上解决了并行问题，LLM 也有两个行为倾向导致串行：

| 行为 | 说明 |
|------|------|
| 一次只派一个 agent | 弱模型（GLM-5）不擅长在一轮响应中返回多个 tool_call，通常是派一个、等结果、再派下一个 |
| 不主动用后台模式 | LLM 倾向于前台同步等结果（默认行为），很少主动设 `run_in_background=true` |

---

## 三、现有的并行能力：后台模式

`dispatch_agent` 支持 `run_in_background=true` 参数，后台模式下：

```typescript
if (runInBackground) {
  // 启动子 Agent → 立即返回 agentId，不等执行完
  runSubAgentInBackground(subLoop, ...)
  return { success: true, output: JSON.stringify({ status: 'async_launched', agentId }) }
}
```

后台子 Agent 在独立的 `void (async () => {...})()` 中运行，**多个后台 SubAgent 是真正并行的**。

但这依赖 LLM 主动在 tool_call 参数中设置 `run_in_background: true`，且后续主动用 `task_output` 去读取结果。

---

## 四、改进方向

### 方向 A：Prompt 引导（短期，低成本）

在 System Prompt 的工作模式中明确引导 LLM 使用后台并行模式。这不解决工程限制，但能让 LLM 更积极地利用现有的 `run_in_background` 能力。

**适用性**：对 Claude 4 等强模型效果好，对 GLM-5 效果有限。

### 方向 B：前台多路复用（中期，大改动）

重构 AgentLoop 的事件消费机制，让多个 StreamableTool 的事件可以交错 yield：

```
方案思路：
1. 多个 dispatch_agent 同时启动
2. 每个子 Agent 的事件推入共享队列
3. AgentLoop 从队列中逐个 yield（交错 merge）
4. 所有子 Agent 完成后，合并 tool_result 推入 history
```

**复杂度**：需要改 AgentLoop 核心循环、classifyToolCalls 分类逻辑、tool_result 合并策略。估计 3-5 天工作量。

**风险**：事件交错可能影响 SessionLogger 的 parentUuid 链条、SubAgentPanel 的渲染顺序。

### 方向 C：自动后台化（中期，中等改动）

在 `classifyToolCalls` 中检测：如果同一轮有多个 `dispatch_agent` tool_call，自动将后续的改为后台模式：

```typescript
// 伪代码
const dispatchCalls = dangerous.filter(tc => tc.toolName === 'dispatch_agent')
if (dispatchCalls.length > 1) {
  // 第一个前台执行，其余自动改为后台
  for (let i = 1; i < dispatchCalls.length; i++) {
    dispatchCalls[i].args.run_in_background = true
  }
}
```

**优点**：改动小，复用现有后台机制。
**缺点**：LLM 不知道被自动改了参数，后续可能不会主动用 `task_output` 读结果。需要配合注入一条提示消息。

---

## 五、决策参考

| 方向 | 工作量 | 效果 | 风险 | 建议优先级 |
|------|--------|------|------|-----------|
| A. Prompt 引导 | 0.5 天 | 对强模型有效 | 无 | **P0 先做** |
| B. 前台多路复用 | 3-5 天 | 彻底解决 | 架构风险高 | P2 长期规划 |
| C. 自动后台化 | 1-2 天 | 中等 | LLM 不感知参数被改 | P1 短期可选 |
