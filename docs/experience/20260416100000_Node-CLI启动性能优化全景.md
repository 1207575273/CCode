# 第一部分：底层原理

## 一、CJS 与 ESM —— 两个时代的模块系统

### 1.1 历史脉络

```
2009  Node.js 诞生 → 发明 CommonJS（当时浏览器没有模块系统，require/module.exports 是 Node 的私有发明）
2015  ES6 标准发布 → ESM (import/export) 成为 JavaScript 官方模块规范
2017  浏览器开始原生支持 ESM（<script type="module">）
2020  Node.js 12+ 正式支持 ESM（之前只有 CJS）
2024  生态完成 CJS → ESM 迁移（ink 6、chalk 5、execa 等纯 ESM 包不再提供 CJS 入口）
```

**CJS 是 Node.js 的"方言"，ESM 是 JavaScript 的"普通话"。** CJS 只能在 Node.js 里跑，ESM 浏览器和 Node.js 都能跑。

### 1.2 核心差异

| | CommonJS | ES Module |
|---|---------|-----------|
| 语法 | `require()` / `module.exports` | `import` / `export` |
| 规范归属 | Node.js 私有（非 ECMA 标准） | **ECMAScript 官方标准** |
| 加载时机 | **运行时**——遇到 `require()` 才加载 | **编译时**——执行前先构建完整依赖图 |
| 静态分析 | 不可能（`require(variable)` 合法） | 可以（import 路径必须是字面量） |
| Tree-shaking | 不支持（无法静态分析哪些导出被用了） | **支持**（打包工具的基础能力） |
| 顶层 await | 不支持 | 支持 |
| 浏览器 | 不支持 | 原生支持 |
| `this` 指向 | `module.exports` | `undefined` |
| 循环依赖 | 返回已加载的部分（可能是不完整对象） | TDZ 报错（更安全） |

### 1.3 ESM 的两种加载方式

**ESM 不是只有"全部预加载"，它支持静态和动态两种 import：**

```typescript
// ─── 静态 import（编译时加载） ───
// 必须在文件顶部，不能放 if 里，不能用变量
// Node.js 在执行任何代码前，先把整棵依赖树全部加载完
import { render } from 'ink'        // 不管用不用，ink 整棵依赖树都被加载

// ─── 动态 import（运行时按需加载） ───
// 可以放在任意位置：if/函数/循环/try-catch 里
// 只有代码执行到这里时才加载，返回 Promise
if (needInk) {
  const { render } = await import('ink')   // 走到这行才加载 ink
}
```

| | 静态 `import x from 'y'` | 动态 `await import('y')` |
|---|--------------------------|--------------------------|
| 位置 | 只能在文件顶部 | 任意位置 |
| 时机 | 模块加载阶段（代码执行前） | 代码运行时（走到才触发） |
| 效果 | **整棵依赖树全部预加载** | **按需加载，不走到不加载** |
| Tree-shaking | 支持 | 不支持（打包工具不知道运行时会不会走到） |
| 返回值 | 直接拿到导出值 | 返回 `Promise<Module>` |
| 典型场景 | 核心依赖（一定会用的） | 可选功能、路由懒加载、条件分支 |

### 1.4 为什么 ESM 是正确方向

即使 ESM 静态 import 在启动时"看起来更慢"（预加载整棵依赖树），ESM 仍然是唯一正确选择：

1. **Tree-shaking 只有 ESM 能做** —— CJS 的 `require()` 是运行时动态调用，打包工具无法在编译期确定哪些导出被使用。
2. **全量 bundle 后差异消失** —— I/O 瓶颈被消除后，V8 对 ESM 有专门的优化路径，bundle 后 ESM 反而略快。
3. **生态单向迁移不可逆** —— ink 6、chalk 5、execa 等主流库已是纯 ESM。
4. **浏览器兼容** —— 同构代码（CLI + Web 共享逻辑）只有 ESM 走得通。
5. **ESM 的"慢"是 Node.js 实现问题** —— 不是规范问题。Bun 里 ESM 模块解析快 5-10 倍。

**总结：CJS 是历史包袱，ESM 是未来。性能问题通过 bundle 解决，而不是退回 CJS。**

## 二、Node.js 启动一个 CLI 到底发生了什么

```
[1] Node.js 进程启动          ~50ms
    ├─ V8 引擎初始化（堆分配、内置对象创建）
    ├─ libuv 事件循环初始化
    └─ 内置模块注册（fs/path/http 等 C++ binding）

[2] 入口文件加载              ~10ms
    ├─ 读取 cli.js 文件内容
    ├─ V8 解析为 AST → 编译为字节码
    └─ 执行顶层代码

[3] 依赖模块解析（瓶颈所在）   ~100ms ~ 数秒
    ├─ 遇到 import/require
    ├─ 模块解析算法（向上遍历 node_modules 目录树）
    ├─ 递归加载所有依赖的依赖
    └─ 每个模块都重复 [读取文件 → 解析 → 编译 → 执行]

[4] 业务代码执行               ~10ms
    └─ 解析参数、执行逻辑、输出结果
```

**阶段 [3] 是唯一可优化的大头。** 其他阶段加起来不超过 100ms。

### 2.1 模块解析的 I/O 开销

每个 `import 'xxx'` 至少 2-3 次文件系统读取（目录查找 + package.json + 入口文件）。32 个直接依赖 × 每个 5-10 个间接依赖 = 数百次文件 I/O。

| 操作 | Linux ext4 | Windows NTFS | macOS APFS |
|------|-----------|-------------|-----------|
| 单次文件 stat | ~0.01ms | ~0.05ms | ~0.02ms |
| 读取 package.json | ~0.03ms | ~0.15ms | ~0.05ms |

**Windows NTFS 的小文件随机读取比 Linux ext4 慢 3-5 倍**，这是同一个项目 Windows 上启动比 Linux 慢的根本原因。

### 2.2 核心瓶颈的数学模型

```
CLI 启动时间 = T_runtime + T_resolve + T_parse + T_execute

T_runtime:  运行时启动（Node 50ms / Bun 20ms）
T_resolve:  模块解析 = N_files × T_stat_per_file        ← 最大瓶颈
T_parse:    V8 解析编译 = Total_bytes × T_parse_per_byte
T_execute:  业务代码执行 ≈ 10ms（忽略不计）
```

**结论：T_resolve 是最大瓶颈。任何能减少文件系统 I/O 次数的方案都有效。**

## 三、各加速技术的核心原理

### 3.1 Bundle（打包）

**核心原理**：把数百个模块文件合并成 1 个文件，消除模块解析的文件系统 I/O。

1 次大文件顺序读取 比 500 次小文件随机读取快 10-50 倍（操作系统层面有预读优化）。

| 工具 | 底层 | 特点 |
|------|------|------|
| **esbuild** | Go | 最快（100ms 级），tree-shaking 能力中等 |
| **rollup** | JS | 最强 tree-shaking，但慢 |
| **webpack** | JS | 生态最全，配置最复杂 |
| **tsup** | esbuild 封装 | TS 友好，零配置 |
| **tsdown** | rollup 封装 | Claude Code 使用的工具 |
| **@vercel/ncc** | webpack 封装 | Vercel CLI 使用，专为 Node CLI 打包 |

**局限**：Native addon（.node）不能 bundle；WASM 不能 bundle；`__dirname`/`import.meta.url` 路径会变。

### 3.2 Tree-Shaking（摇树优化）

**核心原理**：静态分析 import/export 关系，删除未被引用的代码。

**生效条件**：必须是 ESM + 无副作用（side effects）+ package.json `"sideEffects": false`。

**现实效果**：纯工具库（lodash-es）效果极好，框架（react、ink）效果有限。

### 3.3 Code Splitting（代码分割）

**核心原理**：把一个大 bundle 拆成多个 chunk，按需加载。需要配合 `await import()` 动态导入。

### 3.4 V8 Snapshot（快照）

**核心原理**：把 V8 堆内存状态序列化为二进制文件，下次启动直接反序列化恢复，跳过解析和编译。

Node.js 自身的内置模块就是用 snapshot 加载的，`require('fs')` 几乎不花时间。

限制：快照中不能包含外部资源引用（文件句柄、网络连接、Timer）。

### 3.5 单可执行文件（SEA）

**核心原理**：把 JS 代码注入到 Node.js 二进制文件中。分发简单（无需装 Node），但体积大（~50MB）。

### 3.6 Bun 运行时

**核心原理**：用 Zig 重写 Node.js 运行时，底层用 JavaScriptCore（WebKit）替代 V8。

- 模块解析：自研 resolver，内存级 trie 查找（不遍历目录）
- 内置 transpiler：`.ts` 不需要外部 tsx
- **为什么内存更高**：JSC 的 JIT 编译器有 4 层（LLInt → Baseline → DFG → FTL），比 V8 的 3 层更激进地预编译到原生机器码。用空间换时间，这是 JSC 的设计哲学，不是 bug。

### 3.7 tsx / ts-node 运行时转译

**核心原理**：注册 ESM loader hook，拦截 `.ts` 文件即时转译为 JS。tsx 基于 esbuild（Go），比 ts-node（JS TypeScript Compiler）快 50 倍，但无持久缓存。

### 3.8 @vercel/ncc

**核心原理**：基于 webpack 的 Node CLI 打包工具。自动处理 `__dirname` 重写、native addon 复制、动态 `require(variable)` 的 context module。

各方案消除的瓶颈项：

| 方案 | 消除 T_resolve | 消除 T_parse | 消除 T_runtime |
|------|--------------|-------------|---------------|
| Bundle | ✅ | 部分 | ❌ |
| Tree-shaking | 间接 | ✅ | ❌ |
| V8 Snapshot | ✅ | ✅ | ❌ |
| SEA | ✅ | 部分 | ❌ |
| Bun | ✅ | 部分 | ✅ |
| Code Splitting | 部分 | 部分 | ❌ |

## 十一、踩坑实录

### 坑 1：`Could not resolve "react-devtools-core"`

**现象**：加完 `noExternal: [/.*/]` 直接 build，esbuild 报错无法解析 `react-devtools-core`。

**根因**：ink 的 `build/devtools.js` 里有一行静态 import：

```js
import devtools from 'react-devtools-core';
```

但 `react-devtools-core` 是 ink 的 **optional peer dependency**，pnpm 默认不装可选 peer。esbuild 在 bundle 阶段要解析所有静态 import 的实际路径，找不到就直接报错，即使这个代码路径生产永远走不到。

**解法**：用 esbuild plugin 把这个模块替换成空 stub：

```ts
{
  name: 'stub-react-devtools-core',
  setup(build) {
    build.onResolve({ filter: /^react-devtools-core$/ }, (args) => ({
      path: args.path,
      namespace: 'stub-devtools',
    }))
    build.onLoad({ filter: /.*/, namespace: 'stub-devtools' }, () => ({
      contents: 'export default {}',
      loader: 'js',
    }))
  },
}
```

**为什么不装 react-devtools-core**：它纯开发调试用，只在 `useDevTools()` 被调用时才激活，生产代码路径永远走不到。装了徒增 node_modules 体积和 bundle 产物。stub 是最干净的做法。

### 坑 2：`Error: Dynamic require of "os" is not supported`

**现象**：build 成功，`--version` 也正常，但跑 `--web` 立即抛错：

```
Error: Dynamic require of "os" is not supported
    at file:///dist/bin/ccli.js:12:9
    at node_modules/.../fast-glob/out/utils/path.js
    at __require2 (dist/bin/ccli.js:18)
```

**根因**：产物是 ESM（`format: ['esm']`），但 bundle 进来的 `fast-glob` 是 CJS 库，内部有 `const os = require('os')`。esbuild 在 ESM 产物里遇到 CJS 的 `require` 会生成一个**默认抛错的 `__require` shim**：

```js
throw Error('Dynamic require of "' + x + '" is not supported');
```

为什么默认抛错？因为 ESM 模块系统没有同步的 `require` —— Node 的 ESM 里想同步加载内置模块得用 `createRequire(import.meta.url)` 显式构造一个。

**解法**：在 banner 注入 `createRequire`，让全局 `require` 真正可用：

```ts
banner: {
  js: `#!/usr/bin/env node
import { createRequire as __ccodeCreateRequire } from 'node:module';
const require = __ccodeCreateRequire(import.meta.url);`,
}
```

这样 bundle 里的 `require('os')`、`require('path')` 等都能正常工作。

**为什么这个坑比坑 1 出现得晚**：坑 1 是 build-time 错误（esbuild 自己发现），坑 2 是 run-time 错误（只有真正执行到 `fast-glob` 的代码路径才触发）。`--version` 路径上不碰 `fast-glob`，所以短路逃过了这个坑，直到跑 `--web` 进入 glob 工具才爆。

### 坑 3：`Cannot find module '@libsql/win32-x64-msvc'`

**现象**：修完坑 2 再跑 `--web`，又一个新错：

```
Cannot find module '@libsql/win32-x64-msvc'
    at ...requireNative...
    at node_modules/.../libsql@0.5.22/node_modules/libsql/index.js
    at __require2 (dist/bin/ccli.js)
```

**注意错误栈里 `libsql/index.js` 是从 `__require2` 里被调用的** —— 意味着 libsql 自己被 bundle 进去了！

**根因**：我原本写的配置长这样：

```ts
noExternal: [/.*/],
external: ['libsql', 'jieba-wasm'],
```

直觉上以为 `external` 会从 `noExternal` 里扣掉。但 tsup 的实现下 `noExternal: [/.*/]` **优先级更高**，把字段 `external` 的名单直接吃掉了。结果就是 libsql 被强制 inline 进 bundle，而 libsql 运行时还要 `require('@libsql/<platform>-<arch>-<toolchain>')` 去加载平台特定的 native addon —— inline 的 libsql 用我们的 createRequire 从 `dist/bin/` 向上找 node_modules，找不到这个子包。

**验证方式**：直接看产物字符串里是否还有 libsql 的包路径：

```bash
node -e "const c=require('fs').readFileSync('dist/bin/ccli.js','utf8');
         console.log('libsql inlined?', c.includes('libsql@0.5.22/node_modules/libsql'))"
# → true（被 inline 了）
```

**解法**：放弃字段 `external`，改用 esbuild plugin 的 `onResolve` 在解析阶段直接标记为 external，绕过 tsup 的字段合并逻辑：

```ts
{
  name: 'force-external',
  setup(build) {
    build.onResolve({ filter: /^(libsql|jieba-wasm)(\/.*)?$/ }, (args) => ({
      path: args.path,
      external: true,
    }))
  },
}
```

改完之后 `libsql inlined? false`，真正 external 了。

**为什么正则要加 `(\/.*)?$`**：要同时拦住 `libsql`（顶层包）和 `libsql/index.js` 这种子路径 import，否则会漏网。

### 坑 4（未真正遇到但要警惕）：pnpm 非扁平结构 vs npm 扁平

pnpm 的 `node_modules` 是**符号链接结构**：

```
node_modules/
  libsql → .pnpm/libsql@0.5.22/node_modules/libsql
  .pnpm/
    node_modules/
      @libsql/win32-x64-msvc/       ← 提到 .pnpm 下，不在顶层
```

Node 的模块解析算法**不会看** `.pnpm/node_modules/`。从 `dist/bin/` 向上找 `cCli/node_modules/` 顶层时，`@libsql/win32-x64-msvc` 不可见。

本地调试 `node dist/bin/ccli.js` 理论上应该失败，但实测没失败 —— 因为 Node 解析 `libsql` 包时会走到 `node_modules/libsql/index.js`（通过符号链接到 `.pnpm/libsql@0.5.22/...`），libsql 内部的 `require('@libsql/...')` 是**从 libsql 自己的位置向上找**，能找到 `.pnpm/libsql@0.5.22/node_modules/@libsql/` 里的平台包。

**用户发布给 npm 后不用担心**：用户 `npm i -g` 是扁平结构，`@libsql/<platform>` 装在顶层 `node_modules/`，直接可达。

## 十二、最终的 `tsup.config.ts`

```ts
import { defineConfig } from 'tsup'
import { cp } from 'node:fs/promises'

export default defineConfig({
  entry: ['bin/ccli.ts'],
  outDir: 'dist/bin',
  format: ['esm'],
  target: 'node20',
  splitting: false,
  sourcemap: true,
  clean: true,
  banner: {
    // ESM 产物需要 createRequire 才能让 CJS 依赖（如 fast-glob）内部的 require('os') 工作
    // 否则 esbuild 生成的 __require shim 会抛 "Dynamic require of X is not supported"
    js: `#!/usr/bin/env node
import { createRequire as __ccodeCreateRequire } from 'node:module';
const require = __ccodeCreateRequire(import.meta.url);`,
  },
  tsconfig: 'tsconfig.build.json',
  // noExternal: [/.*/] 会覆盖字段 external，真正 external 靠下方 force-external plugin 拦截
  noExternal: [/.*/],
  esbuildPlugins: [
    // native addon / WASM 不能 inline bundle，必须运行时从 node_modules 解析
    {
      name: 'force-external',
      setup(build) {
        build.onResolve({ filter: /^(libsql|jieba-wasm)(\/.*)?$/ }, (args) => ({
          path: args.path,
          external: true,
        }))
      },
    },
    // ink 的 react-devtools-core 是 optional peer，生产永不走到但静态 import 阻止 bundle
    // 用空 stub 替换
    {
      name: 'stub-react-devtools-core',
      setup(build) {
        build.onResolve({ filter: /^react-devtools-core$/ }, (args) => ({
          path: args.path,
          namespace: 'stub-devtools',
        }))
        build.onLoad({ filter: /.*/, namespace: 'stub-devtools' }, () => ({
          contents: 'export default {}',
          loader: 'js',
        }))
      },
    },
  ],
  // bundle 后 src/skills/engine/store.ts 里 join(__dirname, '..', 'builtin')
  // 期望 dist/skills/builtin 存在 —— 构建完成后把源码资源复制过去
  async onSuccess() {
    await cp('src/skills/builtin', 'dist/skills/builtin', { recursive: true })
    console.log('[tsup] copied skills/builtin → dist/skills/builtin')
  },
})
```

## 十四、经验教训

1. **esbuild warning 清零 ≠ 运行时无错**。Build 成功只验证了 build-time 的路径解析，CJS `require` 是运行时才抛。必须真正执行 bundle 产物才算过关。`--version` 走不到的代码路径要靠真实业务场景（`--web` / 交互模式）才能暴露问题。

2. **tsup 字段 `external` 在 `noExternal: [/.*/]` 下会被覆盖**。这个是 tsup 的字段合并行为而非 bug。安全做法是用 `esbuildPlugins` 的 `onResolve({ external: true })` 在解析阶段直接标记，绕过字段合并。

3. **验证 external 生不生效最直接的办法是 grep bundle 产物**，看目标包的路径字符串还在不在。比看文件体积或看 warning 更可靠。

4. **ESM + CJS 混用场景必须注入 createRequire**。只要依赖树里有任何 CJS 包用到 `require()`，ESM 产物就会炸。这是 bundle ESM 的标配模板。

5. **最小改动原则有效**。坚持"源码零改动，只改构建配置"，没有碰任何 `__dirname` / `import.meta.url` 代码，只靠维持 `dist/` 产物的相对目录结构就让 2 处路径敏感代码继续工作。如果一开始就想改源码，代价会大得多。

6. **Windows / pnpm 的差异对发包无影响**。用户实际 `npm i -g` 是扁平 `node_modules`，与我们的 bundle 产物完全兼容。本地 pnpm 调试能跑通，生产就没问题。

7. **实施时间比文档预估快一倍**。文档预估 3-4h，实际 2h。省出的时间主要来自：
   - 真实依赖数比预估少（26 vs 32）
   - 没有隐藏的 native addon（没有 yoga-layout）
   - 路径敏感代码只有 2 处且都不碰
