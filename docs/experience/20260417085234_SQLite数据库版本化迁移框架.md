# SQLite 数据库版本化迁移框架

> 建立日期：2026-04-17
> 代码位置：`cCli/src/persistence/db.ts`
> 关键性等级：⚠️ **高危 · 改错会导致用户本地数据损坏**

---

## 一、背景

### 1.1 CCode 是什么样的应用

CCode 是一个**客户端 CLI 工具**（类 Claude Code），每个用户在自己电脑上独立运行，各自有一份本地数据文件：

```
~/.ccode/
├── data/
│   └── ccode.db              ← SQLite 数据库（本文的主角）
├── sessions/
│   └── {project-slug}/
│       └── *.jsonl           ← 会话事件日志
└── memory/                   ← 用户记忆文件
```

**关键特征**：
- 不是服务端应用，**没有统一的数据库可以 DBA 手动改**
- 用户量 = 装了 CCode 的电脑数量，每台机器一份数据
- 升级代码 = 发布新版本 CLI → 用户下次启动时执行新代码
- **用户本地的 SQLite 文件和代码版本必须对得上**，否则崩溃或丢数据

### 1.2 SQLite 里存了什么

| 表 | 作用 | 数据价值 |
|----|------|---------|
| `usage_logs` | 每次 LLM 调用的 token / 费用 / 性能记录 | ⭐⭐⭐ 用户的计费凭证、使用历史 |
| `pricing_rules` | 各模型的计价规则 | ⭐⭐ 可重建（代码里有 seed） |
| `schema_comments` | 字段注释元数据 | ⭐ 可重建 |
| `memory_meta` | 记忆系统配置（embedding 维度等） | ⭐⭐ 丢了要重新配 |
| `memory_vectors` | 用户记忆的向量索引 | ⭐⭐⭐ 计算成本高，丢了要重算 |

**高星级的表（`usage_logs` / `memory_vectors`）一旦损坏，用户几乎不可能恢复**。费用历史丢了没法和供应商对账，记忆向量丢了要重新 embedding（可能花几小时 + 几美元 API 费）。

### 1.3 为什么选 SQLite（而不是文件 / JSON / 云数据库）

| 选项 | 评估 | 是否采用 |
|------|------|---------|
| JSON 文件 | 并发读写不安全、无法做复杂查询（按时间段聚合费用） | ❌ |
| 文件系统多文件 | 无事务、聚合查询要手写扫描 | ❌ |
| 云数据库 | 需要用户有账号、有网络、暴露数据给服务端 | ❌ 违背 CLI 本地化定位 |
| **SQLite** | 零配置、单文件、完整 SQL、事务、索引 | ✅ |
| PostgreSQL | 太重，用户要装 server | ❌ |

**SQLite 的唯一痛点**：schema 变更不如 PostgreSQL 灵活（不支持 DROP COLUMN、不支持 ALTER TYPE），但对 CCode 这种场景利大于弊。

### 1.4 为什么偏偏今天需要版本管理

**事件驱动**：2026-04-17 实施性能层观测时，要给 `usage_logs` 加三个字段：

```
ttft_ms    — 首 Token 延迟（ms）
tps        — 输出吞吐率（tokens/sec）
duration_ms — 端到端耗时（字段早就存在，但从未写入）
```

这不是第一次改 schema——之前加过 `cache_read` / `cache_write` 等字段。**但这次我们停下来意识到：再这么改下去就要出事了**。

### 1.5 老方案撑到什么时候开始失效

改造前的 `db.ts` 用的是最朴素的 SQLite 惯用法：`CREATE TABLE IF NOT EXISTS`。

```typescript
// ❌ 改造前的脆弱写法
db.exec(`
  CREATE TABLE IF NOT EXISTS usage_logs (
    input_tokens  INTEGER NOT NULL,
    output_tokens INTEGER NOT NULL,
    -- ... 其他字段 ...
  )
`)
```

**这个写法在早期"够用"的原因**：用户量极小、还没升级过代码，全部跑的是同一个 schema。

**但它有三个迟早要爆的地雷**：

1. **只管"表存在不存在"，不管"表结构对不对"**
   - 老用户库里的 `usage_logs` 字段是老版本 schema
   - 代码升级后，新代码想写 `ttft_ms` 字段 → 老库崩溃：`no such column: ttft_ms`
   - `CREATE TABLE IF NOT EXISTS` 看到表已存在就跳过，**完全不会帮你升级结构**

2. **没有版本号，无法判断该升级到哪一步**
   - 用户 A 是上周装的（schema v1），用户 B 是今天装的（schema v3）
   - 同一份代码不知道要给谁执行什么迁移步骤

3. **临时救火方案是 try/catch ALTER TABLE**
   ```typescript
   try { db.exec('ALTER TABLE usage_logs ADD COLUMN ttft_ms INTEGER') } catch {}
   ```
   - 字段越加越多，try/catch 越堆越长
   - 无法追踪"哪个用户跑过了哪些迁移"
   - 迁移失败被吞掉，没有错误提示，最终在写数据时才炸

### 1.6 今天不做系统方案，未来就要还债

当时团队面临两个选择：

**选项 A：继续加 try/catch ALTER TABLE**
- 短期成本：5 分钟
- 长期代价：每加一次字段风险累积一次，最终某个用户的数据库处于"半迁移"状态无人知晓

**选项 B：建立版本化迁移框架**
- 短期成本：半天工作量
- 长期收益：所有后续 schema 变更都有标准流程，可追溯、可审计、可测试

选了 **B**。本文记录这个选择的产物——一套轻量但严谨的版本化迁移系统。

## 二、为什么这事很危险

### 2.1 数据库改动和代码改动的本质区别

| 维度 | 代码改动 | 数据库 schema 改动 |
|------|---------|------------------|
| 作用对象 | 新版本的代码行为 | **用户本地已有的数据文件** |
| 回滚方式 | git revert 一行搞定 | 数据已改，无法回滚 |
| 受害范围 | 只影响新版本功能 | **历史数据可能全部损坏或丢失** |
| 错误代价 | 用户重启 = 修复 | 用户重装 = 丢所有历史记录 |

**核心风险**：一次错误的迁移函数推送到用户手里，可能让所有用户本地 SQLite 的 usage_logs、pricing_rules、memory_vectors 全部损坏。没有第二次机会。

### 2.2 CLI 工具特有的困境

服务端应用改 schema 可以：
- DBA 手动在数据库上执行 SQL
- 蓝绿部署，老版本仍能读老 schema
- 出问题可以回滚到某个备份点

**CLI 工具都没有**：
- 用户本地库是黑盒，你看不见也摸不到
- 没有备份（除非用户自己备份了）
- 你发布的代码一旦跑在用户机器上，**世界上可能有成千上万份不同状态的数据库**同时存在

这意味着 migration 代码必须**幂等、容错、可重试**，写错一行是灾难级事故。

## 三、解决方案：PRAGMA user_version + 迁移函数数组

### 2.1 核心设计

```
┌──────────────────────────────────────────────────┐
│          migrations[] (代码中声明的迁移序列)         │
│                                                    │
│   [0]  v0 → v1：基线建表（首次安装走这里）           │
│   [1]  v1 → v2：加字段 / 新表（未来扩展）           │
│   [2]  v2 → v3：...                               │
│   [N]  v(N) → v(N+1)                             │
└────────────────────────┬─────────────────────────┘
                         │
                         │ runMigrations() 执行器
                         ▼
          ┌──────────────────────────────┐
          │  读取用户库的 user_version    │
          │  → 执行 slice(current) 待跑的 │
          │  → 设置 user_version = N+1    │
          └──────────────────────────────┘
                         │
                         ▼
          ┌──────────────────────────────┐
          │  SQLite DB 文件头             │
          │  user_version = N (整数)      │
          └──────────────────────────────┘
```

### 2.2 为什么选 `PRAGMA user_version`

SQLite 内置的元数据字段，存在 DB 文件头：
- ✅ 无需额外建表
- ✅ 读写零开销
- ✅ 原子性（单条 PRAGMA 语句）
- ⚠️ **但不受事务保护**——这是后面踩坑的根源

### 2.3 双轨版本管理

CCode 的 DB 里有两套 schema，版本管理也分两条线：

```
┌─────────────────────────────────────────────────────────┐
│                    PRAGMA user_version                    │
│                  （主 schema 版本号）                      │
│                                                          │
│  管辖范围：                                               │
│    usage_logs · pricing_rules · schema_comments          │
│    · memory_meta                                         │
│                                                          │
│  迁移方式：migrations[] 数组 + runMigrations()            │
│  版本存储：SQLite 文件头（PRAGMA user_version）            │
├─────────────────────────────────────────────────────────┤
│              memory_meta.vectors_schema_version           │
│                （向量表 schema 版本号）                     │
│                                                          │
│  管辖范围：                                               │
│    memory_vectors                                        │
│                                                          │
│  迁移方式：ensureMemoryVectors() 内部版本判断              │
│  版本存储：memory_meta 表的 vectors_schema_version 键      │
│  特殊性：维度变化时无条件 DROP + 重建（向量列定义不可变）    │
└─────────────────────────────────────────────────────────┘
```

**为什么分两套**：
- 主 schema 在 `createDb()` 时一次性完成，生命周期 = 进程
- `memory_vectors` 在 `LibsqlVectorStore.initialize()` 时延迟创建，维度是运行时参数（F32_BLOB(N) 的 N 不能在静态迁移里硬编码）
- 两者独立演进，互不阻塞

## 五、如何触发迁移（小白实战手册）

> **你只需要改 1 个文件**：`cCli/src/persistence/db.ts`
> **你只需要找 1 个地方**：文件里的 `migrations` 数组（大约在第 42 行）
> **你只需要做 1 件事**：往数组末尾追加一个函数

### 5.1 你要动的文件长这样（认路）

打开 `cCli/src/persistence/db.ts`，你会看到这样的结构（只展示骨架）：

```typescript
// cCli/src/persistence/db.ts

import Database from 'libsql'
// ... 一些类型 import ...

let _db: DatabaseType | null = null

type MigrationFn = (db: DatabaseType) => void

// 👇👇👇 这里就是你要改的地方 👇👇👇
const migrations: MigrationFn[] = [              // ← 第 42 行附近

  // ═══ v0 → v1：基线 schema（已有）═══
  (db) => {
    db.exec('DROP TABLE IF EXISTS usage_logs')
    // ... 建表 ...
    seedDefaultPricing(db)
    seedSchemaComments(db)
  },

  // 👇 你的新代码加在这里 👇
  // （下文会告诉你怎么加）

]                                                 // ← 第 126 行附近

const SCHEMA_VERSION = migrations.length          // ← 第 129 行，别动
                                                   //   （它会自动 +1）

function runMigrations(db: DatabaseType): void { // ← 第 139 行，别动
  // 不用你改，它会自动读出 migrations 并逐个执行
}

export function createDb(dbPath: string): DatabaseType {  // ← 第 161 行，别动
  // 启动时自动调用 runMigrations，不用你管
}
```

**记住 3 条规则**：

1. ✅ **只在 `migrations[]` 数组里追加新函数**（不要改已有的）
2. ❌ **不要碰 `SCHEMA_VERSION`**（它会自动算）
3. ❌ **不要碰 `runMigrations` 和 `createDb`**（框架代码，不用你管）

### 5.2 我想给表加一个字段 —— 最常见场景

**实际需求举例**：Phase 3 要加错误码字段到 `usage_logs` 表。

**具体步骤**：

**① 打开文件**：`cCli/src/persistence/db.ts`

**② 找到 `migrations` 数组末尾**（大概第 125 行附近，紧挨着 `]`）：

```typescript
    seedDefaultPricing(db)
    seedSchemaComments(db)
  },              // ← v0→v1 那个函数的结束花括号 + 逗号
]                 // ← 数组结束的方括号
```

**③ 在 `]` 之前追加一个函数**（这就是你的新 migration）：

```typescript
    seedDefaultPricing(db)
    seedSchemaComments(db)
  },

  // ═══ v1 → v2：加错误码字段 ═══     👈 新加的注释（不必要但推荐）
  (db) => {                           👈 新加的函数
    db.exec('ALTER TABLE usage_logs ADD COLUMN error_code TEXT')
    db.exec('ALTER TABLE usage_logs ADD COLUMN retry_count INTEGER DEFAULT 0')
  },

]
```

**④ 就完了。** 保存文件，跑 `npm run build`，发布。

**⑤ 后台会发生什么**（你不用手动做这些）：
1. `SCHEMA_VERSION` 因为数组长度 +1，自动变成 `2`
2. 用户下次启动 CCode：
   - `createDb` 自动调用 `runMigrations`
   - `runMigrations` 读用户的 `user_version`（是 `1`）
   - 发现 `1 < 2`，执行 `migrations[1]`（你刚加的那个函数）
   - 执行成功后设置 `user_version = 2`
3. 用户的 `usage_logs` 表现在多了两列，**历史数据保留**
4. 再下次启动：`user_version` 已经是 `2` = `SCHEMA_VERSION`，直接跳过

### 5.3 我想加一张全新的表

**操作位置**：一样在 `cCli/src/persistence/db.ts` 的 `migrations` 数组末尾追加。

**示例代码**：

```typescript
  // ═══ v1 → v2：新建告警记录表 ═══
  (db) => {
    db.exec(`
      CREATE TABLE alert_logs (
        id          INTEGER PRIMARY KEY AUTOINCREMENT,
        session_id  TEXT NOT NULL,
        level       TEXT NOT NULL,
        message     TEXT NOT NULL,
        timestamp   TEXT NOT NULL
      );
      CREATE INDEX idx_alert_session ON alert_logs(session_id);
    `)
  },
```

### 5.4 我想加字段并用老数据填初始值

**场景**：加一个 `total_tokens` 字段，要用 `input_tokens + output_tokens + ...` 填进去。

```typescript
  // ═══ v1 → v2：加 total_tokens 字段并回填历史数据 ═══
  (db) => {
    // 先加字段（默认 NULL）
    db.exec('ALTER TABLE usage_logs ADD COLUMN total_tokens INTEGER')

    // 再回填历史数据
    db.exec(`
      UPDATE usage_logs
      SET total_tokens = input_tokens + output_tokens + cache_read + cache_write
    `)
  },
```

### 5.5 我想改列名 / 改列类型（高危场景）

⚠️ **SQLite 不支持直接改列名或改类型**，只能走"建新表 → 复制数据 → 删老表 → 重命名"的套路。

```typescript
  // ═══ v1 → v2：重命名 cache_read → cache_read_tokens ═══
  (db) => {
    db.exec(`
      -- 1. 建新表（新列名）
      CREATE TABLE usage_logs_new (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        session_id TEXT NOT NULL,
        -- ... 其他字段完全照抄旧表 ...
        cache_read_tokens INTEGER NOT NULL DEFAULT 0,  -- 👈 新列名
        -- ...
      );

      -- 2. 复制数据（老列名 → 新列名）
      INSERT INTO usage_logs_new
      SELECT id, session_id, ..., cache_read, ...
      FROM usage_logs;

      -- 3. 删老表
      DROP TABLE usage_logs;

      -- 4. 新表改名
      ALTER TABLE usage_logs_new RENAME TO usage_logs;

      -- 5. 重建索引（老索引随着 DROP TABLE 一起没了）
      CREATE INDEX idx_usage_session ON usage_logs(session_id);
      CREATE INDEX idx_usage_timestamp ON usage_logs(timestamp);
    `)
  },
```

⚠️ 这种迁移务必先在测试环境用真实数据跑过。

### 5.6 checklist：提交 PR 前自查

写完新 migration 后，交 PR 前检查这些：

- [ ] 只往 `migrations[]` 末尾追加了新函数，没动历史函数
- [ ] `SCHEMA_VERSION` 没有被手动改（它应该自动等于 `migrations.length`）
- [ ] 新 migration 函数用了 `ALTER TABLE` 或 `CREATE TABLE` 或 `CREATE TABLE _new + INSERT + DROP + RENAME`，**没有直接 `DROP TABLE usage_logs` 等会丢数据的写法**
- [ ] 本地测过：删掉 `~/.ccode/data/ccode.db` 跑一次（模拟新用户） + 人工把 `user_version` 设成老版本再跑一次（模拟老用户升级）
- [ ] `tests/unit/db.test.ts` 里加了对应测试
- [ ] 如果是高危迁移（场景 5.5），在 PR 描述里明确标注"此迁移不可逆"

### 5.7 我怎么知道改对了？

**最快的验证方式**：

```bash
# 1. 删掉本地库（放心，会自动重建）
rm ~/.ccode/data/ccode.db

# 2. 启动 CCode（什么都行，甚至 ccode --help）
ccode --version

# 3. 查版本号
sqlite3 ~/.ccode/data/ccode.db "PRAGMA user_version;"
# 应该返回：你 migrations.length 的值

# 4. 查表结构有没有新字段
sqlite3 ~/.ccode/data/ccode.db ".schema usage_logs"
# 应该看到你新加的列
```

如果步骤 3 或 4 不符合预期，说明 migration 没跑对。看步骤 2 的启动日志有没有报错。

## 六、红线与反面教材

### 5.1 红线一：永远不要改已发布的 migrations[i]

```typescript
const migrations = [
  (db) => { /* v0 → v1 */ },   // ❌ 不能改！用户可能跑过了
  (db) => { /* v1 → v2 */ },   // ❌ 不能改！
  // ✅ 只能往后追加
]
```

**为什么**：`user_version` 记录的是"已执行到哪一步"。如果改了 `migrations[0]`，用户 `user_version = 1` 的库不会重跑它，新逻辑永远不生效。

**需求变了怎么办**：加一个新的 `migrations[N]` 来修正。

### 5.2 红线二：不要手动改 SCHEMA_VERSION

```typescript
const SCHEMA_VERSION = migrations.length   // ✅ 永远自动推导
// const SCHEMA_VERSION = 5                 // ❌ 手动改会错乱
```

### 5.3 红线三：PRAGMA 不受事务保护

**错误示范**（实际踩过的坑）：

```typescript
function runMigrations(db) {
  const tx = db.transaction(() => {
    migrations[i](db)
    db.pragma(`user_version = ${SCHEMA_VERSION}`)  // ❌ 放事务里
  })
  tx()
}
```

**后果**：如果 migration 中途失败，事务回滚了 DDL，但 `user_version` 已经写成新值（PRAGMA 不参与回滚）。下次启动读到新版本号 → 跳过迁移 → schema 残缺。

**正确写法**（当前实现）：

```typescript
function runMigrations(db) {
  // 迁移在事务外逐个执行
  for (let i = current; i < SCHEMA_VERSION; i++) {
    migrations[i](db)
  }
  // 全部成功后才更新版本号
  db.pragma(`user_version = ${SCHEMA_VERSION}`)
}
```

### 5.4 红线四：migration 内部别嵌套事务

**错误示范**：

```typescript
// runMigrations 外层加了事务
const tx = db.transaction(() => {
  migrations[i](db)  // migration 内部又调用了 seedDefaultPricing
})                   // ↑ seedDefaultPricing 里也开了 db.transaction()

// 结果：SqliteError: cannot start a transaction within a transaction
```

**教训**：SQLite 不支持嵌套事务。当前方案是**外层不包事务**，让每个 migration 和它调用的 seed 函数自己管理事务。

### 5.5 红线五：DROP 重建型迁移只能一次性使用

v0 → v1 的基线迁移里有 `DROP TABLE IF EXISTS usage_logs`，这是早期数据没价值时可以接受的。

**之后所有迁移必须是增量的（ALTER / INSERT / UPDATE）**，不能再 DROP。

## 七、验证方案

### 6.1 本地测试

**场景 1 - 全新安装**：

```bash
# 删掉本地 DB 模拟全新用户
rm ~/.ccode/data/ccode.db
# 启动 CCode
```

## 八、为什么没有回滚机制

### 7.1 不做回滚的原因

传统 web 项目常见 `up() / down()` 迁移对，但 CCode 不做：

1. **客户端应用，不是服务端**——用户本地库状态不统一，没有"一起回滚"的场景
2. **回滚代码本身是风险**——写回滚比写迁移难，错了代价更大
3. **代码降级 ≠ 数据库降级**——用户如果想用老版本代码，直接装老版本 CCode 即可，数据库保持新版本也没问题（老代码读新库会忽略新字段）

### 7.2 真想回滚怎么办

**场景**：发现 v2 迁移有 bug 想撤销

**正确做法**：发布 v3 迁移来"撤销" v2 的效果

```typescript
const migrations = [
  (db) => { /* v0 → v1 */ },
  (db) => {
    db.exec('ALTER TABLE usage_logs ADD COLUMN buggy_field TEXT')  // v1 → v2（有 bug）
  },
  // ↓ 新加这个来修正
  (db) => {
    // 不能直接 DROP COLUMN（SQLite 不支持），但可以忽略这个字段
    // 或走"新表 + 复制 + 删老表"的路子（见 4.4）
    db.exec('CREATE TABLE usage_logs_new AS SELECT id, session_id, ... FROM usage_logs')
    db.exec('DROP TABLE usage_logs')
    db.exec('ALTER TABLE usage_logs_new RENAME TO usage_logs')
  },
]
```

**永远往前修复，不要往回走**。

## 九、故障排查

### 8.1 错误：`no such column: xxx`

**含义**：代码想读/写某列，但用户的库里没有这列。

**原因**：通常是忘了给 schema 改动加 migration。

**排查**：
1. 查代码里写入的字段列表
2. 查 `migrations[]` 里最新 migration 有没有加这个字段
3. 缺了就追加一个 migration

### 8.2 错误：`cannot start a transaction within a transaction`

**含义**：SQLite 嵌套事务了。

**原因**：migration 内部调用的函数自带 `db.transaction()`，外层又包了事务。

**排查**：检查调用链，确保只有一层事务。

### 8.3 用户反馈"数据丢了"

**场景**：用户升级 CCode 后发现历史 usage_logs 为空。

**可能原因**：
1. 用户本地 `user_version = 0`（全新库状态），触发了 v0 → v1 的 DROP 重建
2. 某个 migration 写错了 DROP 语句

**排查**：
1. 先让用户备份 `~/.ccode/data/ccode.db`
2. 查 `PRAGMA user_version` 的历史变化（需要日志）
3. 代码 review 最近添加的 migrations，确认没有 DROP 误操作

**兜底**：任何 DROP 类变更，必须在 PR review 里明确标注并评估影响范围。

## 十、决策备忘

### 9.1 为什么不用 Knex / Prisma / Drizzle 之类的 migration 工具

| 选项 | 评估 | 结论 |
|------|------|------|
| Knex migrations | 需要 CLI + 额外元数据表 | 太重，CCode 是 CLI 工具 |
| Prisma migrate | 强依赖 prisma schema | 重写全部数据层，成本太高 |
| Drizzle | 类型友好，但仍是大型框架 | CCode 数据量小，不值得 |
| 自建 PRAGMA + 数组 | 一个文件搞定，< 200 行 | ✅ 选这个 |

### 9.2 为什么不用 migration 文件（每个 .sql 一个文件）

- CCode 是 CLI 工具，打包进 node_modules 后读 `.sql` 文件路径不稳定
- 数组形式所有迁移在一个代码文件里，review 和 grep 都方便
- 数量不多（预期 10 年内不会超过 50 个）
