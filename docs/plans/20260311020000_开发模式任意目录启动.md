# 开发模式任意目录启动

## 背景

ZCli 开发阶段使用 `tsx` 直接运行 TypeScript 源码，但 `pnpm dev` 要求在 `zCli/` 目录下执行，因为 tsx 需要找到 `tsconfig.json` 来解析路径别名（`@core/*`, `@tools/*` 等）。

这导致开发期间只能在项目目录内使用 ZCli，无法在其他工作目录体验真实的使用场景。

## 解决方案

新增 `zCli/zcli.sh` 启动脚本，通过 `--tsconfig` 参数显式指定配置文件路径，使 tsx 在任意目录都能正确解析路径别名，同时 `process.cwd()` 保持为用户的实际工作目录。

```bash
# zcli.sh 核心逻辑
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
exec npx --prefix "$SCRIPT_DIR" tsx --tsconfig "$SCRIPT_DIR/tsconfig.json" "$SCRIPT_DIR/bin/zcli.ts" "$@"
```

**关键点**：
- `--prefix "$SCRIPT_DIR"` — 让 npx 在项目目录查找 tsx
- `--tsconfig "$SCRIPT_DIR/tsconfig.json"` — 显式指定 tsconfig，解决路径别名解析
- 不 cd 到项目目录 — `process.cwd()` 保持用户当前目录，工具的 cwd 上下文正确

## 使用方式

### 直接调用

```bash
cd ~/Desktop/某个项目
bash /d/a_dev_work/claude_cli_z01/zCli/zcli.sh
```

非交互模式：

```bash
bash /d/a_dev_work/claude_cli_z01/zCli/zcli.sh "分析一下这个项目的结构"
```

### 设置 alias（推荐）

在 `~/.bashrc` 中添加：

```bash
alias zcli='bash /d/a_dev_work/claude_cli_z01/zCli/zcli.sh'
```

生效后任意目录直接使用：

```bash
source ~/.bashrc

cd ~/Desktop/某个项目
zcli                        # 交互模式
zcli "这段代码有什么问题"      # 非交互模式
zcli --resume               # 恢复会话
```

## 与正式安装的区别

| | 开发模式（zcli.sh） | 正式安装（pnpm link --global） |
|---|---|---|
| 需要构建 | 不需要，tsx 直接运行 TS | 需要 `pnpm build` |
| 启动速度 | 较慢（tsx 实时编译） | 较快（预编译 JS） |
| 代码修改生效 | 立即生效 | 需重新 build |
| 适用场景 | 开发调试 | 日常使用 |
