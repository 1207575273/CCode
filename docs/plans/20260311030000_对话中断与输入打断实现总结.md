# F18 对话中断与输入打断 — 实现总结

## 1. 实现目标

LLM 响应期间，用户可通过以下方式中断：
1. **Escape / Ctrl+C** — 立即中止当前流式响应
2. **输入新消息并 Enter** — 中断当前响应并开启新一轮对话

中断后被打断的部分回复保留在消息历史中，并按 Claude Code 模式记录 `[Request interrupted by user]` 到 session 日志。

---

## 2. 核心架构

```
用户按 Escape / Ctrl+C
  │
  ▼ App.tsx useInput hook (isActive: isStreaming && !pendingPermission)
  │
  ▼ abort() → AbortController.abort()
  │
  ├── AgentLoop.#callLLM 捕获 AbortError → yield llm_usage(stopReason: 'abort')
  ├── useChat catch(AbortError) → 保存 accumulated 部分回复 + 记录中断消息
  └── finally → setIsStreaming(false)

用户在 streaming 期间输入新消息并 Enter
  │
  ▼ InputBar handleSubmit → onInterruptSubmit(text)
  │
  ▼ App.tsx → interruptAndSubmit(text)
  │
  ├── abort() — 中止当前流
  └── setTimeout(() => submit(text), 0) — 等状态更新后发起新请求
```

**关键**：AbortController 基础设施（useChat → AgentLoop → Provider → LLM）在 F1 时已完整搭建，本次仅在 UI 层接线。

## 4. 关键实现细节

### 4.1 useChat.ts

**interruptAndSubmit 方法**：
```typescript
const interruptAndSubmit = useCallback((text: string) => {
  abortRef.current?.abort()
  setTimeout(() => submit(text), 0)
}, [submit])
```
- `setTimeout(0)` 确保 abort 触发的 finally 块（`setIsStreaming(false)`）先执行
- `submit()` 内部的 `if (isStreaming) return` 守卫保持不变，靠时序保证安全

**AbortError 中断处理**：
```typescript
} catch (err) {
  if ((err as Error).name === 'AbortError') {
    // 保存已累积的部分回复
    if (accumulated) {
      const partialMsg = { id: randomUUID(), role: 'assistant', content: accumulated }
      setMessages(prev => [...prev, partialMsg])
      sessionLogger.logAssistantMessage(accumulated, currentModel)
    }
    // 按 Claude Code 模式记录中断
    sessionLogger.logUserMessage('[Request interrupted by user]')
  } else {
    setError(err instanceof Error ? err.message : String(err))
  }
}
```

### 4.2 App.tsx

**Streaming 中断 useInput**：
```typescript
useInput((input, key) => {
  if (key.escape || (input === 'c' && key.ctrl)) {
    abort()
    appendSystemMessage('⏹ 已中断响应')
  }
}, { isActive: isStreaming && pendingPermission == null })
```
- `isActive` 排除权限弹窗场景，避免按键冲突
- 中断后追加 system 消息（仅 UI 展示，不发给 LLM）

**InputBar 集成**：
```tsx
<InputBar
  key={inputResetKey}
  value={inputValue}
  onChange={setInputValue}
  onSubmit={handleSubmit}
  onInterruptSubmit={(text) => {
    setInputValue('')
    interruptAndSubmit(text)
  }}
  streaming={isStreaming}
/>
```

### 4.3 InputBar.tsx

`disabled` prop 改为 `streaming`，语义从"禁用"变为"可打断"：
- 视觉上提示符仍然变暗（streaming 状态指示）
- 功能上允许输入和提交
- `handleSubmit` 根据 `streaming` 状态分发到 `onInterruptSubmit` 或 `onSubmit`

## 5. 设计决策

| # | 决策 | 理由 |
|---|------|------|
| 1 | `setTimeout(0)` 延迟 submit | abort 触发后 finally 块需要一个 tick 来更新 `isStreaming`，直接同步调用会被守卫拦截 |
| 2 | Escape 和 Ctrl+C 都支持 | Claude Code 两者都支持，用户习惯不同 |
| 3 | `isActive` 排除 pendingPermission | 权限弹窗有自己的 Escape 处理，不能冲突 |
| 4 | `streaming` 替代 `disabled` | 语义更清晰：streaming 不是"禁用"，是"可打断" |
| 5 | 部分回复保留在消息历史 | 用户可能需要看到部分内容；与 Claude Code 行为一致 |
| 6 | 记录 `[Request interrupted by user]` | 参照 Claude Code 的 session 日志格式，LLM 下一轮能感知上轮被打断 |
| 7 | "⏹ 已中断响应" 作为 system 消息 | 清晰告知用户操作结果，system 消息不发给 LLM |

## 8. 后续演进

- **工具执行期间中断**：当前中断仅在 LLM streaming 期间生效。工具执行（如 bash 长时间运行）期间的中断需要额外的 AbortSignal 传递到工具层
- **中断动画**：可在中断瞬间显示短暂的视觉反馈（如闪烁），目前仅用 system 消息
- **中断确认**：对于工具执行场景，可考虑二次确认（"正在执行工具，确认中断？"）
