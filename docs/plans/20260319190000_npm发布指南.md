## 一、为什么可以发到 npm

npm 发布不需要"创建项目"或在 npm 网站上注册包名。只要满足三个条件：

1. **有 `package.json`** — 包含 `name`（包名）和 `version`（版本号）
2. **包名未被占用** — `npm search <name>` 确认没人用
3. **已登录 npm** — `npm login` 或 `npm whoami` 确认身份

执行 `npm publish` 时，npm 读取当前目录的 `package.json`，把 `.npmignore` 之外的文件打成 tarball 上传到 registry。第一次 publish 就自动注册了这个包名。

## 三、发布步骤（完整流程）

### 3.1 前置条件（只需做一次）

```bash
# 注册 npm 账号（如果没有）
npm adduser

# 或登录已有账号
npm login

# 确认登录状态
npm whoami
# → yang1207575273
```

### 3.2 首次发布

```bash
cd cCli

# 1. 确认包名未被占用
npm search ccode-cli

# 2. 构建（CLI + Web 一体化）
pnpm build:all

# 3. 检查发布内容（重要！确认无源码泄露）
npm pack --dry-run
# 逐行确认：只有 dist/ + package.json + README.md
# 不应出现：src/ bin/ tests/ .ccode/ config.json 等

# 4. 发布
npm publish
# 非 scope 包（不带 @）默认 public，不需要 --access public
# 如果开了 2FA，需要输入 authenticator 验证码

# 5. 验证
npm info ccode-cli
# 或打开 https://www.npmjs.com/package/ccode-cli
```

### 3.3 后续版本更新

```bash
cd cCli

# 1. 改代码 + 跑测试
npx vitest run

# 2. 更新版本号（三种方式）

# 或用 npm version 命令：
```

## 五、实际发布踩坑记录

### 5.1 .ccode/ 运行时目录混入包中

**问题**：首次 `npm pack --dry-run` 发现 `.ccode/settings.local.json` 被打入包内。

**原因**：`.npmignore` 里排除了 `.zcli/`（旧名），但改名后 `.ccode/` 没加。

**修复**：`.npmignore` 增加 `.ccode/` 排除规则。

**教训**：每次改名后检查 `.npmignore` 是否覆盖了新路径。

### 5.2 npm 页面无 README

**问题**：v0.0.2 发布后 npm 页面空白，没有 README。

**原因**：npm 显示的是 `package.json` 同级的 `README.md`。`cCli/` 目录下没有 README（只有项目根目录有）。

**修复**：在 `cCli/README.md` 创建面向 npm 用户的精简版 README。v0.0.3 发布后页面正常。

**教训**：npm 包的 README 和 GitHub 仓库的 README 是独立的，需要分别维护。

### 5.3 bin script name 自动修正

**现象**：`npm publish` 时输出 `npm warn publish "bin[ccode]" script name was cleaned`。

**原因**：npm 对 bin 字段的 key 做了自动清理（去除特殊字符等），不影响功能。

**处理**：忽略，或执行 `npm pkg fix` 消除警告。

## 六、npm 包名选择原则

本项目选择 `ccode-cli` 而非 `ccode` 的原因：

| 包名 | 优劣 |
|------|------|
| `ccode` | 简短，但 npm 上已被占用（2020 年的旧包） |
| `ccode-cli` | 加 `-cli` 后缀，明确表示是 CLI 工具，npm 可用 |
| `@codeyang/ccode` | scope 包，永远不会冲突，但安装命令更长 |

**最终选择 `ccode-cli`**：兼顾可用性和简洁性。用户安装后命令是 `ccode`（由 `bin` 字段决定，和包名无关）。
