> 日期: 2026-03-20

> 检查范围: cCli（`tsc --noEmit`）+ Web（`tsc --noEmit`）

## 一、背景

项目启用了 `exactOptionalPropertyTypes: true`（TypeScript 严格可选属性类型），
该选项要求**可选属性不能被显式赋值为 `undefined`**——必须省略该属性或赋予有效值。

这是比 `strict: true` 更严格的约束，能在编译期发现"意外传入 undefined"的隐患，
但也意味着常见的 `{ key: maybeUndefined }` 写法会报错。

本次对 cCli 和 Web 两个项目执行全量 `tsc --noEmit` 检查，修复所有 24 个错误。

---

## 二、错误分类与修复

### 2.1 exactOptionalPropertyTypes 违规（16 个）

**问题模式**：

```typescript
// ❌ 报错：string | undefined 不能赋给可选 string?
const obj: { key?: string } = { key: maybeUndefined }
```

**修复模式**：

```typescript
// ✅ 条件展开：undefined 时不传该属性
const obj = { ...(maybeUndefined !== undefined ? { key: maybeUndefined } : {}) }
```

**涉及文件**：

| 文件 | 问题属性 | 修复方式 |
|------|----------|----------|
| `src/commands/compact.ts` | `strategy`, `focus` | 条件展开 |
| `src/tools/ext/dispatch-agent.ts` | `resultSummary` | 条件展开（2 处） |
| `src/tools/ext/subagent-store.ts` | `resultSummary`, `currentTool` | 条件展开 + `delete` |
| `src/persistence/session-store.ts` | `args`, `durationMs`, `success`, `resultSummary` | 条件展开（2 处） |
| `src/ui/App.tsx` | compact 参数, Ink Text `color` prop | 条件展开 |
| `src/ui/SubAgentPanel.tsx` | Ink Text `color` prop | 条件展开 |
| `src/ui/useChat.ts` | `systemPrompt`, `strategy`, `focus` | 条件展开（3 处） |

**Ink 组件 color prop 特殊处理**：

```typescript
// ❌ Ink 的 Text color 不接受 undefined
<Text color={isSelected ? 'cyan' : undefined}>

// ✅ 条件展开 props
<Text {...(isSelected ? { color: 'cyan' as const } : {})}>
```

### 2.2 delete vs undefined 赋值（2 个）

**问题**：`exactOptionalPropertyTypes` 下，`obj.key = undefined` 报错。

```typescript
// ❌ 报错

// ✅ 用 delete 移除属性
delete state.currentTool
```

**涉及文件**：`src/tools/ext/subagent-store.ts`（markSubAgentDone + consumeAgentEvent）

### 2.3 模块导入路径残留（2 个）

**问题**：工具分层后 `@tools/types.js` 改为 `@tools/core/types.js`，但 `agent-loop.ts` 内有两处用了 inline `import()` 语法未更新。

```typescript
// ❌ 旧路径
function buildToolContext(...): import('@tools/types.js').ToolContext {

// ✅ 新路径
function buildToolContext(...): import('@tools/core/types.js').ToolContext {
```

**涉及文件**：`src/core/agent-loop.ts`（444-445 行）

### 2.4 类型缺失（2 个）

**问题**：`SubagentSnapshot` 和 `SubagentSnapshotEvent` 类型在 `session-store.ts` 中使用但未导入。

**修复**：在 import 语句中补充导入。

**涉及文件**：`src/persistence/session-store.ts`

### 2.5 Message.content 类型不安全（2 个）

**问题**：`Message.content` 的类型是 `string | MessageContent | MessageContent[]`，
但 context-manager.ts 中直接调用了 `.length`、`.match()` 等 string 方法。

```typescript
// ❌ content 可能不是 string
const msgTokens = Math.ceil(history[i]!.content.length / 4)

// ✅ 类型守卫
const content = history[i]!.content
const contentStr = typeof content === 'string' ? content : JSON.stringify(content)
const msgTokens = Math.ceil(contentStr.length / 4)
```

**涉及文件**：`src/core/context-manager.ts`（3 处）

### 2.6 Node.js API 类型不匹配（2 个）

**问题**：execa 返回的 `child.stdout` 是 `Readable`，没有 `unref()` 方法（`unref` 只在 `Socket` 上有）。

```typescript
// ❌ Readable 没有 unref
child.stdout?.unref()

// ✅ 运行时检查
if (child.stdout && 'unref' in child.stdout) (child.stdout as any).unref()
```

**涉及文件**：`src/tools/core/bash.ts`

## 五、经验教训

### exactOptionalPropertyTypes 的正确应对

这是 TypeScript 最严格的选项之一。核心规则：

```
可选属性 ≠ 可以是 undefined
可选属性 = 可以不存在
```

| 写法 | 含义 | 是否合法 |
|------|------|---------|
| `{ key: 'value' }` | 属性存在，值为 string | ✅ |
| `{}` | 属性不存在 | ✅ |
| `{ key: undefined }` | 属性存在，值为 undefined | ❌ 违规 |

**推荐写法**：用条件展开代替直接赋值

```typescript
// 通用模式：可能为 undefined 的值用条件展开传入
const result = {
  name: 'required',
  ...(optional !== undefined ? { optional } : {}),
}
```

### 清除可选属性用 delete

```typescript
// ❌
obj.optionalKey = undefined

// ✅
delete obj.optionalKey
```

### Ink 组件的 props 展开

Ink 的 `Text` 等组件的 `color` prop 不接受 `undefined`，需要用条件 props 展开：

```tsx
// ❌
<Text color={condition ? 'cyan' : undefined}>

// ✅
<Text {...(condition ? { color: 'cyan' as const } : {})}>
```

### ESM / CJS 混用问题（人工排查发现）

项目配置了 `"type": "module"`（ESM），但部分文件混用了 CJS 的 `require()` 语法。
tsc 不会报这类错误（它只检查类型），需要人工排查。

**排查命令**：
```bash
# 在 ESM 文件中搜索 require()（排除合法的 .cjs 文件）
grep -rn "require(" --include="*.js" --include="*.ts" --include="*.tsx" --include="*.mjs" \
  --exclude-dir=node_modules --exclude-dir=dist | grep -v "\.cjs"
```

**发现的问题**：

| 文件 | 问题 | 修复 |
|------|------|------|
| `web/tailwind.config.js` | `export default`（ESM）+ `require('@tailwindcss/typography')`（CJS）混用 | `require` → `import typography from '@tailwindcss/typography'` |
| `tests/unit/session-store.test.ts` | 函数内 `require('node:fs')` 取 `readdirSync`/`utimesSync` | 移到顶部 ESM `import`，函数内删除 `require` |

**规则总结**：

```
.ts / .tsx / .js / .mjs  → 必须用 ESM（import/export）
.cjs                     → 可以用 CJS（require/module.exports）— 这是合法的

混用判断：同一个文件里出现 export default + require() = 不合法
```

**为什么 tsc 不报这个错？**

TypeScript 编译器只做类型检查，不验证模块系统一致性。`require()` 在 TS 看来只是一个函数调用，
类型上没问题就不报错。但 Node.js 运行时在 ESM 模式下会拒绝执行 `require()`（或行为不符预期）。
这类问题只能靠 **人工代码审查** 或 **ESLint 规则**（如 `import/no-commonjs`）发现。
