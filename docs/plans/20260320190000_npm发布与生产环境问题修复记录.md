## 二、生产环境问题与修复

### 2.1 Web 静态资源 404（v0.0.4 修复）

**现象**：
```
ccode --web
serveStatic: root path './D:/Program_dev/nvm/v20.19.3/node_modules/ccode-cli/dist/web' is not found
```

**根因**：
Hono 的 `serveStatic` 只支持相对 cwd 的路径。全局安装后，cwd（`C:\Users\...`）和 npm 包目录（`D:\Program_dev\nvm\...`）跨盘，`path.relative()` 无法计算出有效的相对路径，产生 `./D:/...` 这种无效路径。

**修复**：
弃用 `serveStatic`，改为手动 `readFileSync` + MIME 映射直接返回文件，使用绝对路径不依赖 cwd。

```typescript
// 修复前：依赖 relative(cwd, distDir)，跨盘失败
const relativeDistDir = relative(process.cwd(), distDir)
app.use('/*', serveStatic({ root: `./${relativeDistDir}` }))

// 修复后：绝对路径手动读文件
app.get('*', (c) => {
  let filePath = join(distDir, urlPath === '/' ? 'index.html' : urlPath)
  if (!existsSync(filePath)) filePath = join(distDir, 'index.html') // SPA fallback
  const body = readFileSync(filePath)
  return new Response(body, { headers: { 'Content-Type': MIME[ext] } })
})
```

**教训**：
- Hono `serveStatic` 在 Windows 跨盘场景下不可靠
- 全局安装的 npm 包不能假设 cwd 和包路径在同一个盘
- 手动读文件虽然不够优雅，但最可靠

### 2.2 敏感目录权限崩溃（v0.0.5 修复）

**现象**：
```
ccode --web    # 在 C:\Users\ThinkPad 下启动
[Error: EPERM: operation not permitted, scandir 'C:\Users\ThinkPad\Application Data']
```

**根因**：
`FileIndex.scan()` 使用 `fast-glob('**/*')` 扫描 cwd 下所有文件。在用户主目录下启动时，会尝试扫描 `Application Data`（Windows 系统保护的 junction 目录），触发 EPERM 权限错误，进程直接崩溃。

**修复（两层防御）**：

**第一层：工作目录信任确认**

新增 `src/core/workspace-trust.ts`，在交互模式启动前检测 cwd 是否为敏感目录：

```typescript
// 敏感目录：用户主目录、根目录、系统目录等
const SENSITIVE_DIRS = new Set([
  resolve(homedir()).toLowerCase(),
  'c:\\', 'c:\\users', 'c:\\windows',
  'd:\\', '/', '/root', '/home', '/tmp',
])

// bin/ccli.ts 启动时
if (isSensitiveDirectory(process.cwd())) {
  const trusted = await confirmWorkspaceTrust(process.cwd())
  if (!trusted) process.exit(0)
}
```

用户在敏感目录下启动会看到：
```
────────────────────────────────────────────────
 Accessing workspace:

 C:\Users\ThinkPad

 This looks like a sensitive directory (home, root, etc.).
 CCode will be able to read, edit, and execute files here.

 Do you trust this folder? (y/N):
```

**第二层：文件扫描容错**

`fast-glob` 加 `suppressErrors: true`，跳过无权限目录而非崩溃：

```typescript
allFiles = await fg('**/*', {
  cwd: this.cwd,
  dot: false,
  onlyFiles: true,
  suppressErrors: true,  // ← 跳过 EPERM 等错误
})
```

**参考**：
Claude Code CLI 在非项目目录启动时也有类似的信任确认机制，提示用户 "Is this a project you created or one you trust?"。

### 2.3 npm 页面无 README（v0.0.3 修复）

**现象**：v0.0.2 发布后 npm 页面空白。

**根因**：npm 显示的是 `package.json` 同级的 `README.md`，而项目的 README 在仓库根目录，`cCli/` 下没有。

**修复**：在 `cCli/README.md` 创建面向 npm 用户的精简版 README。

### 2.4 .ccode/ 运行时目录混入包（v0.0.3 修复）

**现象**：`npm pack --dry-run` 发现 `.ccode/settings.local.json` 被打入包。

**修复**：`.npmignore` 增加 `.ccode/` 排除规则。

### 2.5 生产模式 CLI 注册 Bridge 超时（v0.0.4 修复）

**现象**：`node dist/bin/ccli.js --web` 后 Web 页面显示"CLI 已离线"。

**根因**：`setTimeout(100ms)` 固定延迟不够，生产模式 bootstrapAll 耗时可能超过 100ms，sessionId 尚未创建。

**修复**：改为轮询等待（100ms 间隔，最多 50 次 = 5 秒上限）。

## 三、发布流程沉淀

每次发布的标准步骤：

```bash
cd cCli

# 1. 改代码 + 跑测试
npx vitest run

# 2. 更新版本号（两处）
# - package.json version
# - src/ui/WelcomeScreen.tsx APP_VERSION

# 3. 构建
pnpm build:all

# 4. 检查包内容
npm pack --dry-run
# 确认无 .ccode/ src/ bin/ 等泄露

# 5. 发布
npm publish

# 6. git 提交
cd .. && git add -A && git reset HEAD cCli/.ccode/ && git commit -m "..."
```

**注意事项**：
- `git reset HEAD cCli/.ccode/` — 每次都要排除运行时配置目录
- 版本号必须同步两处（package.json + WelcomeScreen），否则 CLI 显示和 npm 版本不一致
- `pnpm build:all` 必须在 publish 前执行，否则包里的 dist 是旧的

## 四、Windows 生产环境特殊问题速查

| 问题 | 根因 | 修复 |
|------|------|------|
| `serveStatic` 路径 404 | cwd 和 npm 包跨盘，`relative()` 失效 | 手动 `readFileSync` + 绝对路径 |
| `EPERM: scandir` 崩溃 | 扫描系统保护的 junction 目录 | `suppressErrors: true` + 信任确认 |
| `sessionId` 为 null | bootstrapAll 耗时超过固定延迟 | 轮询等待替代固定 setTimeout |
| `not a git repository` | 在非 git 目录下启动 | 容错处理，不影响功能 |
