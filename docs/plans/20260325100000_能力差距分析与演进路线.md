## 二、核心 Agent 能力缺口

### 2.1 错误恢复 / 自动重试（影响：高）

**现状**：工具执行失败或 LLM 调用 429/500 时，直接 yield error 结束，用户需要手动重试。

**差距**：
- ProviderWrapper 文档注释里设计了"429 限流自动重试"，但标记为 TODO 未实现
- 工具执行失败后没有 retry 策略，LLM 也没有被引导去换方法
- Claude Code CLI 有指数退避重试 + 模型降级

**涉及文件**：`src/providers/wrapper.ts`（ProviderWrapper）、`src/core/agent-loop.ts`

**建议方案**：
- ProviderWrapper 加 429/500/503 指数退避重试（最多 3 次，间隔 1s/2s/4s）
- 工具执行失败时在 history 中追加带上下文的错误消息，引导 LLM 自行调整

---

### 2.2 安全护栏 Guardrails（影响：高）

**现状**：LLM 生成的 bash 命令/文件操作全靠用户手动确认（permission_request），没有系统级防护。

**差距**：
- 无命令黑名单（`rm -rf /`、`format`、`mkfs`、`dd if=/dev/zero` 等破坏性命令无自动拦截）
- 无路径保护（系统目录 `/etc`、`C:\Windows` 可被写入）
- 无输出过滤（敏感信息如 API Key 可能出现在工具输出中传回 LLM）
- Claude Code CLI 有 `deny` 规则列表和系统目录保护

**涉及文件**：`src/core/agent-loop.ts`（权限检查链）、新增 `src/core/guardrails.ts`

**建议方案**：
- bash 命令黑名单（正则匹配 `rm -rf /`、`> /dev/sda` 等模式），命中时自动拒绝 + 告知 LLM
- 文件操作路径白名单：只允许操作 cwd 及其子目录，系统目录直接拒绝
- 可配置：`~/.ccode/config.json → guardrails: { enabled, denyPatterns, protectedPaths }`

---

### 2.3 WebFetch / WebSearch 工具（影响：高）

**现状**：CCode 没有任何网络获取能力。"帮我看看这个 issue"、"查一下 React 18 的 API" 这类高频场景完全做不了（除非用户手动配 MCP web-search server）。

**差距**：
- Claude Code CLI 有 `WebFetch`（抓取 URL 内容）和 `WebSearch`（搜索引擎）两个内置工具
- 这是开箱即用 vs 需要额外配置的差距

**涉及文件**：新增 `src/tools/ext/web-fetch.ts`、`src/tools/ext/web-search.ts`

**建议方案**：
- `web_fetch`：接收 URL，使用 Node.js fetch 获取内容，HTML 转纯文本（去标签），限制响应体大小
- `web_search`：可选接入搜索引擎 API（SerpAPI/Brave Search），或 fallback 到 DuckDuckGo HTML 解析
- 两个工具归类为 ext 扩展工具，标记为安全工具（只读操作）

---

### 2.4 长期记忆 / RAG（影响：中）

**现状**：只有会话内上下文（JSONL），跨会话知识完全丢失。每次启动新会话都要重新理解代码库。

**差距**：
- 架构文档 5.2 节明确标注"Memory: 短期 → 长期记忆（向量搜索 + RAG）"
- Claude Code CLI 有 Memory 工具，可以保存和检索跨会话知识
- 需求文档待办 T8 已列入"上下文管理 Phase 3 — Memory 工具"

**涉及文件**：新增 `src/tools/ext/memory.ts`、`src/persistence/memory-store.ts`

**建议方案**（分阶段）：
- Phase 1：简单 KV 存储（JSON 文件），`memory_save(key, content)` / `memory_search(query)` 模糊匹配
- Phase 2：向量嵌入 + 语义搜索（需要 embedding API）
- Phase 3：自动摘要（每次会话结束自动提取关键知识保存）

---

### 2.5 多模态输入 Vision（影响：中）

**现状**：所有输入只能是文本。无法处理截图、图片文件、UI 设计稿。

**差距**：
- Claude（Vision）和 GPT-4o 都支持图片输入
- Claude Code CLI 支持截图粘贴和图片文件读取
- CCode 的 `read_file` 只读文本文件，遇到图片返回错误

**涉及文件**：`src/tools/core/read-file.ts`、`src/core/types.ts`（Message content 需支持 image block）

**建议方案**：
- `read_file` 检测图片文件（png/jpg/gif/webp）→ base64 编码 → 作为 image content block 发给 LLM
- Message content 类型扩展：支持 `{ type: 'image', source: { type: 'base64', media_type, data } }`
- 两个协议层（Anthropic / OpenAI）分别适配 image block 格式

## 三、Agent 架构模式缺口

### 3.1 Reflection 自我反思（影响：中）

**现状**：LLM 生成代码/执行操作后没有任何自动验证环节。

**架构文档定位**：5.2 节"Reflection: 无 → 添加输出自检（代码生成后自动 review）"

**建议方案**：
- 利用已有的 Hooks 系统（`PostToolUse`）实现：
  - `write_file` 后自动触发 lint / tsc 检查
  - `bash` 执行后检查 exit code 和 stderr
  - `edit_file` 后自动 diff 审查
- 不需要额外 LLM 调用，用本地工具做验证，结果追加到 history 引导 LLM 修正

---

### 3.2 显式 Planning 任务分解（影响：中）

**现状**：有 `todo_write` 工具但只是简单的待办列表，没有真正的任务分解 + 依赖管理 + 进度追踪。

**架构文档定位**：5.2 节"Planning: 隐式 → 显式 Task 系统（可视化任务分解）"

**建议方案**：
- 增强 `todo_write`：支持子任务、依赖关系、状态流转（pending → in_progress → completed/blocked）
- 或新增 `plan` 工具：LLM 先输出结构化计划（JSON），然后按计划逐步执行
- Web Dashboard 可视化任务树

---

### 3.3 自动路由 Router（影响：低）

**现状**：dispatch_agent 完全由 LLM 自行决定何时派发和用什么模型。

**架构文档定位**：5.2 节"Routing: 手动派发 → 自动路由到最合适的模型/Agent"

**建议方案**：
- 基于任务类型自动选模型：探索任务用便宜模型，编码任务用强模型
- 基于 token 消耗动态降级：接近预算时自动切到便宜模型
- 优先级低，当前 LLM 自行决策已够用

---

## 四、工程健壮性缺口

### 4.1 LLM 429/限流自动重试（影响：高）

**现状**：ProviderWrapper 注释写了 TODO，但代码只做了 stopReason 标准化。

**建议方案**：
```
ProviderWrapper.chat():
  try:
    yield* inner.chat(request)
  catch 429/503:
    wait 1s → retry
    catch again: wait 2s → retry
    catch again: wait 4s → retry
    catch again: yield error
```

---

### 4.2 模型降级链（影响：中）

**现状**：当一个 provider 不可用时，整个会话直接报错。

**建议方案**：
- config 中可配置 fallback 链：`"fallback": ["glm-5", "deepseek-chat"]`
- 主模型 3 次重试失败后自动切到 fallback 模型
- 切换时通知用户（yield 一个 info 事件）

---

### 4.3 Token 预算控制（影响：中）

**现状**：没有消费上限，长会话可能不知不觉跑出几十美元。

**建议方案**：
- `config.json → budget: { maxPerSession: 5.00, maxPerDay: 20.00, currency: "USD" }`
- TokenMeter 累计到阈值时 yield 告警事件（80% 警告 → 100% 阻断并提示 /compact 或结束会话）
- Web Dashboard 展示消费趋势图

---

### 4.4 单元测试覆盖（影响：中）

**现状**：`tests/` 目录有 Vitest 框架但核心模块测试很少。

**缺失测试的关键模块**：
- `agent-loop.ts` — 多轮执行、工具调度、中断处理
- `parallel-executor.ts` — 并行/串行分组、错误处理
- `context-manager.ts` — 三种压缩策略
- `session-store.ts` — JSONL 读写、loadMessages 还原
- `providers/` — mock LLM 响应测试 stream chunk 解析

## 五、优先级总览

| 级别 | 能力 | 类型 | 工作量估计 |
|------|------|------|-----------|
| **P0** | LLM 429 自动重试 | 工程健壮性 | 小（ProviderWrapper 加 retry） |
| **P0** | WebFetch 工具 | 工具能力 | 小（Node.js fetch + HTML 转文本） |
| **P1** | Guardrails 基础版 | 安全 | 中（命令黑名单 + 路径保护） |
| **P1** | Token 预算告警 | 工程健壮性 | 小（TokenMeter 加阈值检查） |
| **P1** | 多模态输入 | Agent 能力 | 中（read_file 图片支持 + 协议层适配） |
| **P1** | 错误恢复策略 | Agent 能力 | 小（错误消息增强引导 LLM 自行调整） |
| **P1** | 模型降级链 | 工程健壮性 | 中（ProviderWrapper fallback 链） |
| **P2** | Reflection hook | 架构模式 | 小（PostToolUse hook 触发 lint） |
| **P2** | 长期记忆 Phase 1 | Agent 能力 | 中（KV 存储 + 模糊搜索） |
| **P2** | 显式 Planning | 架构模式 | 中（增强 todo_write 或新增 plan 工具） |
| **P2** | 单元测试覆盖 | 工程质量 | 大（核心模块逐个补测试） |
| **P3** | WebSearch 工具 | 工具能力 | 中（需要搜索引擎 API） |
| **P3** | 自动路由 Router | 架构模式 | 大（任务分类 + 模型选择策略） |
