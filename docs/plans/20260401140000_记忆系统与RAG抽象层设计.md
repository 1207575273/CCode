## 一、背景与目标

### 1.1 现状

cCli 当前不具备跨会话的记忆能力：
- JSONL 会话持久化仅用于 resume/fork，不支持语义检索
- CCODE.md 指令文件是静态的，不支持动态记忆
- 上下文压缩（Compact）是有损的，信息随压缩丢失

### 1.2 目标

构建一个三层记忆系统 + RAG 抽象层，实现：

1. **跨会话持久记忆**：用户偏好、项目决策、技术债等信息跨会话保留
2. **语义检索**：用自然语言检索相关记忆，而非依赖精确匹配
3. **RAG 基础设施**：可复用的 RAG 引擎，为后续代码库索引等场景提供基础
4. **与现有架构无缝融合**：通过 Tool、System Prompt、Bootstrap、Hook 等扩展点集成

### 1.3 设计原则

- **人类可读可编辑**：记忆文件用 Markdown 存储，用户可直接编辑
- **渐进增强**：Embedding 可插拔，向量索引可选
- **复用现有基础设施**：SQLite (libsql)、Provider 系统、Tool 系统
- **不引入重依赖**：不依赖 LangChain RAG 链、外部向量数据库服务
- **外部 IO 不上热路径**：Embedding API 等外部网络调用**永远异步后台**执行，同步路径只做本地操作（文件读写、内存 BM25、SQLite 读写）。这条原则贯穿整个系统——启动时 initialize 分同步/异步两阶段、memory_write 先写后 embed、MemoryWatcher 增量索引后台跑。用户感知到的延迟 = 本地操作耗时，与 Embedding API 响应速度无关

## 二、三层记忆模型

| 层级 | 存储位置 | 生命周期 | 内容类型 |
|------|---------|---------|---------|
| **全局记忆** | `~/.ccode/memory/` | 永久，跨项目 | 用户偏好、编码风格、通用知识 |
| **项目记忆** | `<project>/.ccode/memory/` | 永久，项目内 | 架构决策、技术债、项目约定 |
| **会话记忆** | JSONL（已有） | 会话内 | 当前对话上下文 |

### 2.1 目录结构

```
# 全局记忆
~/.ccode/memory/
  ├── MEMORY.md                    # 索引文件（自动维护，列出所有记忆条目摘要）
  ├── user_preferences.md          # 用户偏好
  ├── coding_style.md              # 编码风格
  └── ...                          # 按语义主题自由组织

# 项目级记忆
<project-root>/.ccode/memory/
  ├── MEMORY.md                    # 索引文件
  ├── insights/                    # 语义主题目录（架构决策、技术债等）
  │   ├── auth-middleware.md
  │   └── performance-baseline.md
  ├── feedback/                    # 用户反馈与指导
  │   ├── testing-approach.md
  │   └── code-review-prefs.md
  └── sessions/                    # 会话自动总结（日期驱动）
      ├── 2026-04-01-project-setup.md
      └── 2026-04-02-bug-fix-auth.md
```

**组织原则**：
- insights/ 和 feedback/ 按**语义主题**组织，日期在 frontmatter 中
- sessions/ 按**日期**组织（日期前缀），因为会话总结天然按时间线回溯
- 全局记忆层级较浅，直接平铺即可
- MEMORY.md 是给人看的索引，不参与 RAG 检索

### 2.2 记忆文件格式

单个记忆文件 = **YAML frontmatter + Markdown body**：

```markdown
---
type: project | user | feedback | reference | session-summary
created: 2026-04-01T10:00:00Z
updated: 2026-04-01T10:00:00Z
tags: [auth, middleware, compliance]
source: user | agent | auto-summarize
---

# 认证中间件重写

重写 auth middleware 是因为 legal 标记了 session token 存储
不满足新合规要求。

- 范围决策应优先合规而非易用性
- 预计完成时间：2026-04-15
```

**关键设计决策**：
- 每个文件是一个语义完整的记忆单元（文件即 chunk）
- frontmatter 提供元数据用于标签过滤 + 向量索引
- 文件粒度由写入者控制——LLM 倾向细粒度，用户自由编辑
- 大文件（>1000 tokens）在索引时由 Chunker 切分为多个 chunk

**YAML frontmatter 解析**：
- 自实现轻量解析器（不引入 gray-matter 依赖），逻辑简单：
  1. 检测文件开头 `---\n`
  2. 找到第二个 `---\n` 的位置
  3. 中间部分用正则逐行解析 `key: value`（支持字符串、数组 `[a, b]`、日期）
  4. 剩余部分作为 Markdown body
- 写入时用模板字符串拼接 frontmatter，不需要 YAML 序列化库

### 2.3 MEMORY.md 索引文件

自动维护的索引，供人类快速浏览：

```markdown
# Memory Index

- [认证中间件重写](insights/auth-middleware.md) — 合规驱动的 auth 中间件重构
- [用户偏好 React](user_preferences.md) — React 17+ 函数组件风格
```

每次记忆条目增删改时自动更新。

### 2.4 记忆文件 `.gitignore` 策略

项目级记忆文件包含**个人知识**（用户偏好、反馈等），默认不应被 git 跟踪。

**策略**：`MemoryManager.initialize()` 时自动在 `<project>/.ccode/memory/` 下生成 `.gitignore`：

```gitignore
# 记忆文件为个人知识，默认不跟踪
*
!.gitignore
!MEMORY.md
```

- `MEMORY.md` 索引文件可选跟踪（团队共享记忆场景，手动取消注释）
- 全局记忆 `~/.ccode/memory/` 不在任何 git 仓库中，无需处理
- MVP 阶段不创建 `sessions/` 子目录，P1 会话自动总结时按需创建

### 3.2 核心接口

```typescript
// === types.ts ===

type MemoryScope = 'global' | 'project'

type MemoryType = 'user' | 'feedback' | 'project' | 'reference' | 'session-summary'

interface MemoryEntry {
  id: string                      // scope + 相对路径组合，保证全局唯一
                                  // 格式: `${scope}:${relativePath}`
                                  // 例: 'project:insights/auth-middleware' / 'global:user_preferences'
  scope: MemoryScope
  title: string
  content: string                 // Markdown body
  type: MemoryType
  tags: string[]
  source: 'user' | 'agent' | 'auto-summarize'
  created: string                 // ISO 8601
  updated: string
  filePath: string                // 原始文件路径
}

interface MemoryQuery {
  query: string
  scope?: MemoryScope | 'all'     // 默认 'all'
  tags?: string[]
  type?: MemoryType
  topK?: number                   // 默认 5
  dateRange?: { from?: string; to?: string }
}

interface MemorySearchResult {
  entry: MemoryEntry
  score: number                   // 0-1 相关性分数
  snippet: string                 // 匹配的文本片段
  matchedChunks: MemoryChunk[]    // 匹配的具体 chunk
}

interface MemoryChunk {
  id: string
  entryId: string
  text: string
  chunkIndex: number              // 同一文件的第几个 chunk
  embedding?: number[]            // 向量
}

// === embedding/types.ts ===

/**
 * EmbeddingProvider — 独立于 LLMProvider 的 Embedding 接口。
 *
 * 不扩展 LLMProvider（后者只有 chat/countTokens），而是独立体系。
 * 仅复用 providers 配置中的 apiKey + baseURL，通过 fetch 直接调 /embeddings 端点。

interface EmbeddingProvider {
  readonly name: string
  readonly dimension: number

// === MemoryManager 核心接口 ===

  search(query: MemoryQuery): Promise<MemorySearchResult[]>

  delete(id: string): Promise<void>
  list(scope?: MemoryScope): Promise<MemoryEntry[]>
  getRelevantContext(query: string): Promise<string>  // System Prompt 注入用

// === storage/types.ts — 存储层抽象接口 ===

/**
 * 文件存储抽象 — 管理记忆条目的 CRUD
 * 实现类：FileStore（Markdown 文件系统）
 */
interface IFileStore {
  /** 扫描指定目录下所有记忆文件 */
  scan(basePath: string): Promise<MemoryEntry[]>
  /** 读取单个记忆文件 */
  read(filePath: string): Promise<MemoryEntry | null>
  /** 写入/创建记忆文件，返回完整条目 */
  save(entry: Omit<MemoryEntry, 'created' | 'updated'>): Promise<MemoryEntry>
  /** 更新已有记忆文件 */
  update(filePath: string, content: string, tags?: string[]): Promise<MemoryEntry>
  /** 删除记忆文件 */
  delete(filePath: string): Promise<void>
  /** 更新 MEMORY.md 索引文件 */
  updateIndex(entries: MemoryEntry[]): Promise<void>

/**
 * 向量存储抽象 — 管理向量索引的 CRUD + 相似度检索
 * 实现类：LibsqlVectorStore | MemoryVectorStore
 *
 * 设计意图：将向量存储的具体实现（libsql 内置向量）与业务逻辑解耦。
 * 后续可替换为 FAISS、ChromaDB、Qdrant 等实现，无需改动 Retriever 和 Indexer。
 */
interface IVectorStore {
  /** 向量维度，由 Embedding Provider 决定 */
  readonly dimension: number

  /** 初始化存储（建表/建索引） */
  initialize(): Promise<void>

  /** 批量写入向量 */
  upsert(chunks: Array<{
    id: string
    entryId: string
    embedding: number[]
    text: string
    chunkIndex: number
    scope: MemoryScope
    projectSlug?: string
    tags: string[]
    type: MemoryType
    source: string
    created: string
    updated: string
  }>): Promise<void>

  /** 删除某个记忆条目的所有 chunk */
  deleteByEntryId(entryId: string): Promise<void>

  /** 向量相似度检索，返回 Top-K 结果（按相似度降序） */
  similaritySearch(
    queryEmbedding: number[],
    options: {
      topK: number
      scope?: MemoryScope | 'all'
      projectSlug?: string
      tags?: string[]
      type?: MemoryType
    }
  ): Promise<Array<{
    chunkId: string
    entryId: string
    text: string
    score: number          // 0-1 相似度分数
  }>>

  /** 获取指定记忆条目的所有 chunk 文本（BM25 构建用） */
  getChunkTexts(scope?: MemoryScope): Promise<Array<{ chunkId: string; text: string; entryId: string }>>

  /** 清空所有向量数据 */
  clear(): Promise<void>

  /** 关闭连接/释放资源 */
  close(): Promise<void>
}

// === CompactBridge — Compact 与记忆融合接口 ===

interface ICompactBridge {

  extractAndSave(messages: Message[]): Promise<MemoryEntry[]>

}
```

### 3.4 Embedding 配置

**配置位置**：`~/.ccode/config.json`

```json
{
  "memory": {
    "enabled": true,
    "embedding": {
      "provider": "glm",          // 必填，已配置的供应商名
      "model": "embedding-3"      // 必填，embedding 模型名
    }
  }
}
```

**规则**：
- `memory.enabled` 默认 `false`，用户主动开启
- `embedding` 配置块**可选**（不配置 → 纯 BM25 关键词检索，记忆功能仍可用）
- 配置了 embedding 时 `provider` + `model` 必填
- provider 必须支持 `/embeddings` 端点（openai-compat 协议）
- Anthropic 协议不支持 Embedding → 启动时提示，自动降级为纯 BM25

**实现方式**：
- `ProviderEmbedding`：复用现有 providers 配置中的 `apiKey` + `baseURL`，调用 `/embeddings` 端点
- 无需新增 SDK 依赖，直接用 fetch 调用兼容 API
- 支持的协议：`openai-compat`（GLM、DeepSeek、OpenAI 等）
- 不支持：`native-anthropic`（无 Embedding API）

**降级链**：
```
配置的 provider → 检查协议是否支持 embedding
  ├── 支持（openai-compat）→ 使用向量 + BM25 混合检索
  ├── 不支持（anthropic）→ 警告，降级为纯 BM25 检索（记忆功能仍可用，只是检索精度下降）
  └── API 不可达 → 启动时报错，降级为纯 BM25
```

**注意**：`ProviderEmbedding` 是独立于 `LLMProvider` 的实现类，只复用 providers 配置中的 `apiKey` + `baseURL`，不复用 Provider 实例。参见 `embedding/types.ts` 中 `EmbeddingProvider` 接口的注释。

### 3.5 向量索引

**存储**：扩展现有 `ccode.db` SQLite 数据库（`src/persistence/db.ts`），新增 `memory_vectors` 表。

**核心优势**：libsql 内置向量数据类型和相似度函数，**无需任何外部扩展**，直接复用现有 `libsql` 连接。

```sql
-- 建表：使用 libsql 内置的 F32_BLOB 向量列类型
-- 维度由 EmbeddingProvider.dimension 动态决定，建表时拼接到 DDL 中
-- 例：dimension=1024 → F32_BLOB(1024)，dimension=1536 → F32_BLOB(1536)
-- 维度信息同时写入 memory_meta 表，后续启动时校验是否与当前 Provider 一致
CREATE TABLE IF NOT EXISTS memory_meta (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
-- INSERT INTO memory_meta VALUES ('embedding_dimension', '1024');

CREATE TABLE IF NOT EXISTS memory_vectors (
  id TEXT PRIMARY KEY,                    -- chunk ID: {entryId}_{chunkIndex}
  entry_id TEXT NOT NULL,                 -- MemoryEntry.id
  scope TEXT NOT NULL,                    -- 'global' | 'project'
  project_slug TEXT,                      -- 项目标识（全局记忆为 NULL）
  embedding F32_BLOB(${dimension}),       -- 维度从 EmbeddingProvider.dimension 动态注入
  chunk_text TEXT NOT NULL,               -- 切分后的文本
  chunk_index INTEGER NOT NULL,           -- 同一文件的第几个 chunk
  tags TEXT,                              -- JSON array
  type TEXT NOT NULL,                     -- MemoryType
  source TEXT NOT NULL,
  created TEXT NOT NULL,
  updated TEXT NOT NULL
);

-- 创建向量索引（DiskANN 算法，加速 ANN 近邻检索）
CREATE INDEX IF NOT EXISTS idx_memory_vec
  ON memory_vectors(libsql_vector_idx(embedding));

-- 插入向量
INSERT INTO memory_vectors (id, entry_id, scope, embedding, chunk_text, ...)
VALUES ('entry1_0', 'entry1', 'project', vector32('[0.1, 0.2, ...]'), '文本内容', ...);

-- ANN 近邻检索（走索引，推荐）
SELECT v.id, mv.entry_id, mv.chunk_text
FROM vector_top_k('idx_memory_vec', vector32('[0.15, 0.25, ...]'), 20) AS v
JOIN memory_vectors AS mv ON mv.rowid = v.id;

-- 暴力余弦距离计算（小数据量 / 精确检索场景）
SELECT id, entry_id, chunk_text,
       vector_distance_cos(embedding, vector32('[0.15, 0.25, ...]')) AS dist
FROM memory_vectors
WHERE scope = 'project'
ORDER BY dist ASC
LIMIT 20;
```

**关键 API**：
- `F32_BLOB(dim)` — 声明向量列，dim 为维度
- `vector32('[...]')` — 从 JSON 数组构造 32-bit 向量
- `vector_distance_cos(a, b)` — 余弦**距离**（0=相同，2=相反），**注意不是相似度**
- `vector_top_k(index_name, query, k)` — 表值函数，ANN 近邻检索
- `vector_extract(v)` — 提取向量为文本表示（调试用）

**距离→相似度转换**：`IVectorStore.similaritySearch` 返回 0-1 相似度分数，
LibsqlVectorStore 实现中需转换：`score = 1 - (distance / 2)`
（余弦距离范围 [0,2]，转换后：0距离→1相似度，2距离→0相似度）

**向量维度**：由 Embedding Provider 的 `dimension` 属性决定。由于 SQLite 不支持动态列类型，建表时需要确定维度。如果用户更换 Embedding 模型导致维度变化，需要重建表（`/remember rebuild`）。维度信息持久化到 `memory_meta` 表，启动时校验一致性，不一致则提示 rebuild。

**project_slug 来源**：复用现有 `src/persistence/session-utils.ts` 中的 `toProjectSlug()` 函数（基于 cwd 路径生成短标识），全局记忆的 project_slug 为 NULL。

### 3.6 混合检索（向量 + BM25 + RRF）

**检索流水线**：

```
用户查询 (query) + 过滤条件 (tags, type, scope, dateRange)
       │
       ├── 元数据过滤下推到检索层（先缩小候选集，再做融合）
       │
       ├── BM25 关键词检索（带 scope/tags 过滤）→ 候选集 A (Top-K chunks)
       ├── Dense 向量检索（WHERE scope/tags 过滤）→ 候选集 B (Top-K chunks)
       │         │
       │         ▼
       │   RRF (Reciprocal Rank Fusion) 融合
       │         │
       │         ▼
       │   Entry 级聚合去重：
       │     同一 entry_id 的多个 chunk 合并为一条结果，取最高分
       │     snippet = 最高分 chunk 的文本
       │         │
       │         ▼
       └── Top-K 结果 → 返回 MemorySearchResult[]
```

**为什么过滤在融合之前**：如果先融合再过滤，Top-20 融合结果中过滤掉 15 条，最终只剩 5 条，检索质量崩塌。过滤下推后，BM25 和 Vector 各自在缩小的候选集中取 Top-K，融合质量有保障。

**Entry 级去重**：一个 MemoryEntry 被 Chunker 切成 N 个 chunk，多个 chunk 可能同时命中。不去重会导致同一篇记忆占据多个结果位。聚合策略：按 entry_id 分组，取组内最高 RRF 分数，matchedChunks 收集该 entry 的所有命中 chunk。

**RRF 公式**：`RRF_score(d) = Σ 1/(k + rank_i(d))`，k = 60

**BM25 实现**：纯 TypeScript，零外部依赖。基于内存的倒排索引，启动时从 `memory_vectors` 表中加载 chunk_text 构建。

**中文分词**（重要）：BM25 依赖分词建倒排索引。英文空格分词即可，但中文没有词边界——朴素空格分词会把整段中文当成一个 token，导致 BM25 对中文完全失效。

**分词器选型对比**：

| 维度 | CJK Bigram + Latin | jieba-wasm |
|------|-------------------|------------|
| 依赖 | 零，纯 TS ~50 行 | ~4MB wasm 文件，纯 JS 运行，无 native addon |
| 精度 | 差："认证中间件" → 9 个 token，含"证中""间件"噪音 | **好**："认证中间件" → ["认证", "中间件"] |
| BM25 排序质量 | 差：噪音 token 稀释 TF-IDF 权重，小数据量下尤其致命 | **好**：精准词频，Top-K 结果干净 |
| 召回率 | 高：bigram 组合多 | 中：未登录词可能切错（Embedding 通道兜底） |
| 初始化 | 无 | wasm 加载 ~50-100ms（一次性） |
| 停用词 | 不需要（bigram 本身就是噪音） | **需要**：过滤"的、了、和、是"等高频无意义词 |

**选型决策：MVP 用 jieba-wasm，CJK Bigram 作为降级兜底**。

关键推理——**数据量越小，精度越重要**：
- 记忆系统数据量在百~千级（chunk 后可能到万级），远非百万级
- 这个量级下，BM25 Top-5 里混进 1 个噪音结果 = 20% 上下文被污染 → 直接给 LLM 带偏路
- Bigram 的"证中""间件"等无意义 token 在小数据量下不会被 IDF 统计稀释，会实实在在地干扰排名
- jieba-wasm 是纯 wasm 模块（非 native addon），跨平台零问题，~4MB 对 CLI 工具可接受
- 这不算"重依赖"——重依赖指的是 LangChain、ChromaDB 这类改变架构走向的东西

```typescript
// rag/tokenizer.ts — 可插拔分词策略

interface Tokenizer {
  tokenize(text: string): string[]
}

/** MVP 默认：jieba-wasm 分词 + 停用词过滤 */
class JiebaTokenizer implements Tokenizer {
  private stopWords = new Set(['的', '了', '和', '是', '在', '有', '我', '这', '个', '就', '不', '也', '都', '一', '你', '他', '她', '它', '们', '要', '会', '对', '说', '而', '但', '被', '到', '从', '把', '那', '很', '让', '又', '才', '只'])

  tokenize(text: string): string[] {
    const words = jieba.cut(text)
    return words
      .filter(w => w.trim().length > 0)
      .filter(w => !this.stopWords.has(w))        // 过滤停用词
      .map(w => /^[\x00-\x7f]+$/.test(w) ? w.toLowerCase() : w)  // 英文小写
  }
}

/** 降级兜底：CJK Bigram + Latin 空格分词（jieba-wasm 不可用时） */
class BigramTokenizer implements Tokenizer {
  tokenize(text: string): string[] {
    const tokens: string[] = []
    for (const segment of splitByScript(text)) {
      if (segment.isCJK) {
        for (let i = 0; i < segment.text.length; i++) {
          tokens.push(segment.text[i])                         // unigram
          if (i + 1 < segment.text.length)
            tokens.push(segment.text.slice(i, i + 2))          // bigram
        }
      } else {
        tokens.push(...segment.text.toLowerCase().split(/\s+/).filter(Boolean))
      }
    }
    return tokens
  }
}

/** 工厂函数：优先 jieba-wasm，不可用时降级为 Bigram */
function createTokenizer(): Tokenizer {
  try {
    require.resolve('jieba-wasm')
    return new JiebaTokenizer()
  } catch {
    console.warn('[Memory] jieba-wasm 未安装，BM25 降级为 CJK Bigram（中文检索精度下降）')
    return new BigramTokenizer()
  }
}
```

**避坑要点**：
- **停用词必须过滤**：jieba 切出的"的""了""是"等词在 BM25 中占据不必要的权重，影响排名
- **混合检索是安全网**：jieba 切不到的未登录词，Embedding 向量通道能兜底——这就是坚持 RRF 混合检索的意义
- **libsql 本地向量查询极快**：千级数据量下 vector_top_k 在 10ms 以内，体验极佳

```typescript
// rag/tokenizer.ts
function tokenize(text: string): string[] {
  const tokens: string[] = []
  // 按 unicode 分段：CJK 走 bigram，Latin 走空格分词
  for (const segment of splitByScript(text)) {
    if (segment.isCJK) {
      // 字符 unigram + bigram
      for (let i = 0; i < segment.text.length; i++) {
        tokens.push(segment.text[i])
        if (i + 1 < segment.text.length) {
          tokens.push(segment.text.slice(i, i + 2))
        }
      }
    } else {
      // 英文/数字：空格分词 + 小写
      tokens.push(...segment.text.toLowerCase().split(/\s+/).filter(Boolean))
    }
  }
  return tokens
}
```

**为什么这是关键**：BM25 是记忆系统的**兜底检索通道**——Embedding 不可用时降级为纯 BM25、memory_write "先写后 embed" 一致性窗口期只有 BM25 能搜、RRF 混合中 BM25 贡献关键词精确匹配能力。如果 BM25 对中文失效，这三个场景全部崩塌。

**向量检索**：libsql 内置的 `vector_top_k()` 表值函数（走 DiskANN 索引），或 `vector_distance_cos()` 暴力计算。

```typescript
  const baseDir = args.scope === 'global'
    ? join(homedir(), '.ccode', 'memory')
    : join(process.cwd(), '.ccode', 'memory')

  // type → 子目录映射
```

### 4.2 System Prompt 注入

在 `<instructions>` 块之后注入记忆上下文：

```typescript
/**
 * 启动时注入记忆上下文（冷启动，无用户查询）。
 *
 * 冷启动策略：不走向量检索（没有语义查询），而是：
 * 1. 加载 MEMORY.md 索引内容（人类可读的记忆摘要列表）
 * 2. 按 updated 降序取最近 N 条记忆的 title + 摘要
 * 3. 合并为 system prompt 段落
 *
 * 对话中的语义检索由 LLM 主动调用 memory_search 工具完成。
 */
async function loadMemoryContext(cwd: string): Promise<string> {
  // 冷启动：加载 MEMORY.md 索引 + 最近更新的记忆摘要
  const recentEntries = await memoryManager.list('all')  // 按 updated 降序
  const top = recentEntries.slice(0, 10)

  if (top.length === 0) return ''

  // Token 预算上限：记忆上下文不超过 1500 tokens（约 3000 字符）
  // 超出预算时截断条目数量，保证 system prompt 体积可控
  const TOKEN_BUDGET_CHARS = 3000
  let totalChars = 0
  const budgeted: typeof top = []
  for (const e of top) {
    const line = `- **${e.title}** (${e.scope}, ${e.type}) — ${e.content.slice(0, 100)}`
    if (totalChars + line.length > TOKEN_BUDGET_CHARS) break
    budgeted.push(e)
    totalChars += line.length
  }
  if (budgeted.length === 0) return ''

  return `<memory-context>
以下是记忆系统中的近期记录（按更新时间排序）：
${budgeted.map(e => `- **${e.title}** (${e.scope}, ${e.type}) — ${e.content.slice(0, 100)}...`).join('\n')}

可通过 memory_search 工具按语义检索更多记忆。
可通过 memory_write 工具保存重要信息。
</memory-context>`
}
```

### 4.4 Compact 与记忆融合

**核心理念**：Compact（上下文压缩）本质上是有损的"选择性遗忘"。融合记忆系统后，Compact 变为"先归档再压缩"——信息从短期记忆转移到长期记忆，而非直接丢失。

**融合流程**：

```
用户输入 /compact（或自动触发）
  │
  ▼
  ① 分析即将被压缩的上下文
     → 识别关键信息：用户偏好、项目决策、技术发现、未完成任务
     → 用 LLM 提取结构化记忆条目
  │
  ▼
  ② 自动写入长期记忆（去重检测）
     → 检查是否与已有记忆重复（向量余弦相似度 > 0.95）
     → 不重复：写入 insights/ 或 feedback/ 目录
     → 重复但内容更新：合并或覆盖
     → 更新向量索引
  │
  ▼
  ③ 执行原有的上下文压缩
     → 复用现有 3 种压缩策略
     → 压缩后的上下文 + 记忆已归档 = 信息损失最小化
  │
  ▼
  ④ 压缩后追加提示（可选）
     → 在压缩后的上下文中提示："部分历史信息已归档到记忆系统，可通过 memory_search 检索"
```

**实现要点**：

```typescript
// memory-compact-bridge.ts
// 注入到现有的 compact 流程中

interface CompactBridge {
  // 在压缩前调用，提取并保存关键信息
  extractAndSave(messages: Message[]): Promise<MemoryEntry[]>
  // 在压缩后调用，注入记忆提示
  getCompactHint(): string
}
```

**与现有 ContextManager 的集成**：
- 现有 `context-manager.ts` 有 3 种压缩策略（可插拔）
- ContextManager 有两个执行入口，都需要注入 CompactBridge：
  - `prepare()` — 自动触发（上下文超阈值），在级联策略执行前调 `extractAndSave()`
  - `compact()` — 手动 `/compact` 命令，在策略执行前调 `extractAndSave()`
- 具体改动：ContextManager 新增 `setCompactBridge(bridge: ICompactBridge)` 方法
- 压缩策略本身不变，只是前置了一个"记忆提取"步骤
- 记忆提取的 LLM 调用可以复用当前会话的 Provider，不额外配置

**触发时机**：
- 用户手动 `/compact`
- 上下文超阈值自动触发压缩
- 两种场景都走同一套融合逻辑

**成本控制**：
- 记忆提取的 LLM 调用**输入 = 完整 history + 提取指令**（与 compact 摘要的输入量相当，约 50K-100K tokens），输出约 300-800 tokens（结构化记忆条目）
- 这相当于多一次 LLM 调用，成本与 compact 本身相当。可接受：compact 本身就是一次 LLM 调用，多一次提取换来信息不丢失
- 优化：提取和摘要可以**合并为一次 LLM 调用**（同一个 prompt 同时输出摘要 + 记忆条目），避免双倍成本
- 仅在记忆系统启用时才执行提取步骤
- 可通过配置关闭此融合行为：`memory.compactExtraction: false`

**原子性保障**：
- `extractAndSave()` 写入的记忆统一标记 `source: 'auto-summarize'`
- 如果提取成功但后续压缩失败（LLM 超时等），信息会同时存在于原始 history 和记忆中
- 下次压缩时去重检测识别 `source: 'auto-summarize'` 来源，跳过已归档内容，避免重复写入
- 如果压缩失败，已写入的记忆保留（有益无害：即使 history 没被压缩，多一份记忆副本不影响正确性）

- 只对新增/修改的文件重新 embed
- 删除的文件从索引中移除

**启动延迟控制**（重要）：
- Embedding 批量调用是外部 HTTP 请求，50 个文件可能需要 5-10 秒
- **首次建索引**采用后台异步：先用已有索引（或纯 BM25）响应首次对话，embed 在后台跑完后热更新索引
- **增量索引**（后续启动）：只 embed 新增/修改文件，大多数情况 < 1 秒
- Bootstrap 链 D' 中，`MemoryManager.initialize()` 分两阶段：
  1. 同步阶段：扫描文件 + 加载已有向量索引 + 构建 BM25（毫秒级，不阻塞 SystemPrompt）
  2. 异步阶段：增量 embed 新/改文件（后台，不阻塞启动）

### 5.2 在线检索（对话中）

```
LLM 调用 memory_search({ query, scope, tags })
  → Retriever.search(query, { scope, tags })
    → 元数据过滤下推到检索层（scope/tags/type 在 SQL WHERE 和 BM25 过滤中前置）
    → libsql vector_top_k 向量检索 Top-20（带 scope/tags WHERE 过滤）
    → BM25 关键词检索 Top-20（带 scope/tags 内存过滤）
    → RRF 融合 → Entry 级聚合去重 → Top-K
    → 返回 MemorySearchResult[]
```

### 5.3 在线学习（对话中写入）

```
LLM 调用 memory_write({ title, content, tags })
  → MemoryManager.write()
    → FileStore.save()              # 写入 Markdown 文件（同步，<1ms）
    → BM25 索引即时更新             # 内存操作（同步，<1ms）
    → 更新 MEMORY.md 索引文件       # 文件写入（同步，<5ms）
    → 工具返回结果（LLM 不等 embed）
    → 后台：Indexer.embedAndUpsert() # embed API 调用 + 写向量表（异步，200-500ms）
    → MemoryWatcher 感知变更但跳过（自己写的，不重复索引）
```

**写入延迟优化**：embed API 调用（200-500ms 网络延迟）不应阻塞 memory_write 工具返回。
采用"先写后 embed"策略：文件 + BM25 索引同步完成即返回结果，向量索引在后台异步更新。
在 embed 完成前，该记忆只能被 BM25 检索到（关键词匹配），embed 完成后才支持语义检索。

**去重机制**（分级降级）：
- **Embedding 可用**：写入前检查向量余弦相似度 > 0.95 的记忆
- **Embedding 不可用**：title 完全匹配 或 BM25 相似度超阈值
- **最低保障**：title 完全匹配时一律提示覆盖
- 相似时提示 LLM 选择覆盖或跳过
- 用户通过 /remember 命令写入时，交互式确认
- Compact 自动提取的记忆标记 `source: 'auto-summarize'`，去重时识别此来源避免重复写入

## 九、风险与缓解

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| Embedding API 延迟 | 启动建索引慢、memory_write 卡顿 | initialize 分同步/异步两阶段；memory_write 先写后 embed；增量索引只处理变更文件 |
| Embedding 不可用 | Anthropic 协议无 /embeddings 端点 | 降级为纯 BM25 检索（记忆功能仍可用，检索精度下降）；去重降级为 title 匹配 + BM25 相似度 |
| 向量维度变更 | 用户更换 Embedding 模型导致维度不匹配 | `/remember rebuild` 重建表 + 全量重新 embed |
| LLM 写入过多噪音 | 记忆质量下降 | 去重检测 + system prompt 中指导 LLM 仅保存重要信息 |
| 记忆文件冲突 | 用户编辑与 LLM 写入冲突 | 文件锁 + MemoryWatcher selfWritePaths 跳过自身写入 |
| BM25 内存占用 | 大量记忆条目时内存压力 | 持久化到 SQLite FTS5（P1） |
| Compact 提取重复 | 压缩失败后重试时重复归档 | `source: 'auto-summarize'` 标记 + 去重识别 |
| /remember rebuild 成本 | 全量重新 embed 消耗 API 额度 | 执行前提示确认（显示预估文件数 + 预估 API 调用次数） |

从启动延迟、热路径性能、运行时可靠性角度审视。本轮提炼出贯穿全系统的核心原则：

> **外部 IO 不上热路径** — Embedding API 等外部网络调用永远异步后台执行，同步路径只做本地操作。用户感知延迟 = 本地操作耗时，与 Embedding API 响应速度无关。

| # | 问题 | 修复 |
|---|------|------|
| 1 | **启动时 Embedding 批量调用阻塞 Bootstrap**（50 个文件 → 5-10 秒） | initialize 分两阶段：同步加载已有索引（毫秒级）+ 异步后台增量 embed |
| 2 | **Compact 提取成本估算错 100 倍**（写的 200 tokens，实际 50K-100K） | 修正估算 + 建议与 compact 摘要合并为一次 LLM 调用 |
| 3 | **System Prompt 注入无 token 预算上限** | 加 3000 字符（~1500 tokens）硬上限，超出截断 |
| 4 | **元数据过滤在 RRF 融合之后**（先融合再过滤 → 结果不足） | 过滤下推到 BM25/Vector 查询层，先缩小候选集再融合 |
| 5 | **搜索结果同一 Entry 重复出现**（多 chunk 命中不聚合） | 按 entry_id 分组聚合，取组内最高分 |
| 6 | **memory_write 同步等 embed**（200-500ms 阻塞工具返回） | "先写后 embed"：文件 + BM25 同步完成即返回，向量后台异步 |

### 10.5 第五轮：分词器选型修正

第四轮扫描选定 CJK Bigram 作为 MVP 分词方案（零依赖优先）。经进一步分析后推翻该决策，改为 **jieba-wasm 为默认、CJK Bigram 为降级兜底**。

**推翻原方案的核心论据**：

> 数据量越小，精度越重要。万级数据下，BM25 Top-5 里混进 1 个噪音结果 = 20% 上下文被污染 → 直接给 LLM 带偏路。

| 场景 | CJK Bigram 表现 | jieba-wasm 表现 |
|------|----------------|----------------|
| 搜索"算法" | "算盘""办法"等文档被误排高位（"算"bigram 命中） | 精确匹配"算法"，无噪音 |
| 搜索"中间件重写" | "中间""间件""件重"等无意义 bigram 干扰排名 | 精确切分为"中间件""重写" |
| 小数据量（千~万级 chunk） | 噪音 token 的 IDF 值不够低（数据量不足以稀释），排名被严重干扰 | 精准词频统计，Top-K 干净 |

**历史决策轨迹**：

```
第四轮（初始）：CJK Bigram 做 MVP 默认
  ↓ 理由：零依赖，符合"不引入重依赖"
  ↓
分析反驳：小数据量下 bigram 噪音不可接受
  ↓ 关键认知：jieba-wasm 是纯 wasm（非 native addon），不算"重依赖"
  ↓
第五轮（修正）：jieba-wasm 做 MVP 默认，CJK Bigram 降级兜底
  ↓ 理由：精度 > 零依赖，4MB wasm 对 CLI 工具可接受
```

**包体积影响**：
- jieba-wasm npm 安装体积 16 MB（4 份 wasm 冗余），实际有效载荷 3.9 MB（nodejs 单份）
- 对比 libsql 已有体积 8.6 MB，属于同一量级
- 如需优化：postinstall 删除不需要的 bundler/deno/web 三份 wasm

**同步修复**：
- jieba-wasm 作为 `dependencies` 引入
- 停用词过滤加入 JiebaTokenizer 实现
- 混合检索的意义强调：jieba 未登录词切不到时，Embedding 向量通道兜底

### 10.6 评审结论

经过 5 轮审查，共发现并修复 **29 个问题**（致命 1 / 高 8 / 中 12 / 低 8）。核心收获：

1. **最小依赖**：删除 sqlite-vec（libsql 内置向量足够）；唯一新增 jieba-wasm（3.9 MB 有效载荷，精度不可替代）
2. **外部 IO 不上热路径**：贯穿 initialize、memory_write、MemoryWatcher 三个场景的核心设计原则
3. **中文分词是 BM25 的生死线**：jieba-wasm 做 MVP 默认（精度优先），CJK Bigram 做降级兜底（零依赖保障）
4. **数据量越小精度越重要**：千级数据下 Top-5 每个噪音结果 = 20% 上下文污染，直接影响 LLM 生成质量
5. **过滤下推 + Entry 去重**：保证检索质量不因 chunk 切分而崩塌
6. **设计与工程对齐**：每处架构决策都标注了具体改动的文件和函数签名

Phase 1-5 施工完成后，在集成和端到端验证过程中做了以下设计调整：

### 11.1 Embedding 配置模式变更

**原设计**：复用 providers 配置（从 `memory.embedding.provider` 字段找到已配置的 provider，复用其 apiKey + baseURL）。

**实际调整**：改为**完全独立配置**，不依赖 providers。

```jsonc
// 原设计（已放弃）
"memory": {
  "enabled": true,
  "embedding": {
    "provider": "glm",        // 引用 providers.glm 的 apiKey + baseURL
    "model": "embedding-3"
  }
}

// 实际落地
"memory": {
  "enabled": false,
  "embedding": {
    "apiKey": "your-embedding-api-key",
    "baseURL": "https://your-embedding-api-base-url/v4",
    "model": "your-embedding-model",
    "dimension": 1024
  }
}
```

**调整原因**：
- Chat API 和 Embedding API 的 baseURL 往往不同（如 GLM 的 chat 走 `coding/paas/v4`，embedding 走 `paas/v4`）
- 复用 provider 配置导致路径混淆，调试困难
- 独立配置语义更清晰：embedding 就是 embedding，不跟 chat 耦合

### 11.2 配置模板与降级链

**新增**：初始化时在 config.json 中生成 memory 配置模板（默认值占位）。

**完整降级链**：
```
memory.enabled = false → 记忆系统不启动

memory.enabled = true
  → embedding 值是模板默认（your-embedding-api-key 等）→ 纯 BM25
  → embedding 配置了真实值
    → 探测 isAvailable()
      → 可达 → 向量 + BM25 混合检索
      → 不可达 → 警告提示（黄色 ⚠，8 秒自动消失）→ 降级纯 BM25
        → 向量存储初始化失败 → 降级纯 BM25
```

**降级警告**：通过 `BootstrapResult.warnings` 传递到 App.tsx 渲染，黄色 `⚠` 提示 8 秒后自动消失。

### 11.3 CompactBridge 实施细节

**原设计**：提取和摘要合并为一次 LLM 调用。

**实际落地**：MVP 阶段分开做（两次 LLM 调用），简单可控。合并优化留后续迭代。

**注入方式**：
- `ContextManager.setCompactBridge(bridge)` — bootstrap 中 MemoryManager 初始化后注入
- `prepare()` 和 `compact()` 两个入口都在策略执行前调 `extractAndSave()`
- `compact()` 额外在压缩后注入 `getCompactHint()` 提示

### 11.4 /remember 命令

**原设计**：提到了 `/remember` 但未详细定义子命令。

**实际落地**：
```
/remember <content>       — 快速写入 project 记忆（标题自动提取）
/remember list [scope]    — 列出记忆（可选 global/project）
/remember search <query>  — 搜索记忆
/remember delete <id>     — 删除记忆
/remember rebuild         — 重建索引
```

别名：`/mem`、`/memory`。
