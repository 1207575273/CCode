## 一、背景与目标

### 1.1 现状

cCli 当前不具备跨会话的记忆能力：
- JSONL 会话持久化仅用于 resume/fork，不支持语义检索
- CCODE.md 指令文件是静态的，不支持动态记忆
- 上下文压缩（Compact）是有损的，信息随压缩丢失

### 1.2 目标

构建一个三层记忆系统 + RAG 抽象层，实现：

1. **跨会话持久记忆**：用户偏好、项目决策、技术债等信息跨会话保留
2. **语义检索**：用自然语言检索相关记忆，而非依赖精确匹配
3. **RAG 基础设施**：可复用的 RAG 引擎，为后续代码库索引等场景提供基础
4. **与现有架构无缝融合**：通过 Tool、System Prompt、Bootstrap、Hook 等扩展点集成

### 1.3 设计原则

- **人类可读可编辑**：记忆文件用 Markdown 存储，用户可直接编辑
- **渐进增强**：Embedding 可插拔，向量索引可选
- **复用现有基础设施**：SQLite (libsql)、Provider 系统、Tool 系统
- **不引入重依赖**：不依赖 LangChain RAG 链、外部向量数据库服务

## 二、三层记忆模型

| 层级 | 存储位置 | 生命周期 | 内容类型 |
|------|---------|---------|---------|
| **全局记忆** | `~/.ccode/memory/` | 永久，跨项目 | 用户偏好、编码风格、通用知识 |
| **项目记忆** | `<project>/.ccode/memory/` | 永久，项目内 | 架构决策、技术债、项目约定 |
| **会话记忆** | JSONL（已有） | 会话内 | 当前对话上下文 |

### 2.1 目录结构

```
# 全局记忆
~/.ccode/memory/
  ├── MEMORY.md                    # 索引文件（自动维护，列出所有记忆条目摘要）
  ├── user_preferences.md          # 用户偏好
  ├── coding_style.md              # 编码风格
  └── ...                          # 按语义主题自由组织

# 项目级记忆
<project-root>/.ccode/memory/
  ├── MEMORY.md                    # 索引文件
  ├── insights/                    # 语义主题目录（架构决策、技术债等）
  │   ├── auth-middleware.md
  │   └── performance-baseline.md
  ├── feedback/                    # 用户反馈与指导
  │   ├── testing-approach.md
  │   └── code-review-prefs.md
  └── sessions/                    # 会话自动总结（日期驱动）
      ├── 2026-04-01-project-setup.md
      └── 2026-04-02-bug-fix-auth.md
```

**组织原则**：
- insights/ 和 feedback/ 按**语义主题**组织，日期在 frontmatter 中
- sessions/ 按**日期**组织（日期前缀），因为会话总结天然按时间线回溯
- 全局记忆层级较浅，直接平铺即可
- MEMORY.md 是给人看的索引，不参与 RAG 检索

### 2.2 记忆文件格式

单个记忆文件 = **YAML frontmatter + Markdown body**：

```markdown
---
type: project | user | feedback | reference | session-summary
created: 2026-04-01T10:00:00Z
updated: 2026-04-01T10:00:00Z
tags: [auth, middleware, compliance]
source: user | agent | auto-summarize
---

# 认证中间件重写

重写 auth middleware 是因为 legal 标记了 session token 存储
不满足新合规要求。

- 范围决策应优先合规而非易用性
- 预计完成时间：2026-04-15
```

**关键设计决策**：
- 每个文件是一个语义完整的记忆单元（文件即 chunk）
- frontmatter 提供元数据用于标签过滤 + 向量索引
- 文件粒度由写入者控制——LLM 倾向细粒度，用户自由编辑
- 大文件（>1000 tokens）在索引时由 Chunker 切分为多个 chunk

### 2.3 MEMORY.md 索引文件

自动维护的索引，供人类快速浏览：

```markdown
# Memory Index

- [认证中间件重写](insights/auth-middleware.md) — 合规驱动的 auth 中间件重构
- [用户偏好 React](user_preferences.md) — React 17+ 函数组件风格
```

每次记忆条目增删改时自动更新。

### 3.2 核心接口

```typescript
// === types.ts ===

type MemoryScope = 'global' | 'project'

type MemoryType = 'user' | 'feedback' | 'project' | 'reference' | 'session-summary'

interface MemoryEntry {
  id: string                      // 文件名（不含扩展名）作为 ID
  scope: MemoryScope
  title: string
  content: string                 // Markdown body
  type: MemoryType
  tags: string[]
  source: 'user' | 'agent' | 'auto-summarize'
  created: string                 // ISO 8601
  updated: string
  filePath: string                // 原始文件路径
}

interface MemoryQuery {
  query: string
  scope?: MemoryScope | 'all'     // 默认 'all'
  tags?: string[]
  type?: MemoryType
  topK?: number                   // 默认 5
  dateRange?: { from?: string; to?: string }
}

interface MemorySearchResult {
  entry: MemoryEntry
  score: number                   // 0-1 相关性分数
  snippet: string                 // 匹配的文本片段
  matchedChunks: MemoryChunk[]    // 匹配的具体 chunk
}

interface MemoryChunk {
  id: string
  entryId: string
  text: string
  chunkIndex: number              // 同一文件的第几个 chunk
  embedding?: number[]            // 向量
}

// === embedding/types.ts ===

interface EmbeddingProvider {
  readonly name: string
  readonly dimension: number
  readonly maxBatchSize: number
  embed(text: string): Promise<number[]>
  embedBatch(texts: string[]): Promise<number[][]>
  isAvailable(): Promise<boolean>
}

// === MemoryManager 核心接口 ===

interface IMemoryManager {
  initialize(): Promise<void>                    // 扫描 + 建索引
  search(query: MemoryQuery): Promise<MemorySearchResult[]>
  write(entry: Omit<MemoryEntry, 'id' | 'created' | 'updated'>): Promise<MemoryEntry>
  update(id: string, content: string, tags?: string[]): Promise<MemoryEntry>
  delete(id: string): Promise<void>
  list(scope?: MemoryScope): Promise<MemoryEntry[]>
  getRelevantContext(query: string): Promise<string>  // System Prompt 注入用
}

// === storage/types.ts — 存储层抽象接口 ===

/**
 * 文件存储抽象 — 管理记忆条目的 CRUD
 * 实现类：FileStore（Markdown 文件系统）
 */
interface IFileStore {
  /** 扫描指定目录下所有记忆文件 */
  scan(basePath: string): Promise<MemoryEntry[]>
  /** 读取单个记忆文件 */
  read(filePath: string): Promise<MemoryEntry | null>
  /** 写入/创建记忆文件，返回完整条目 */
  save(entry: Omit<MemoryEntry, 'created' | 'updated'>): Promise<MemoryEntry>
  /** 更新已有记忆文件 */
  update(filePath: string, content: string, tags?: string[]): Promise<MemoryEntry>
  /** 删除记忆文件 */
  delete(filePath: string): Promise<void>
  /** 更新 MEMORY.md 索引文件 */
  updateIndex(entries: MemoryEntry[]): Promise<void>
}

/**
 * 向量存储抽象 — 管理向量索引的 CRUD + 相似度检索
 * 实现类：SqliteVectorStore | MemoryVectorStore
 *
 * 设计意图：将向量存储的具体实现（sqlite-vec）与业务逻辑解耦。
 * 后续可替换为 FAISS、ChromaDB、Qdrant 等实现，无需改动 Retriever 和 Indexer。
 */
interface IVectorStore {
  /** 向量维度，由 Embedding Provider 决定 */
  readonly dimension: number

  /** 初始化存储（建表/建索引） */
  initialize(): Promise<void>

  /** 批量写入向量 */
  upsert(chunks: Array<{
    id: string
    entryId: string
    embedding: number[]
    text: string
    chunkIndex: number
    scope: MemoryScope
    projectSlug?: string
    tags: string[]
    type: MemoryType
    source: string
    created: string
    updated: string
  }>): Promise<void>

  /** 删除某个记忆条目的所有 chunk */
  deleteByEntryId(entryId: string): Promise<void>

  /** 向量相似度检索，返回 Top-K 结果（按相似度降序） */
  similaritySearch(
    queryEmbedding: number[],
    options: {
      topK: number
      scope?: MemoryScope | 'all'
      projectSlug?: string
      tags?: string[]
      type?: MemoryType
    }
  ): Promise<Array<{
    chunkId: string
    entryId: string
    text: string
    score: number          // 0-1 相似度分数
  }>>

  /** 获取指定记忆条目的所有 chunk 文本（BM25 构建用） */
  getChunkTexts(scope?: MemoryScope): Promise<Array<{ chunkId: string; text: string; entryId: string }>>

  /** 清空所有向量数据 */
  clear(): Promise<void>

  /** 关闭连接/释放资源 */
  close(): Promise<void>
}

// === CompactBridge — Compact 与记忆融合接口 ===

interface ICompactBridge {
  /** 在压缩前调用，从即将被压缩的上下文中提取并保存关键信息 */
  extractAndSave(messages: Message[]): Promise<MemoryEntry[]>
  /** 在压缩后调用，返回可追加到上下文的记忆提示 */
  getCompactHint(): string
}
```

### 3.3 存储实现类

| 接口 | 实现 | 说明 |
|------|------|------|
| `IFileStore` | `FileStore` | 基于 Markdown 文件系统，读写 YAML frontmatter |
| `IVectorStore` | `SqliteVectorStore` | 基于 sqlite-vec，复用现有 libsql 连接 |
| `IVectorStore` | `MemoryVectorStore` | 纯内存实现，用于测试和 sqlite-vec 不可用时降级 |
| `ICompactBridge` | `CompactBridge` | 注入到现有 ContextManager 压缩流程 |

### 3.4 Embedding 配置

**配置位置**：`~/.ccode/config.json`

```json
{
  "memory": {
    "enabled": true,
    "embedding": {
      "provider": "glm",          // 必填，已配置的供应商名
      "model": "embedding-3"      // 必填，embedding 模型名
    }
  }
}
```

**规则**：
- `memory.enabled` 默认 `false`，用户主动开启
- 开启时 `embedding.provider` + `embedding.model` **必填**
- provider 必须支持 `/embeddings` 端点
- Anthropic 协议不支持 Embedding → 记忆功能不启用，启动时提示

**实现方式**：
- `ProviderEmbedding`：复用现有 providers 配置中的 `apiKey` + `baseURL`，调用 `/embeddings` 端点
- 无需新增 SDK 依赖，直接用 fetch 调用兼容 API
- 支持的协议：`openai-compat`（GLM、DeepSeek、OpenAI 等）
- 不支持：`native-anthropic`（无 Embedding API）

**降级链**：
```
配置的 provider → 检查协议是否支持 embedding
  ├── 支持（openai-compat）→ 使用
  ├── 不支持（anthropic）→ 警告 + 记忆功能禁用
  └── API 不可达 → 启动时报错，提示检查配置
```

### 3.5 向量索引

**存储**：扩展现有 `ccode.db` SQLite 数据库，新增 `memory_vectors` 表。

```sql
CREATE TABLE IF NOT EXISTS memory_vectors (
  id TEXT PRIMARY KEY,            -- chunk ID: {entryId}_{chunkIndex}
  entry_id TEXT NOT NULL,         -- MemoryEntry.id
  scope TEXT NOT NULL,            -- 'global' | 'project'
  project_slug TEXT,              -- 项目标识（全局记忆为 NULL）
  embedding BLOB NOT NULL,        -- Float32Array 序列化
  chunk_text TEXT NOT NULL,       -- 切分后的文本
  chunk_index INTEGER NOT NULL,   -- 同一文件的第几个 chunk
  tags TEXT,                      -- JSON array
  type TEXT NOT NULL,             -- MemoryType
  source TEXT NOT NULL,
  created TEXT NOT NULL,
  updated TEXT NOT NULL
);
```

**索引技术**：sqlite-vec 扩展，与现有 libsql 共享连接。

**向量维度**：由 Embedding Provider 的 `dimension` 属性决定，建表时记录。

### 3.6 混合检索（向量 + BM25 + RRF）

**检索流水线**：

```
用户查询 (query)
  ├── BM25 关键词检索 → 候选集 A (Top-K)
  ├── Dense 向量检索 → 候选集 B (Top-K)
       │
       ▼
  RRF (Reciprocal Rank Fusion) 融合
       │
       ▼
  元数据过滤（tags, type, dateRange, scope）
       │
       ▼
  Top-K 结果 → 返回 MemorySearchResult[]
```

**RRF 公式**：`RRF_score(d) = Σ 1/(k + rank_i(d))`，k = 60

**BM25 实现**：纯 TypeScript，零外部依赖。基于内存的倒排索引，启动时从 sqlite-vec 中加载 chunk_text 构建。

**向量检索**：sqlite-vec 的 `vec_distance_cosine` 函数。

### 3.7 文档切分

使用递归字符切分器（RAG 文档推荐的最佳策略）：

```typescript
class RecursiveCharacterChunker {
  chunk(text: string, options: {
    maxChunkSize: number    // 默认 512 tokens（约 2000 字符）
    overlap: number         // 默认 100 字符
    separators: string[]    // 默认 ['\n## ', '\n### ', '\n\n', '\n', ' ']
  }): MemoryChunk[]
}
```

**策略**：按层级分隔符递归切分，优先在标题/段落边界切割。小于 maxChunkSize 的文件不切分。

### 4.3 Bootstrap 集成

调整 `bootstrapAll()` 的初始化拓扑：

```
调整前：
  链 A: Skills → Instructions → Hooks → SessionStartHooks → SystemPrompt
  链 B: FileIndex 扫描
  链 C: Runtime Plugin 发现

调整后：
  链 A': Skills → Instructions → Hooks → SessionStartHooks
  链 B': FileIndex 扫描 ∥ MemoryManager.initialize()
  屏障: 等链 A' + 链 B' 都完成

  链 D': Runtime Plugin 发现（独立，可与 C' 并行）
```

关键点：MemoryManager.initialize() 需要在 SystemPrompt 构建前完成，以便注入记忆上下文。

### 4.4 Compact 与记忆融合

**核心理念**：Compact（上下文压缩）本质上是有损的"选择性遗忘"。融合记忆系统后，Compact 变为"先归档再压缩"——信息从短期记忆转移到长期记忆，而非直接丢失。

**融合流程**：

```
用户输入 /compact（或自动触发）
  │
  ▼
  ① 分析即将被压缩的上下文
     → 识别关键信息：用户偏好、项目决策、技术发现、未完成任务
     → 用 LLM 提取结构化记忆条目
  │
  ▼
  ② 自动写入长期记忆（去重检测）
     → 检查是否与已有记忆重复（向量余弦相似度 > 0.95）
     → 不重复：写入 insights/ 或 feedback/ 目录
     → 重复但内容更新：合并或覆盖
     → 更新向量索引
  │
  ▼
  ③ 执行原有的上下文压缩
     → 复用现有 3 种压缩策略
     → 压缩后的上下文 + 记忆已归档 = 信息损失最小化
  │
  ▼
  ④ 压缩后追加提示（可选）
     → 在压缩后的上下文中提示："部分历史信息已归档到记忆系统，可通过 memory_search 检索"
```

**实现要点**：

```typescript
// memory-compact-bridge.ts
// 注入到现有的 compact 流程中

interface CompactBridge {
  // 在压缩前调用，提取并保存关键信息
  extractAndSave(messages: Message[]): Promise<MemoryEntry[]>
  // 在压缩后调用，注入记忆提示
  getCompactHint(): string
}
```

**与现有 ContextManager 的集成**：
- 现有 `context-manager.ts` 有 3 种压缩策略（可插拔）
- 在 `compress()` 方法执行前，先调用 `CompactBridge.extractAndSave()`
- 压缩策略本身不变，只是前置了一个"记忆提取"步骤
- 记忆提取的 LLM 调用可以复用当前会话的 Provider，不额外配置

**触发时机**：
- 用户手动 `/compact`
- 上下文超阈值自动触发压缩
- 两种场景都走同一套融合逻辑

**成本控制**：
- 记忆提取的 LLM 调用使用轻量 prompt（~200 tokens 输入，~300 tokens 输出）
- 仅在记忆系统启用时才执行提取步骤
- 可通过配置关闭此融合行为：`memory.compactExtraction: false`

### 4.5 /remember 命令

作为用户主动管理记忆的快捷入口，定位为 **显式交互方式**（与 Compact 的隐式提取互补）：

```typescript
// commands/remember.ts
// 用法：
//   /remember <content>           — 快速写入记忆（交互式收集 title/tags）
//   /remember list [scope]        — 列出记忆
//   /remember search <query>      — 搜索记忆
//   /remember delete <id>         — 删除记忆
//   /remember rebuild             — 重建向量索引
```

**三种记忆管理方式对比**：

| 方式 | 触发者 | 适用场景 | 文件来源 |
|------|--------|---------|---------|
| **Compact 自动提取** | 系统 | 压缩上下文时隐式归档重要信息 | `source: 'auto-summarize'` |
| **LLM 自主写入** | Agent | LLM 在对话中判断信息重要时主动保存 | `source: 'agent'` |
| **/remember 命令** | 用户 | 用户显式要求"记住这个" | `source: 'user'` |
| **直接编辑文件** | 用户 | 直接编辑 Markdown 文件 | （无系统记录，文件直接变更） |

### 4.6 文件变更监听

```typescript
// memory-watcher.ts
// 监听 memory/ 目录的文件变更
// 新增文件 → 增量索引（embed + 写入 sqlite-vec + 更新 BM25）
// 修改文件 → 更新索引（删除旧 chunk → 重新 embed → 写入新 chunk）
// 删除文件 → 删除索引（删除对应 chunk）
```

使用 Node.js `fs.watch` 递归监听，debounce 300ms。

### 4.7 Hook 集成（P1）

```typescript
// hooks/session-memory-hook.ts
// 注册为会话结束 Hook
// 触发时机：会话正常结束或被 compact 时
// 行为：用 LLM 总结本次会话的关键决策、发现的问题、学到的经验
// 写入 sessions/ 目录，自动添加日期前缀
```

## 五、离线与在线 RAG 分工

### 5.1 离线索引（项目启动时）

```
bootstrapAll()
  → MemoryManager.initialize()
    → FileStore.scan()
        → 扫描 ~/.ccode/memory/ (全局)
        → 扫描 <project>/.ccode/memory/ (项目)
    → Indexer.buildIndex()
        → 遍历 MemoryEntry[]
        → RecursiveCharacterChunker.chunk()
        → EmbeddingProvider.embedBatch()
        → 写入 sqlite-vec 向量表
        → 构建 BM25 倒排索引（内存）
    → MemoryWatcher.start()
```

**增量策略**：
- 基于 `updated` 时间戳判断是否需要重新索引
- 只对新增/修改的文件重新 embed
- 删除的文件从索引中移除

### 5.2 在线检索（对话中）

```
LLM 调用 memory_search({ query, scope, tags })
  → Retriever.search(query)
    → sqlite-vec 向量相似度检索 Top-20
    → BM25 关键词检索 Top-20
    → RRF 融合 → Top-10
    → 元数据过滤（tags, type, dateRange, scope）
    → 返回 MemorySearchResult[]
```

### 5.3 在线学习（对话中写入）

```
LLM 调用 memory_write({ title, content, tags })
  → MemoryManager.write()
    → FileStore.save()         # 写入 Markdown 文件
    → Indexer.upsert()         # 即时向量化 + 索引更新
    → 更新 MEMORY.md 索引文件
    → MemoryWatcher 感知变更但跳过（自己写的，不重复索引）
```

**去重机制**：
- 写入前检查是否存在 title 相同或内容高度相似（向量余弦相似度 > 0.95）的记忆
- 相似时提示 LLM 选择覆盖或跳过
- 用户通过 /remember 命令写入时，交互式确认

## 九、风险与缓解

| 风险 | 影响 | 缓解策略 |
|------|------|---------|
| sqlite-vec 生态不够成熟 | 向量索引可能需要迁移 | 抽象 VectorStore 接口，后续可替换为其他实现 |
| Embedding API 延迟 | 启动建索引慢 | 批量 embed + 增量索引 + 异步后台建索引 |
| LLM 写入过多噪音 | 记忆质量下降 | 去重检测 + system prompt 中指导 LLM 仅保存重要信息 |
| 记忆文件冲突 | 用户编辑与 LLM 写入冲突 | 文件锁 + 跳过 MemoryWatcher 自身写入 |
| BM25 内存占用 | 大量记忆条目时内存压力 | 持久化到 SQLite FTS5（P1） |
