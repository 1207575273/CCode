### 验证说明

#### 2.1 分词器（tokenizer.test.ts — 17 个测试）

**验证什么**：BM25 的质量完全取决于分词质量。验证 jieba-wasm 能正确切分中文、过滤停用词、英文小写化、降级可用。

| 用例 | 输入 | 期望 | 为什么这是正确的 |
|------|------|------|----------------|
| 中文精准分词 | "认证中间件重写" | 包含"认证""中间件""重写" | jieba 切出语义完整的词，不是无意义的 bigram |
| 不含噪音 | "认证中间件重写" | 不包含"证中" | bigram 会产生"证中"这种噪音，jieba 不会 |
| 停用词过滤 | "这是一个认证中间件的重写" | 不包含"这""是""一""个""的" | 停用词在 BM25 中只会占据无意义权重 |
| 英文小写化 | "Hello World" | 包含"hello""world" | 搜索不区分大小写 |
| Bigram 降级 | "认证middleware" | 包含"认""证""认证""middleware" | jieba 不可用时有兜底方案 |

**为什么这证明了质量**：
- 搜"算法"时，jieba 只会匹配包含"算法"这个词的文档，不会误匹配"算盘"或"办法"
- 停用词过滤确保"的""了""是"不会干扰 BM25 的 TF-IDF 权重计算

#### 2.2 文档切分器（chunker.test.ts — 7 个测试）

**验证什么**：大文件能被正确切分为语义完整的 chunk。

| 用例 | 场景 | 期望 | 为什么重要 |
|------|------|------|-----------|
| 小文本不切分 | < maxChunkSize | 返回 1 个 chunk | 文件即 chunk 的设计原则 |
| 按标题切分 | 含 `## 标题` 的长文本 | 多个 chunk | 优先在语义边界切割 |
| 按段落切分 | 含 `\n\n` 的长文本 | 多个 chunk | 次优先级切分 |
| 强制切割 | 无分隔符的超长文本 | 每个 chunk ≤ maxChunkSize | 最后手段，保证不超限 |
| chunkEntry | 任意文本 | chunk 带 entryId + chunkIndex | 切分结果能关联回原始 entry |

#### 2.3 文件存储（file-store.test.ts — 13 个测试）

**验证什么**：记忆文件的完整生命周期。

| 用例 | 场景 | 期望 | 为什么重要 |

#### 3.1 BM25 倒排索引（bm25.test.ts — 11 个测试）

**验证什么**：关键词检索能力 + 过滤下推的正确性。

| 用例 | 场景 | 期望 | 为什么重要 |

### 验证说明

#### 4.1 混合检索 + Indexer（retriever.test.ts — 10 个测试）

**验证什么**：完整的 RAG 检索链路——BM25 + 向量 RRF 融合、Entry 级去重、Indexer 同步/异步分阶段。

| 用例 | 场景 | 期望 | 为什么重要 |
|------|------|------|-----------|
| 纯 BM25 检索 | Embedding 不可用（NoopEmbedding） | 仍能搜到结果，e1 排第一 | **降级模式的生命线**——Anthropic 用户或 API 不可达时，记忆系统不废 |
| RRF 融合 | BM25 + MockEmbedding 双通道 | e1 排第一（两个通道都命中） | 两个通道的结果能正确融合而非互相覆盖 |
| score 归一化 | 任意查询 | 最高分 = 1.0，所有分数在 [0,1] | 不同查询的 score 可比较 |
| tags 过滤 | 限 tags=['db'] | 只返回带 db 标签的结果 | 过滤下推到两个通道 |
| Entry 去重 | 同一 entry 被切成 2 个 chunk，两个都命中 | 只返回 1 条结果，matchedChunks=2 | 不去重会导致同一篇记忆占据多个 Top-K 位置 |
| Indexer buildBM25 | 2 个 entry 同步建索引 | bm25.size=2，chunks=2 | BM25 同步阶段毫秒级完成，不阻塞启动 |
| Indexer upsertEntry | 写入 1 个 entry | BM25 **立即**可搜 + 向量 **await 后**可搜 | "先写后 embed"原则验证 |
| Indexer removeEntry | 删除后清理 | BM25 和向量存储都为空 | 删除的一致性 |

**为什么 Entry 去重是关键**：
- 一篇 2000 字的记忆被切成 3 个 chunk
- 搜"认证中间件"时，chunk 0 和 chunk 1 都命中
- 不去重 → Top-5 里 2 个位置被同一篇记忆占据，浪费 40% 结果位
- 去重后 → 合并为 1 条结果（matchedChunks=[chunk0, chunk1]），score 取最高

**为什么"先写后 embed"很重要**：
- memory_write 工具调用时，LLM 在等结果返回
- embed API 调用需要 200-500ms 网络延迟
- 如果同步等 embed，LLM 每次写记忆都卡半秒
- 测试验证：BM25 **立即可搜**（同步），向量 **await embedPromise 后可搜**（异步）
