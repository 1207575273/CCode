## 一、概述

在 Web UI 的 ChatPage header 右侧（Context 标签旁）新增一个 🧠 按钮，点击后从右侧滑出抽屉面板，展示 System Prompt 组成概览 + 向量记忆散点图的统一视图。用户可以边聊天边查看记忆的语义分布和系统提示词构成。

---

## 二、核心决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 视图形式 | 统一视图（System Prompt + 向量记忆在同一界面） | 展示两者之间的关系 |
| 入口交互 | ChatPage 右侧滑出抽屉面板 | 可以边聊天边查看，不打断工作流 |
| 降维计算 | 纯前端 PCA | 记忆量级几十~几百条，浏览器毫秒级完成，不增加后端复杂度 |
| 散点粒度 | 每个 chunk 一个点 | 同一记忆的多个 chunk 用连线关联 |
| 着色维度 | 按 scope（global 蓝 / project 绿） | 简洁直观 |
| System Prompt | 固定标注节点，不做 embedding | 不消耗 API 调用，位置标记在散点图中 |
| 数据传输 | API 返回原始高维向量，前端降维 | 灵活，几百 KB 局域网无压力 |

### 3.1 区域一：System Prompt 概览

- 横向排列小卡片，展示各组成部分的 token 占比
- 卡片内容：名称 + token 数
- 组成部分：CLAUDE.md、MCP 工具描述、记忆上下文、其他指令
- 各段落作为 ★ 固定标注节点显示在散点图中（灰色，不参与聚类）

### 3.2 区域二：向量散点图

- 每个 chunk 一个 ● 点
- 同一 MemoryEntry 的多个 chunk 用细线连接
- 按 scope 着色：global = 蓝色，project = 绿色
- ★ 标记 System Prompt 固定节点（灰色）
- 支持鼠标滚轮缩放、拖拽平移
- PCA 降维在前端完成

### 3.3 区域三：选中详情

- 默认隐藏，点击/hover 散点后展开
- 展示：标题、类型（type）、标签（tags）、scope、内容摘要
- chunk 级别显示匹配的文本片段

## 四、数据接口

### 4.1 API 定义

```
GET /api/memory/vectors
```

### 4.2 响应格式

```typescript
interface MemoryVectorsResponse {
  chunks: Array<{
    id: string
    entryId: string
    title: string
    scope: 'global' | 'project'
    type: string
    tags: string[]
    chunkText: string
    chunkIndex: number
    embedding: number[]  // 原始高维向量（如 1024 维）
  }>
  systemPrompt: {
    totalTokens: number
    sections: Array<{
      name: string     // "Instructions" / "Skills" / "Hooks" / "Memory" 等
      tokens: number
      source: string   // 来源标识，如文件路径或组件名
    }>
  }
  dimension: number     // 向量维度（前端 PCA 需要知道）
}
```

### 4.3 数据来源

- `chunks`：从 `ccode.db` 的 `memory_vectors` 表读取
  - F32_BLOB 读出为 Buffer，通过 `new Float32Array(buf.buffer, buf.byteOffset, buf.byteLength / 4)` 转为数组
  - 需要 JOIN `memory_meta` 获取 title 等信息，或从 MemoryManager.entryMap 获取
- `systemPrompt.sections`：**关键问题 — 见 §9.1**
- `dimension`：从 `memory_meta` 表读取 `embedding_dimension`

## 五、技术选型

| 模块 | 方案 | 说明 |
|------|------|------|
| 散点图渲染 | Canvas 2D | 轻量，几百个点不需要 WebGL |
| PCA 降维 | `ml-pca` 库（~8KB gzip） | **不手写** — 见 §9.2 关于数值稳定性的分析 |
| 抽屉动画 | Tailwind `translate-x` + `transition` | 从右侧滑入，300ms 过渡 |
| 缩放拖拽 | 原生 wheel + mousedown/mousemove | 不引入额外交互库 |

**依赖变更**：仅新增 `ml-pca`（轻量 PCA 库，基于 SVD 分解，数值稳定），不引入 TensorFlow 等重型依赖。

## 七、交互细节

### 7.1 入口按钮

- 位置：ChatPage header 右侧，Context 标签和"关闭 Bridge"按钮之间
- 样式：与 Context 标签一致的小按钮风格
- 文案：`🧠 记忆` 或仅图标 `🧠`
- 无记忆数据时按钮灰显，hover 提示"暂无记忆数据"

### 7.2 抽屉面板

- 宽度：400px
- 从右侧滑入，背景半透明遮罩（可选）
- ESC 键 或 点击 × 按钮关闭
- 打开时 ChatPage 内容不位移，面板浮在上层

### 7.3 散点图交互

- hover 某点：高亮该点 + 同 entry 的其他 chunk，底部详情区显示信息
- 点击某点：锁定选中状态，再次点击取消
- 滚轮缩放：以鼠标位置为中心
- 拖拽平移：mousedown 后 mousemove
- 空白区域双击：重置视图到初始状态

### 7.4 无数据状态

- 无向量数据（未配置 Embedding 或无记忆）：散点图区域显示空状态提示
- System Prompt 概览仍可正常显示（不依赖 Embedding）

## 九、架构审视 — 关键风险与细节打磨

> 以下是从架构师和工程师角度对设计方案的二次审视，标注了 5 个必须在实施前解决的问题。

### 9.1 [关键] System Prompt sections 数据从哪来？

**问题**：设计文档说"从 ContextManager 获取 System Prompt 各段的 token 计数"，但实际代码中：

```typescript
// bootstrap.ts:253
export function buildSystemPrompt(hookContext: string, memoryContext?: string): void {
  const instructionsPrompt = getInstructionsPrompt()
  const skillsPrompt = getSkillsSystemPrompt()
  const parts = [instructionsPrompt, skillsPrompt, hookContext, memoryContext].filter(Boolean)
  cachedSystemPrompt = parts.length > 0 ? parts.join('\n\n') : undefined
}
```

拼接后只保留了最终字符串 `cachedSystemPrompt`，**各段落的边界信息丢失了**。Dashboard API 运行在 Bridge Server 进程中，而 `cachedSystemPrompt` 在 CLI 进程内存里，Bridge 无法直接访问。

**解决方案**：在 `buildSystemPrompt()` 中保存各段的结构化信息：

```typescript
// 新增：保存各段元信息
interface SystemPromptSection {
  name: string       // "Instructions" / "Skills" / "Hooks" / "Memory"
  charLength: number // 字符长度（用于估算 token）
  source: string     // 来源标识
}
let cachedSections: SystemPromptSection[] = []
```

然后通过两种方式之一暴露给 Web：
- **方案 A**：CLI 通过 WebSocket 推送 `system_prompt_info` 事件给 Bridge → 前端
- **方案 B**：新增 API `/api/memory/system-prompt`，CLI 在 Bootstrap 完成后写入 DB 或内存共享

**推荐方案 A**，复用现有 EventBus → Bridge → Web 的推送链路，不需要落库。

### 9.2 [关键] PCA 手写 50 行 ≠ 数值稳定

**问题**：原方案说"手写 ~50 行矩阵运算"做 PCA。但对 1024 维向量：

1. **协方差矩阵** = 1024 × 1024 = 100 万个元素
2. **特征值分解**（Eigendecomposition）的朴素实现容易有数值精度问题
3. 手写 Jacobi 旋转或 QR 分解既复杂又容易出 bug

**修正方案**：使用 `ml-pca` 库（~8KB gzip，纯 JS，基于 SVD 分解）。理由：
- SVD 比特征值分解数值更稳定
- 库经过社区验证，edge case 覆盖好
- 8KB 体积可以接受，不算重型依赖
- 备选：`ml-matrix`（ml-pca 的底层依赖）单独使用也可以

### 9.3 [中等] 传输体积估算与优化

**实测**：
- 200 个 chunk × 1024 维 × 每个浮点数平均 8 字符（JSON） ≈ **1.6 MB**
- 500 个 chunk → **4 MB**

对局域网虽然不大，但 JSON 序列化 float 效率差。

**优化选项**（按必要性排序，非必须全做）：
1. **截断精度**：`embedding.map(v => +v.toFixed(4))` — 4 位小数足够 PCA 降维，体积减半
2. **Base64 编码**：传 Float32Array 的 Base64 而不是 JSON 数组 — 体积减 60%
3. **分页加载**：超过 300 chunk 时分页（但 PCA 需要全量数据，分页反而麻烦）

**推荐**：先做精度截断（一行代码），体积问题不大就不做 Base64。

### 9.4 [中等] System Prompt "固定节点"的定位问题

**问题**：设计说 System Prompt 各段作为 ★ 固定标注节点显示在散点图中，但它们没有 embedding，没有坐标。放在哪里？

**解决方案**：

- **不放在散点图坐标系内**，而是作为**散点图外围的标注标签**
- 散点图上方/左侧放 System Prompt 段落标签，每个标签标注名称 + token 数
- 与散点图之间用虚线边框分隔
- 这样避免了"没有 embedding 怎么定位"的尴尬，语义上也更清晰（它们是"上下文容器"，不是"记忆点"）

### 9.5 [低] 实时更新策略

**问题**：用户在对话中写入新记忆后，面板是否自动刷新？

**建议**：
- 面板打开时首次加载一次
- 不做实时 WebSocket 推送更新（过度设计）
- 面板顶部加一个 🔄 刷新按钮，手动重新拉取
- 面板关闭再打开时自动重新加载

## 十、实施优先级

基于以上审视，建议实施顺序：

1. **先解决 §9.1**：Bootstrap 保存 sections 结构 + EventBus 推送

4. **PCA + 散点图**：引入 ml-pca + Canvas 散点图
5. **交互打磨**：hover/click 详情 + 缩放拖拽 + 空状态
