## 二、踩坑记录

### 坑 1：jieba-wasm ESM 兼容（require 在 ESM 下静默失败）

**现象**：启动时打印 `[Memory] jieba-wasm 未安装或加载失败，BM25 降级为 CJK Bigram`，但 `npm ls jieba-wasm` 确认已安装。

**根因**：项目 `package.json` 设置了 `"type": "module"`，ESM 模式下 `require()` 不可用。原代码用 `require('jieba-wasm')` 同步加载，被 `try-catch` 吞掉后走了 Bigram 降级。

**修复**：
- `JiebaTokenizer.ensureInit()` 改为 `async`，用 `await import('jieba-wasm')` 替代 `require()`
- `getTokenizer()` 改为 `async function`，返回 `Promise<Tokenizer>`
- `MemoryManager` 构造函数中的 BM25/Indexer/Retriever 创建移到 `initialize()`（已是 async）
- 测试加 `beforeAll(async () => await tokenizer.ensureInit())`

**教训**：ESM 项目中不能用 `require()`，且 `try-catch` 静默吞异常会让问题极难发现。

---

### 坑 2：Embedding 维度配置错误（1024 vs 2048）

**现象**：`memory_vectors` 表始终为空，Web 面板显示「暂无向量数据」。

**根因**：`~/.ccode/config.json` 中 `memory.embedding.dimension` 配置为 1024，但 GLM `embedding-3` 实际返回 2048 维向量。`embedPending()` 在插入时被 libsql 拒绝：
```
SqliteError: vector index(insert): dimensions are different: 2048 != 1024
```
异常被 Bootstrap 中的 `catch(() => {})` 静默吞掉。

**修复**：将配置 dimension 改为 2048。DB 的 `ensureMemoryTables` 自动检测维度变化并 DROP + 重建表。

**教训**：
- Embedding 模型的维度必须查官方文档确认，不能凭记忆填
- `catch(() => {})` 是非常危险的模式，至少应该 `console.warn`
- 可以考虑在初始化时自动探测实际维度（调一次 embed API 看返回长度）

---

### 坑 3：F32_BLOB Buffer 共享导致 embedding 空数组

**现象**：API 返回的 chunks 中 `"embedding": []`，所有向量数组都是空的。dimension=2048，chunks 有 4 条，但 embedding 字段为空。

**根因**：Node.js 的 `Buffer` 底层共享同一个 `ArrayBuffer` pool。代码中 `new Float32Array(buf.buffer, buf.byteOffset, buf.byteLength / 4)` 在序列化过程（Hono 的 `c.json()` 内部）中，`buf.buffer` 指向的共享 ArrayBuffer 可能已经被回收或偏移不正确。

**修复**：先 copy 到独立 ArrayBuffer 再转换：
```typescript
// Before（有 bug）
const f32 = new Float32Array(buf.buffer, buf.byteOffset, buf.byteLength / 4)

// After（修复）
const copy = new Uint8Array(buf).buffer
const f32 = new Float32Array(copy)
```

**教训**：Node.js Buffer 的 `.buffer` 属性指向共享内存池，直接用 `byteOffset` 构造 TypedArray 在跨作用域传递时不安全。安全做法是先 `new Uint8Array(buf)` copy 出独立副本。

---

### 坑 4：System Prompt sections 数据跨进程不可达

**现象**：设计时计划"从 ContextManager 获取 System Prompt 各段的 token 计数"，但实现发现 `buildSystemPrompt()` 拼接后只保留最终字符串，各段边界信息丢失。

**根因**：`cachedSystemPrompt` 是一个拼接后的纯字符串，无法反向拆分回各段。

**修复**：在 `buildSystemPrompt()` 中同步保存 `cachedSections: SystemPromptSection[]`，记录每段的 name 和 charLength。新增 `getSystemPromptSections()` 导出。API 通过 `charLength / 3.5` 粗略估算 token 数。

---

### 坑 5：PCA 手写 vs 库

**设计时的误判**："手写 ~50 行矩阵运算做 PCA"。

**实际问题**：1024/2048 维向量的协方差矩阵特征值分解不是 50 行能稳定实现的。朴素 Jacobi 旋转或 QR 分解有数值精度问题。

**修正**：使用 `ml-pca` 库（~8KB gzip，基于 SVD 分解，数值稳定），一行 `new PCA(data).predict(data, { nComponents: 2 })` 搞定。

---

### 坑 6：API 只读向量表，忽略文件系统记忆

**现象**：有记忆文件（.md）但没有向量数据时，面板显示「暂无向量数据（未配置 Embedding 或无记忆）」。

**根因**：API 只从 `memory_vectors` 表读数据。没配置 Embedding 或 embed 失败时，向量表为空，但文件系统中的记忆是存在的。

**修复**：API 增加 FileStore 扫描 `~/.ccode/memory/` 和 `<项目>/.ccode/memory/`，返回 `entries` 列表。前端根据有无向量数据切换显示模式：
- 有向量 → PCA 散点图
- 无向量但有文件 → 记忆条目列表
- 都没有 → 空状态提示
