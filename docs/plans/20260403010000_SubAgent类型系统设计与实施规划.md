> 日期: 2026-04-03
> 状态: 待评审
> 关联: F13 SubAgent、dispatch_agent 工具
> 版本: v4（新增输入输出 Schema 对比分析与结构化输出设计）

## 一、背景与目标

### 1.1 现状痛点

cCli 的 `dispatch_agent` 工具当前是"无类型、无名称、无结构化输出"的单一模式：

```typescript
// 当前输入 — 只有 3 个字段
{ description: string, prompt: string, run_in_background?: boolean }

// 当前输出 — 通用 ToolResult，信息全靠纯文本拼接
{ success: boolean, output: string, error?: string }
```

**问题**：
1. LLM 无法在**结构化层面**约束子 Agent 行为（只能靠 prompt 自然语言暗示）
2. 所有子 Agent 共享同一个工具集（全量 - dispatch_agent - ask_user），无法按任务类型收窄
3. 子 Agent 只有 agentId（17 位 hex），没有**人类可读名称**，日志和 UI 辨识度差
4. **输出是纯文本**：agentId、status、result 全部打散在 output 字符串里，LLM 只能靠自然语言解析
5. 无统一的 Agent 定义接口，自定义 Agent / 插件 Agent 无处挂载
6. 子 Agent 代码散落在 `tools/ext/`，与普通工具混杂

### 1.2 Claude Code AgentTool 参照

```typescript
// Claude Code 输入 — 9 个参数，结构化
{
  description: z.string(),
  prompt: z.string(),
  subagent_type: z.string().optional(),
  model: z.enum(['sonnet', 'opus', 'haiku']).optional(),
  run_in_background: z.boolean().optional(),
  name: z.string().optional(),
  team_name: z.string().optional(),
  mode: permissionModeSchema().optional(),
  isolation: z.enum(['worktree']).optional(),
}

// Claude Code 输出 — 结构化，区分同步/异步
{ status: 'completed', agentId, name, prompt, result }     // 同步完成
{ status: 'async_launched', agentId, name, description }    // 异步启动
```

### 1.3 差距矩阵

| 维度 | cCli 现有 | Claude Code | 本次目标 |
|------|-----------|-------------|---------|
| subagent_type | ❌ 注释预留 | ✅ | **做** |
| name | ❌ | ✅ | **做** |
| model | ❌ 实现了但没暴露 | ✅ enum | **做**（全名格式） |
| 结构化输出 | ❌ 纯文本拼接 | ✅ status 区分 | **做** |
| mode 权限 | ❌ | ✅ | 预留接口 |
| isolation | ❌ | ✅ | 预留接口 |
| team_name | ❌ | ✅ | 预留接口 |

### 1.4 设计目标

1. **三种内置类型**：general / explore / plan
2. **命名子 Agent**：每个子 Agent 有 name，人类可读
3. **结构化输出**：同步返回 `{ status, agentId, name, result }`，异步返回 `{ status, agentId, name }`
4. **统一 Agent 定义接口**：三种来源共享 `AgentDefinition`
5. **tools 下独立子目录**：`src/tools/agent/`
6. **硬性约束**：所有子 Agent 禁止 dispatch_agent（不可递归）

### 1.5 不做什么（接口预留）

| 能力 | 原因 |
|------|------|
| SendMessage | 复杂度高，name 字段已预留 |
| mode 权限模式 | 需权限分类器 |
| isolation: worktree | 需 git worktree 管理 |
| team_name | 需 Agent Teams 架构 |

## 二、核心架构 — AgentDefinition 统一接口

### 2.1 三层架构

```
┌──────────────────────────────────────────────────┐
│                dispatch_agent 工具                 │  ← LLM 调用入口
│  输入: subagent_type / name / model / prompt      │
│  输出: { status, agentId, name, result }          │
├──────────────────────────────────────────────────┤
│            AgentDefinitionRegistry                │  ← 类型注册表
│  ┌───────────┐  ┌───────────┐  ┌──────────────┐  │
│  │ built-in  │  │  custom   │  │   plugin     │  │
│  │ general   │  │ .ccode/   │  │ (预留接口)    │  │
│  │ explore   │  │ agents/   │  │              │  │
│  │ plan      │  │ *.md      │  │              │  │
│  └───────────┘  └───────────┘  └──────────────┘  │
├──────────────────────────────────────────────────┤
│            AgentDefinition 统一接口                │
│  agentType / source / getSystemPrompt()           │
│  toolPolicy / maxTurns / modelHint / whenToUse    │
├──────────────────────────────────────────────────┤
│     AgentLoop + ToolRegistry + SessionLogger      │  ← 执行层（现有）
└──────────────────────────────────────────────────┘
```

### 2.2 AgentDefinition 接口

```typescript
// src/tools/agent/types.ts

export type AgentSource = 'built-in' | 'custom' | 'plugin'

export type ToolPolicy =
  | { mode: 'exclude'; tools: string[] }
  | { mode: 'include'; tools: string[] }

/**
 * Agent 定义 — 描述一种 Agent 的"模板"。
 *
 * 当 LLM 调用 dispatch_agent({ subagent_type: 'explore' }) 时，
 * 框架查找对应 AgentDefinition 来配置系统提示、工具集、轮次上限。
 */
export interface AgentDefinition {
  readonly agentType: string
  readonly source: AgentSource
  readonly whenToUse: string
  getSystemPrompt(): string
  readonly toolPolicy: ToolPolicy
  readonly maxTurns: number
  readonly modelHint?: 'fast' | 'balanced' | 'strong'

  // ── 预留（注释保留语义） ──
  // readonly skills?: string[]
  // readonly background?: boolean
  // readonly isolation?: 'worktree'
  // readonly omitInstructions?: boolean
  // readonly hooks?: HooksSettings
}

/** 内置 Agent */
export interface BuiltInAgentDefinition extends AgentDefinition {
  readonly source: 'built-in'
}

/** 自定义 Agent — .ccode/agents/*.md */
export interface CustomAgentDefinition extends AgentDefinition {
  readonly source: 'custom'
  readonly filePath: string
}

/** 插件 Agent — 预留 */
export interface PluginAgentDefinition extends AgentDefinition {
  readonly source: 'plugin'
  readonly pluginName: string
}
```

### 2.3 AgentDefinitionRegistry

```typescript
// src/tools/agent/definition-registry.ts

export class AgentDefinitionRegistry {
  readonly #definitions = new Map<string, AgentDefinition>()

  register(def: AgentDefinition): void
  get(agentType: string): AgentDefinition | undefined
  getAll(): AgentDefinition[]
  getTypeNames(): string[]
  buildTypeDescriptions(): string
}

export const agentDefinitionRegistry = new AgentDefinitionRegistry()
```

注册顺序（后覆盖先）：内置 → 全局自定义 → 项目自定义 → 插件

## 三、dispatch_agent 输入输出 Schema 重新设计

### 3.1 输入 Schema

```typescript
// src/tools/agent/dispatch-agent.ts

parameters: {
  type: 'object',
  properties: {
    description: {
      type: 'string',
      description: 'Short task description (3-5 words)',
    },
    prompt: {
      type: 'string',
      description: 'Complete instructions for the sub-agent',
    },
    subagent_type: {
      type: 'string',
      enum: ['general', 'explore', 'plan'],
      description:
        'Agent type — determines tools and behavior:\n' +
        '- general (default): Full toolset, complex multi-step tasks\n' +
        '- explore: Read-only, fast code search and analysis\n' +
        '- plan: Read-only, outputs implementation plans',
    },
    name: {
      type: 'string',
      description:
        'Human-readable name for this sub-agent (e.g. "search-auth", "plan-refactor"). ' +
        'Used in logs, UI, and progress tracking. Auto-generated if omitted.',
    },
    model: {
      type: 'string',
      description:
        'Override model (e.g. "glm-5", "claude-sonnet-4-6"). Inherits parent model if omitted.',
    },
    run_in_background: {
      type: 'boolean',
      description: 'Run in background, return immediately with agentId',
    },
  },
  required: ['description', 'prompt'],
}
```

### 3.2 输出 Schema — 结构化

**现有问题回顾**：

```typescript
// 现有：同步完成 — agentId、status 全部藏在 output 纯文本里
return {
  success: true,
  output: `[Sub-agent completed: ${description}]\n\n${finalText}\n\n[INSTRUCTION: ...]`,
}

// 现有：异步启动 — agentId 也是拼在字符串里
return {
  success: true,
  output: `Sub-agent started in background (agentId: ${agentId}, task: ${description}).`,
}
```

**改造方案**：

ToolResult 的 `output` 字段改为**结构化 JSON 字符串**（保持 ToolResult 接口不变，但 output 是可 parse 的 JSON）：

```typescript
/** SubAgent 同步完成的结构化输出 */
interface AgentCompletedOutput {
  status: 'completed'
  agentId: string
  name: string
  agentType: string
  model: string
  prompt: string           // 回显传入的 prompt（LLM 上下文用）
  result: string           // 子 Agent 最终文本输出
}

/** SubAgent 异步启动的结构化输出 */
interface AgentAsyncLaunchedOutput {
  status: 'async_launched'
  agentId: string
  name: string
  agentType: string
  model: string
  description: string
}

/** SubAgent 失败的结构化输出 */
interface AgentErrorOutput {
  status: 'error'
  agentId: string
  name: string
  agentType: string
  error: string
  partialResult?: string   // 失败前已产出的部分文本
}

type AgentOutput = AgentCompletedOutput | AgentAsyncLaunchedOutput | AgentErrorOutput
```

**实际返回**：

```typescript
// 同步完成
return {
  success: true,
  output: JSON.stringify({
    status: 'completed',
    agentId,
    name,
    agentType,
    model: modelName,
    prompt,
    result: finalText,
  } satisfies AgentCompletedOutput),
}

// 异步启动
return {
  success: true,
  output: JSON.stringify({
    status: 'async_launched',
    agentId,
    name,
    agentType,
    model: modelName,
    description,
  } satisfies AgentAsyncLaunchedOutput),
}

// 失败
return {
  success: false,
  output: JSON.stringify({
    status: 'error',
    agentId,
    name,
    agentType,
    error: errorMsg,
    partialResult: finalText || undefined,
  } satisfies AgentErrorOutput),
  error: `子 Agent 执行异常: ${errorMsg}`,
}
```

### 3.3 为什么用 JSON 字符串而不改 ToolResult 接口

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: output 改为 JSON 字符串 | ToolResult 接口不变，其他工具不受影响 | LLM 需要解析 JSON |
| B: ToolResult 新增 `data` 字段 | 强类型，LLM 直接拿到结构化数据 | 改接口影响面大，所有工具和消费者 |
| C: 用 ToolResultMeta | 已有 meta 字段 | meta 设计为 UI 展示元数据，语义不匹配 |

**选择方案 A**：最小改动，LLM 天然能解析 JSON。

但要在 description 中提示输出格式：

```
Output is a JSON object with fields:
- status: 'completed' | 'async_launched' | 'error'
- agentId: unique agent ID
- name: human-readable agent name
- result: agent output text (when completed)
```

### 3.4 wrapSubagentResult 的去留

现有 `wrapSubagentResult()` 在 output 里塞了一段"不要重复验证"的指令文本：

```typescript
'[INSTRUCTION: The sub-agent has already executed all steps and verified the result.',
'Do NOT repeat any of the above actions. Simply relay this result to the user.]',
```

**改造**：
- 这段指令从 output 移到 **description**（工具描述）— 这是全局约束，不应每次输出都带
- output 只放纯净的 JSON 结构
- description 已有："IMPORTANT: The sub-agent result is ALREADY VERIFIED."

## 四、三种内置类型

### 4.1 general — 通用型

| 属性 | 值 |
|------|-----|
| agentType | `'general'` |
| whenToUse | 通用型，处理复杂多步任务：代码实现、文件修改、编译验证等 |
| toolPolicy | `{ mode: 'exclude', tools: [] }` |
| maxTurns | 25 |
| modelHint | `'balanced'` |

**系统提示**：

```
You are a sub-agent. Complete the assigned task autonomously.
Constraints:
- Focus solely on the given task
- Do NOT dispatch further sub-agents
- Do NOT ask questions — you have no user interaction
- When done, output your final result as plain text
```

### 4.2 explore — 探索型

| 属性 | 值 |
|------|-----|
| agentType | `'explore'` |
| whenToUse | 只读探索：代码搜索、定义查找、调用链分析、目录结构梳理 |
| toolPolicy | `{ mode: 'include', tools: ['read_file', 'grep', 'glob', 'bash', 'task_output'] }` |
| maxTurns | 15 |
| modelHint | `'fast'` |

**系统提示**：

```
You are a code exploration specialist. Your job is to search, read, and analyze code — never modify it.

Rules:
- Use only read-only tools: read_file, grep, glob, bash (read-only commands only)
- Do NOT create, edit, or delete any files
- Be fast and focused — stop as soon as you have the answer
- Output: file paths with line numbers + relevant code snippets + concise analysis

If bash is needed, only run read-only commands (cat, find, git log, wc, etc).
Never run commands that modify state (rm, mv, npm install, git commit, etc).
```

### 4.3 plan — 规划型

| 属性 | 值 |
|------|-----|
| agentType | `'plan'` |
| whenToUse | 架构分析和实施规划：设计方案、影响评估、重构计划。只读不执行 |
| toolPolicy | `{ mode: 'include', tools: ['read_file', 'grep', 'glob'] }` |
| maxTurns | 15 |
| modelHint | `'strong'` |

**系统提示**：

```
You are a software architect. Your job is to analyze requirements, read existing code, and produce implementation plans.

Rules:
- Read code and documentation to understand the current state
- Do NOT modify any files or execute any commands
- Output a structured plan: steps, file list, key design decisions, risks, dependencies
- Be specific: reference exact file paths and line numbers when relevant
- Identify what to change, what to add, what to leave alone
```

### 4.4 工具矩阵

```
硬编码排除（所有类型）：dispatch_agent、ask_user_question

                        read  grep  glob  bash  edit  write  todo  kill  verify  task_output  skill  memory_*
general (exclude [])      ✓     ✓     ✓     ✓     ✓     ✓     ✓     ✓     ✓        ✓          ✓       ✓
explore (include 白名单)  ✓     ✓     ✓     ✓     ✗     ✗     ✗     ✗     ✗        ✓          ✗       ✗
plan    (include 白名单)  ✓     ✓     ✓     ✗     ✗     ✗     ✗     ✗     ✗        ✗          ✗       ✗
```

MCP 工具：白名单模式下默认不可用（可能有副作用）。

## 五、SubAgent 命名机制

### 5.1 双标识

| 标识 | 用途 | 示例 |
|------|------|------|
| **agentId** | 系统唯一 ID，17 位 hex | `a3f8e2b01c9d7e4f2` |
| **name** | 人类可读名称 | `search-eventbus`、`plan-dashboard` |

### 5.2 name 生成规则

```
1. LLM 传入 name       → 规范化（trim、kebab-case、截断 40 字符）+ 会话内去重
2. LLM 未传 name       → 自动生成：`{agentType}-{agentId 前 6 位}`
                           例: explore-a3f8e2、general-7b2c1d
```

会话内同名 → 自动追加序号：`search-code`、`search-code-2`

### 5.3 使用场景

```
CLI SubAgentPanel:
  ⠋ [explore] search-eventbus  turn 3/15  ▸ grep
  ✓ [general] refactor-config   25.3s

ToolStatusLine:
  ⠋ Agent[search-eventbus] turn 3/15 ▸ grep

task_output 支持 name 查找:
  task_output({ agent_id: "search-eventbus" })    // 按 name
  task_output({ agent_id: "a3f8e2b01c9d7e4f2" })  // 按 agentId

结构化输出 JSON:
  { "status": "completed", "agentId": "a3f8e2...", "name": "search-eventbus", ... }
```

cloneWith(...names: string[]): ToolRegistry {
  const includeSet = new Set(names)
  const cloned = new ToolRegistry()
  for (const tool of this.#tools.values()) {
    if (includeSet.has(tool.name)) {
      cloned.register(tool)
    }
  }
  return cloned
}

## 九、构建受限 ToolRegistry

```typescript
/** 硬编码排除 — 所有子 Agent 必须遵守 */
const ALWAYS_EXCLUDE = ['dispatch_agent', 'ask_user_question']

function buildSubRegistry(parentRegistry: ToolRegistry, toolPolicy: ToolPolicy): ToolRegistry {
  if (toolPolicy.mode === 'include') {
    const allowed = toolPolicy.tools.filter(t => !ALWAYS_EXCLUDE.includes(t))
    return parentRegistry.cloneWith(...allowed)
  } else {
    const allExclude = [...new Set([...ALWAYS_EXCLUDE, ...toolPolicy.tools])]
    return parentRegistry.cloneWithout(...allExclude)
  }
}
```

## 十、执行流程

```
LLM 调用 dispatch_agent({
  subagent_type: 'explore',
  name: 'search-eventbus',
  prompt: '...',
  model: 'glm-5',
  run_in_background: false,
})
  │
  ├─ 1. 解析 subagent_type → agentDefinitionRegistry.get('explore')
  │     └─ 找不到 → 回退 general + warn
  │
  ├─ 2. 生成 agentId（17 位 hex）
  │
  ├─ 3. 解析 name（传入 → 规范化 + 去重；未传 → 自动生成）
  │
  ├─ 4. 解析 model（显式 > 继承父 Agent）
  │
  ├─ 5. 构建受限 ToolRegistry（buildSubRegistry）
  │
  ├─ 6. 获取系统提示（definition.getSystemPrompt()）
  │
  ├─ 7. 注册到 store（agentId + name + agentType + modelName）
  │
  ├─ 8. 创建 AgentLoop 并执行
  │
  └─ 9. 返回结构化 JSON 输出
        ├─ 同步: { status: 'completed', agentId, name, agentType, model, prompt, result }
        ├─ 异步: { status: 'async_launched', agentId, name, agentType, model, description }
        └─ 失败: { status: 'error', agentId, name, agentType, error, partialResult? }
```

## 十七、踩坑记录

### 17.1 libsql DiskANN 向量索引 `INSERT OR REPLACE` shadow row 冲突

**现象**：memory 模块写入向量时反复报错：

```
SqliteError: vector index(insert): failed to insert shadow row
  code: 'SQLITE_ERROR', rawCode: 1
```

错误发生在 `LibsqlVectorStore.upsert()` 调用 `INSERT OR REPLACE INTO memory_vectors` 时，每次 embed 都会触发。

**根因**：libsql 的 DiskANN 向量索引维护了内部 shadow table（`idx_memory_vec_shadow`）用于跟踪索引状态。`INSERT OR REPLACE` 在 SQLite 内部实际执行 `DELETE` + `INSERT`，但 DiskANN 索引的 shadow table 在这个内部 DELETE 阶段未正确清理残留记录，导致后续 INSERT 写入 shadow row 时发现已有冲突而报错。

**修复**：将 `INSERT OR REPLACE` 拆为显式的 `DELETE` + `INSERT`，在同一事务中执行：

```typescript
// 修复前（触发 bug）
const stmt = db.prepare(`INSERT OR REPLACE INTO memory_vectors ...`)
stmt.run(...)

// 修复后（绕开 shadow table 冲突）
const delStmt = db.prepare('DELETE FROM memory_vectors WHERE id = ?')
const insStmt = db.prepare(`INSERT INTO memory_vectors ...`)
delStmt.run(c.id)
insStmt.run(...)
```

**额外处理**：如果 shadow table 已损坏，需要手动清理数据库：

```sql
DROP INDEX IF EXISTS idx_memory_vec;
DROP TABLE IF EXISTS idx_memory_vec_shadow;
DROP TABLE IF EXISTS libsql_vector_meta_shadow;
DROP TABLE IF EXISTS memory_vectors;
DELETE FROM memory_meta WHERE key = 'embedding_dimension';
```

数据库位于 `~/.ccode/data/ccode.db`，清理后重启应用会自动重建。memory 原始条目在 `.ccode/memory/` 目录下是 markdown 文件，不受影响。

**文件**：`src/memory/storage/libsql-vector-store.ts`  
**提交**：`07d0633 fix: libsql 向量索引 INSERT OR REPLACE shadow row 冲突`  
**教训**：libsql 的向量索引功能尚不成熟，对带向量索引的表避免使用 `INSERT OR REPLACE` / `UPSERT`，改用显式 `DELETE` + `INSERT`。

### 17.2 Dev 模式 Web Dashboard 反代 Vite 失败（502 "未就绪"）

**现象**：dev 模式（`pnpm run dev:web`）启动 cCli 后，访问 `http://localhost:9800/session/:id` 页面显示 `Vite dev server 未就绪，请稍等...`（HTTP 502）。尤其在用户同时运行自己的前端项目时必现。

**背景**：Bridge Server 的 dev 模式架构为"反向代理"——`ccli.ts` 用 `execa` 启动独立的 Vite dev server 进程，Bridge Server（9800 端口）将非 `/ws`、`/api` 的 HTTP 请求 `fetch` 转发到 Vite（5173 端口）。`fetch` 失败时返回 502 错误文本。

**根因分析**（两个独立问题叠加）：

1. **端口冲突**：Vite 端口硬编码为 5173。用户自己的前端项目（如 Vue/React 脚手架）默认也用 5173，导致 cCli 的 Vite 进程要么启动失败（端口被占），要么反代到了错误的 Vite 实例。
2. **启动时序**：`execa('npx', ['vite'])` 是异步启动，ccli.ts 没有等待 Vite 进程就绪就继续执行。Bridge Server 立刻开始监听 9800 端口，此时用户访问页面，反代 `fetch('http://localhost:5173/...')` 因 Vite 尚未监听而抛连接拒绝错误，被 `catch` 捕获返回 502。

**探索过程**：

最初尝试了 **Vite middleware 模式**（`createViteServer({ server: { middlewareMode: true } })`），将 Vite 内嵌到 Bridge Server 进程内，消除独立端口和反代。但遇到以下问题：
- Vite 是 web 子项目的依赖，主项目未安装，需要 `createRequire` 从 web 目录动态解析
- Windows 下 `import()` 不接受裸绝对路径（`D:\...`），需要 `pathToFileURL` 转换
- `server.ts` 中 `scriptDir` 到 web 目录的相对路径计算错误（`../../web` 少了一级应为 `../../../web`）
- Vite middleware 的 connect 中间件与 Hono + `@hono/node-server` + `@hono/node-ws` 在 request listener 层面集成复杂，替换 `server.listeners('request')` 影响 WebSocket upgrade
- Vite middleware 模式不自带 SPA fallback，`/session/:id` 等客户端路由返回 404

综合来看 middleware 模式改动面大、与现有 Hono 生态集成成本高，**不如在原有反代架构上做最小修复**。

**最终方案**：保持反代架构，解决端口冲突和启动时序：

```
改动前：
  ccli.ts → execa('npx', 'vite')                    // 固定 5173，不等就绪
  server.ts → fetch(`http://localhost:5173/...`)     // 硬编码端口

改动后：
  ccli.ts → net.createServer().listen(0) 找空闲端口   // 动态端口，避免冲突
          → execa('npx', 'vite', '--port', port)     // 用空闲端口启动
          → 轮询 net.createConnection 等端口就绪       // 等 Vite ready，最多 15s
          → startBridgeServer({ dev: true, vitePort })
  server.ts → fetch(`http://localhost:${vitePort}/...`)  // 用传入端口
```

具体改动：

1. **`server.ts`**：删除 `VITE_DEV_PORT = 5173` 硬编码常量，`BridgeServerOptions` 新增 `vitePort?: number`，反代 URL 改用 `options.vitePort`。

2. **`ccli.ts`**：
   - `net.createServer().listen(0)` 让操作系统分配空闲端口，取到后立即关闭监听
   - `execa('npx', ['vite', '--port', String(vitePort)])` 用该端口启动 Vite
   - 轮询 `net.createConnection({ port })` 检测端口连通性（100ms 间隔，最多 150 次 = 15 秒）
   - 就绪后再调用 `startBridgeServer({ dev: true, vitePort })`

**文件**：`src/server/bridge/server.ts`、`bin/ccli.ts`  
**教训**：
- 反代外部进程时，必须解决端口分配（动态 > 硬编码）和启动时序（等就绪 > 即发即忘）两个基本问题
- Vite middleware 模式虽然理论上更优（单端口、无反代），但与 Hono node-server + WebSocket 的集成成本高，在现有架构上做最小改动是更务实的选择
- Windows 环境下 ESM 动态 `import()` 不接受裸绝对路径，必须用 `pathToFileURL()` 转换——这是 Node.js 在 Windows 上的特殊限制
