# SubAgent 类型系统设计与实施规划

> 日期: 2026-04-03
> 状态: 待评审
> 关联: F13 SubAgent、dispatch_agent 工具

---

## 一、背景

### 1.1 现状

cCli 的 `dispatch_agent` 工具当前只有一种子 Agent 模式：

```typescript
// 参数
{ description: string, prompt: string, run_in_background?: boolean }

// 行为
// - 统一系统提示词（通用的 "You are a sub-agent"）
// - 统一工具集（全量工具 - dispatch_agent - ask_user_question）
// - 统一 maxTurns = 25
// - 模型继承父 Agent（model 参数已实现但注释掉未开放）
```

LLM 在调用 `dispatch_agent` 时，**只能通过 prompt 自然语言描述来隐式表达意图**。对于"只需要搜索代码不需要修改"、"只需要分析不需要执行"这类场景，LLM 无法在结构化层面约束子 Agent 的行为边界。

### 1.2 Claude Code 的做法

Claude Code 将 Agent 工具参数化为 7 个维度：

| 参数 | 说明 |
|------|------|
| `subagent_type` | 预定义类型：general-purpose / Explore / Plan / code-reviewer / test-runner / debugger |
| `model` | 模型覆盖：sonnet / opus / haiku |
| `name` | 可寻址命名，支持 SendMessage 继续通信 |
| `mode` | 权限模式：acceptEdits / bypassPermissions / plan / auto / dontAsk |
| `isolation` | 隔离模式：worktree（git 工作树隔离） |
| `run_in_background` | 后台异步执行 |
| `description` / `prompt` | 任务描述和完整指令 |

每种 `subagent_type` 背后对应：
- **专属系统提示词**（引导子 Agent 聚焦特定任务模式）
- **工具白名单/黑名单**（Explore 只有只读工具，Plan 不能修改代码）
- **默认 maxTurns**（Explore 快速收敛，general 允许更多轮次）
- **默认模型建议**（Explore 用快速模型，code-reviewer 用强模型）

### 1.3 差距总结

| 维度 | cCli | Claude Code | 优先级 |
|------|------|-------------|--------|
| Agent 类型体系 | 无 | 6+ 预定义类型 | **P0** |
| model 参数 | 已实现未开放 | 开放 | **P0** |
| 专属系统提示 | 统一一个 | 每类型独立 | **P0** |
| 工具受限策略 | 统一 | 按类型白名单/黑名单 | **P0** |
| name 命名 | 无 | 支持 SendMessage | P2 |
| mode 权限模式 | 全部 isSidechain | 6 种模式 | P2 |
| isolation 隔离 | 无 | worktree | P2 |
| 自定义 Agent 定义 | 无 | `.claude/agents/*.md` | P1 |

## 二、设计方案

### 2.1 SubAgent 类型注册表

新增文件 `src/tools/ext/subagent-types.ts`，定义类型注册表：

```typescript
/** 子 Agent 类型标识 */
export type SubAgentTypeName =
  | 'general'        // 通用型：完整工具集，处理复杂多步任务
  | 'explore'        // 探索型：只读工具，快速搜索和代码分析
  | 'plan'           // 规划型：只读 + 分析，输出方案不写代码
  | 'code-reviewer'  // 代码审查：只读 + 分析，输出审查报告
  | 'test-runner'    // 测试执行：只读 + bash，运行测试并报告
  | 'debugger'       // 调试型：全量工具，聚焦错误定位和修复

/** 子 Agent 类型配置 */
export interface SubAgentTypeConfig {
  name: SubAgentTypeName
  /** 一行描述，供 LLM 选择类型时参考 */
  label: string
  /** 系统提示词 */
  systemPrompt: string
  /** 工具限制策略 */
  toolPolicy: ToolPolicy
  /** 默认最大轮次 */
  maxTurns: number
  /** 建议使用的模型强度（hint，不强制） */
  modelHint?: 'fast' | 'balanced' | 'strong'
}

/** 工具限制策略 */
export type ToolPolicy =
  | { mode: 'exclude'; tools: string[] }  // 排除指定工具
  | { mode: 'include'; tools: string[] }  // 只保留指定工具
```

### 2.2 预定义类型详细设计

#### (1) general — 通用型

```
系统提示: 完成指派任务，可以使用所有工具，自主决策
工具策略: exclude ['dispatch_agent', 'ask_user_question']（现有逻辑）
maxTurns: 25
modelHint: balanced
```

适用场景：复杂多步实现任务、文件创建/修改、需要编译验证的工作。

#### (2) explore — 探索型

```
系统提示:
  你是代码探索专家。任务是搜索和分析代码，回答问题。
  约束：
  - 只使用只读工具（read_file、grep、glob、bash 只跑只读命令）
  - 不修改任何文件
  - 快速收敛，找到答案就停
  - 输出要简洁：文件路径 + 行号 + 关键代码片段

工具策略: include ['read_file', 'grep', 'glob', 'bash', 'task_output']
maxTurns: 15（搜索不需要太多轮）
modelHint: fast
```

适用场景：
- "帮我找一下 XXX 在哪定义的"
- "这个模块的调用链是什么"
- "搜索所有使用了 YYY 的地方"

#### (3) plan — 规划型

```
系统提示:
  你是软件架构师。任务是分析需求并输出实施方案。
  约束：
  - 可以阅读代码和文档理解现状
  - 不修改任何文件、不执行任何命令
  - 输出结构化的实施计划（步骤、文件清单、关键设计决策）
  - 识别风险点和依赖关系

工具策略: include ['read_file', 'grep', 'glob']
maxTurns: 15
modelHint: strong（规划需要强推理能力）
```

适用场景：
- "帮我设计这个功能的方案"
- "评估一下这个重构的影响范围"

#### (4) code-reviewer — 代码审查

```
系统提示:
  你是资深代码审查者。审查给定的代码变更。
  关注维度：
  - 正确性：逻辑错误、边界条件
  - 安全性：注入、XSS、敏感信息泄露
  - 性能：N+1 查询、内存泄漏、不必要的循环
  - 可维护性：命名、重复代码、过度复杂
  输出格式：按文件列出问题，严重程度标注 [Critical/Warning/Info]

工具策略: include ['read_file', 'grep', 'glob', 'bash']
maxTurns: 15
modelHint: strong
```

适用场景：
- 完成一个功能后自动审查
- "帮我审查一下 src/xxx 的代码质量"

#### (5) test-runner — 测试执行

```
系统提示:
  你是测试工程师。运行测试并分析结果。
  约束：
  - 运行指定的测试命令
  - 分析测试输出，总结通过/失败/跳过数量
  - 对失败的测试，给出失败原因和建议修复方向
  - 不修改源代码（只运行测试）

工具策略: include ['bash', 'read_file', 'grep', 'glob']
maxTurns: 10
modelHint: fast
```

适用场景：
- 代码变更后跑一遍测试
- "帮我跑一下 memory 模块的单元测试"

#### (6) debugger — 调试型

```
系统提示:
  你是调试专家。定位和修复指定的问题。
  工作方式：
  - 先复现问题（运行命令、查看日志）
  - 定位根因（阅读相关代码、添加临时调试）
  - 修复问题（最小化改动）
  - 验证修复（重新运行）
  输出：根因分析 + 修复 diff + 验证结果

工具策略: exclude ['dispatch_agent', 'ask_user_question']（同 general）
maxTurns: 25
modelHint: strong
```

适用场景：
- "这个测试为什么失败了"
- "帮我定位这个 bug 的根因"

### 2.5 工具描述更新

dispatch_agent 的 description 需要更新，让 LLM 知道可以选择类型：

```
Dispatch a sub-agent to handle a task independently.

Agent types:
- general (default): Full toolset, complex multi-step tasks
- explore: Read-only, fast code search and analysis
- plan: Read-only analysis, outputs implementation plans
- code-reviewer: Code quality and security review
- test-runner: Run tests and analyze results  
- debugger: Locate and fix bugs

The sub-agent runs autonomously with its own context.
Set run_in_background=true for parallel execution.

IMPORTANT: The sub-agent result is ALREADY VERIFIED.
Do NOT re-verify or re-run commands the sub-agent completed.
```

### 2.6 ToolRegistry 增强

当前 `cloneWithout` 只支持排除模式。需新增 `cloneWith` 支持白名单模式：

```typescript
// registry.ts 新增方法
cloneWith(...names: string[]): ToolRegistry {
  const includeSet = new Set(names)
  const cloned = new ToolRegistry()
  for (const tool of this.#tools.values()) {
    if (includeSet.has(tool.name)) {
      cloned.register(tool)
    }
  }
  return cloned
}
```

## 三、自定义 Agent 定义（Phase 2）

### 3.1 文件格式

`.ccode/agents/<name>.md`（项目级）或 `~/.ccode/agents/<name>.md`（全局级）：

```markdown
---
name: security-reviewer
label: 安全审查专家
tools:
  mode: include
  list: [read_file, grep, glob, bash]
maxTurns: 15
modelHint: strong
---

你是安全代码审查专家。重点关注：
- SQL 注入
- XSS
- 路径遍历
- 敏感信息泄露
- 认证/授权缺陷

输出格式：
[CRITICAL] 文件:行号 - 问题描述
[WARNING] 文件:行号 - 问题描述
```

### 3.2 加载优先级

```
1. 项目级 .ccode/agents/ → 优先
2. 全局级 ~/.ccode/agents/ → 次之
3. 内置预定义类型 → 兜底

同名时项目级覆盖全局级，自定义覆盖内置。
```

### 3.3 加载时机

启动时扫描 agents 目录，解析 frontmatter，注册到类型注册表。
dispatch_agent 的 `subagent_type` enum 动态生成（内置 + 自定义）。

## 四、远期能力（Phase 3，仅记录不实施）

### 4.1 name + SendMessage

```typescript

{ name: 'my-explorer', description: '...', prompt: '...' }
```

### 4.2 mode 权限模式

```typescript
// dispatch_agent 新增 mode 参数

```

当前 isSidechain=true 等价于 bypassPermissions。引入 mode 体系需要先有权限分类器，暂不做。

## 七、决策待确认

| 决策点 | 选项 | 建议 |
|--------|------|------|
| explore 类型是否允许 bash | A: 允许（搜索命令） B: 不允许（纯文件读取） | **A**：grep/find 的 bash 替代很常用 |
| plan 类型是否允许 bash | A: 允许 B: 不允许 | **B**：规划不需要执行命令 |
| subagent_type 不传时默认类型 | A: general B: 按 prompt 内容自动推断 | **A**：简单可靠，不引入额外推断成本 |
| model 参数格式 | A: 模型全名 "glm-5" B: 别名 "fast"/"strong" | **A**：与 config.json 一致，无歧义 |
| Phase 2 自定义 Agent 是否做 | A: 做 B: 先不做 | 取决于实际需求频率 |
| 工具策略冲突处理（MCP 工具） | A: MCP 工具对所有类型可用 B: MCP 工具只对 general/debugger 可用 | **A**：MCP 工具是用户显式启用的，不应被类型限制 |

## 八、风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM 不选 subagent_type | 退化到 general，无功能损失 | description 中明确列出类型和场景 |
| 弱模型选错类型 | explore 该用 general | 类型约束只限制工具集，不阻断执行；LLM 会在无法完成时报告 |
| 白名单遗漏 MCP 工具 | explore 子 Agent 无法调 MCP 工具 | include 策略额外保留所有 MCP 前缀工具 |
| 系统提示词过长 | 占用子 Agent 上下文 | 每个类型的 prompt 控制在 200 token 以内 |
