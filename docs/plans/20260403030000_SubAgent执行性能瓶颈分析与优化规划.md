# cCli SubAgent 执行性能瓶颈分析与优化规划

> 日期：2026-04-03
> 背景：cCli 的 SubAgent 干活效率明显低于 Claude Code CLI，本文基于两侧源码逐行审计，给出落地到具体文件和行号的改造方案
> 注意：本文所有时间数据为基于代码分析的估算值，非实测数据。实施前后应使用 `performance.now()` 打点对比

---

## 一、问题定性

SubAgent 干活慢的本质是**三类开销的叠加**：

1. **每次工具调用的固定税**：shell 查找、进程创建、MSYS2 初始化等，每次 bash 命令 ~195ms（Windows），20 次工具调用累积 ~3.9 秒
2. **每次 LLM 请求的重复传输**：无 Prompt Cache，系统提示 + 工具定义每轮全量发送，5 个并行 SubAgent 差距可达 5-10 秒
3. **每次 submit / dispatch 的重复初始化**：配置文件重读、Provider 重建、Registry 重建，本应一次性完成的工作被反复执行

三者叠加后，一个 SubAgent 的总体感知延迟比 Claude Code 慢 **3-5 倍**。

## 三、逐项分析与改造方案

### 3.1 resolveShell() 无缓存

**现状**（`src/platform/shell-resolver.ts` 全文 54 行）：

```
findGitBash()          → :20-31  每次遍历 GIT_BASH_PATHS + PATH.split(';') + existsSync() × N
resolveShell()         → :33-53  每次调用 findGitBash() + detectPlatform()
```

`BashTool.execute()` 每次调用都走 `resolveShell()`（`bash.ts:42`），Windows 上 `existsSync()` 遍历可能 5-8 个路径。

**改造方案**：

```typescript
// src/platform/shell-resolver.ts — 改造后
import { existsSync } from 'node:fs'
import { detectPlatform } from './detector.js'

const GIT_BASH_PATHS = [/* ... 不变 */]

/** 模块级缓存，进程生命周期内只解析一次 */
let _cached: ResolvedShell | undefined

function doResolveShell(preferred?: ShellType): ResolvedShell {
  const { isWindows } = detectPlatform()
  if (isWindows) {
    if (preferred !== 'powershell') {
      const gitBash = findGitBash()
      if (gitBash) return { type: 'gitbash', path: gitBash, args: ['-c'] }
    }
    return { type: 'powershell', path: 'powershell.exe', args: ['-NoProfile', '-NonInteractive', '-Command'] }
  }
  const bash = existsSync('/bin/bash') ? '/bin/bash' : '/bin/sh'
  return { type: bash.endsWith('bash') ? 'bash' : 'sh', path: bash, args: ['-c'] }
}

export function resolveShell(preferred?: ShellType): ResolvedShell {
  // preferred 参数目前代码中从未传入非 undefined 值，安全缓存
  if (!_cached) _cached = doResolveShell(preferred)
  return _cached
}
```

**改动量**：5 行 | **收益**：每次 bash 调用 -50~100ms (Windows)

同理 `detectPlatform()`（`src/platform/detector.ts:18-30`）也应缓存：

```typescript
// src/platform/detector.ts — 加缓存
let _cached: PlatformInfo | undefined
export function detectPlatform(): PlatformInfo {
  if (!_cached) {
    const p = os.platform()
    _cached = { platform: p, arch: os.arch(), isWindows: p === 'win32', ... }
  }
  return _cached
}
```

### 3.2 ToolRegistry 每次 submit 重建

**现状**（`src/core/bootstrap.ts:78-103`）：

`buildRegistry()` 每次调用创建 1 个 `new ToolRegistry()` + 15 个 `new XxxTool()` 实例。

调用点：
- `useChat.ts:225` — **每次用户发消息**
- `pipe-runner.ts:53` — 每次 pipe 调用

子 Agent 执行时（`dispatch-agent.ts:174`）还要 `cloneWith/Without()` 创建新的 Map。

**但工具实例是无状态的**。`ReadFileTool`、`GrepTool`、`BashTool` 等没有任何实例级别的可变状态，它们的 `execute()` 方法只依赖传入的 `args` 和 `ctx`。

**改造方案**：

```typescript
// src/core/bootstrap.ts — 改造

/** 会话级 registry 单例，MCP 工具注册后失效重建 */
let _registryCache: ToolRegistry | null = null
let _registryDirty = false

export function getRegistry(): ToolRegistry {
  if (!_registryCache || _registryDirty) {
    _registryCache = buildRegistry()
    _registryDirty = false
  }
  return _registryCache
}

/** MCP 工具注册后调用，标记 registry 需要重建 */
export function invalidateRegistry(): void {
  _registryDirty = true
}

// buildRegistry() 函数体不变，只是不再被外部直接调用
```

调用点改造：

```typescript
// src/ui/useChat.ts:225 — 改前
const registry = buildRegistry()
// 改后
const registry = getRegistry()

// src/core/pipe-runner.ts:53 — 同理
const registry = getRegistry()
```

子 Agent 的 `buildSubRegistry()`（`dispatch-agent.ts:324-332`）保持 `cloneWith/Without()`，因为子 Agent 确实需要裁剪工具集。但 clone 的是缓存的单例 registry，不再每次从头构建。

**风险点**：`DispatchAgentTool.description`（`:54-70`）和 `parameters`（`:73-110`）是 getter，每次访问都调用 `agentDefinitionRegistry.buildTypeDescriptions()` 和 `getTypeNames()` 重新生成。如果 registry 被缓存为单例，这两个 getter 的动态生成行为不受影响——它们读的是 `agentDefinitionRegistry`（独立的全局注册表），不是 `ToolRegistry`。`toToolDefinitions()` 缓存（见 3.4 节）需要在 `agentDefinitionRegistry` 变化时失效，但实际上 `registerBuiltInAgents()` 是幂等的，agent 定义不会动态变化，所以安全。

**改动量**：15 行 | **收益**：每次 submit -10~30ms，子 Agent 创建时避免重复的 15 个 `new Tool()`

### 3.3 LLM Provider 每次 submit 新建 + OpenAI 兼容模式双重创建

**现状有三个问题**：

**问题 A**：`useChat.ts:224` 每次 submit 都 `createProvider()`，创建新的 Anthropic SDK 客户端。

**问题 B**：`configManager.load()`（`config-manager.ts:47-58`）每次调用都 `readFileSync()` 读磁盘——无内存缓存。`useChat.ts:222` 和 `dispatch-agent.ts:456` 都在调它。

**问题 C**：`openai-compat.ts:25-31` 更严重——每次 `chat()` 调用都 `new ChatOpenAI()`，再 `bindTools()`：

```typescript
// openai-compat.ts:24-39 — 现状
async *chat(request: ChatRequest): AsyncIterable<StreamChunk> {
  const baseModel = new ChatOpenAI({       // ⚠️ 每次 chat() 都新建
    apiKey: this.#config.apiKey,
    model: request.model,                  // ← 注意：model 每次可能不同
    ...
  })
  const model = (request.tools && request.tools.length > 0)
    ? baseModel.bindTools(request.tools.map(t => ({...})))  // ⚠️ 每次都 bind
    : baseModel
```

一个子 Agent 执行 5 轮 LLM 调用 = 5 次 `new ChatOpenAI()` + 5 次 `bindTools()`。

**改造方案**：

**A. Provider 缓存**（`src/providers/registry.ts`）：

```typescript
const _providerCache = new Map<string, LLMProvider>()

export function getOrCreateProvider(providerName: string, config: CCodeConfig): LLMProvider {
  const cfg = config.providers[providerName]
  if (!cfg) throw new Error(`Provider "${providerName}" 未配置`)
  // 缓存 key：provider 名 + 完整 apiKey hash（apiKey 可能在设置页面切换）
  const cacheKey = `${providerName}|${cfg.baseURL ?? ''}|${cfg.apiKey}`

  let cached = _providerCache.get(cacheKey)
  if (!cached) {
    cached = createProvider(providerName, config)
    _providerCache.set(cacheKey, cached)
  }
  return cached
}

/** 配置变更时清空缓存（设置页保存后调用） */
export function clearProviderCache(): void {
  _providerCache.clear()
}
```

**B. configManager 加内存缓存**（`src/config/config-manager.ts`）：

```typescript
export class ConfigManager {
  readonly #configPath: string
  #cached: CCodeConfig | null = null
  #cachedMtime: number = 0

  load(): CCodeConfig {
    // 检查文件修改时间，未变则返回缓存
    try {
      const mtime = statSync(this.#configPath).mtimeMs
      if (this.#cached && mtime === this.#cachedMtime) return this.#cached
      this.#cachedMtime = mtime
    } catch {
      // 文件不存在，走原逻辑
    }

    // 原有逻辑...
    const raw = readFileSync(this.#configPath, 'utf-8')
    this.#cached = JSON.parse(raw) as CCodeConfig
    return this.#cached
  }

  save(config: CCodeConfig): void {
    this.#ensureDir()
    this.#write(config)
    this.#cached = config  // 写入后更新缓存
    this.#cachedMtime = statSync(this.#configPath).mtimeMs
  }
}
```

**C. OpenAI ChatOpenAI 复用 — Session Provider 模式**

详见第九节「Session Provider 架构设计」。这是一个涉及 Provider 接口的架构变更，不宜放在 P0 中，移至 P1。

调用点改造（P0 部分，Provider 缓存 + configManager 缓存）：

```typescript
// src/ui/useChat.ts:222-224 — 改前
const config = configManager.load()
const provider = createProvider(currentProvider, config)
// 改后
const config = configManager.load()  // 现在有 mtime 缓存，不重复读磁盘
const provider = getOrCreateProvider(currentProvider, config)

// src/tools/agent/dispatch-agent.ts:461 — resolveSubAgentProvider() 内
// 改前
const provider = createProvider(name, config)
// 改后
const provider = getOrCreateProvider(name, config)
```

**附带收益**：子 Agent 如果继承父 Agent 的 model，`resolveSubAgentProvider()` 在 `:451-453` 直接返回 `ctx.provider!`，不创建新实例——这条路径已经是对的，无需改。

**改动量**：45 行（A + B 部分） | **收益**：每次 submit -20~50ms，configManager.load() 避免重复磁盘读取

### 3.4 toToolDefinitions() 每轮 LLM 调用都序列化

**现状**（`agent-loop.ts:189`）：

```typescript
const chatRequest = {
  tools: this.#registry.toToolDefinitions(),  // 每轮都调用
  // ...
}
```

`ToolRegistry.toToolDefinitions()`（`src/tools/core/registry.ts:55-67`）遍历所有工具，为每个生成 `{ name, description, parameters }` 对象。15 个工具 × 每轮调用 = 不必要的重复序列化。

**改造方案**：

```typescript
// src/tools/core/registry.ts — 加缓存
export class ToolRegistry {
  #tools = new Map<string, Tool | StreamableTool>()
  #cachedDefinitions: ToolDefinition[] | null = null

  register(tool: Tool | StreamableTool): void {
    this.#tools.set(tool.name, tool)
    this.#cachedDefinitions = null  // 注册新工具时失效缓存
  }

  toToolDefinitions(): ToolDefinition[] {
    if (!this.#cachedDefinitions) {
      this.#cachedDefinitions = Array.from(this.#tools.values()).map(t => ({
        name: t.name,
        description: t.description,
        parameters: t.parameters,
      }))
    }
    return this.#cachedDefinitions
  }
}
```

**改动量**：8 行 | **收益**：每轮 LLM 调用减少一次数组遍历和对象创建

### 3.5 Shell 快照机制（Windows 每次 bash 命令省 ~80ms）

**现状**：`bash.ts:59` 每次通过 execa 调用 `bash -c "command"`，Git Bash 每次都走完整的 login shell 初始化。

**Claude Code 的做法**（`ShellSnapshot.ts` + `bashProvider.ts`，已读取源码确认）：

1. **CLI 启动时**：用 `child_process.execFile` 执行一次 `bash -c -l <snapshot-script>`，source 用户 `.bashrc`，将函数（base64 编码）、shopt 选项、别名、PATH 导出到 `~/.ccode/shell-snapshots/snapshot-bash-<ts>.sh`
2. **每次命令执行时**：`bash -c "source <snapshot> && eval <command> && pwd -P >| <cwd-file>"`——不传 `-l`，跳过 login shell 初始化
3. **退出时**：清理快照文件

**cCli 改造方案**：

新增 `src/platform/shell-snapshot.ts`（~120 行）：

```typescript
import { execFile } from 'node:child_process'
import { mkdir, unlink, access } from 'node:fs/promises'
import { join } from 'node:path'
import { homedir, tmpdir } from 'node:os'
import { resolveShell } from './shell-resolver.js'
import { detectPlatform } from './detector.js'

const SNAPSHOT_TIMEOUT = 10_000

let _snapshotPath: string | undefined
let _snapshotPromise: Promise<string | undefined> | undefined

/**
 * 启动时异步创建 Shell 快照。
 * 返回快照文件路径；失败返回 undefined（降级为 login shell）。
 */
export function startSnapshotCreation(): Promise<string | undefined> {
  if (_snapshotPromise) return _snapshotPromise

  _snapshotPromise = (async () => {
    const shell = resolveShell()
    if (shell.type === 'powershell') return undefined  // PowerShell 不需要快照

    const { isWindows } = detectPlatform()
    const snapshotsDir = join(homedir(), '.ccode', 'shell-snapshots')
    await mkdir(snapshotsDir, { recursive: true })

    const ts = Date.now()
    const rand = Math.random().toString(36).slice(2, 8)
    const snapshotFile = join(snapshotsDir, `snapshot-${shell.type}-${ts}-${rand}.sh`)

    // 检测用户 shell 配置文件
    const configFile = join(homedir(), shell.type === 'bash' || shell.type === 'gitbash' ? '.bashrc' : '.profile')

    // 构建快照脚本
    const script = buildSnapshotScript(snapshotFile, configFile, isWindows)

    return new Promise<string | undefined>((resolve) => {
      execFile(shell.path, ['-c', '-l', script], {
        timeout: SNAPSHOT_TIMEOUT,
        maxBuffer: 1024 * 1024,
        env: { ...process.env, GIT_EDITOR: 'true' },
      }, async (error) => {
        if (error) {
          resolve(undefined)
          return
        }
        try {
          await access(snapshotFile)
          _snapshotPath = snapshotFile
          resolve(snapshotFile)
        } catch {
          resolve(undefined)
        }
      })
    })
  })()

  return _snapshotPromise
}

/** 获取已创建的快照路径（同步，创建完成后可用） */
export function getSnapshotPath(): string | undefined {
  return _snapshotPath
}

/** 清理快照文件 */
export async function cleanupSnapshot(): Promise<void> {
  if (_snapshotPath) {
    try { await unlink(_snapshotPath) } catch { /* ignore */ }
    _snapshotPath = undefined
  }
}

function buildSnapshotScript(snapshotFile: string, configFile: string, isWindows: boolean): string {
  const esc = (s: string) => s.replace(/'/g, "'\\''")
  return `
SNAPSHOT_FILE='${esc(snapshotFile)}'
[ -f "${configFile}" ] && source "${configFile}" < /dev/null 2>/dev/null

echo "# Shell snapshot" >| "$SNAPSHOT_FILE"
echo "unalias -a 2>/dev/null || true" >> "$SNAPSHOT_FILE"

# 导出函数（base64 编码，避免特殊字符问题）
echo "# Functions" >> "$SNAPSHOT_FILE"
declare -F | cut -d' ' -f3 | grep -vE '^_[^_]' | while read func; do
  encoded=$(declare -f "$func" | base64)
  echo "eval \\"\\$(echo '$encoded' | base64 -d)\\" > /dev/null 2>&1" >> "$SNAPSHOT_FILE"
done

# 导出 shell 选项
echo "# Shell Options" >> "$SNAPSHOT_FILE"
shopt -p | head -n 200 >> "$SNAPSHOT_FILE"
set -o | grep "on" | awk '{print "set -o " $1}' | head -n 100 >> "$SNAPSHOT_FILE"
echo "shopt -s expand_aliases" >> "$SNAPSHOT_FILE"

# 导出别名（Windows 过滤 winpty）
echo "# Aliases" >> "$SNAPSHOT_FILE"
${isWindows
    ? "alias | grep -v \"='winpty \" | sed 's/^alias //g' | sed 's/^/alias -- /' | head -n 200 >> \"$SNAPSHOT_FILE\""
    : "alias | sed 's/^alias //g' | sed 's/^/alias -- /' | head -n 200 >> \"$SNAPSHOT_FILE\""
}

# 导出 PATH
echo "export PATH='$PATH'" >> "$SNAPSHOT_FILE"
`.trim()
}
```

**修改 `bash.ts`** — 仅修改前台模式的命令构建逻辑，**保留 execa 和后台模式 `#runBackground()` 不变**：

```typescript
// src/tools/core/bash.ts — 改造点仅在 execute() 的前台分支
import { getSnapshotPath } from '@platform/shell-snapshot.js'

async execute(args: Record<string, unknown>, ctx: ToolContext): Promise<ToolResult> {
  const command = String(args['command'] ?? '')
  if (!command.trim()) return { success: false, output: '', error: '命令不能为空' }

  const cwd = String(args['cwd'] ?? ctx.cwd)
  const shell = resolveShell()  // 已有缓存，~0ms
  const runInBackground = args['run_in_background'] === true
  const rawTimeout = Number(args['timeout'])
  const timeout = !Number.isFinite(rawTimeout) || rawTimeout <= 0
    ? DEFAULT_TIMEOUT_MS : Math.min(rawTimeout, MAX_TIMEOUT_MS)

  // ---- 后台模式不变 ----
  if (runInBackground) {
    return this.#runBackground(shell, command, cwd)  // 保留 execa（需要 detached + unref）
  }

  // ---- 前台模式：注入快照 ----
  const snapshotPath = getSnapshotPath()
  const finalCommand = snapshotPath
    ? `source '${snapshotPath}' 2>/dev/null || true && ${command}`
    : command

  const shellArgs = snapshotPath
    ? ['-c', finalCommand]              // 有快照：不传 -l，省掉 login shell
    : [...shell.args, finalCommand]     // 无快照：保持原行为

  try {
    // 前台仍用 execa（保留 timeout 和 timedOut 检测能力）
    const { stdout, stderr } = await execa(shell.path, shellArgs, {
      cwd,
      timeout,
      reject: true,
      windowsHide: true,
    })
    const output = [stdout, stderr].filter(Boolean).join('\n')
    return { success: true, output: output || '(no output)' }
  } catch (err: unknown) {
    // ... 原有错误处理不变（timedOut 检测等）
  }
}
```

> **为什么保留 execa 而非换 child_process.spawn**：
> 1. execa 提供 `timedOut` 属性，spawn 没有原生超时支持（需要手动 `setTimeout` + `kill`）
> 2. `#runBackground()` 依赖 execa 的 `detached` + `unref()` + Promise-like API 和 process tracker 集成
> 3. 快照机制的收益主要在跳过 `-l` login shell 初始化，与用什么进程库无关
> 4. 完全替换 execa 可以作为 P2 单独做，收益有限（模块已加载，后续调用无额外开销）

**在 bootstrap 中启动快照创建**：

```typescript
// src/core/bootstrap.ts — bootstrapAll() 内加一条并行链
import { startSnapshotCreation, cleanupSnapshot } from '@platform/shell-snapshot.js'

// 链 E'：Shell 快照创建（完全独立，不阻塞其他初始化）
startSnapshotCreation()

// 进程退出时清理
process.on('exit', () => { void cleanupSnapshot() })
```

**改动量**：~150 行新增 + ~30 行修改 | **收益**：每次 bash 命令 -80ms (Windows)，-30ms (Linux/macOS)

### 3.7 Prompt Cache：SubAgent systemPrompt 传递

**现状**：每个 SubAgent 通过 `definition.getSystemPrompt()` 独立构建 systemPrompt（`dispatch-agent.ts:184`），与主 Agent 的 systemPrompt 完全不同，无法命中 Anthropic API 的 Prompt Caching。

**Claude Code 的做法**：Fork 模式的子 Agent 直接继承父 Agent 的 `renderedSystemPrompt` 字节，确保 API 请求前缀一致 → 缓存命中。

**cCli 改造方案**（中等投入，收益最大）：

```typescript
// src/tools/agent/dispatch-agent.ts — stream() 内
// 改前：
const subLoop = new AgentLoop(subProvider, subRegistry, {
  systemPrompt: definition.getSystemPrompt(),  // 独立 prompt
})

// 改后：将主 agent 的 systemPrompt 作为基础，子 agent 指令作为追加
const parentSystemPrompt = ctx.systemPrompt  // 通过 ToolContext 传入
const subDirective = definition.getSystemPrompt()
const subLoop = new AgentLoop(subProvider, subRegistry, {
  // 保持前缀不变 → Anthropic Prompt Cache 命中
  systemPrompt: parentSystemPrompt
    ? `${parentSystemPrompt}\n\n---\n\nSubAgent directive:\n${subDirective}`
    : subDirective,
})
```

同时确保工具定义列表**顺序稳定**（当前 `ToolRegistry` 基于 `Map` 插入顺序，已满足）。

**改动量**：15 行 | **收益**：Anthropic API 缓存命中后延迟降低 50-80%，token 费用降低 90%（缓存前缀部分）

## 九、Session Provider 架构设计（P1 #5 核心变更）

### 9.1 问题：全局 Provider 单例 vs 并发 SubAgent 的矛盾

P0 #4 将 Provider 缓存为全局单例（按 providerName + apiKey）。这对 Anthropic Provider 是安全的——`new Anthropic()` 只是存配置，`chat()` 每次创建独立的 HTTP 流，无共享状态。

但对 OpenAI 兼容 Provider，问题出现了：

```
如果在全局单例上缓存 ChatOpenAI（按 model key）：

Provider("glm") ← 全局单例
  └─ modelCache["glm-4-flash"] ← 一个 ChatOpenAI 实例
       ├─ SubAgent-A 第 1 轮 chat() ← 并发
       ├─ SubAgent-B 第 3 轮 chat() ← 并发
       └─ SubAgent-C 第 2 轮 chat() ← 并发
                                       ↑ 共享 httpAgent、连接状态等
```

LangChain 的 `ChatOpenAI` 内部持有 `httpAgent`、请求队列等有状态资源。多个并发 SubAgent 共享同一实例，可能出现连接池耗尽、请求交叉等不可预测行为。

### 9.2 Claude Code 的做法：不缓存客户端

Claude Code 源码确认（`src/services/api/withRetry.ts:170-253`）：

```typescript
// Claude Code — 每个请求链路创建临时客户端
let client: Anthropic | null = null

for (let attempt = 1; attempt <= maxRetries + 1; attempt++) {
  if (client === null || isAuthError || isStaleConnection) {
    client = await getClient()  // 按需创建，不缓存
  }
  return await operation(client, attempt, retryContext)
}
```

- **不做全局缓存**——客户端在每个 `withRetry` 循环内创建，循环结束后 GC 回收
- SubAgent 通过 `ToolUseContext.options` 继承父 Agent 的**配置参数**（不是客户端实例）
- 认证令牌等昂贵资源单独缓存在 `bootstrap/state.ts`，与客户端对象解耦

Claude Code 能这样做是因为 `new Anthropic()` 极轻量。cCli 的问题在于 `new ChatOpenAI()` 成本高（LangChain 内部初始化链长），所以需要缓存——但缓存粒度不能是全局的。

### 9.3 设计方案：Session Provider

**核心思想**：Provider 分两层——全局层（存配置、缓存 apiKey）和会话层（绑定到 AgentLoop 生命周期、持有 ChatOpenAI 实例）。

```
全局层（P0 #4 缓存）：
  Provider("glm") ← 单例，只存 config，不持有 ChatOpenAI

会话层（每个 AgentLoop 一个）：
  ├─ SessionProvider-A（主 Agent）    ← provider.createSession()
  │    └─ #chatModel: ChatOpenAI      ← 主 Agent 5 轮 chat() 复用
  │    └─ AgentLoop 结束 → 无引用 → GC
  │
  ├─ SessionProvider-B（SubAgent-1）  ← provider.createSession()
  │    └─ #chatModel: ChatOpenAI      ← SubAgent-1 独占
  │    └─ SubAgent 结束 → 无引用 → GC
  │
  └─ SessionProvider-C（SubAgent-2）  ← provider.createSession()
       └─ #chatModel: ChatOpenAI      ← SubAgent-2 独占
       └─ SubAgent 结束 → 无引用 → GC
```

### 9.4 接口变更

```typescript
// src/providers/provider.ts — LLMProvider 接口新增可选方法

export interface LLMProvider {
  readonly name: string
  readonly protocol: ProviderProtocol
  chat(request: ChatRequest): AsyncIterable<StreamChunk>
  isModelSupported(model: string): boolean

  /**
   * 创建绑定到 AgentLoop 生命周期的会话级 Provider。
   *
   * 会话级 Provider 可以在内部缓存有状态资源（如 LangChain ChatOpenAI 实例），
   * 这些资源的生命周期跟随 AgentLoop，AgentLoop 结束时必须调用 dispose() 释放。
   *
   * 如果 Provider 本身无状态（如 Anthropic SDK），可不实现此方法或返回 this。
   */
  createSession?(): LLMProvider

  /**
   * 释放会话级资源（httpAgent、连接池等）。
   * AgentLoop 结束时（无论正常完成还是异常退出）必须调用。
   * 全局 Provider 的 dispose() 应为空操作。
   */
  dispose?(): void
}
```

### 9.5 各 Provider 实现

**Anthropic Provider — 无需 session（SDK 无状态）**：

```typescript

export class AnthropicProvider implements LLMProvider {
  // ... 现有代码不变

  createSession(): LLMProvider {
    return this  // Anthropic SDK 无状态，安全共享
  }

  dispose(): void {

  }

```

**OpenAI 兼容 Provider — 会话级缓存 ChatOpenAI**：

```typescript
// src/providers/openai-compat.ts — 改造
export class OpenAICompatProvider implements LLMProvider {
  readonly name: string
  readonly protocol: ProviderProtocol = 'openai-compat'
  readonly #config: ProviderConfig

  /** 会话级缓存：仅由 createSession() 创建的实例使用 */
  #chatModel: ChatOpenAI | null = null
  #chatModelName: string | null = null
  #disposed = false

  constructor(providerName: string, config: ProviderConfig) {
    this.name = providerName
    this.#config = config
  }

  /** 为每个 AgentLoop 创建独立的会话级 Provider */
  createSession(): OpenAICompatProvider {
    // 共享 config 引用（只读），但 ChatOpenAI 缓存独立
    return new OpenAICompatProvider(this.name, this.#config)
  }

  /** 释放 ChatOpenAI 实例及其内部 httpAgent、连接池 */
  dispose(): void {
    if (this.#disposed) return
    this.#disposed = true
    this.#chatModel = null
    this.#chatModelName = null
  }

  isModelSupported(model: string): boolean {
    return this.#config.models.includes(model)
  }

  async *chat(request: ChatRequest): AsyncIterable<StreamChunk> {
    if (this.#disposed) {
      throw new Error('SessionProvider already disposed')
    }

    // 同一个 session 内，model 通常不变 → 复用 ChatOpenAI
    // 如果 model 变了（理论上不会，但防御性处理），重建
    if (!this.#chatModel || this.#chatModelName !== request.model) {
      this.#chatModel = new ChatOpenAI({
        apiKey: this.#config.apiKey,
        model: request.model,
        ...(this.#config.baseURL !== undefined && {
          configuration: { baseURL: this.#config.baseURL, apiKey: this.#config.apiKey },
        }),
      })
      this.#chatModelName = request.model
    }

    // bindTools 仍需每次调用（SubAgent 工具列表可能与主 Agent 不同）
    const model = (request.tools && request.tools.length > 0)
      ? this.#chatModel.bindTools(request.tools.map(t => ({
          type: 'function' as const,
          function: { name: t.name, description: t.description, parameters: t.parameters },
        })))
      : this.#chatModel

    // ... 其余流式处理逻辑不变
  }
}
```

### 9.6 调用点改造

**核心原则：`createSession()` 与 `dispose()` 必须成对出现，用 `try/finally` 保证。**

**主 Agent**（`src/ui/useChat.ts` submit 回调内）：

```typescript
const provider = getOrCreateProvider(currentProvider, config)
const sessionProvider = provider.createSession?.() ?? provider

try {
  const loop = new AgentLoop(sessionProvider, registry, {
    model: currentModel,
    provider: currentProvider,
    // ...
  })

  for await (const event of loop.run(optimizedHistory)) {
    // ... 事件处理
  }
} finally {
  sessionProvider.dispose?.()  // 无论正常结束还是异常/中断，都释放
}
```

### 9.7 生命周期保证

| 事件 | Provider（全局） | SessionProvider（会话级） | ChatOpenAI |
|------|----------------|------------------------|------------|
| CLI 启动 | 首次 `getOrCreateProvider()` 创建 | — | — |
| 用户 submit | 从缓存获取 | `createSession()` 创建 | — |
| AgentLoop 首次 chat() | 不变 | 不变 | **延迟创建** |
| AgentLoop 后续 chat() | 不变 | 不变 | **复用** |
| dispatch_agent | 不变 | 子 Agent `createSession()` 创建独立实例 | 子 Agent 首次 `chat()` 时创建 |
| AgentLoop 结束（正常/异常/中断） | 不变（仍在缓存中） | **`finally` 中 `dispose()` 释放** | **置 null，断开引用** |
| 设置页切换 apiKey | `clearProviderCache()` 清空 | 当前运行的 session 不受影响 | 下次 submit 创建新 session |

**显式清理，不依赖 GC**：

- `createSession()` 和 `dispose()` 成对出现，通过 `try/finally` 保证
- `dispose()` 将 `#chatModel` 置 null + 设置 `#disposed = true`
- dispose 后再调用 `chat()` 会抛出明确错误（防止野指针式的静默 bug）
- 全局 Provider 的 `dispose()` 是空操作（它的生命周期跟随进程，不需要清理）

**防泄漏检查**（可选，debug 模式下启用）：

```typescript
// 在 OpenAICompatProvider 构造函数中，debug 模式下注册 FinalizationRegistry
// 如果 session 被 GC 但没被 dispose，打印警告
if (process.env.CCODE_DEBUG) {
  const registry = new FinalizationRegistry((name: string) => {
    console.warn(`[LEAK] SessionProvider "${name}" was GC'd without dispose()`)
  })
  registry.register(this, this.name)
}
```

### 9.8 为什么不按 model 缓存在全局 Provider 上

| 方案 | 并发安全 | 生命周期管理 | 内存 | 复杂度 |
|------|---------|------------|------|--------|
| ❌ 全局 Provider + model Map 缓存 | 不安全（共享 httpAgent） | 无清理时机 | 长期驻留 | 低 |
| ❌ 按 SubAgent ID 缓存到全局 Map | 安全 | 需手动 register/unregister，容易遗漏 | 泄漏风险高 | 高 |
| ✅ Session Provider + dispose（本方案） | 安全（独立实例） | try/finally 显式清理 + dispose 防护 | 确定性释放 | 中 |
