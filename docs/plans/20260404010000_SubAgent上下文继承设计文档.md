# SubAgent 上下文继承设计文档

> 日期：2026-04-04
> 类型：架构设计
> 状态：待评审

---

## 一、问题

### 现状

cCli 的 SubAgent 启动时只收到一条 user 消息（`dispatch_agent` 工具的 `prompt` 参数），**完全没有主 Agent 的对话历史**。

```typescript
// dispatch-agent.ts:229（前台）/ :367（后台）
subLoop.run([{ role: 'user', content: prompt }])
//           ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
//           只有一条消息
```

### 实际影响（已通过 JSONL 验证）

以会话 `019d542a` 为例——用户要求构建前后端项目，主 Agent 做了大量前期工作后派发 SubAgent：

- **主 Agent JSONL**：443 行事件，包含用户需求、目录创建、文件读写等丰富上下文
- **SubAgent JSONL**：第 1 行 `session_start`，第 2 行直接就是一条 `user` 消息（仅 prompt），然后就开始 `llm_call_start`

SubAgent **不知道**：
- 用户的完整需求描述
- 主 Agent 之前做了什么（创建了哪些文件、改了哪些配置）
- 其他并行 SubAgent 的任务分工（前端 SubAgent 不知道后端 API 契约）

### Claude Code 对比

Claude Code 有两种模式向 SubAgent 传递上下文：

**Fork 模式**（`forkSubagent.ts:107-169`）——子 Agent 继承父 Agent 的完整对话历史：
```
initialMessages = [
  ...父 Agent 完整历史（经 filterIncompleteToolCalls 清理）,
  父 Assistant 消息（含所有 tool_use blocks）,
  User 消息（统一占位符 tool_results + 子 Agent 指令）
]
```

**普通模式**（`runAgent.ts:370-373`）——子 Agent 继承筛选后的上下文：
```
contextMessages = filterIncompleteToolCalls(forkContextMessages)
initialMessages = [...contextMessages, ...promptMessages]
```

---

## 二、设计目标

1. **SubAgent 能看到主 Agent 的对话历史**——知道用户要什么、主 Agent 做了什么
2. **可控的上下文量**——不是无脑传全量历史，避免 token 爆炸
3. **Prompt Cache 友好**——多个并行 SubAgent 共享相同的历史前缀
4. **向后兼容**——不破坏现有 AgentLoop 和工具接口

---

## 三、方案设计

### 3.1 核心改动：ToolContext 新增 history 字段

当前 `ToolContext`（`src/tools/core/types.ts`）不包含 history。需要新增：

```typescript
export interface ToolContext {
  // ... 现有字段
  /** 主 Agent 的对话历史快照（只读，供 dispatch_agent 传给子 Agent） */
  history?: ReadonlyArray<Message>
}
```

用 `ReadonlyArray` 防止工具意外修改主 Agent 的 history。

### 3.2 AgentLoop 透传 history 到 ToolContext

当前 `buildToolContext()`（`agent-loop.ts:482-497`）不传 history。改为：

```typescript
// agent-loop.ts — #executeOneTool() 中
const ctx = buildToolContext(this.#provider, this.#registry, this.#config, history)

// buildToolContext() 新增 history 参数
function buildToolContext(
  provider: LLMProvider,
  registry: ToolRegistry,
  config: AgentConfig,
  history?: ReadonlyArray<Message>,
): ToolContext {
  const ctx: ToolContext = {
    cwd: process.cwd(),
    provider,
    providerName: config.provider,
    model: config.model,
    registry,
  }
  // ... 现有字段
  if (history !== undefined) { ctx.history = history }
  return ctx
}
```

同样，并行执行安全工具的 `executeSafeToolsInParallel()` 也需要接收 history 并传入 ToolContext。

### 3.3 dispatch-agent 消费 ctx.history 构建子 Agent 初始消息

这是核心改动。两种策略：

#### 策略 A：直接传全量历史（简单，但 token 开销大）

```typescript
// dispatch-agent.ts stream() 内
const contextMessages = ctx.history ? [...ctx.history] : []
const initialMessages: Message[] = [
  ...contextMessages,
  { role: 'user', content: prompt },
]

subLoop.run(initialMessages)
```

**优点**：实现最简单，子 Agent 拥有完整上下文
**缺点**：历史可能很长（几十轮对话），token 爆炸，尤其并行 5 个 SubAgent 时

#### 策略 B：裁剪历史 + 摘要（推荐）

```typescript
// dispatch-agent.ts stream() 内
const contextMessages = ctx.history
  ? trimHistoryForSubAgent(ctx.history, {
      maxMessages: 20,       // 最多保留最近 20 条消息
      maxTokenEstimate: 8000, // 估算 token 不超过 8k
      keepSystemPrompt: true, // 保留 system 消息
      keepLastNToolResults: 5, // 保留最近 5 个工具结果
    })
  : []

const initialMessages: Message[] = [
  ...contextMessages,
  { role: 'user', content: prompt },
]

subLoop.run(initialMessages)
```

**裁剪策略**：
1. 始终保留第一条 user 消息（用户原始需求）
2. 保留最近 N 条消息（保证最新上下文）
3. 中间消息按 token 预算裁剪，超过预算的旧消息替换为摘要
4. 工具结果中的超长输出截断（只保留 resultSummary，不保留 resultFull）

#### 策略 C：Fork 模式（Claude Code 风格，Cache 友好但复杂）

```typescript
// 只在同一轮 LLM 调用中有多个 dispatch_agent 时使用
// 所有 fork 共享相同的历史前缀 + 统一的 tool_result 占位符
const forkedMessages = buildForkedMessages(
  prompt,
  history,         // 完整历史
  assistantMessage, // 当前 assistant 消息（含所有 tool_use blocks）
)

subLoop.run(forkedMessages)
```

**优点**：Prompt Cache 命中率最高
**缺点**：实现复杂，需要构造 tool_result 占位符、处理 assistant 消息结构

### 3.4 推荐方案：策略 B（裁剪历史）为默认，策略 A 为 fallback

理由：
- 策略 B 在 token 开销和上下文完整性之间取得平衡
- 策略 C 的 Prompt Cache 优化在 P1 #9 中已通过 systemPrompt 前缀继承部分实现
- 策略 A 作为 `subagent_type` 级别的配置选项（某些 agent type 可能需要全量历史）

### 3.5 消息清理：filterIncompleteToolCalls

从 Claude Code 学到的关键教训——**不能直接传原始 history**，需要清理：

1. **不完整的 tool_call**：assistant 消息中有 tool_call 但对应的 tool_result 还没执行完（比如当前正在执行 dispatch_agent 的那一轮），API 会报错
2. **清理策略**：截断到最后一个"完整轮次"（assistant 的所有 tool_call 都有对应 tool_result）

```typescript
/**
 * 过滤不完整的工具调用轮次。
 * 如果最后一条 assistant 消息包含 tool_call，但对应的 tool_result 不全，
 * 则移除该 assistant 消息和后续所有消息。
 */
function filterIncompleteToolCalls(history: ReadonlyArray<Message>): Message[] {
  const result = [...history]

  // 从后往前找最后一条 assistant 消息
  for (let i = result.length - 1; i >= 0; i--) {
    const msg = result[i]
    if (msg.role !== 'assistant') continue

    // 提取 assistant 消息中的 tool_call
    const toolCalls = extractToolCalls(msg)
    if (toolCalls.length === 0) break  // 纯文本回复，无需清理

    // 检查后续 user 消息中是否有对应的 tool_result
    const toolResultIds = new Set<string>()
    for (let j = i + 1; j < result.length; j++) {
      const r = result[j]
      if (r.role === 'user' && Array.isArray(r.content)) {
        for (const block of r.content) {
          if (block.type === 'tool_result') toolResultIds.add(block.toolCallId)
        }
      }
    }

    const allComplete = toolCalls.every(tc => toolResultIds.has(tc.toolCallId))
    if (!allComplete) {
      // 不完整，截断到这条 assistant 之前
      result.length = i
    }
    break
  }

  return result
}
```

### 3.6 trimHistoryForSubAgent 裁剪函数

```typescript
interface TrimOptions {
  /** 最多保留的消息条数（默认 20） */
  maxMessages?: number
  /** 估算 token 上限（默认 8000） */
  maxTokenEstimate?: number
  /** 是否保留第一条 user 消息（用户原始需求） */
  keepFirstUser?: boolean
}

function trimHistoryForSubAgent(
  history: ReadonlyArray<Message>,
  options: TrimOptions = {},
): Message[] {
  const { maxMessages = 20, maxTokenEstimate = 8000, keepFirstUser = true } = options

  // 1. 先 filterIncompleteToolCalls
  const clean = filterIncompleteToolCalls(history)
  if (clean.length === 0) return []

  // 2. 保留第一条 user 消息
  const first = keepFirstUser ? clean.find(m => m.role === 'user') : undefined

  // 3. 取最近 maxMessages 条
  let recent = clean.slice(-maxMessages)

  // 4. 如果 first 不在 recent 中，插到开头
  if (first && !recent.includes(first)) {
    recent = [first, ...recent]
  }

  // 5. 按 token 预算裁剪（从旧到新移除）
  let estimatedTokens = estimateTokens(recent)
  while (estimatedTokens > maxTokenEstimate && recent.length > 2) {
    // 保留第一条和最后一条，移除中间最旧的
    recent.splice(1, 1)
    estimatedTokens = estimateTokens(recent)
  }

  // 6. 工具结果截断（只保留 summary，去掉超长 output）
  return recent.map(msg => truncateToolResults(msg))
}

function estimateTokens(messages: Message[]): number {
  let total = 0
  for (const m of messages) {
    const content = typeof m.content === 'string'
      ? m.content
      : Array.isArray(m.content)
        ? m.content.map(c => 'text' in c ? c.text : JSON.stringify(c)).join('')
        : ''
    total += Math.ceil(content.length / 4)
  }
  return total
}

function truncateToolResults(msg: Message): Message {
  if (msg.role !== 'user' || !Array.isArray(msg.content)) return msg
  return {
    ...msg,
    content: msg.content.map(block => {
      if (block.type === 'tool_result' && typeof block.result === 'string' && block.result.length > 500) {
        return { ...block, result: block.result.slice(0, 500) + '\n... (truncated)' }
      }
      return block
    }),
  }
}
```

### 3.7 Agent 类型级别的上下文策略配置

不同类型的 SubAgent 需要不同的上下文量：

```typescript
// src/tools/agent/types.ts — AgentDefinition 扩展
export interface AgentDefinition {
  // ... 现有字段
  /** 上下文继承策略 */
  contextPolicy?: {
    /** 'full' = 全量历史，'trimmed' = 裁剪（默认），'none' = 不继承 */
    mode: 'full' | 'trimmed' | 'none'
    /** trimmed 模式的参数 */
    maxMessages?: number
    maxTokenEstimate?: number
  }
}
```

```typescript
// src/tools/agent/built-in.ts — 内置 Agent 配置
const EXPLORE_AGENT: AgentDefinition = {
  agentType: 'explore',
  // explore 是只读搜索，不需要太多历史，给最近 10 条足矣
  contextPolicy: { mode: 'trimmed', maxMessages: 10, maxTokenEstimate: 4000 },
  // ...
}

const GENERAL_AGENT: AgentDefinition = {
  agentType: 'general',
  // general 需要较完整的上下文（要知道之前做了什么）
  contextPolicy: { mode: 'trimmed', maxMessages: 20, maxTokenEstimate: 8000 },
  // ...
}

const PLAN_AGENT: AgentDefinition = {
  agentType: 'plan',
  // plan 需要知道全貌
  contextPolicy: { mode: 'trimmed', maxMessages: 30, maxTokenEstimate: 12000 },
  // ...
}
```

## 六、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Token 爆炸 | 历史太长导致 API 费用飙升 | trimHistoryForSubAgent 有 maxTokenEstimate 硬限制 |
| 不完整 tool_call 导致 API 报错 | assistant 消息有 tool_call 但无对应 tool_result | filterIncompleteToolCalls 截断到最后完整轮次 |
| 并行 SubAgent 重复发送大量历史 | 5 个 SubAgent × 8k tokens 历史 = 40k tokens | systemPrompt 前缀继承（P1 #9 已实现）利于缓存命中 |
| history 传递引入竞态 | SubAgent 读到的 history 在主 Agent 继续执行时被修改 | ToolContext.history 是 ReadonlyArray，且 trimHistoryForSubAgent 返回新数组 |
| 工具结果包含敏感信息 | 子 Agent 看到不该看到的内容 | 当前所有 SubAgent 都是 isSidechain=true（跳过权限检查），安全模型与主 Agent 一致 |

## 七、测试策略

1. **单元测试** — `filterIncompleteToolCalls()` 和 `trimHistoryForSubAgent()`：
   - 空历史 → 返回空
   - 全量历史在 token 预算内 → 原样返回
   - 超预算 → 裁剪中间消息，保留首尾
   - 不完整 tool_call → 正确截断
   - 超长工具结果 → 截断到 500 字符

2. **集成测试** — dispatch_agent 调用：
   - 子 Agent 的 LLM 请求中包含历史消息
   - 不同 contextPolicy 配置下历史条数正确

3. **手动验证** — 对比改造前后的 SubAgent JSONL：
   - 改造前：第 2 行只有 1 条 user 消息
   - 改造后：第 2 行有历史消息 + 新 prompt
