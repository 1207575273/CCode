## 一、问题

### 1.1 表面问题：SubAgent 没有上下文

cCli 的 SubAgent 启动时只收到一条 user 消息：

```typescript
// dispatch-agent.ts:229
subLoop.run([{ role: 'user', content: prompt }])
```

通过 JSONL 验证：SubAgent 的第 2 行直接就是孤立的 user 消息，然后立即 `llm_call_start`。

### 1.2 深层问题：history 中没有 assistant 消息

审查 `agent-loop.ts` 的 `run()` 方法发现，**LLM 的回复（text + tool_calls）从未被追加到 history**：

```typescript
// agent-loop.ts:163-172 — 主循环
for (let turn = 0; turn < maxTurns; turn++) {
  const llmResult = yield* this.#callLLM(history)  // LLM 回复通过 yield 流出
  // ❌ llmResult 中的 text 和 toolCalls 没有 push 到 history
  yield* this.#executeToolCalls(llmResult.toolCalls, history)
  // ↑ 工具结果以 role:'user' 字符串追加到 history
}
```

当前 history 的实际结构：

```
[user] "用户的原始消息"
[user] "[Tool bash result]: ls output..."        ← 工具结果，字符串格式
[user] "[Tool read_file result]: file content..." ← 工具结果，字符串格式
[user] "[Tool bash error]: command not found"     ← 工具错误
...全部是 role:'user'
```

**没有 assistant 消息，没有 tool_call 结构化内容**。这和 Anthropic API 期望的 `user → assistant(tool_use) → user(tool_result)` 严格交替格式完全不同。

### 1.3 为什么现在能工作

因为 `toAnthropicMessages()`（`anthropic.ts:19-73`）将 history 按原样发给 API——连续的 `role:'user'` 消息被 API 合并处理。LLM 每轮收到的是「完整历史 + 上一轮所有工具结果」，虽然格式不标准但功能正确。

但这意味着：**直接把主 Agent 的 history 传给子 Agent 是安全的**——格式已经是扁平的 user 消息序列，不存在不完整的 tool_call 配对问题。

### 1.4 Claude Code 对比

Claude Code 的 history 是严格的 Anthropic API 格式：
```
[user] 用户消息
[assistant] { text + tool_use[] }      ← 结构化
[user] { tool_result[] }               ← 结构化
[assistant] { text + tool_use[] }
...
```

所以 Claude Code 需要 `filterIncompleteToolCalls()` 清理不完整的工具调用。**cCli 不需要这个函数**——因为 history 中根本没有 assistant 消息和结构化 tool_call。

---

## 二、设计目标

1. **SubAgent 能看到主 Agent 的对话历史**——知道用户要什么、做了什么
2. **可控的上下文量**——有 token 预算限制，避免爆炸
3. **不同 agent type 不同策略**——explore 少、general 多、plan 最多
4. **向后兼容**——不改变 history 的现有格式（全 user 消息）

---

## 三、方案设计

### 3.1 核心思路

由于 cCli 的 history 是扁平的 `role:'user'` 字符串序列，传递给 SubAgent 非常简单——不需要 Claude Code 那样复杂的 fork/filter 机制。

核心改动：
1. `ToolContext` 新增 `history` 字段
2. `buildToolContext()` 透传 history
3. `dispatch-agent` 从 `ctx.history` 取历史，裁剪后拼接到 SubAgent 的初始消息

### 3.2 ToolContext 扩展

```typescript
// src/tools/core/types.ts
```

- `#executeToolCalls()` 的并行执行路径中：`buildToolContext(..., history)` — 透传给 `executeSafeToolsInParallel`

实际上 `parallel-executor` **不需要改**——它已经接收 `ctx: ToolContext` 并传给 `tool.execute(args, ctx)`。只要 `ctx` 中有 `history`，工具就能访问。

但需要确认：当前 `#executeToolCalls()` 的并行路径中 `buildToolContext` 没有传 history：

```typescript

// ❌ 缺少 history 参数
```

改为：
```typescript

```

### 3.6 trimHistoryForSubAgent 裁剪函数

由于 cCli 的 history 全是 `role:'user'` 字符串消息，裁剪逻辑比 Claude Code 简单得多——**不需要处理 assistant 消息、tool_call 配对、role 交替**。

```typescript
// 新增 src/tools/agent/context-utils.ts

import type { Message } from '@core/types.js'

export interface TrimOptions {
  /** 最多保留的消息条数（默认 20） */
  maxMessages?: number
  /** 估算 token 上限（默认 8000） */
  maxTokenEstimate?: number
}

/**
 * 裁剪主 Agent 历史，为 SubAgent 构建精简上下文。
 *
 * 策略：
 *   1. 始终保留第一条 user 消息（用户原始需求）
 *   2. 保留最近 maxMessages 条消息（最新上下文）
 *   3. 超过 token 预算时从旧端移除
 *   4. 工具结果中的超长输出截断
 *
 * 由于 cCli 的 history 全是 role:'user' 字符串消息（无 assistant 消息、
 * 无结构化 tool_call 配对），不需要 Claude Code 那样的 filterIncompleteToolCalls。
 */
export function trimHistoryForSubAgent(
  history: ReadonlyArray<Message>,
  options: TrimOptions = {},
): Message[] {
  const { maxMessages = 20, maxTokenEstimate = 8000 } = options

  if (history.length === 0) return []

  // 第一条 user 消息（用户原始需求）
  const firstUser = history.find(m => m.role === 'user')

  // 取最近 maxMessages 条
  let recent = history.slice(-maxMessages).map(m => truncateLongContent(m))

  // 如果 firstUser 不在 recent 中，插到开头
  if (firstUser && !recent.includes(firstUser)) {
    recent = [truncateLongContent(firstUser), ...recent]
  }

  // 按 token 预算裁剪（从旧端移除，保留首尾）
  while (estimateTokens(recent) > maxTokenEstimate && recent.length > 2) {
    recent.splice(1, 1)
  }

  return recent
}

/** 截断单条消息中的超长内容（工具结果经常很长） */
function truncateLongContent(msg: Message): Message {
  if (typeof msg.content !== 'string') return msg
  if (msg.content.length <= 1000) return msg
  return { ...msg, content: msg.content.slice(0, 1000) + '\n... (truncated)' }
}

/** 粗略估算 token 数（字符数 / 4） */
function estimateTokens(messages: ReadonlyArray<Message>): number {
  let total = 0
  for (const m of messages) {
    const text = typeof m.content === 'string'
      ? m.content
      : JSON.stringify(m.content)
    total += Math.ceil(text.length / 4)
  }
  return total
}
```

## 四、不做什么（YAGNI）

| 不做的事 | 理由 |
|---------|------|
| filterIncompleteToolCalls | cCli 的 history 中没有 assistant 消息和结构化 tool_call，不存在不完整配对问题 |
| Fork 模式 / buildForkedMessages | 需要改造 history 格式为严格 user/assistant 交替，改动量巨大，ROI 不值 |
| Prompt Cache 占位符 | 依赖 Fork 模式，同上 |
| history 格式改造（补 assistant 消息） | 这是独立的架构改进，不在本次范围内。当前格式虽不标准但能工作 |
| 消息摘要 / 压缩 | 裁剪 + 截断已经足够控制 token，摘要引入额外 LLM 调用得不偿失 |

## 六、数据流图

```
主 Agent 的 history（全是 role:'user' 字符串消息）：
  [user] "用户原始需求"
  [user] "[Tool bash result]: mkdir backend..."
  [user] "[Tool write_file result]: created pyproject.toml"
  [user] "[Tool bash result]: uv sync..."
  [user] "[Tool read_file result]: ..."
  ...（可能几十条）

dispatch_agent 被调用时：
  ctx.history = 上面的完整 history（ReadonlyArray）
      │
      ▼
  trimHistoryForSubAgent(ctx.history, contextPolicy)
      │
      ├─ 保留第一条：[user] "用户原始需求"
      ├─ 保留最近 N 条：最新的工具结果
      ├─ 超长截断：每条 ≤ 1000 字符
      └─ token 预算：总计 ≤ 8000
      │
      ▼
  initialMessages = [
    [user] "用户原始需求"               ← 第一条
    [user] "[Tool bash result]: ..."    ← 最近的上下文
    [user] "[Tool write_file result]..."← 最近的上下文
    [user] "你需要完成前端项目..."        ← SubAgent 的 prompt
  ]
      │
      ▼
  subLoop.run(initialMessages)
      │
      └─ SubAgent 现在知道：
           ✅ 用户要做什么（原始需求）
           ✅ 主 Agent 做了什么（工具结果）
           ✅ 自己要做什么（prompt）
```

### 单元测试（context-utils.test.ts）

| 用例 | 输入 | 预期 |
|------|------|------|
| 空历史 | `[]` | 返回 `[]` |
| 短历史在预算内 | 5 条消息 | 原样返回（截断超长内容） |

## 八、风险与缓解

| 风险 | 缓解 |
|------|------|
| Token 开销增加 | maxTokenEstimate 硬限制 + truncateLongContent 截断 |
| 历史消息干扰子 Agent 判断 | prompt 放在最后一条，LLM 倾向于关注最新消息；contextPolicy.mode='none' 可关闭 |
| 并行 SubAgent 重复传历史 | systemPrompt 前缀已共享（P1 #9），历史消息是 input token（Anthropic 有缓存） |
| ReadonlyArray 不防深拷贝修改 | trimHistoryForSubAgent 返回新数组 + 新 Message 对象（truncateLongContent 用 spread 创建新对象）|
