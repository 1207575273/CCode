# SubAgent 提前终止与结果回传修复设计

> 日期：2026-04-04
> 类型：架构设计（P0）
> 状态：待评审

---

## 一、问题

### 1.1 SubAgent 只执行 1 步就停了

**实际 JSONL 数据**（会话 `019d5470`）：

用户要求创建前后端项目，主 Agent 派发 2 个后台 SubAgent。两个 SubAgent 各自只执行了 **1 次 bash 命令**，然后 LLM 返回纯文本"我来开始创建前端项目。"—— `toolCalls.length === 0`，AgentLoop 判定完成并退出。

```
SubAgent 前端（10 行事件）：
  line 1: user（852 字符 prompt）
  line 4: bash → ok（仅 1 次工具调用）
  line 8: assistant "我来开始创建前端项目。"  ← LLM 认为自己说完了
  line 9: session_end                        ← AgentLoop 结束

SubAgent 后端（10 行事件）：同样只做了 1 步
```

**根因**：`agent-loop.ts:167-170` 的 done 条件：

```typescript
if (llmResult.toolCalls.length === 0) {
  yield { type: 'done', reason: 'complete' }
  return  // ← 立即退出，不管任务是否真正完成
}
```

LLM（GLM-5.1）在第一轮工具调用后返回了纯文本而非继续调用工具。这在能力较弱的模型上尤其常见——模型"说"了要做什么，但没有"做"。

### 1.2 后台 SubAgent 结果回传不完整

主 Agent 通过 `task_output` 查看后台 SubAgent 状态时，只能看到：
- `finalText`：LLM 最后的文本回复（如"我来开始创建前端项目。"）
- 工具调用数量汇总

**看不到**：
- SubAgent 具体执行了哪些工具、结果是什么
- 创建/修改了哪些文件
- 是否真正完成了任务

## 二、方案设计

### 2.1 问题一：防止 SubAgent 提前终止

**核心思路**：不能仅靠 `toolCalls.length === 0` 判断任务完成。需要结合 **systemPrompt 强化** + **续跑机制** 双管齐下。

#### 2.1.1 方案 A：systemPrompt 强化（最简单、最有效）

当前 general agent 的 systemPrompt：
```
You are a sub-agent. Complete the assigned task autonomously.
Constraints:
- Focus solely on the given task
- Do NOT dispatch further sub-agents
- Do NOT ask questions — you have no user interaction
- When done, output your final result as plain text
```

**问题**：最后一句"When done, output your final result as plain text"被弱模型理解为"说一句话就算 done"。

**改造后**：
```
You are a sub-agent. Complete the assigned task autonomously.

CRITICAL RULES:
- Execute ALL steps of the task using tools. Do NOT just describe what you will do.
- Keep calling tools until the task is FULLY COMPLETE.
- Do NOT output text without calling tools first — text alone is NOT completion.
- Do NOT say "I will do X" — actually DO X by calling the appropriate tool.
- Only output your final summary AFTER all tool calls are done and verified.
- If a tool call fails, diagnose and retry with a different approach.
- Do NOT dispatch further sub-agents.
- Do NOT ask questions — you have no user interaction.
```

关键改动：
- 明确禁止"说了不做"（Do NOT output text without calling tools first）
- 强调"持续调用工具直到完成"（Keep calling tools until FULLY COMPLETE）
- 禁止"描述性回复"作为完成标志

explore 和 plan agent 也做类似强化。

#### 2.1.2 方案 B：续跑机制（兜底防御）

即使 systemPrompt 强化后，弱模型仍可能提前停。加一个**续跑检测**：如果 SubAgent 执行轮次太少（< minTurns）且 LLM 停了，自动注入一条"继续"消息让 LLM 继续执行。

```typescript
// agent-loop.ts — run() 主循环修改
async *run(messages: Message[]): AsyncIterable<AgentEvent> {
  const history: Message[] = [...messages]
  const maxTurns = this.#config.maxTurns ?? DEFAULT_MAX_TURNS
  const minTurns = this.#config.minTurns ?? 0

  for (let turn = 0; turn < maxTurns; turn++) {
    const llmResult = yield* this.#callLLM(history)
    if (llmResult.aborted) return

    // 追加 assistant 消息到 history
    // ... 现有逻辑 ...

    if (llmResult.toolCalls.length === 0) {
      // 续跑检测：轮次不足 minTurns 时，注入继续消息
      if (turn < minTurns && this.#config.isSidechain) {
        history.push({
          role: 'user',
          content: 'You have not completed the task yet. Continue executing tools to finish. Do NOT just describe — actually call tools.',
        })
        continue  // 强制进入下一轮
      }
      yield { type: 'done', reason: 'complete' }
      return
    }

    yield* this.#executeToolCalls(llmResult.toolCalls, history)
  }
}
```

**minTurns 配置**：
```typescript
// AgentConfig 新增
minTurns?: number  // 最少执行轮次（仅 isSidechain 模式生效）

// 内置 agent 配置
general:  minTurns: 3   // 至少 3 轮（复杂任务通常需要多轮）
explore:  minTurns: 2   // 至少 2 轮（搜索 + 分析）
plan:     minTurns: 2   // 至少 2 轮（读代码 + 输出方案）
```

**续跑触发条件**：
- `toolCalls.length === 0`（LLM 想停）
- `turn < minTurns`（还没到最少轮次）
- `isSidechain === true`（仅 SubAgent 触发，主 Agent 不受影响）

**为什么不影响主 Agent**：主 Agent 的 `isSidechain` 为 false/undefined，续跑条件不满足，行为不变。

#### 2.1.3 推荐：A + B 结合

- 方案 A（systemPrompt 强化）解决 80% 的场景——大部分模型能理解"持续执行"的指令
- 方案 B（续跑机制）作为兜底——弱模型仍然提前停时，自动注入继续消息

两者互不依赖，可以分步实施。

### 2.2 问题二：后台 SubAgent 结果回传增强

**核心思路**：SubAgent 完成时，构建一个**结构化的执行摘要**，包含工具调用详情，而不只是 LLM 的最后一句话。

#### 2.2.1 增强 SubAgentState 数据模型

当前 `markSubAgentDone()` 只存 `finalText`（LLM 文本回复）。改为同时存储工具执行摘要：

```typescript
// src/tools/agent/store.ts — markSubAgentDone 增强
export function markSubAgentDone(
  agentId: string,
  finalText: string,
  status: 'done' | 'error',
  toolSummary?: ToolExecutionSummary,  // 新增
): void {
  const state = agents.get(agentId)
  if (!state) return
  state.status = status
  state.finalText = finalText
  if (toolSummary) state.toolSummary = toolSummary
}

export interface ToolExecutionSummary {
  totalCalls: number
  successCount: number
  failCount: number
  details: Array<{
    toolName: string
    success: boolean
    /** 工具结果摘要（截断到 200 字符） */
    resultSummary: string
  }>
}
```

#### 2.2.2 dispatch-agent 在 SubAgent 完成时收集工具摘要

```typescript
// dispatch-agent.ts — 前台和后台模式都收集

// 收集工具调用详情
const toolDetails: Array<{ toolName: string; success: boolean; resultSummary: string }> = []

for await (const event of subLoop.run(initialMessages)) {
  // ... 现有事件处理 ...
  if (event.type === 'tool_done') {
    toolDetails.push({
      toolName: event.toolName,
      success: event.success,
      resultSummary: event.resultSummary?.slice(0, 200) ?? '',
    })
  }
}

const toolSummary: ToolExecutionSummary = {
  totalCalls: toolDetails.length,
  successCount: toolDetails.filter(d => d.success).length,
  failCount: toolDetails.filter(d => !d.success).length,
  details: toolDetails,
}

markSubAgentDone(agentId, finalText, 'done', toolSummary)
```

#### 2.2.3 task_output 返回增强的结构化信息

```typescript
// task-output.ts — 返回工具执行详情
if (state.toolSummary) {
  const ts = state.toolSummary
  output += `\n\n工具执行汇总: ${ts.successCount}/${ts.totalCalls} 成功`
  for (const d of ts.details) {
    output += `\n  ${d.success ? '✓' : '✗'} ${d.toolName}: ${d.resultSummary}`
  }
}
```

#### 2.2.4 前台模式的返回结果也增强

```typescript
// dispatch-agent.ts — 前台完成时
const output: AgentCompletedOutput = {
  status: 'completed',
  agentId,
  name: agentName,
  agentType,
  model: modelName,
  prompt,
  result: finalText,
  toolSummary,  // 新增
}
```

## 五、风险

| 风险 | 缓解 |
|------|------|
| 续跑注入的"继续"消息可能导致 LLM 无限循环 | minTurns 有上限（general=3，explore=2），超过后正常结束；maxTurns 仍是硬限制 |
| systemPrompt 强化后 token 略增 | 增量 ~100 token / agent，可忽略 |
| 工具摘要收集增加内存占用 | 每个 tool_done 只存 200 字符摘要，几十个工具调用也只是 KB 级 |
| 续跑机制影响主 Agent | 仅在 `isSidechain === true` 时触发，主 Agent 不受影响 |

## 六、实测验证发现的新问题（会话 019d5483）

### 6.1 实测数据

对改造后的代码进行实测（同一 prompt："创建 uv+python+fastapi 后端 + vue 前端"），JSONL 分析结果：

**主 Agent（58 行事件）**：
```
line 14: todo_write
line 18: assistant "好的，我用 subagent 并行创建！" ← 说了不做
line 19: 用户 "？？？？"
line 22-25: 派发 2 个 SubAgent
line 33: 用户 "？？？？？？？"
line 36-39: 又派发 2 个 SubAgent（重复！同样的任务）
line 57: session_end
```

**4 个 SubAgent（各 18-22 行事件，3-5 次工具调用后结束）**：
- 续跑机制（minTurns=3）生效了：从之前的 1 次工具调用提升到 3-5 次
- 但仍然不够：只做了脚手架（mkdir + uv init / pnpm create），没有写业务代码
- 模型在 minTurns 轮次用完后仍然选择输出文本而非继续调用工具

### 6.2 问题三：AbortSignal MaxListeners 内存泄漏

```
(node:23824) MaxListenersExceededWarning: Possible EventTarget memory leak detected.
11 abort listeners added to [AbortSignal]. MaxListeners is 10.
```

**根因**：所有 SubAgent 共享主 Agent 的 `ctx.signal`（同一个 AbortSignal 实例）。每个 SubAgent 的 AgentLoop 在 `provider.chat()` 调用时给这个 signal 添加 abort listener。4 个 SubAgent × 多轮 LLM 调用 = 11+ listener → 超过 Node.js 默认 MaxListeners=10。

**当前代码**：
```typescript
// dispatch-agent.ts — 直接传父级 signal
const subLoop = new AgentLoop(sessionProvider, subRegistry, {
  signal: ctx.signal,  // ← 所有 SubAgent 共享同一个 AbortSignal
  // ...
})
```

**修复**：每个 SubAgent 创建独立的 AbortController，链接到父级 signal：
```typescript
const subController = new AbortController()
const onParentAbort = () => subController.abort()
ctx.signal?.addEventListener('abort', onParentAbort, { once: true })

const subLoop = new AgentLoop(sessionProvider, subRegistry, {
  signal: subController.signal,  // 独立 signal
})

// 完成后清理
ctx.signal?.removeEventListener('abort', onParentAbort)
```

### 6.3 问题四：主 Agent 重复派发 SubAgent

主 Agent 在 line 22-25 派发了 2 个后台 SubAgent，但 SubAgent 很快完成（3-5 步）后，用户不满意催促，主 Agent 又在 line 36-39 派发了同样的 2 个任务。导致：
- 4 个 SubAgent 做 2 个任务（浪费）
- 前后两次的 SubAgent 可能互相冲突（写同一目录）

**这是模型决策问题，非代码 bug**。但可以通过以下方式缓解：
- SubAgent 完成后给主 Agent 更详细的结果摘要（已在第三步实现）
- 主 Agent 在下一轮 LLM 调用时能看到 SubAgent 的完成结果，避免重复派发

### 6.4 问题五：minTurns=3 不够，需要提高

实测显示 SubAgent 在 3-5 轮后仍然选择停下。对于"创建整个项目"这类复杂任务，3 轮远远不够。

**改造**：general agent 的 minTurns 从 3 提升到 5。
