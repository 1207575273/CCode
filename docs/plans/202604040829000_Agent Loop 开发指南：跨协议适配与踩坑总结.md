Agent Loop 开发指南：跨协议适配与踩坑总结本文档旨在总结大模型 Agent Loop（智能体循环）的底层机制，特别是 OpenAI 与 Anthropic 协议的核心差异，以及如何利用 LangChain Core 优雅地实现手搓 Agent 循环，避免在工程优化中出现的“智商下降”或“突然停止”问题。1. OpenAI vs Anthropic：底层协议核心差异虽然当前大模型都支持 Tool Calling (Function Calling)，但两大巨头在底层协议实现上存在显著分歧。不使用适配层直接手写 JSON 容易在多模型切换时踩坑。1.1 角色定义与交替规则OpenAI:System Prompt 作为一条独立消息混合在 messages 数组中（{"role": "system"}）。消息顺序较宽容，允许连续发送同角色的消息。Anthropic (Claude):System Prompt 是 API 的顶层参数，绝对不能放在 messages 数组内。messages 数组必须严格遵循 user -> assistant -> user 的交替顺序。违反即报错。1.2 工具调用（Tool Calling）闭环逻辑OpenAI:模型发起请求：返回 assistant 角色消息，包含 tool_calls 列表。客户端反馈：执行完成后，必须追加一条角色为 tool 的新消息，且必须携带一一对应的 tool_call_id。Anthropic:模型发起请求：返回 assistant 角色消息，内容是一个包含 tool_use 块的数组。客户端反馈：执行完成后，必须将结果包装在 user 角色消息中返回（为了满足交替规则），内容块类型为 tool_result，并关联对应的 ID。2. LangChain 的定位：纯粹的“协议翻译官”在手搓 Agent 循环时，不推荐使用 LangChain 臃肿的 AgentExecutor，但强烈推荐引入其底层 langchain-core 模块。LangChain 在此架构中的唯一职责是抹平底层 JSON 协议的差异：统一消息结构：开发者只操作 SystemMessage, HumanMessage, AIMessage, ToolMessage。LangChain 负责在发包时将其编译为目标 API 接受的合法格式。统一工具绑定：通过 @tool 装饰器定义 Python 函数，bind_tools() 自动生成 OpenAI 或 Anthropic 对应的 JSON Schema。3. 手搓 Agent Loop 最佳实践架构以下是规避了底层协议坑点的最小化可用（MVP）Agent 循环模板：from langchain_core.messages import SystemMessage, HumanMessage, ToolMessage
from langchain_core.tools import tool
# 根据需要导入具体的模型包装器
from langchain_openai import ChatOpenAI
# from langchain_anthropic import ChatAnthropic

# 1. 定义工具 (LangChain 自动处理 Schema)
@tool
def read_file(filepath: str) -> str:
"""读取本地文件内容"""
try:
with open(filepath, 'r') as f:
return f.read()
except Exception as e:
# 重要：错误信息必须返回给模型，触发其自我纠错
return f"Error reading file: {str(e)}"

# 2. 初始化与绑定
tools = [read_file]
llm = ChatOpenAI(model="glm-4") # 可无缝切换其它兼容模型
llm_with_tools = llm.bind_tools(tools)

messages = [
SystemMessage(content="你是代码助手。"),
HumanMessage(content="读取 main.py")
]

# 3. 控制循环 (开发者完全把控)
max_iterations = 5
for _ in range(max_iterations):
# 发送请求，LangChain 自动处理格式适配
response_msg = llm_with_tools.invoke(messages)

    # 必须将模型的响应（即使只有 tool_calls 没有文本）加入上下文
    messages.append(response_msg)
    
    # 无工具调用，任务结束
    if not response_msg.tool_calls:
        print("Final Answer:", response_msg.content)
        break 
        
    # 处理工具调用
    for tool_call in response_msg.tool_calls:
        tool_name, tool_args, tool_id = tool_call["name"], tool_call["args"], tool_call["id"]
        
        # 路由执行工具逻辑（可在此加入人类确认逻辑）
        tool_map = {t.name: t for t in tools}
        selected_tool = tool_map.get(tool_name)
        tool_result = selected_tool.invoke(tool_args) if selected_tool else f"Error: No tool named {tool_name}"
            
        # 核心：使用 ToolMessage 封装结果并提供对应的 tool_call_id
        messages.append(ToolMessage(content=str(tool_result), tool_call_id=tool_id))
4. 工程优化排雷指南：为什么 Sub-agent 会“变傻”或“停机”？在进行上下文裁剪、消息合并等工程优化时，极易破坏大模型的推理闭环。排查以下四个“高危雷区”：雷区一：破坏工具调用的上下文闭环 (最致命)现象：模型报错、死循环调用同一工具、或突然停止生成。规则：绝对不要拆散 AIMessage（包含 tool_calls）及其对应的 ToolMessage。建议：如果要清理历史消息节省 Token，必须将“请求工具 -> 执行工具 -> 返回结果”的完整成对链路一起删除，或者将其总结为一条纯文本的 HumanMessage 进行替换。雷区二：吞噬错误堆栈（Error Handling 断裂）现象：工具执行失败后，Agent 失去响应或不知道下一步该做什么。规则：如果代码崩溃或工具执行异常，绝对不能直接中断 CLI 或返回空字符串。建议：必须将 Exception 的字符串（Traceback）作为 ToolMessage 的 content 传回给模型。大模型具备根据错误日志自我修正参数重试的能力。雷区三：System Prompt 权重被稀释现象：随着对话加深或上下文被粗暴截断，Sub-agent 忘记自己的核心任务指令。建议：无论如何裁剪历史对话，必须确保高权重的 SystemMessage 始终保持在 messages 列表的最顶端（对于 OpenAI 协议）。雷区四：Stop Reason 误判现象：模型请求了工具，但本地循环提前退出了。建议：循环退出的唯一条件应是“模型返回的消息中没有 tool_calls”（如上述代码示例），而不是简单地判断是否生成了文本内容。
