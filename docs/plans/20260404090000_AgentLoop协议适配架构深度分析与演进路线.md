## 一、全局视角：Agent 的三大支柱

一个 Agent 的质量取决于三件事：**感知、决策、执行**。

- **感知**：LLM 看到了什么（上下文中的信息质量）
- **决策**：LLM 基于感知做出什么判断（模型能力 + 提示词）
- **执行**：工具系统是否可靠地完成指令（工具层）

我们的问题集中在**感知层**——不是工具执行不了，也不是模型太笨，而是 **LLM 看到的上下文信息不完整、不准确、甚至丢失**。

本文聚焦上下文工程（Context Engineering）：如何确保 LLM 在每一轮调用时都有足够的、正确的信息来做决策。

| 考量 | 判断 |

## 六、P0 架构缺陷：History 所有权错位

### 6.1 现象

用户说"继续"后，主 Agent 重复之前内容。JSONL 显示 inputTokens 振荡：9K → 24K → 9K → 18K。

### 6.2 根因

`useChat.ts:238-241` — 每次 submit 从 UI 的 `ChatMessage[]` 重建 LLM history：

```typescript
// 只保留 user 和 assistant，过滤掉所有 role='tool'
const history = llmMessages
  .filter(m => m.role === 'user' || m.role === 'assistant')
  .map(m => ({ role: m.role, content: m.content }))  // content 是纯文本
```

**两层丢失**：
1. `role: 'tool'` 被过滤 → 工具执行过程和结果全部丢失
2. `assistant.content` 只保留纯文本 → 结构化 `tool_call` 信息丢失

LLM 在"继续"时实际看到的：

```
[user]      "帮我创建项目"
[assistant] "subagent 完成了但不完整，让我检查并补全"   ← 只有文本，没有工具过程
[user]      "继续"
```

**中间所有的 bash 命令、SubAgent 结果、文件创建全部不可见。** LLM 以为什么都没做过 → 重复。

### 6.3 架构分析：为什么会这样

这不是一个简单的 filter 写错了——**是 history 所有权的架构问题**。

```
当前：
  UI ChatMessage[] ──有损重建──→ LLM history
  ↑ 渲染数据模型                   ↑ 推理数据模型
  两种数据模型不同构，有损转换不可避免

正确：
  ContextManager.#history ──原样提供──→ LLM history
  ↓ 投影                                ↑ 唯一 source of truth
  UI ChatMessage[]（只负责渲染）
```

ChatMessage 和 Message 是两种不同的数据模型，服务于不同目的：

| | ChatMessage（UI 用） | Message（LLM 用） |
|---|---|---|
| 目的 | 渲染聊天气泡、工具卡片 | 驱动 LLM 多轮推理 |
| content | 纯文本 string | 结构化 `TextContent / ToolCallContent / ToolResultContent` |
| 工具信息 | 独立的 `toolCall` 对象 | 内嵌在 assistant content 数组中 |
| role | user / assistant / system / tool | user / assistant（严格交替） |

**用渲染模型反向重建推理模型，是信息必然丢失的。**

### 6.4 方案：ContextManager 持有 history

LLM history 是上下文管理的一部分。ContextManager 已经在管压缩和裁剪——**让它也管存储，是职责内聚的自然延伸，不需要另造抽象**。

**扩展现有 `ContextManager`**：

```typescript
export class ContextManager {
  // 原有
  #strategyName: string
  #compactBridge: ICompactBridge | null

  // 新增：LLM 完整 history（唯一 source of truth）
  #history: Message[] = []

  /** 返回内部引用。AgentLoop.run() 直接在上面追加，零同步成本 */
  getHistoryRef(): Message[] { return this.#history }

  /** submit 时追加用户消息 */
  pushUser(content: string): void { this.#history.push({ role: 'user', content }) }

  /** compact 后原地替换（保持数组引用不变，run 中途不会断开） */
  replaceHistory(compacted: Message[]): void {
    this.#history.length = 0
    this.#history.push(...compacted)
  }

  /** /clear */
  clearHistory(): void { this.#history.length = 0 }

  /** /resume（从 JSONL 重建） */
  restoreHistory(messages: Message[]): void {
    this.#history.length = 0
    this.#history.push(...messages)
  }
}
```

**关键设计决策**：

| 决策 | 理由 |
|---|---|
| `getHistoryRef()` 返回引用而非副本 | AgentLoop.run() 在执行过程中追加 assistant + tool_result；返回引用 = 修改自动反映到 ContextManager = 零同步成本 |
| `replaceHistory()` 用 `length=0 + push` 原地替换 | 保持数组引用不变。如果 auto-compact 在两次 run() 之间触发，不会断开引用链 |
| 不改 `prepare()` 签名 | 保持 `prepare(rawHistory)` 参数传入方式，调用方传 `getHistoryRef()`。改动最小，后续有需要再收拢 |

**agent-loop.ts 配套改动**（1 行）：

```typescript
// 修改前：复制后与外部断开
const history: Message[] = [...messages]

// 修改后：直接引用，ContextManager 能看到所有追加
const history = messages
```

### 6.5 改动范围

| 文件 | 改动 | 行数 |
|---|---|---|
| `src/core/context-manager.ts` | 新增 `#history` + 5 个方法 | ~30 |
| `src/core/agent-loop.ts` | 去掉 `[...messages]` 复制 | 1 |
| `src/ui/useChat.ts` | submit/compact/clear/resume 走 contextManager | ~20 |

### 6.6 数据流（修改后）

```
submit("继续")
  → contextManager.pushUser("继续")               // 追加到 #history
  → contextManager.prepare(getHistoryRef(), ...)   // 完整 history，含所有 tool_call + tool_result
  → AgentLoop.run(getHistoryRef())                 // 引用，run() 内部追加直接反映到 ContextManager
  → run 结束                                       // #history 已自动包含所有新增消息
  → UI: setMessages(prev => [...prev, assistantMsg]) // 单独追加渲染用 ChatMessage
```

### 6.7 同步矩阵

| 操作 | UI messages | ContextManager #history |
|---|---|---|
| submit | `setMessages(+userMsg)` | `pushUser(text)` |
| AgentLoop 运行中 | `setMessages(+toolMsg)` per tool_done | run() 内部自动追加 |
| AgentLoop 完成 | `setMessages(+assistantMsg)` | 已自动包含 |
| /compact | `setMessages(compactedMsgs)` | `replaceHistory(compacted)` |
| /clear | `setMessages([])` | `clearHistory()` |
| /resume | `setMessages(restored)` | `restoreHistory(structured)` |

### 6.8 风险

| 风险 | 缓解 |
|---|---|
| 引用传递是隐式副作用 | 方法名 `getHistoryRef()` 明确标注；run() 注释说明会修改传入数组 |
| /resume 恢复时 JSONL 可能不含结构化消息 | 需确认 `session-store.loadMessages()` 的重建逻辑 |
| UI messages 和 #history 漂移 | ContextManager 是 source of truth；UI 只做追加投影，不反向影响 |

## 九、LangChain + GLM 流式 tool_calls 丢失问题

### 9.1 现象

主 Agent 干到一半突然停止。JSONL 显示最后一轮 `llm_call_end` 的 `stopReason = "tool_calls"`（LLM 认为自己返回了工具调用），但紧接着就是 `assistant` + `session_end`，**没有任何 tool_call_start**。

```
L37 LLM_END: in=20688 out=226 stop=tool_use   ← LLM 说有工具调用
L38 ASSISTANT: "好的...让我检查一下..."           ← 但只有文本
L39 SESSION_END                                   ← 直接结束了，没有执行任何工具
```

AgentLoop 内部走了 `toolCalls.length === 0` → 判定完成 → yield done。**LLM 明明想调工具，但 Agent 却认为任务结束了。**

### 9.2 根因

**debug.log 实锤**：LangChain `ChatOpenAI.stream()` + GLM-5.1 在多轮工具调用场景下，流式 chunk 中工具调用数据**完全缺失**。

第一轮正常（chunk 中有 `tool_call_chunks` + `tool_calls`）：

```
chunk[33]: tool_call_chunks=[{"name":"bash","args":"{...}","id":"call_xxx"}]
           tool_calls=[{"name":"bash","args":{...},"id":"call_xxx"}]   ← 正常
```

第二轮异常（chunk 中工具数据全部为空，但 finish_reason 说有）：

```
chunk[0-20]: content="" tool_call_chunks=[] tool_calls=[]   ← 全是空 chunk
chunk[最后]: tool_calls=[] finish_reason="tool_calls"        ← finish_reason 说有，但数据没有
```

**关键事实**：
- GLM-5.1 的 `finish_reason` 报告自己返回了工具调用
- 但流式 chunk 中 `tool_call_chunks` 和 `tool_calls` 全部为空
- `additional_kwargs.tool_calls` 也为空
- **数据在流式传输中完全丢失了**

### 9.3 验证：GLM API 本身没问题

用 `tests/integration/glm-stream-debug.ts` 直接调 GLM API 测试了三种场景：

| 场景 | stream 模式 | invoke 模式 |
|---|---|---|
| 第一轮：单轮工具调用 | ✅ tool_calls 正常 | ✅ 正常 |
| 第二轮：带历史的多工具调用 | ✅ tool_calls 正常 | ✅ 正常 |
| 第三轮：直接要求 dispatch_agent | ✅ tool_calls 正常 | ✅ 正常 |

**API 本身在独立测试中完全正常**。问题只在 cCli 实际运行时（长 systemPrompt 13K + 多轮历史累积 20K+ tokens）的特定条件下偶发。

### 9.4 可能的触发条件（待进一步验证）

| 因素 | 说明 |
|---|---|
| systemPrompt 很长（13K 字符） | 可能影响流式返回的缓冲/分片行为 |
| history 累积到 20K+ tokens | 大上下文可能导致 GLM API 流式返回不稳定 |
| LangChain ChatOpenAI 的 chunk 聚合逻辑 | `AIMessageChunk.concat()` 在某些 chunk 类型混合时（`ChatMessageChunk` + `AIMessageChunk`）可能丢失 `tool_call_chunks` |
| `ChatMessageChunk` 混入 | debug.log 中观察到第二轮开头有 `ChatMessageChunk`（不是 `AIMessageChunk`），LangChain 的 concat 可能不兼容混合类型 |

### 9.5 当前修复：invoke fallback（临时方案）

在 `openai-compat.ts` 中加了两级 fallback：

```
stream 聚合后 tool_calls 为空 + finish_reason = "tool_calls"
  ↓
fallback 1: 从 additional_kwargs.tool_calls 提取（部分 API 放在这里）
  ↓ 仍为空
fallback 2: 用 invoke（非流式）重试一次，绕过流式聚合 bug
  ↓ 从 invoke 结果提取 tool_calls
```

**优点**：
- 实测有效，invoke 模式下 GLM API 稳定返回 tool_calls
- 不影响正常路径（tool_calls 在 stream 中正常时不触发 fallback）
- debug.log 有详细日志便于追踪

**代价**：
- invoke fallback 时该轮 LLM 调用被执行了**两次**（stream + invoke），token 和时间双倍消耗
- 流式输出的文本已经 yield 过了，invoke 重试只用于提取 tool_calls，文本不会重复
- 理论上 invoke 返回的 tool_calls 和 stream 想返回的可能不同（概率极低但存在）

### 9.6 根本解决方案（待评估）

| 方案 | 改动量 | 风险 | 说明 |
|---|---|---|---|
| **A. 绕过 LangChain 流式，直接用 OpenAI SDK** | 大（重写 openai-compat.ts） | 中 | 用 `openai` npm 包的原生流式，自己解析 SSE delta，不依赖 LangChain 的 chunk 聚合 |
| **B. 默认使用 invoke 模式** | 小（改 1 行） | 低 | 牺牲流式输出体验（用户看不到逐字生成），但 100% 可靠 |
| **C. invoke fallback（当前方案）** | 已完成 | 低 | 正常时走 stream，丢数据时自动 invoke 补救。双倍 token 是代价 |
| **D. 等 LangChain 修复** | 无 | 不可控 | 提 issue 给 LangChain，等上游修复 ChatMessageChunk 混合类型的 concat 问题 |
| **E. 自定义 chunk 聚合** | 中 | 低 | 不用 `allChunks.reduce((a,b) => a.concat(b))`，而是自己遍历 chunk 提取 `tool_call_chunks`，手动拼接 |

**推荐**：短期继续用 **C（invoke fallback）**，中期切到 **E（自定义 chunk 聚合）** 彻底避免 LangChain concat 的兼容性问题。

### 9.7 方案 E 设计思路

不依赖 LangChain 的 `AIMessageChunk.concat()`，而是自己在遍历 chunk 时就地提取工具调用：

```typescript
// 替代 allChunks.reduce((a,b) => a.concat(b))

const collectedToolCalls = new Map<string, { id: string; name: string; argsJson: string }>()
let finishReason = 'stop'

for await (const chunk of stream) {
  // 文本照常 yield
  if (chunk.content) yield { type: 'text', text: chunk.content }

  // 就地提取 tool_call_chunks（不等聚合）
  for (const tc of (chunk.tool_call_chunks ?? [])) {
    if (tc.id && tc.name) {
      // 新工具调用开始
      collectedToolCalls.set(tc.id, { id: tc.id, name: tc.name, argsJson: tc.args ?? '' })
    } else if (tc.index !== undefined) {
      // 增量 args（某些 API 分多个 chunk 传参数）
      const existing = [...collectedToolCalls.values()][tc.index]
      if (existing) existing.argsJson += tc.args ?? ''
    }
  }

  // 也检查 additional_kwargs（GLM 兼容）
  for (const tc of (chunk.additional_kwargs?.tool_calls ?? [])) {
    if (tc.id && tc.function?.name && !collectedToolCalls.has(tc.id)) {
      collectedToolCalls.set(tc.id, {
        id: tc.id,
        name: tc.function.name,
        argsJson: tc.function.arguments ?? '{}',
      })
    }
  }

  if (chunk.response_metadata?.finish_reason) {
    finishReason = chunk.response_metadata.finish_reason
  }
}

// 最终提取
for (const tc of collectedToolCalls.values()) {
  yield {
    type: 'tool_call',
    toolCall: { type: 'tool_call', toolCallId: tc.id, toolName: tc.name, args: JSON.parse(tc.argsJson) },
  }
}
```

**优点**：不依赖 `concat()`，逐 chunk 提取，不可能丢失。
**代价**：需要处理分片传参（args 分多个 chunk 传递的场景），逻辑比 `reduce + concat` 复杂一些。

### 9.12 轮次与 MaxListeners 配置调整

基于实测反馈：默认轮次太低导致复杂任务做不完，MaxListeners 需与轮次同步。

**调整决策**：

| 配置 | 调整前 | 调整后 | 理由 |
|---|---|---|---|
| 主 Agent maxTurns | 50 | **100** | 复杂任务（多文件创建 + SubAgent 协调 + 验证）50 轮不够 |
| general SubAgent maxTurns | 25 | **50** | 创建完整项目（脚手架 + 配置 + 业务代码 + 验证）需要更多轮次 |
| explore SubAgent maxTurns | 15 | **50** | 深度代码搜索（多次 grep + read_file 迭代）15 轮太紧 |
| plan SubAgent maxTurns | 15 | **50** | 大型架构分析需要读很多文件 |
| SubAgent 兜底 maxTurns | 25 | **50** | 与定义对齐 |
| 主 Agent maxListeners | 100 | **200** | 100 轮 × 每轮可能 2 个 listener = 需要 200 |
| SubAgent maxListeners | 50 | **100** | 50 轮 × 2 = 100 |

**MaxListeners 计算公式**：`maxListeners ≥ maxTurns × 2`。每轮 LLM 调用 LangChain 可能给 signal 加 1-2 个 listener（stream 开始时加，结束时清理但存在延迟），所以需要 2 倍余量。

**关联文件**：
- `src/core/agent-loop.ts` — `DEFAULT_MAX_TURNS = 100`
- `src/tools/agent/built-in.ts` — 三个 agent 的 `maxTurns: 50`
- `src/tools/agent/dispatch-agent.ts` — `DEFAULT_MAX_TURNS = 50` + `setMaxListeners(100)`
- `src/ui/useChat.ts` — `setMaxListeners(200)`

### 9.13 RepetitionDetector — 工具调用循环防御

**问题背景**：

GLM-5.1 在 SubAgent 场景下会对同一文件反复调用 write_file（完全相同的参数），即使每次都返回 success。

实测数据（两次 session）：

| 场景 | 修复前 | 加"软约束"后 | 说明 |
|------|--------|-------------|------|
| backend write_file | 19 次 | 1 次 ✅ | 简单任务，提示词+反馈增强有效 |
| frontend write_file | 9 次 | 48 次 ❌ | 复杂任务，提示词完全无效，反而恶化 |

尝试过的"软约束"（对复杂任务无效）：
1. write_file tool_result 增强反馈："✅ 无需重复写入，请继续下一步"
2. SubAgent systemPrompt 加入 "NEVER call the same tool with the same arguments twice"

**结论**：纯提示词方案对弱模型在复杂任务场景下不可靠，必须在 agent-loop 层面做硬性拦截。

**实现方案 — RepetitionDetector**：

独立模块 `src/core/repetition-detector.ts`，三级响应：

| 连续次数 | 级别 | 行为 |
|---------|------|------|
| 1-2 次 | allow | 正常执行（可能是合理重试） |
| 3-4 次 | warn | 正常执行，但在 tool_result 后注入警告到 history |
| ≥5 次 | block | **跳过工具执行**，直接返回拦截消息作为 tool_result |

指纹计算：`toolName + JSON.stringify(args, sortedKeys)`，当模型调用不同工具/不同参数时自动重置。

**集成位置**：`agent-loop.ts` 的 `#executeOneTool()` 方法，在权限检查之前执行（block 级别不需要走后续流程）。

**关联文件**：
- `src/core/repetition-detector.ts` — 独立模块，RepetitionDetector 类
- `src/core/agent-loop.ts` — 集成检测器，#executeOneTool 中调用

**多层防御体系**（从软到硬）：
1. write_file tool_result 增强反馈 → 对简单任务有效
2. SubAgent systemPrompt 防重复指令 → 对强模型有效
3. RepetitionDetector warn → 第 3 次注入警告消息
4. RepetitionDetector block → 第 5 次硬性拦截，跳过执行
