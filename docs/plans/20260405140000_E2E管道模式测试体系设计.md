# E2E 管道模式测试体系设计

## 一、设计初衷

cCli 的 SubAgent 是核心差异化能力——主 Agent 派发多个子 Agent 并行执行独立任务。但此前只在交互模式下手动验证，缺乏可重复、可量化的自动化测试手段。

**核心诉求**：一条命令启动，自动驱动 cCli 完成一个真实全栈项目搭建，然后验证产出物是否正确。不 mock，不模拟，真实端到端。

## 二、背景

### 2.1 管道模式（Pipe Mode）

cCli 支持非交互模式：`ccode -p "prompt" --yes --verbose`。

- 不启动 React/Ink UI，纯文本输出到 stdout
- `--yes` 自动批准所有工具执行（无需人工确认）
- `--verbose` 在 stderr 输出工具执行进度
- 执行完自动退出，适合脚本驱动和 CI 集成

管道模式与交互模式共享相同的 AgentLoop、Provider、SessionLogger、TokenMeter 全链路。主 Agent 和 SubAgent 的 session 都会持久化到 JSONL（`~/.ccode/sessions/<projectSlug>/<sessionId>/`）。

### 2.2 已有验证基础

在此之前，项目已积累了若干手动调试脚本：

- `tests/integration/glm-stream-debug.ts` — 验证 GLM 流式 tool_calls 行为
- `tests/integration/glm-stream-usage.ts` — 验证 GLM 流式 token usage 返回
- `tests/integration/glm-usage-e2e.ts` — 端到端模拟 openai-compat.ts 的 usage 提取链路

但缺少**真正驱动完整 Agent 工作流**的测试。

## 三、测试场景设计

选择"全栈项目搭建"作为 E2E 测试场景，因为它天然覆盖 SubAgent 的核心能力矩阵：

| 维度 | 覆盖点 |
|------|--------|
| dispatch_agent 派发 | 主 Agent 能否正确拆解任务并派发 3 个 SubAgent |
| 工具调用 | bash（命令执行）、write_file（文件创建/修改） |
| SubAgent 隔离 | 每个 SubAgent 只做自己的任务，不越界 |
| RepetitionDetector | 不出现循环调用（之前 GLM-5.1 的 48 次 write_file 问题） |
| 端口/路径配置 | 前后端通过约定端口通信，验证配置是否正确穿透 |

### 测试任务

1. **SubAgent 1（后端）**：uv + FastAPI，端口 9091，GET /hello 返回 helloworld
2. **SubAgent 2（前端）**：Vue + Vite，端口 9092，axios 调后端显示数据
3. **SubAgent 3（运维）**：交互式菜单脚本 run.bat / run.sh（1.启动 2.停止 3.状态 4.退出）

每个 SubAgent 完成后需执行自验证（如 `bash -n run.sh` 检查语法）。

## 四、开发过程中遇到的问题

### 4.1 tsx 路径别名解析失败

**现象**：`ERR_MODULE_NOT_FOUND: Cannot find package '@ui/format-utils.js'`

**根因**：tsx 根据 cwd 查找 tsconfig.json。当 e2e 测试将 cwd 切到目标工作目录（桌面）时，tsx 找不到项目的 tsconfig，所有 `@ui/`、`@config/` 等 paths 别名全部解析失败。

**修复**：
1. `pipe-runner.ts` 原来 import `@ui/ToolStatusLine.js`（.tsx 文件），把 `buildArgsSummary` 纯函数从 React 组件迁移到 `format-utils.ts`（纯 .ts），消除对 .tsx 的依赖
2. e2e 脚本中使用项目本地 tsx + `--tsconfig` 显式指定 tsconfig 路径：
   ```bash
   "$PROJECT_ROOT/node_modules/.bin/tsx" --tsconfig "$PROJECT_ROOT/tsconfig.json" "$CLI_ENTRY" ...
   ```

### 4.2 TS 版 prompt 传参被 Windows cmd.exe 截断

**现象**：cCli 执行完后 0 个验证通过（文件没创建），但无报错。

**根因**：TS 版使用 `spawn(..., { shell: true })` 将超长多行 prompt（含反引号、中文、换行）作为命令行参数传递。Windows cmd.exe 有 ~8192 字符限制且特殊字符转义不完整。Shell 版用 bash 变量传递无此问题。

**修复**：TS 版改为 prompt 写临时文件 → 通过 stdin fd 管道传入 cCli → pipe-runner 的 `readStdin()` 自动读取：
```typescript
const promptFile = join(tmpdir(), `ccode-e2e-${Date.now()}.txt`)
writeFileSync(promptFile, prompt, 'utf-8')
const stdinFd = openSync(promptFile, 'r')
spawn(TSX, [..., '-p', '按照 stdin 中的任务描述执行', ...], {
  stdio: [stdinFd, 'pipe', 'pipe'], shell: true,
})
```

### 4.3 Git Bash 兼容性

| 问题 | 修复 |
|------|------|
| `(( PASSED++ ))` 在 `set -e` 下值为 0 时退出 | 改为 `PASSED=$((PASSED + 1))` |
| `$0` 在 source 场景不可靠 | 改为 `${BASH_SOURCE[0]}` |
| heredoc 赋值变量转义复杂 | prompt 写临时文件 + `trap` 清理 |

### 4.4 TS vs Shell 性能差异

TS 版比 Shell 版慢数秒（启动阶段），因为多了一层：Node.js 启动 → tsx 编译 → spawn 子进程（再次 Node.js + tsx）。Shell 版直接 fork。

对 e2e 测试来说启动开销（秒级）相比任务执行时间（分钟级）可忽略，但符合第一性原理——Shell 更原生，实际跑测试优先用 Shell 版。

## 五、脚本规范

### 5.1 文件位置

```
tests/
├── e2e/
│   ├── test-subagent-fullstack.sh   # Shell 版（推荐，CI 用）
│   └── test-subagent-fullstack.ts   # TS 版（复杂断言场景备用）
├── integration/
│   ├── glm-stream-debug.ts          # GLM tool_calls 调试
│   ├── glm-stream-usage.ts          # GLM token usage 诊断
│   └── glm-usage-e2e.ts             # usage 提取链路验证
└── unit/
    └── ...                          # Vitest 单元测试
```

### 5.2 运行方式

```bash
# 从项目根目录运行（推荐）
bash tests/e2e/test-subagent-fullstack.sh
npx tsx tests/e2e/test-subagent-fullstack.ts

# 从脚本所在目录运行
cd tests/e2e
bash test-subagent-fullstack.sh
npx tsx test-subagent-fullstack.ts
```

### 5.3 脚本结构约定

每个 e2e 脚本遵循统一结构：

```
1. 头部说明（目的、用法）
2. 路径常量（PROJECT_ROOT / TSX / TSCONFIG / CLI_ENTRY / WORKSPACE）
3. Prompt 定义
4. 执行 cCli（记录开始/结束时间）
5. 验证产出物（check_file / check_content）
6. 输出统计（passed / failed / 耗时）
```

### 5.4 验证检查模式

```bash
# 文件存在性
check_file "描述" "文件路径"

# 文件内容包含关键字
check_content "描述" "文件路径" "关键字"
```

统一输出格式：
```
  ✓ backend/main.py 存在
  ✗ frontend/vite.config.ts 包含端口 9092
```

### 5.5 关键约束

- **tsx 必须用项目本地版本**：`$PROJECT_ROOT/node_modules/.bin/tsx`，不用 `npx tsx`（避免从 npm 缓存拉全局版本）
- **必须带 `--tsconfig`**：显式指定项目 tsconfig 路径，确保 paths 别名解析正确
- **prompt 不走命令行参数**：Shell 版用变量，TS 版写临时文件走 stdin，避免被 cmd.exe 截断
- **统计耗时**：记录开始时间、结束时间、总耗时，方便对比不同模型/配置的执行效率

## 六、结论

### 6.1 管道模式验证完整性

| 能力 | 管道模式是否覆盖 |
|------|:---:|
| AgentLoop 多轮调用 | ✓ |
| 工具调用 + 权限（--yes 自动批准） | ✓ |
| dispatch_agent 派发 SubAgent | ✓ |
| SubAgent 独立 JSONL 持久化 | ✓ |
| TokenMeter 写入 SQLite | ✓ |
| RepetitionDetector 防循环 | ✓ |
| systemPrompt + scopeFence 注入 | ✓ |
| Hook 系统 | ✓ |

管道模式与交互模式共享全部核心链路，唯一差异是 UI 层（不启动 Ink）和权限层（`--yes` 跳过弹窗）。

### 6.2 测试体系定位

```
单元测试 (Vitest)        → 函数级，快，纯逻辑验证
集成测试 (integration/)   → 模块级，验证 LLM API 交互
E2E 测试 (e2e/)          → 系统级，真实驱动完整 Agent 工作流
```

E2E 不替代单元测试，但它是唯一能验证"主 Agent → SubAgent → 工具调用 → 产出物"完整链路的手段。

### 6.3 后续可扩展方向

- **多模型对比**：同一 prompt 切不同模型，对比成功率和耗时
- **CI 集成**：Shell 版天然适合 GitHub Actions，退出码即测试结果
- **回归防护**：代码变更后跑一遍 e2e，确保 SubAgent 能力不退化
- **`--json` 模式**：配合管道模式的 JSON 输出，可做程序化结果分析
