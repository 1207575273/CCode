# 统一底部状态栏（StatusBar）设计 v2

## 一、背景与目标

### 1.1 现状问题

1. **信息分散**：token 统计、Context%、费用等散落在 App.tsx 内联 `<Text>` 节点中
2. **缺乏资源视角**：没有系统级和进程级的内存/CPU 监控
3. **CLI 独占**：Web 端无法看到运行时状态

### 1.2 目标

在 InputBar 下方新增**两行固定状态栏**：
- 第一行：**系统总资源**（SYS）— 全局 MEM 和 CPU
- 第二行：**cCli 进程资源**（PROC）— 进程 MEM/CPU + 运行时间 + token/context/cost
- CLI 和 Web 端共享同一数据源，通过 eventBus 推送到 Web

## 二、布局

### 2.1 CLI 终端

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│  ChatView / WelcomeScreen                                                               │
│  Dialog zone / TodoPanel / SubAgent 指示器 / "Esc to interrupt"                         │
│  InputBar                                                                               │
│  SYS  MEM ████████░░ 62% 10.2/16GB | CPU ██████░░░░ 45%                                │
│  PROC MEM ████░░░░░░  3%  256MB    | CPU ██░░░░░░░░  8% | ⏱ 03:25 | 1.2K/800 tok | Ctx 65% | $0.02 │
└─────────────────────────────────────────────────────────────────────────────────────────┘
```

- 两行固定在 InputBar 正下方，是 UI 最底部元素
- 没有对话消息时留空不渲染
- SYS 行前缀 `SYS ` + PROC 行前缀 `PROC`，确保 MEM/CPU 进度条对齐

### 2.2 Web 端

位置与 CLI 一致——输入框下方两行。数据通过 eventBus → Bridge Server → WebSocket 推送。

Web 端渲染由前端组件自行实现（HTML/CSS 进度条），本设计只负责数据传输协议。

## 三、配置

不变，仍然是一个布尔开关：

```jsonc
{
  "statusBar": true  // 默认 true，设 false 关闭（CLI + Web 同时关闭）
}
```

## 四、指标采集

### 4.1 数据源总览

| 指标 | 层级 | 采集方式 | 刷新频率 |
|------|------|----------|----------|
| 系统 MEM | SYS | `(os.totalmem() - os.freemem()) / os.totalmem()` | 3 秒 |
| 系统 CPU | SYS | `os.cpus()` 两次采样 idle/busy 差值 | 3 秒 |
| 进程 MEM | PROC | `process.memoryUsage().rss / os.totalmem()` | 3 秒 |
| 进程 CPU | PROC | `process.cpuUsage()` 差值 / `os.cpus().length` | 3 秒 |
| 运行时间 | PROC | `accumulatedMs + (Date.now() - sessionStartTime)` | 1 秒 |
| Token | PROC | `TokenMeter.getSessionStats()` | 事件驱动 |
| Context | PROC | `contextTracker` 状态 | 事件驱动 |
| 费用 | PROC | `TokenMeter.getSessionStats().costByCurrency` | 事件驱动 |

### 4.2 系统 CPU 计算（新增）

```typescript
// 采集 os.cpus() 每个核心的 idle / total 时间
function sampleCpuTimes(): { idle: number; total: number } {
  const cpuList = os.cpus()
  let idle = 0, total = 0
  for (const cpu of cpuList) {
    idle += cpu.times.idle
    total += cpu.times.user + cpu.times.nice + cpu.times.sys + cpu.times.irq + cpu.times.idle
  }
  return { idle, total }
}

// 两次采样差值
const prev = sampleCpuTimes()
// ... 3 秒后 ...
const curr = sampleCpuTimes()
const idleDelta = curr.idle - prev.idle
const totalDelta = curr.total - prev.total
const sysCpuPercent = totalDelta > 0 ? ((totalDelta - idleDelta) / totalDelta) * 100 : 0
```

### 4.3 系统内存计算（新增）

```typescript
const totalMem = os.totalmem()
const freeMem = os.freemem()
const usedMem = totalMem - freeMem
const sysMemPercent = (usedMem / totalMem) * 100
```

### 4.4 进程 CPU / 进程 MEM

与 v1 相同：
- 进程 CPU：`process.cpuUsage()` 差值 / 核心数 / 采样间隔
- 进程 MEM：`process.memoryUsage().rss / os.totalmem()`

### 4.5 运行时间（跨 resume 累加）

与 v1 相同，`session_resume` / `session_end` 事件的 `accumulatedMs` 字段。

## 五、视觉规范

### 5.1 两行格式

```
SYS  MEM ████████░░ 62% 10.2/16GB | CPU ██████░░░░ 45%
PROC MEM ████░░░░░░  3%  256MB    | CPU ██░░░░░░░░  8% | ⏱ 03:25 | 1.2K/800 tok | Ctx 65% | $0.02
```

**SYS 行**：

| 段 | 格式 | 说明 |
|----|------|------|
| 前缀 | `SYS ` | 4 字符，灰色 dimColor |
| MEM | `MEM {bar} {pct}% {used}/{total}` | 如 `10.2/16GB` |
| CPU | `CPU {bar} {pct}%` | 系统总 CPU 使用率 |

**PROC 行**：

| 段 | 格式 | 说明 |
|----|------|------|
| 前缀 | `PROC` | 4 字符，灰色 dimColor |
| MEM | `MEM {bar} {pct}% {abs}` | 如 `256MB`，只显示进程占用 |
| CPU | `CPU {bar} {pct}%` | 进程 CPU 使用率 |
| elapsed | `⏱ MM:SS` / `⏱ HH:MM:SS` | 秒级跳动 |
| token | `{in}/{out} tok` | K/M 自动缩写 |
| context | `Ctx {pct}%` | 色阶同 MEM/CPU |
| cost | `${amount}` / `¥{amount}` | 多币种拼接 |

### 5.2 进度条

- 宽度：硬编码 10 格（两行对齐更整齐）
- 填充字符：`█`（实心）/ `░`（空心）
- 色阶：0-60% 绿色、60-85% 黄色、85%+ 红色

### 5.3 对齐

SYS 和 PROC 的 MEM/CPU 进度条纵向对齐：
- 两行前缀都是 4 字符（`SYS ` / `PROC`）
- MEM 和 CPU 段宽度固定，百分比右对齐（用空格 pad）

### 5.4 窄终端截断

**SYS 行**：宽度不足时先丢 CPU 段，只保留 MEM。

**PROC 行**：从右向左截断：cost → context → token → elapsed → CPU → MEM。

## 六、Web 端数据推送

### 6.1 新增 eventBus 事件类型

```typescript
// event-bus.ts BridgeEvent 新增
| { type: 'status_bar'; data: StatusBarPayload }
```

```typescript
interface StatusBarPayload {
  sys: {
    memPercent: number
    memUsedBytes: number
    memTotalBytes: number
    cpuPercent: number
  }
  proc: {
    memPercent: number
    memUsedBytes: number
    cpuPercent: number
    elapsedMs: number
  }
  token: {
    inputTokens: number
    outputTokens: number
    costByCurrency: Record<string, number>
    callCount: number
  } | null
  context: {
    usedPercentage: number
    level: string
  } | null
}
```

### 6.2 推送时机

`useStatusBar` hook 每次采样后，通过 `eventBus.emit({ type: 'status_bar', data })` 广播。Web 端收到后渲染。

推送频率与采样频率一致（3 秒），不额外增加流量。运行时间由 Web 端本地计算（收到 `elapsedMs` 基线后自行 +1s），减少推送频率。

### 6.3 Bridge Server 处理

Bridge Server 已有对 BridgeEvent 的 WebSocket 转发逻辑，新增 `status_bar` 类型后自动透传，无需额外处理。

## 七、组件架构

### 7.1 文件变更

| 文件 | 操作 | 职责 |
|------|------|------|
| `src/ui/StatusBar.tsx` | **修改** | 从单行改为两行渲染，新增 SYS 行 |
| `src/ui/useStatusBar.ts` | **修改** | 新增系统 CPU/MEM 采集，输出 `StatusBarData` 增加 sys 字段，新增 eventBus 推送 |
| `src/core/event-bus.ts` | **修改** | `BridgeEvent` 新增 `status_bar` 事件类型 |
| `tests/unit/StatusBar.test.ts` | **修改** | 补充 SYS 行相关测试 |
| `tests/unit/useStatusBar.test.ts` | **修改** | 补充系统 CPU 采样函数测试 |

### 7.2 数据结构

```typescript
// useStatusBar.ts 输出
export interface StatusBarData {
  // SYS 行
  sysMemPercent: number
  sysMemUsedBytes: number
  sysMemTotalBytes: number
  sysCpuPercent: number

  // PROC 行
  procMemPercent: number
  procMemUsedBytes: number
  procCpuPercent: number
  elapsedMs: number
  cpuCoreCount: number
}
```

Token/Context/Cost 数据仍由 App.tsx 传入 StatusBar props（事件驱动，不在 hook 中采集）。

### 7.3 数据流

```
┌──────────────────────────────────────────────────────────┐
│ useStatusBar Hook                                         │
│                                                           │
│  每 3s ──→ os.cpus() 差值        → sysCpuPercent          │
│        ──→ os.freemem()/totalmem → sysMemPercent          │
│        ──→ process.cpuUsage()    → procCpuPercent         │
│        ──→ process.memoryUsage() → procMemPercent         │
│                                                           │
│  每 1s ──→ Date.now() - start    → elapsedMs              │
│                                                           │
│  ──→ StatusBarData ──→ StatusBar.tsx 渲染（CLI）          │
│  ──→ eventBus.emit('status_bar') → Bridge → Web 端渲染   │
└──────────────────────────────────────────────────────────┘
```

## 八、边界场景

| 场景 | 处理 |
|------|------|
| 首次采样无 CPU 前值 | SYS/PROC CPU 均显示 `0%` |
| `os.totalmem()` 返回 0 | MEM 进度条显示 `--` |
| 终端宽度 < 40 列 | SYS 行只保留 MEM，PROC 行只保留 MEM |
| `config.statusBar = false` | 不渲染、不采样、不推送 eventBus，零开销 |
| 管道模式 | 无 Ink UI，不加载 StatusBar，零开销 |
| Web 端未连接 | eventBus 广播无订阅者，emit 无副作用 |
| resume 后时间累加 | 从 `session_end.accumulatedMs` 恢复，链式累加 |

## 九、性能考量

| 关注点 | 策略 |
|--------|------|
| `os.cpus()` 开销 | 同步调用，单次 < 1ms，3s 间隔可忽略 |
| `os.freemem()` 开销 | 同步调用，纳秒级 |
| 两行渲染 vs 一行 | Ink 渲染开销与行数无关（都是一次 stdout.write） |
| eventBus 推送频率 | 3s 一次，每次约 200 字节 JSON，可忽略 |
| Web 端运行时间 | 客户端本地 +1s，不靠服务端推秒级更新 |

## 十、不做的事

- 不监控 bash 工具 spawn 的临时外部命令
- 不做历史资源曲线/图表
- 不做 GPU 监控
- 不做状态栏拖拽/可调位
- 不在 Web 端实现完整 UI（本期只负责数据推送协议，Web 渲染由前端组件后续实现）
