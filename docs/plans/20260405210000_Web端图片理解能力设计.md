# Web 端图片理解能力设计（v2）

> 设计时间：2026-04-05 21:00  |  v2 修订：2026-04-05 22:00
> 目标：Web 端支持粘贴截图，上传到本地项目目录，发送给支持多模态的 LLM 进行理解

---

## 一、需求概述

### 1.1 核心场景

用户在 Web 端粘贴截图（Ctrl+V），图片上传到本地项目目录，作为附件随消息发送给 LLM。LLM 理解图片内容后辅助工作（如 UI 还原、Bug 定位、设计稿分析）。

### 1.2 设计决策

| 决策项 | 选择 | 理由 |
|-------|------|------|
| 输入方式 | 仅粘贴（Ctrl+V） | 最高频场景，截图即粘贴 |
| 存储位置 | `~/.ccode/images/` | 本地即服务器，无需远程存储 |
| 文件命名 | UUIDv7 | 时间有序，便于管理和清理 |
| 视觉能力开关 | `visionModels: string[]`（模型维度） | 同一 provider 内不同模型能力不同 |
| 持久化 | JSONL 只记录 imageId 引用，不存 base64 | 避免日志膨胀 |
| 发送格式 | Provider 层读文件 → base64 inline | 延迟加载，各 Provider 都支持 |
| 前端压缩 | 最长边 ≤ 2048px | 控制 token 成本和传输体积 |

## 二、分层架构总览

### 2.1 核心原则：Web 是薄容器，CLI 是服务端

```
┌─────────────────────────────────────────────────────────────────────┐
│  Web (薄容器)                                                        │
│  职责：展示 + 采集。不做任何业务判断。                                   │
│  ┌────────────┐  ┌────────────────┐  ┌──────────────┐               │
│  │ InputBar   │  │ ImageAttach    │  │ ChatBubble   │               │
│  │ 文本输入    │  │ 粘贴→压缩→上传 │  │ 图片内联展示  │               │
│  └────────────┘  └────────────────┘  └──────────────┘               │
│        ↕ HTTP (upload)    ↕ WS (imageIds)    ↕ HTTP (GET /images)   │
├─────────────────────────────────────────────────────────────────────┤
│  Dashboard API (HTTP 资源层)                                         │
│  职责：图片文件的上传/下载端点，纯 I/O 管道，不含业务逻辑。              │
│  ┌──────────────────┐  ┌──────────────────┐                         │
│  │ POST /images/up  │  │ GET /images/:id  │                         │
│  │  → ImageStore    │  │  → 静态文件服务   │                         │
│  └──────────────────┘  └──────────────────┘                         │
├─────────────────────────────────────────────────────────────────────┤
│  Bridge (WS 传输层)                                                  │
│  职责：透明中继。imageIds 只是消息字段的一部分，Bridge 不解释不转换。     │
│  ┌──────────────────────────────────────────────┐                   │
│  │ { type:'chat', text, imageIds } → 原样路由    │                   │
│  └──────────────────────────────────────────────┘                   │
├─────────────────────────────────────────────────────────────────────┤
│  CLI 业务层 (useChat → ContextManager → AgentLoop)                   │
│  职责：消息构建、Vision 门控、上下文管理。这里做所有业务决策。             │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐              │
│  │ useChat      │  │ContextMgr    │  │ AgentLoop    │              │
│  │ 接收imageIds │→│ pushUser()   │→│ run(msgs)    │              │
│  │ 门控检查     │  │ 管理history  │  │ 调用Provider │              │
│  └──────────────┘  └──────────────┘  └──────────────┘              │
├─────────────────────────────────────────────────────────────────────┤
│  Provider (协议适配层)                                                │
│  职责：将 ImageContent → 供应商特定格式。读 base64 在这里（延迟加载）。  │
│  ┌──────────────────┐  ┌──────────────────┐                         │
│  │ Anthropic        │  │ OpenAI Compat    │                         │
│  │ image block      │  │ image_url block  │                         │
│  │ (base64 source)  │  │ (data: URL)      │                         │
│  └──────────────────┘  └──────────────────┘                         │
├─────────────────────────────────────────────────────────────────────┤
│  Infrastructure (基础设施层)                                          │
│  职责：纯 I/O，无业务逻辑。被上层按需调用。                              │
│  ┌──────────────────┐  ┌──────────────────┐                         │
│  │ ImageStore       │  │ SessionLogger    │                         │
│  │ write/read/clean │  │ 记录imageId引用   │                         │
│  └──────────────────┘  └──────────────────┘                         │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 各层职责边界（严格遵守）

| 层 | 可以做 | 不能做 |
|----|-------|--------|
| **Web** | 压缩、上传、展示缩略图、发 imageIds | 不能判断是否支持 vision、不能读配置、不能构建 Message |
| **Dashboard API** | 写文件、读文件、返回配置查询结果 | 不能修改消息、不能调用 Provider |
| **Bridge** | 透传 imageIds 字段 | 不能解析 imageIds、不能读文件 |
| **useChat** | 接收 imageIds、查配置做门控、构建 MessageContent[] | 不能读 base64、不能知道 Provider 协议差异 |
| **ContextManager** | 管理 Message history | 不关心 content 内部结构 |
| **AgentLoop** | 传递 messages 给 Provider | 不关心图片，透传即可 |
| **Provider** | 读 base64（调用 ImageStore）、转换为供应商格式 | 不能做门控决策（那是 useChat 的事） |
| **ImageStore** | 纯文件 I/O | 不能知道谁在调用、不能做任何业务判断 |

### 2.3 为什么 base64 读取在 Provider 层而不是 useChat？

**延迟加载原则**：imageId 在系统中流转时始终是轻量级引用（几十字节），只有在真正要发 HTTP 请求给 LLM 时（Provider.chat()），才读文件转 base64（可能几百 KB）。

好处：
1. ContextManager 管理的 history 始终轻量，不会因图片膨胀
2. 上下文裁剪时，裁掉一条带图片的消息不会产生内存 GC 压力
3. 同一张图片在多轮对话中被多次发给 LLM 时，每次都读最新文件（未来支持图片编辑/替换）

## 三、Vision 能力配置（模型维度）

### 3.1 问题：为什么不能用 provider 级别的 boolean？

同一个 provider 内，不同模型的多模态能力不同：

| Provider | 支持视觉 | 不支持视觉 |
|----------|---------|-----------|
| Anthropic | claude-opus-4-6, claude-sonnet-4-6 | （实际都支持，但用户可能选择关闭以省 token） |
| OpenAI | gpt-4o, gpt-4o-mini | （同上） |
| GLM | glm-4v, glm-4v-plus | glm-4-flash, glm-4-air, glm-4, glm-5-turbo |

**结论**：vision 能力必须精确到 `provider + model` 维度。

### 3.2 配置数据结构

```typescript
// src/config/config-manager.ts

export interface ProviderConfig {
  apiKey: string
  baseURL?: string
  protocol?: 'anthropic' | 'openai'
  models: string[]
  /** 支持多模态图片理解的模型子集，必须是 models 的子集。为空或不填 = 全部不支持。 */
  visionModels?: string[]
}
```

**设计理由**：
- `visionModels` 是 `models` 的**白名单子集**，不是独立列表
- 默认 `undefined`/`[]` = 无模型支持视觉 = **默认关闭**
- 用户在 Web 配置页手动勾选哪些模型支持视觉
- 不需要改 `CCodeConfig` 顶层结构，完全内聚在 ProviderConfig 中

**配置文件示例（~/.ccode/config.json）**：
```json
{
  "providers": {
    "anthropic": {
      "apiKey": "sk-...",
      "models": ["claude-opus-4-6", "claude-sonnet-4-6", "claude-haiku-4-5-20251001"],
      "visionModels": ["claude-opus-4-6", "claude-sonnet-4-6"]
    },
    "glm": {
      "apiKey": "...",
      "models": ["glm-4v", "glm-4-flash", "glm-4"],
      "visionModels": ["glm-4v"]
    }
  }
}
```

### 3.3 能力查询函数

```typescript
// src/config/config-manager.ts 新增

/** 检查指定 provider + model 是否支持图片理解 */
isVisionEnabled(provider: string, model: string): boolean {
  const config = this.load()
  const prov = config.providers[provider]
  if (!prov?.visionModels?.length) return false
  return prov.visionModels.includes(model)
}
```

**调用方**：useChat（门控决策）、Web API（查询展示）。

### 3.4 Web 配置 UI（SettingsPage ProviderCard）

模型列表中每个模型标签旁增加一个**眼睛图标 👁 / 🚫**，点击切换 vision 开关：

```
模型列表        点击选中测试 · 拖拽排序 · 首位为默认模型
┌─────────────────────┐ ┌─────────────────────┐ ┌──────────────┐
│ ⠿ ★ claude-opus-4-6 │ │ ⠿ claude-sonnet-4-6 │ │ ⠿ claude-hai │
│   👁 ×               │ │   👁 ×               │ │   🚫 ×       │
└─────────────────────┘ └─────────────────────┘ └──────────────┘
   ↑ vision 开启           ↑ vision 开启           ↑ vision 关闭
```

- 👁（蓝色）= 该模型已启用 vision，在 `visionModels` 中
- 🚫（灰色）= 该模型未启用 vision
- 点击切换：添加/移除 `visionModels` 数组中的条目

### 3.5 Vision 门控流程

```
useChat.submit(text, 'web', imageIds)
    │
    ├─ imageIds 为空 → 正常纯文本流程（不变）
    │
    ├─ imageIds 非空
    │   │
    │   ├─ configManager.isVisionEnabled(provider, model) === true
    │   │   → 构建 [TextContent, ...ImageContent[]] 发给 ContextManager
    │   │
    │   └─ === false
    │       → 剥离 imageIds，仅发纯文本
    │       → eventBus.emit 一条提示事件 → Web 端显示 toast 警告
    │         "当前模型 {model} 未启用图片理解，图片已忽略"
```

**门控发生在 useChat（CLI 业务层）**，不在 Web 端、不在 Provider 端。Web 只是展示警告信息。

### 5.2 各层传递的数据形态

| 层级 | 数据形态 | 体积 |
|------|---------|------|
| Web state | `imageIds: string[]` | ~100B |
| WS 传输 | `{ type:'chat', text:'...', imageIds:['uuid1','uuid2'] }` | ~200B |
| EventBus | `{ type:'user_input', text, imageIds, source:'web' }` | ~200B |
| ContextManager history | `{ role:'user', content: [TextContent, ImageContent] }` | ~300B（imageId 引用） |
| AgentLoop → Provider | 同上，ImageContent 只含 imageId | ~300B |
| Provider 内部 | **在这里** 调用 `ImageStore.read()` 加载 base64 | ~300KB（base64） |
| LLM API 请求 | Anthropic image block / OpenAI image_url | ~300KB |

**关键**：base64（大体积）只在最后一步出现，前面全程轻量引用。

## 六、模块设计

### 6.1 ImageStore（基础设施层）

**文件**：`src/core/image-store.ts`

```typescript
/**
 * ImageStore — 图片文件的读写管理。
 *
 * 纯 I/O 操作，无业务逻辑。被 Dashboard API（写入）和 Provider（读取）调用。
 * 存储路径：~/.ccode/images/{uuidv7}.{ext}
 */

import { randomUUIDv7 } from 'node:crypto'  // Node 20.12+

const IMAGES_DIR = join(homedir(), '.ccode', 'images')

/** 图片元信息（write 返回，用于 API 响应） */
export interface ImageMeta {
  id: string          // UUIDv7
  fileName: string    // {id}.jpg
  mediaType: string   // image/jpeg | image/png | image/webp
  sizeBytes: number
}

/** 写入图片文件，返回元信息 */
export function writeImage(buffer: Buffer, mediaType: string): ImageMeta

/** 读取图片为 base64（Provider 发送时调用） */
export function readImageBase64(imageId: string): { base64: string; mediaType: string } | null

/** 检查图片文件是否存在（UI 展示降级用） */
export function imageExists(imageId: string): boolean

/** 清理超过 retentionDays 的图片，返回删除数量 */
export function cleanupImages(retentionDays: number): number
```

**实现要点**：
- `writeImage`：`mkdirSync(IMAGES_DIR, { recursive: true })` + `writeFileSync` 原子写
- `readImageBase64`：`readFileSync` → `buffer.toString('base64')`，文件不存在返回 null
- `cleanupImages`：遍历目录，UUIDv7 前 48 位是时间戳，可直接比较判断过期
- 不暴露文件路径，只暴露 id，路径封装在内部

#### OpenAI 兼容（message-converter.ts toLangChainMessages）

```typescript
// user 消息中有 image 块时，需要用 LangChain 的多模态 content 格式
case 'user': {
  const images = blocks.filter((b): b is ImageContent => b.type === 'image')
  const text = extractText(blocks)

  if (images.length > 0) {
    // LangChain HumanMessage 支持 content 数组（文本 + 图片混合）
    const parts: Array<{ type: string; [k: string]: unknown }> = []
    if (text) parts.push({ type: 'text', text })
    for (const img of images) {
      const data = readImageBase64(img.imageId)
      if (data) {
        parts.push({
          type: 'image_url',
          image_url: { url: `data:${data.mediaType};base64,${data.base64}` },
        })
      }
    }
    result.push(new HumanMessage({ content: parts }))
  } else {
    // 无图片：保持原有纯文本逻辑
    result.push(new HumanMessage(text))
  }
  break
}
```

### 6.5 useChat 扩展（CLI 业务层）

```typescript
// submit 签名扩展
const submit = useCallback((text: string, _source: 'cli' | 'web' = 'cli', imageIds?: string[]) => {
  // ...（现有逻辑不变）

  // 构建用户消息：有图片 + vision 启用时用结构化格式
  const hasImages = imageIds && imageIds.length > 0
  const visionEnabled = hasImages && configManager.isVisionEnabled(currentProvider, currentModel)

  if (hasImages && visionEnabled) {
    // 构建结构化内容：文本 + 图片引用
    const content: MessageContent[] = [
      { type: 'text', text },
      ...imageIds.map(id => ({
        type: 'image' as const,
        imageId: id,
        mediaType: 'image/jpeg',  // 前端统一压缩为 JPEG
      })),
    ]
    contextManager.pushUserContent(content)
  } else {
    // 纯文本（无图片 or vision 未启用）
    contextManager.pushUser(text)

    // vision 未启用但有图片 → 发一条提示事件给 Web
    if (hasImages && !visionEnabled) {
      eventBus.emit({
        type: 'system_notice' as any,   // 或用现有的 appendSystemMessage
        message: `当前模型 ${currentModel} 未启用图片理解，${imageIds.length} 张图片已忽略`,
      })
    }
  }

  // ...（后续 AgentLoop 启动逻辑不变）
})

// EventBus 监听扩展：接收 imageIds
useEffect(() => {
  const off = eventBus.onType('user_input', (event) => {
    if (event.source === 'web' && event.text !== '__abort__') {
      submit(event.text, 'web', event.imageIds)  // ← 透传 imageIds
    }
  })
  return off
}, [submit])
```

## 七、前端压缩策略

### 7.1 Canvas 压缩函数

```typescript
// web/src/utils/image-compress.ts

const MAX_DIMENSION = 2048
const JPEG_QUALITY = 0.85

export async function compressImage(blob: Blob): Promise<Blob> {
  const img = await createImageBitmap(blob)
  const { width, height } = img

  // 最长边不超过 MAX_DIMENSION
  const scale = Math.min(1, MAX_DIMENSION / Math.max(width, height))
  const targetW = Math.round(width * scale)
  const targetH = Math.round(height * scale)

  const canvas = new OffscreenCanvas(targetW, targetH)
  const ctx = canvas.getContext('2d')!
  ctx.drawImage(img, 0, 0, targetW, targetH)

  // 统一转 JPEG（截图 PNG 转 JPEG 体积缩小 5-10 倍）
  return canvas.convertToBlob({ type: 'image/jpeg', quality: JPEG_QUALITY })
}
```

### 7.2 体积预估

| 原始 | 压缩后 | base64 | token 估算（Anthropic） |
|------|-------|--------|----------------------|
| 1920×1080 PNG ~2MB | JPEG ~200KB | ~270KB | ~2700 |
| 2560×1440 PNG ~4MB | 2048×1152 JPEG ~300KB | ~400KB | ~3200 |
| 3840×2160 PNG ~8MB | 2048×1152 JPEG ~300KB | ~400KB | ~3200 |

## 九、风险与兜底

| 风险 | 影响 | 兜底 |
|------|------|------|
| base64 过大导致 LLM 请求超时 | 调用失败 | 前端压缩 ≤ 2048px；Provider 层设独立超时 |
| 图片文件被 GC 删除后 LLM 引用 | Provider 读到 null | readImageBase64 返回 null → 静默跳过该 image block |
| 粘贴非图片内容 | 误触发 | paste 事件只处理 `clipboardData.types` 含 `image/*` |
| WS 消息过大 | 传输慢 | imageIds 只是 UUID 字符串数组，不含图片数据，不会大 |
| visionModels 配置不在 models 中 | 逻辑矛盾 | isVisionEnabled 内部检查 `models.includes(model) && visionModels.includes(model)` |
| Session 恢复时图片已过期 | 展示缺失 | 气泡中检查 imageExists()，不存在则显示"图片已过期"占位 |
| LangChain HumanMessage content 数组兼容性 | 格式不对 | 参考 LangChain 文档 `MessageContentComplex` 类型验证 |

## 十、暂不做但备忘的事

### 10.1 图片专有提示词注入（Deferred）

**考虑过的方案**：用户粘贴图片但文字很少时（≤ 20 字），自动在消息中注入一段引导 hint，帮助模型聚焦于"给出行动方案"而非"泛泛描述图片内容"。例如：

```
请仔细分析附带的截图，结合用户描述给出具体可执行的建议。
- UI 界面截图 → 关注布局、组件、样式细节
- 报错截图 → 提取关键错误信息并分析原因
- 设计稿 → 关注与当前代码的差异
```

**决定暂不做的理由**：
- 当前主流模型（Claude Opus/Sonnet、GPT-4o、GLM-4V）的图片理解能力已经足够强，不需要过度引导
- 用户说"这个页面有什么问题"+ 截图，模型自己就知道该分析 UI 问题
- 过度引导反而可能干扰模型的自主判断，限制其发挥
- 注入位置（useChat 构建消息时条件追加）和实现方式已明确，需要时可以快速补上

**触发条件（未来如果要做）**：
- 上线后实测发现模型在图片场景下频繁发散、给出无用描述
- 特定模型（如较弱的视觉模型）需要引导才能给出有效回答
- 用户反馈"发了截图但 AI 不知道我想干什么"

**预留接口**：在 useChat.submit() 构建 MessageContent[] 后，可以插入引导文本块。不需要改架构，只是加一个条件判断。

## 十一、不做的事

- **CLI 端图片输入**：终端无法粘贴图片，仅 Web 端支持
- **图片生成/编辑**：只做理解，不做 DALL-E 类生成
- **OCR 降级**：不支持视觉的模型直接忽略图片，不做 OCR 降级
- **拖拽上传 / 文件选择器**：一期只做粘贴，后续按需扩展
- **外部 URL 图片**：一期只支持本地上传
- **图片缓存/CDN**：本地文件直接读，无需缓存层
- **Provider 层智能降级**：不自动切换到支持 vision 的模型，用户自行选择
