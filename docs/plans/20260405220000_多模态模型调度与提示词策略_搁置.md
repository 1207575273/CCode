### 1.2 发现的问题

模型看懂了图片，但随后**调用 MCP 工具（puppeteer、web_search）去"进一步获取信息"**，而不是直接基于图片内容回答用户。

**debug 日志中的模型思考链**：
```
用户询问当前页面内容，我需要分析页面显示的信息。
从截图可以看到，这是一个智谱AI的BigModel官方网站页面...（正确识别）
我会调用 mcp__puppeteer__puppeteer_evaluate 来获取页面的HTML内容...（不应该）
我会调用 web_search_prime 工具...（不应该）
```

**根因**：system prompt 中的行为引导（"Complete the task fully... use tools"）让模型倾向于"用工具做事"，把图片理解当作信息收集的第一步而不是最终答案。

## 二、待解决的核心问题

### 2.1 多模型组合调度

当前架构中，一个 session 只绑定一个 provider + model。但实际使用场景中：

- **文本推理**：用 glm-5.1（便宜、快、推理强）
- **图片理解**：用 GLM-4.6V-Flash（支持多模态）
- **复杂任务**：用 Claude Opus 4.6（最强综合能力）

**问题**：用户发了图片，但当前 model 是 glm-5.1（不支持视觉），怎么办？

**方案空间**：

| 方案 | 描述 | 复杂度 | 风险 |
|------|------|--------|------|
| A. 手动切换 | 用户自己 /model 切到视觉模型 | 低 | 体验差 |
| B. 自动降级 | 检测到图片 + 当前模型不支持 → 自动切换到同 provider 的视觉模型 | 中 | 需要"视觉降级模型"配置 |
| C. 混合调度 | 图片消息单独用视觉模型处理，结果注入回主模型上下文 | 高 | 上下文拼接复杂 |
| D. 预处理 | 图片先用视觉模型生成描述文本，再把文本发给主模型 | 中 | 丢失图片细节 |

### 2.2 提示词与工程的结合

简单加一句"看到图片就直接回答"不够，因为：

1. 有些模型不支持多模态 → 提示词说"用视觉能力"但模型没有这个能力
2. 图片理解和工具使用不是互斥的 → 用户可能说"看这个截图，帮我修复这个 bug"，需要先看图再调工具
3. 多轮对话中图片可能在历史消息中 → 不是每条消息都有图片

**正确的方向**：提示词应该是**动态的**，根据当前 model 的能力和消息是否含图片来条件注入。

### 2.3 提示词策略草案（待验证）

```typescript
// 在 AgentLoop 或 useChat 构建 system prompt 时
if (currentMessageHasImages && modelSupportsVision) {
  // 注入视觉引导：先用视觉能力分析图片，再决定是否需要工具辅助
  behaviorGuidance += '\n- The user has attached image(s). Analyze them using your vision capability FIRST, then decide if tools are needed for follow-up actions.'
}

if (currentMessageHasImages && !modelSupportsVision) {
  // 注入降级引导：告知模型图片内容不可见
  behaviorGuidance += '\n- Note: The user attached image(s) but your current model does not support vision. Ask the user to describe the image content or switch to a vision-capable model.'
}
```

这个方向需要进一步思考：
- 提示词注入的时机（每轮都注入？只在有图片的轮次注入？）
- 多模型切换的用户体验
- 上下文管理（视觉模型的理解结果如何回传给主模型）
