### 1.3 实际运行测试

```
bun run bin/ccli.ts
→ WelcomeScreen 正常渲染（Ink 渲染能力 OK）
```

**Ink 的 `useInput` hook 在 mount 时调用 `setRawMode(true)`，这是 Ink 框架的硬性要求。**

## 二、核心问题：Bun stdin raw mode

### 2.1 问题本质

Ink 框架（终端 UI）需要 raw mode 来：
- 逐字符接收键盘输入（而非行缓冲）
- 处理特殊按键（Ctrl+C、方向键、Escape 等）
- 实现 `useInput` hook 的按键回调

Bun 1.3 的 `process.stdin` 是一个普通的 `ReadStream`，**没有实现 `setRawMode` 方法**。这是 Bun 对 Node.js TTY 子系统的不完整实现。

### 2.2 Bun 官方状态

- Bun GitHub Issue: `process.stdin.setRawMode is not a function`
- 状态：已知问题，Bun 团队在推进 TTY 完整支持
- Bun 1.4+ 可能修复（需跟踪）

### 2.3 可能的 workaround

| 方案 | 描述 | 可行性 |
|------|------|--------|
| **A. 等 Bun 修复** | 升级到支持 setRawMode 的 Bun 版本 | 高（等就行） |
| **B. Polyfill setRawMode** | 在入口文件中 monkey-patch process.stdin | 中（风险高，可能导致键盘行为异常） |
| **C. 不用 Ink，换 blessed/terminal-kit** | 替换整个 UI 框架 | 低（改动太大） |
| **D. Bun 只跑非交互模式（pipe mode）** | 管道模式不需要 raw mode | 高（作为过渡方案） |

## 三、内存对比（切 Bun 的原始动机）

| 场景 | Node.js v20 | Bun v1.3 | 结论 |
|------|------------|----------|------|
| 空进程 | 30MB | 101MB | Bun 高 3.4× |
| 加载主要依赖 | 92MB | 309MB | Bun 高 3.4× |

**Bun 内存比 Node.js 高 3 倍以上。** 切 Bun 的动机不应该是内存优化，而应该是：
- 原生 TypeScript 执行（省掉 tsx/ts-node 开销）
- 更快的启动速度
- 更好的开发体验（bun run 直接执行 .ts）

## 四、适配工作量评估

### 必须解决（Blocker）
- [ ] Bun stdin raw mode 问题 — 等 Bun 版本更新或实现 polyfill

### 可提前做的兼容准备
- [ ] 入口文件运行时检测：`typeof Bun !== 'undefined'` 判断运行时
- [ ] pipe mode 适配验证：`bun run bin/ccli.ts --pipe` 管道模式是否能跑通
- [ ] 性能基准：Bun 下启动耗时 vs Node.js（Bun 启动通常快 2-3 倍）

### 不需要做的
- npm 依赖全部兼容，无需替换
- Node.js API（除 stdin raw mode 外）全部兼容
- 文件系统、网络、加密等基础模块无问题

---

## 五、建议路径

**短期**：继续用 Node.js + tsx。Bun stdin 问题是框架级 blocker，不值得 workaround。

**中期**：跟踪 Bun 版本更新（1.4+），等 TTY setRawMode 支持后重新评估。

**长期**：如果 Bun TTY 完善，切换只需改启动命令（`bun run` 替代 `tsx`），代码层面零改动。
