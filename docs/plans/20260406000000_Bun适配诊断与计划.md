### 1.3 TTY / stdin raw mode（纠正误判）

**初始误判**：认为 Bun 不支持 `process.stdin.setRawMode`，Ink 无法运行。

**实际验证**：在 Claude Code 的 Bash 工具中运行，Node 和 Bun **完全相同的报错**：

```
Node:  process.stdin.isTTY = undefined, setRawMode = undefined → Ink 报 Raw mode not supported
Bun:   process.stdin.isTTY = undefined, setRawMode = undefined → Ink 报 Raw mode not supported（完全相同的错误）
```

**根因**：Claude Code 的 Bash 工具在子进程中执行命令，stdin 被重定向为管道而非 TTY。这不是 Bun 的问题——**任何运行时在非 TTY 环境下都没有 `setRawMode`**。

用户在真正的终端中用 `pnpm dev`（Node + tsx）正常运行 cCli，说明真正的 TTY 下 Node 有 `setRawMode`。**Bun 在真正的 TTY 中大概率也能正常工作**（Bun 0.8+ 已实现 `setRawMode`）。

**已通过 polyfill 解决**：`bin/bun-compat.ts` 在非 TTY 环境下补齐 stdin 的 `isTTY`、`setRawMode`、`ref`、`unref`，让 Ink 不崩溃。

## 三、内存对比

| 场景 | Node.js v20 | Bun v1.3 | 差距 |
|------|------------|----------|------|
| 空进程 | 30MB | 101MB | Bun 高 3.4× |
| 加载主要依赖 | 92MB | 309MB | Bun 高 3.4× |

**Bun 内存比 Node.js 高 2-3 倍**（JavaScriptCore 的 JIT 策略空间换时间）。切 Bun 的动机应是：
- 原生 TypeScript（省掉 tsx 转译开销）
- **更快的启动速度**（实际体感明显）
- 更好的开发体验（`bun run` 直接执行 .ts）

**不应该是内存优化。** 但速度提升值得。

### 用户体感

> "用 bun 跑虽然内存大，但速度确实挺快的"

## 三-B、Bun 包管理器速查

Bun 自带包管理器，兼容 npm/pnpm 的 `package.json`。

```bash
# 安装依赖（等同 pnpm install）
bun install

# 添加/删除包
bun add express
bun add -d vitest        # devDependencies
bun remove express

# 运行脚本
bun run dev
bun run build
```

| 对比 | pnpm | bun |
|------|------|-----|
| lockfile | `pnpm-lock.yaml` | `bun.lockb`（二进制） |
| node_modules | 内容寻址 store + 硬链接 | 原生扁平结构 |
| 安装速度 | 快 | **更快**（通常 2-5 倍） |
| 兼容性 | 完全兼容 npm 生态 | 99% 兼容 |

**当前项目不需要切换包管理器**——`bun run` 直接复用 pnpm 装好的 `node_modules`，两者的 `node_modules` 结构互相兼容，无需 `bun install` 重装。

## 四、适配计划

### 用户需要做的验证（优先）

在真正的终端中执行：
```bash
cd D:\a_dev_work\claude_cli_z01\cCli
bun tests/integration/bun-tty-check.ts
```

**结果 A**：`isTTY=true, setRawMode=function` → ✅ 直接可用，零代码改动
**结果 B**：`isTTY=true, setRawMode=function` 但按键行为异常 → 需要处理 Windows raw mode 差异
**结果 C**：`isTTY=undefined` → Bun Windows TTY 实现不完整，需等版本更新

### 如果结果 A（最乐观）

切换只需改启动命令：
```json
// package.json
"scripts": {
  "dev": "bun run bin/ccli.ts",          // 替代 tsx bin/ccli.ts
  "dev:web": "bun run bin/ccli.ts --web"
}
```

### 如果结果 B（需要适配）

可能的问题和处理：
- 方向键序列不同 → InputBar stdin raw listener 需适配
- Ctrl+C 行为差异 → App.tsx 退出逻辑需验证
- Home/End 键 → 已有跨平台兼容代码

### 如果结果 C（等版本）

跟踪 Bun 版本更新，等 Windows TTY 完善后重新评估。
