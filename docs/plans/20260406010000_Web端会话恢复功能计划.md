# Web 端会话恢复功能计划

> 创建时间：2026-04-06 01:00
> 目标：在 Web 端历史会话列表和详情页增加"恢复对话"按钮，等同于 CLI 的 /resume 命令

---

## 一、现状分析

### 1.1 当前 Web 端历史会话的交互

```
对话历史列表（/conversations）
  ├── 每条 session：sessionId + 首条消息预览 + 时间
  ├── 点击 → 跳转到详情回放页（/conversations/{id}）
  └── 详情页：只读回放，播放/暂停/速度控制
       └── 无法恢复对话继续聊天 ❌
```

### 1.2 CLI 端 /resume 的完整流程

```
用户输入 /resume
  → ResumePanel 展示历史 session 列表
  → 用户选择一个 session
  → useChat.loadSession(sessionId) 执行：
      1. sessionStore.loadMessages() → 恢复消息快照
      2. sessionLogger.bind() → 绑定日志
      3. contextManager.restoreHistory() → 恢复 LLM 上下文
      4. sessionStore.append(session_resume) → 标记恢复点
      5. updateBridgeSession(sessionId) → 通知 Web 端
  → CLI 进入该 session 的对话状态，可继续提问
```

### 1.3 缺什么

Web 端没有触发 `loadSession` 的入口。需要：
1. Web → CLI 发送"请恢复到这个 session"的指令
2. CLI 执行 loadSession
3. Web 跳转到 `/session/{sessionId}` 实时聊天页

---

## 二、方案设计

### 2.1 通信机制

复用现有的 Bridge WebSocket 通道，新增一个事件类型：

```
Web 端                    Bridge                   CLI 端
  │                         │                        │
  │ { type:'resume',        │                        │
  │   sessionId:'xxx' }     │───── 路由到 CLI ──────→│
  │                         │                        │ useChat.loadSession()
  │                         │                        │
  │                         │←── session_init ───────│ (恢复完成，推送新 session 数据)
  │←── session_init ────────│                        │
  │                         │                        │
  │ 跳转到 /session/{id}    │                        │
```

### 2.2 改动点

#### 层 1：类型定义

**web/src/types.ts** — ClientMessage 新增 resume 类型：
```typescript
| { type: 'resume'; sessionId: string }
```

**src/core/event-bus.ts** — BridgeEvent 新增：
```typescript
| { type: 'resume_session'; sessionId: string; source: 'web' }
```

#### 层 2：Bridge 透传

**src/server/bridge/client.ts** — handleBridgeMessage 新增 case：
```typescript
case 'resume':
  eventBus.emit({
    type: 'resume_session',
    sessionId: String(msg['sessionId'] ?? ''),
    source: 'web',
  })
  break
```

#### 层 3：CLI 执行恢复

**src/ui/useChat.ts** — 监听 resume_session 事件：
```typescript
useEffect(() => {
  const off = eventBus.onType('resume_session', (event) => {
    if (event.source === 'web' && event.sessionId) {
      loadSession(event.sessionId)
    }
  })
  return off
}, [loadSession])
```

loadSession 内部已有 `updateBridgeSession(sessionId)`，恢复完成后 Bridge Server 会自动向 Web 端推送 session_init。

#### 层 4：Web UI — 历史列表加"恢复"按钮

**web/src/pages/ConversationsPage.tsx** — ConversationList 列表项：

```tsx
{/* 每条 session 右侧加恢复按钮 */}
<button
  onClick={(e) => {
    e.stopPropagation()
    send({ type: 'resume', sessionId: s.sessionId })
    navigate(`/session/${s.sessionId}`)
  }}
  className="px-2 py-1 text-xs bg-blue-600 text-white rounded hover:bg-blue-500"
  title="恢复此对话"
>
  恢复
</button>
```

**问题**：ConversationsPage 当前没有 WebSocket 连接（它用 HTTP API 获取数据）。需要通过 API 端点中转，或直接在列表页也建立 WS 连接。

**更简单的方案**：用 HTTP API 触发恢复。

#### 层 4-B（替代方案）：HTTP API 触发恢复

**src/server/dashboard/api.ts** — 新增端点：
```typescript
api.post('/session/resume', async (c) => {
  const { sessionId } = await c.req.json() as { sessionId: string }
  // 通过 eventBus 通知 CLI 恢复
  eventBus.emit({ type: 'resume_session', sessionId, source: 'web' })
  return c.json({ success: true })
})
```

Web 端调用：
```typescript
await apiPost('/api/session/resume', { sessionId })
navigate(`/session/${sessionId}`)
```

**优势**：ConversationsPage 不需要 WS 连接，直接 HTTP POST 即可。

#### 层 5：Web UI — 详情页顶部加"恢复"按钮

**web/src/pages/ConversationsPage.tsx** — ConversationDetail 回放控制栏：

```tsx
{/* 回放控制栏最左侧加恢复按钮 */}
<button
  onClick={async () => {
    await apiPost('/api/session/resume', { sessionId: id })
    navigate(`/session/${id}`)
  }}
  className="px-3 py-1.5 bg-green-600 text-white text-sm rounded hover:bg-green-500"
>
  恢复对话
</button>
```

#### 层 6：ChatPage 跳转后等待恢复

ChatPage 跳转到 `/session/{id}` 后，WebSocket 连接会触发 Bridge 的 `session_init`。如果 CLI 已恢复到该 session，消息会自动推过来。如果 CLI 还没恢复（异步延迟），ChatPage 等待 `session_init` 即可（已有此逻辑）。

## 三、交互流程

### 3.1 从列表页恢复

```
用户在 /conversations 列表页
  → 点击某条 session 右侧的 [恢复] 按钮
  → POST /api/session/resume { sessionId }
  → 页面跳转到 /session/{sessionId}
  → CLI 执行 loadSession → 恢复消息/上下文/provider
  → Bridge 推送 session_init → ChatPage 渲染历史消息
  → 用户可以继续输入提问
```

### 3.2 从详情回放页恢复

```
用户在 /conversations/{id} 详情回放页
  → 看了回放，想继续这个对话
  → 点击顶部控制栏的 [恢复对话] 按钮
  → POST /api/session/resume { sessionId }
  → 页面跳转到 /session/{sessionId}
  → 后续同上
```

## 五、风险

| 风险 | 影响 | 兜底 |
|------|------|------|
| CLI 未运行，Web 发 resume 无人处理 | 恢复失败 | API 返回前检查 CLI 是否连接，未连接则提示"请先启动 CLI" |
| 恢复耗时长（大 session） | 页面等待 | ChatPage 已有 loading 状态，等 session_init 即可 |
| 当前正在 streaming | loadSession 中断流 | loadSession 内部先 abort 再恢复 |
| session 文件已删除（GC） | 加载失败 | loadSession 已有 try-catch，前端显示错误提示 |
