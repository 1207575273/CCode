## 1. 背景与问题分析

### 1.1 现状

当前 SubAgent 在 Web 端有两种展示入口：

| 入口 | 位置 | 体验问题 |
|------|------|---------|
| `SubAgentCard` | 消息流内嵌（`MessageBubble` 中） | 埋没在消息流中，需要滚动定位 |
| `ToolStatus` | 对话底部工具状态区 | 仅显示 `dispatch_agent` 的总耗时，无内部细节 |

CLI 端已有 `SubAgentPanel`（Ctrl+B 呼出悬浮面板），提供列表 + 详情双层视图，但 Web 端缺少等效体验。

### 1.2 核心痛点

1. **信息割裂** — SubAgent 事件散落在消息流中，无法一览全局（谁在跑、谁完成、谁出错）
2. **后台模式盲区** — `run_in_background=true` 的 SubAgent 几乎无视觉反馈，用户感知不到它在工作
3. **缺乏常驻入口** — 没有一个随时可打开的面板来查看 SubAgent 实时状态

### 1.3 影响范围

```
受影响用户场景：
  主 Agent 派发 2~5 个并行 SubAgent → 用户想实时追踪每个 Agent 的进度
  → 当前只能在消息流中逐个翻找 SubAgentCard → 效率低、不直观
```

---

## 2. 目标与非目标

### 2.1 目标

- 在 Web 端新增**悬浮按钮 + 右侧抽屉面板**，提供全局常驻的 SubAgent 状态视图
- 实时展示所有 SubAgent 的状态、进度、事件流（工具调用、文本输出、错误）
- 复用现有 WebSocket 事件流，**零后端改动**
- 与消息流中原有的 `SubAgentCard` 并存，互不干扰

### 2.2 非目标（本期不做）

- 不做 SubAgent 的中断/取消操作（后端无对应接口）
- 不做 SubAgent 历史回放（已有 JSONL 持久化，后续可按需实现）
- 不做虚拟滚动（SubAgent 通常 < 10 个，性能无压力）

## 3. 现有架构分析

### 3.1 数据链路（已有，无需修改）

```
dispatch_agent (后端)
  ├─ AgentLoop.run() → yield AgentEvent
  ├─ consumeAgentEvent() → 内存 store (SubAgentState)
  ├─ subLogger → 独立 JSONL (持久化)
  └─ eventBus.emit('subagent_progress' / 'subagent_event' / 'subagent_done')
       ↓
  BridgeClient (CLI 端 WebSocket 客户端)
    → ws.send({ type: 'event', payload })
       ↓
  Bridge Server (Hono WebSocket 路由)
    → 广播给同 session 的所有 Web 客户端
       ↓
  ChatPage.handleServerEvent (Web 端)
    → setSubAgents(prev => ...) → Map<string, SubAgentInfo>
```

### 3.2 已有事件类型（完备）

| 事件 | 用途 | 推送时机 |
|------|------|---------|
| `subagent_progress` | 进度更新（turn、currentTool） | 每次 LLM 调用/工具开始时 |
| `subagent_event` | 详细事件（tool_start/tool_done/text/error） | 每次 SubAgent 内部产生事件时 |
| `subagent_done` | 完成通知（success/failure） | SubAgent 执行结束时 |

### 3.3 前端已有状态

`ChatPage.tsx` 已维护 `subAgents: Map<string, SubAgentInfo>`，其中 `SubAgentInfo` 包含：

```typescript
// web/src/components/SubAgentCard.tsx 中定义
interface SubAgentInfo {
  agentId: string
  name?: string
  agentType?: string
  description: string
  status: 'running' | 'done' | 'error'
  turn: number
  maxTurns: number
  currentTool?: string
  events: SubAgentDetailEvent[]  // 完整事件流
}
```

**结论：数据层完全就绪，只需在 UI 层新增消费端。**

## 4. 技术方案

### 4.1 架构概览

```
┌────────────────────────────────────────────────────────┐
│  ChatPage                                               │
│                                                         │
│  ┌──────────────────────────────┐                       │
│  │                              │    ┌────────────────┐ │
│  │     消息流区域                 │    │ SubAgentDrawer │ │
│  │  (MessageBubble + Card)      │    │ (右侧抽屉)      │ │
│  │                              │    │                │ │
│  │                              │    │  ┌ Agent 1 ─┐ │ │
│  │                              │    │  │ ✓ done   │ │ │
│  │                              │    │  ├ Agent 2 ─┤ │ │
│  │                              │    │  │ ⟳ running│ │ │
│  │                              │    │  └──────────┘ │ │
│  └──────────────────────────────┘    └────────────────┘ │
│                                                         │
│                                              [🤖 FAB]   │
│  ┌──────────────────────────────────────────────────┐   │
│  │ StatusBar                                         │   │
│  └──────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────┘
```

### 4.2 类型复用策略

**问题**：`SubAgentInfo` 和 `SubAgentDetailEvent` 当前定义在 `SubAgentCard.tsx` 中，Drawer 组件需要复用。

**方案**：将类型提取到 `web/src/types.ts`（已有的全局类型文件），`SubAgentCard` 和 `SubAgentDrawer` 都从 `types.ts` 导入。这是低风险的纯重构，不影响运行时行为。

```typescript
// web/src/types.ts 新增
export interface SubAgentInfo {
  agentId: string
  name?: string
  agentType?: string
  description: string
  status: 'running' | 'done' | 'error'
  turn: number
  maxTurns: number
  currentTool?: string
  events: SubAgentDetailEvent[]
}

export interface SubAgentDetailEvent {
  type: 'tool_start' | 'tool_done' | 'text' | 'error'
  toolName?: string
  durationMs?: number
  success?: boolean
  resultSummary?: string
  text?: string
  error?: string
}
```

### 4.3 组件设计

#### 4.3.1 文件变更清单

| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `web/src/types.ts` | **修改** | 提取 `SubAgentInfo`、`SubAgentDetailEvent` 类型定义 |
| `web/src/components/SubAgentCard.tsx` | **修改** | 改为从 `types.ts` 导入类型（删除本地定义） |
| `web/src/components/SubAgentDrawer.tsx` | **新增** | 悬浮按钮 + 抽屉面板 |
| `web/src/pages/ChatPage.tsx` | **修改** | 导入并挂载 `SubAgentDrawer`，传入 `subAgents` |

**不需要改动的文件：**

- 后端所有文件（Bridge Server、dispatch-agent、event-bus、store） — 事件协议已完备
- WebSocket 协议类型（`types.ts` 中 `ServerEvent`） — 已有 `subagent_*` 三种事件
- CLI 端 — 独立的终端 UI，不受影响

#### 4.3.2 SubAgentDrawer 接口

```typescript
interface SubAgentDrawerProps {
  /** 所有 SubAgent 实时状态（来自 ChatPage 的 subAgents state） */
  agents: Map<string, SubAgentInfo>
}
```

#### 4.3.3 组件内部结构

```
SubAgentDrawer
│
├── FAB 悬浮按钮 (fixed, right-bottom)
│   ├── 显示条件：agents.size > 0
│   ├── 视觉：机器人图标 + 右上角 badge（运行中数量）
│   ├── 动画：有 running agent 时边框脉冲 animate-pulse
│   └── 点击 → 切换抽屉开关
│
└── Drawer 抽屉面板 (fixed, right-side, w-96)
    ├── 遮罩层 bg-black/40（点击关闭）
    ├── 标题栏："SubAgents (N)" + 关闭按钮
    ├── Agent 列表（按 startedAt 排序，running 置顶）
    │   ├── Agent 行：状态图标 + type badge + name + 耗时 + turn
    │   ├── 点击展开 → 事件流（复用 SubAgentCard 的事件渲染逻辑）
    │   └── 运行中 agent 有 Spinner 动画
    └── 空状态：仅 session_init 恢复的已完成 agent（可折叠）
```

### 4.4 UI 规格

#### FAB 悬浮按钮

| 属性 | 值 |
|------|---|
| 定位 | `fixed right-6 bottom-20`（在 StatusBar 正上方） |
| 尺寸 | `48×48px` 圆形 |
| 背景 | `bg-gray-800/90 border border-gray-600` |
| 图标 | 🤖 emoji（零依赖，无需引入图标库） |
| Badge | 右上角红色圆点 + 数字（运行中数量） |
| 动画 | 有 running agent 时 `animate-pulse` |
| 隐藏 | `agents.size === 0` 时 `hidden` |
| z-index | `z-50`（高于对话内容，低于 PermissionCard 的 z-40） |

#### Drawer 抽屉面板

| 属性 | 值 |
|------|---|
| 定位 | `fixed top-0 right-0 h-full` |
| 宽度 | `w-96`（384px），移动端 `w-full` |
| 背景 | `bg-gray-900 border-l border-gray-700` |
| 遮罩 | `bg-black/40` 覆盖主内容区 |
| 动画 | `transform translate-x` + `transition-transform duration-300 ease-in-out` |
| z-index | `z-40`（遮罩 z-30） |
| Agent 行高 | 折叠态 ~56px，展开态自适应（max-h-80 overflow-y-auto） |
| 排序 | running 置顶 → done → error，同状态按出现顺序 |

### 4.5 交互矩阵

| 场景 | FAB | Drawer | 说明 |
|------|-----|--------|------|
| 无 SubAgent | 隐藏 | - | 不展示任何入口 |
| 首个 SubAgent 注册 | 淡入 + badge `1` | - | 用户感知到有 Agent 在工作 |
| 多个 SubAgent 运行中 | badge 显示 running 数 | 实时更新 | 点击 FAB 打开抽屉查看全部 |
| Agent 运行中（抽屉内） | - | 黄色 `⟳` + 脉冲 + 当前工具名 | 与 CLI SubAgentPanel 一致的视觉语言 |
| Agent 完成（抽屉内） | - | 绿色 `✓` + 耗时 | 点击可展开查看结果 |
| Agent 出错（抽屉内） | - | 红色 `✗` + 错误摘要 | 点击可展开查看错误详情 |
| 点击 Agent 行 | - | 展开/折叠事件流 | 一次只展开一个（手风琴模式） |
| 点击遮罩 / 关闭按钮 | - | 关闭抽屉 | 同步 FAB 状态 |
| 全部完成 | FAB 保留（灰色 badge） | 可查看历史 | 切换会话后清空 |

### 4.6 状态管理

**不引入新状态管理方案**，直接 props drilling：

```typescript
// ChatPage.tsx 改动
import { SubAgentDrawer } from '../components/SubAgentDrawer'

// JSX 末尾新增（在 StatusBar 之后、MemoryPanel 之前）
<SubAgentDrawer agents={subAgents} />
```

SubAgentDrawer 内部自管理两个状态：
- `open: boolean` — 抽屉开关
- `expandedId: string | null` — 当前展开的 Agent ID

### 4.7 性能考量

| 关注点 | 分析 | 方案 |
|--------|------|------|
| 事件高频推送 | SubAgent 事件通过 `setSubAgents` 更新整个 Map | 抽屉关闭时仅渲染 FAB badge 数字，不渲染事件列表 |
| Map 引用变更 | ChatPage 每次 `setSubAgents` 都创建新 Map | React 的 props 浅比较足够（Map 引用变了才重渲染） |
| 大量事件累积 | 长时间运行的 SubAgent 可能累积数百条事件 | 事件列表虚拟滚动暂不需要（max-h-80 截断 + overflow-y-auto） |
| 遮罩层性能 | 半透明遮罩可能触发 GPU 合成 | 使用 `backdrop-blur-sm` 替代纯黑半透明，视觉效果更好且现代浏览器已优化 |

### 4.8 移动端适配

| 场景 | 处理 |
|------|------|
| 屏幕 < 768px | 抽屉宽度从 `w-96` 切换为 `w-full`（全屏覆盖） |
| FAB 位置 | `right-4 bottom-20`，避免被手势导航遮挡 |

## 5. 改动影响分析

### 5.1 风险矩阵

| 风险 | 概率 | 影响 | 兜底方案 |
|------|------|------|---------|
| 类型提取引入编译错误 | 低 | 编译失败 | 提取后立即 `pnpm build` 验证 |
| 抽屉遮挡聊天内容 | 低 | 体验下降 | 抽屉使用 overlay 模式，不挤压主内容区宽度 |
| WebSocket 事件丢包导致状态不同步 | 低 | Drawer 显示过时数据 | 用户可关闭重开抽屉（数据已在 ChatPage state 中） |
| z-index 冲突 | 低 | UI 层叠错乱 | FAB z-50、Drawer z-40，低于 PermissionCard（已有明确层级） |
| MessageBubble 中 SubAgentCard 受影响 | 低 | 消息流中卡片异常 | 类型提取是纯重构，不改组件逻辑 |

### 5.2 回滚方案

全部改动集中在前端 4 个文件，回滚只需 `git revert`，无后端/协议变更。

## 7. 验收标准

| # | 标准 | 验证方式 |
|---|------|---------|
| 1 | 有 SubAgent 时 FAB 出现，无 SubAgent 时隐藏 | 观察有无 dispatch_agent 调用 |
| 2 | FAB badge 数字与实际 running 数量一致 | 派发多个并行 SubAgent 验证 |
| 3 | 点击 FAB 打开抽屉，点击遮罩/关闭按钮关闭 | 手动交互 |
| 4 | 抽屉内实时展示所有 SubAgent 状态和事件 | 观察 WebSocket 推送与抽屉内容同步 |
| 5 | 后台模式 SubAgent 在抽屉中可见且实时更新 | `run_in_background=true` 验证 |
| 6 | Agent 行可展开查看详细事件流 | 点击 Agent 行验证 |
| 7 | 消息流中原有 SubAgentCard 不受影响 | 对比改造前后消息流中的卡片渲染 |
| 8 | `pnpm build` 无编译错误 | 构建验证 |
