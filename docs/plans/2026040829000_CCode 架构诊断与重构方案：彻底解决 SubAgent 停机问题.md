# CCode 架构诊断与重构方案：彻底解决 SubAgent 停机问题

基于对 `CCode` 代码仓库目录结构的深度分析，你目前的架构是标准的手写 Agent 框架。你碰到的“工程优化后 SubAgent 活干得很差、容易停止”的问题，其根源基本可以精确定位到你代码中的以下几个模块：

## 🕵️ 诊断报告：Bug 在哪里？

1. **`cCli/src/core/context-manager.ts` 或 `compact.ts` (上下文裁剪的破坏)** 这几乎是 100% 的案发现场。你在这个文件里做的“工程优化”（截断老消息），极大概率把大模型返回的带有 `tool_calls` 的 `Assistant` 消息，和它对应的 `Tool` 结果消息给**拆散了**。一旦大模型在上下文中看到一个没有结果的 `tool_call`，或者一个没有来源的 `tool` 结果，它就会“智商崩溃”或直接停止输出。
2. **`cCli/src/tools/agent/dispatch-agent.ts` (SubAgent 异常吞噬)** 你的子代理在执行具体 Tool（比如 `edit-file.ts` 或是 `bash.ts`）时，如果发生异常，很可能只在终端打印了错误，却没有把 `Error.stack` 作为一条合法的 `Tool` 消息塞回给 LLM。这就导致大模型在等待工具结果时“卡死”。
3. **`cCli/src/providers/message-converter.ts` (协议适配漏洞)** 你自己手写的格式转换器，在处理嵌套工具调用、多模态消息（图片+文本）、以及 Anthropic 严格的角色交替规则时，难以覆盖所有边缘 Case。

## 🛠️ 重构方案 (不惜代价的彻底修复)

既然决定要改，我们直接引入 Node.js/TypeScript 生态下的 `@langchain/core`（不需要引入庞大的 `langchain` 主包），把脏活累活外包出去。

### 🔪 第一刀：废弃手写转换器，统一消息底座

**涉及文件**：

- 删除或大幅重构 `cCli/src/providers/message-converter.ts`
- 修改 `cCli/src/persistence/session-types.ts`
- 修改 `cCli/src/providers/wrapper.ts`

**动作**： 不再使用自定义的 JSON 结构来表示消息，全面拥抱 LangChain 的 `BaseMessage`。

```
import { HumanMessage, AIMessage, SystemMessage, ToolMessage } from "@langchain/core/messages";

// 你的 session-types.ts 应该直接存储或转换这些标准对象
```

将底层的模型调用直接替换为 `@langchain/openai` 的 `ChatOpenAI` 和 `@langchain/anthropic` 的 `ChatAnthropic`。这样你就可以删掉你手写的协议适配代码，它们会自动处理 `tool_call_id` 的配对问题。

### 🔪 第二刀：重写上下文截断逻辑（保持原子性）

**涉及文件**：

- `cCli/src/core/context-manager.ts`
- `cCli/src/commands/compact.ts`

**核心法则（必须遵守）**：**Tool Calls 是不可分割的原子块！** 在计算 Token 和裁剪历史记录时，如果你要删除某条历史消息：

1. 如果它是一条 `ToolMessage`，你**必须**连同上面那条发起调用的 `AIMessage` 一起删掉。
2. 如果它是一条包含 `tool_calls` 的 `AIMessage`，你**必须**把它后面对应的所有 `ToolMessage` 一起删掉。

**改造伪代码建议**：

```
// context-manager.ts 中的裁剪逻辑示例

```

### 🔪 第三刀：强化 SubAgent 的闭环与错误反馈

**涉及文件**：

- `cCli/src/tools/agent/dispatch-agent.ts`
- `cCli/src/core/agent-loop.ts`

如果 SubAgent 活干得差，通常是因为它处于“盲人摸象”状态。修改 Agent Loop，确保无论发生什么，都要给大模型一个交代。

```
// cCli/src/core/agent-loop.ts 的强化核心
import { ToolMessage } from "@langchain/core/messages";

async function runAgentLoop(messages) {
    while (true) {
        const response = await llm.invoke(messages);
        messages.push(response);
        
        if (!response.tool_calls || response.tool_calls.length === 0) {
            break; // 只有明确没有工具调用时，才退出循环
        }
        
        for (const toolCall of response.tool_calls) {
            let toolResultStr = "";
            try {
                // 执行你的真实工具 (bash, read_file 等)
                const result = await executeTool(toolCall.name, toolCall.args);
                toolResultStr = typeof result === 'string' ? result : JSON.stringify(result);
            } catch (error) {
                // 【救命稻草】绝对不能在这里 throw 终止循环！
                // 必须把大红色的错误堆栈喂给模型，让它知道自己传错参数了
                toolResultStr = `[TOOL EXECUTION ERROR]: ${error.message}\n${error.stack}`;
            }
            
            // 严格配对 ID
            messages.push(new ToolMessage({
                content: toolResultStr,
                tool_call_id: toolCall.id
            }));
        }
    }
}
```

### 🔪 第四刀：完美集成你现有的 JSONL 存储

**涉及文件**：

- `cCli/src/persistence/session-store.ts`
- `cCli/src/observability/session-logger.ts`

你之前提到做了很多自定义的 `jsonl` 格式持久化。直接使用 LangChain 的序列化 API，这会让你在 `session-store.ts` 里的代码精简 70%：

```
import { messageToDict, messagesFromDict } from "@langchain/core/messages";

// 序列化写入 DB/JSONL
const rawDict = messageToDict(langchainMsg);
rawDict.custom_metadata = { 
    token_usage: 120, 
    cost: 0.002, 
    workspace: "/path/to/project" 
}; // 塞入你的自定义监控数据 (对应 token-meter.ts)
writeToJsonl(rawDict);

// 反序列化读取
const dicts = readFromJsonl();
const pureDicts = dicts.map(d => {
    const { custom_metadata, ...rest } = d;
    return rest; // 剥离自定义字段
});
const messages = messagesFromDict(pureDicts); // 完美复原带复杂 tool_calls 的对象
```

## 🎯 总结与预期收益

不要害怕这次重构的代码量（主要集中在 `providers/` 和 `core/` 目录的删减）。当你完成这套改造后：

1. **多模型无缝切换**：你的 `commands/model.ts` 切换 GLM-5 或 Claude 3.5 时，底层协议完全由框架抹平，绝对不会再报格式错误。
2. **SubAgent 智商恢复**：因为 `context-manager.ts` 的原子化裁剪和详细的 Error 反馈机制，SubAgent 能够在一个安全的沙盒里持续试错、自我纠正，再也不会半路“罢工”。
3. **代码大幅瘦身**：抛弃自己维护的 `MessageConverter`，专注于 `CCode` 真正的核心价值：你的插件系统 (`plugin/`) 和技能引擎 (`skills/`)。
