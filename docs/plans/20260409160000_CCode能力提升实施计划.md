### 任务 1.1：工具描述标准化重写 ✅ 已完成（2026-04-09）

**现状问题**：
- 中英混合（task_output 中文，其他英文）
- 缺乏使用示例和边界说明
- 参数约束不够明确（bash timeout 上限、edit_file 唯一性要求等）

**改进方案**：

统一采用 **中文描述 + 结构化格式**（因为主力模型 GLM 对中文理解更好）：

```
[一句话功能说明]

参数：
- param1: 说明（类型，默认值）
- param2: 说明

注意事项：
- 边界条件 1
- 边界条件 2

示例：
- 典型用法描述
```

**逐工具改进清单**：

| 工具 | 当前问题 | 改进要点 |
|------|---------|---------|
| `read_file` | 描述过简 | 补充行号范围参数说明、大文件建议 |
| `write_file` | 无覆盖提示 | 强调"覆盖已有文件"、"优先用 edit_file 修改" |
| `edit_file` | 唯一性约束不够突出 | 突出 old_str 必须唯一、失败时扩大上下文重试 |
| `bash` | 参数列表堆砌 | 分组说明、timeout 上限 600s、后台模式用法 |
| `grep` | 过简 | 补正则语法提示、context 参数用法 |
| `glob` | 过简 | 补常用模式示例（`**/*.ts`、排除 node_modules） |
| `kill_shell` | 英文 | 统一中文 |
| `task_output` | 中英混合 | 统一格式 |
| `dispatch_agent` | 动态描述较长 | 精简三种类型的适用场景对比 |

**文件**：`src/tools/core/*.ts`、`src/tools/ext/*.ts`、`src/tools/agent/dispatch-agent.ts`

### 任务 1.2：按模型系列分策略 System Prompt

**现状问题**：
- 单一 system prompt 服务所有模型
- GLM-5 和 Claude 4 对同一指令的遵循度差异很大
- 弱模型（GLM-4-flash）需要更强的约束，强模型（Claude Opus）需要更少的限制

**改进方案**：

在 `bootstrap.ts` 的 `buildSystemPrompt` 中引入 **模型适配层**：

```typescript
interface PromptStrategy {
  behaviorGuidance: string    // 行为指导（可按模型强度调整措辞）
  toolUsageHints: string      // 工具使用提示（弱模型加更多示例）
  thinkingGuidance: string    // 思维链引导（弱模型需显式引导）
}
```

三种策略：

| 策略 | 适用模型 | 特点 |
|------|---------|------|
| `strong` | Claude 4.x、GPT-4o | 精简指导，信任模型判断力，减少 token 浪费 |
| `standard` | GLM-5、DeepSeek-chat | 标准指导 + 工具使用示例 + 思维链引导 |
| `weak` | GLM-4-flash、小参数模型 | 强约束 + 大量示例 + 步骤分解引导 + 更严格的重复检测 |

**策略匹配方式**：在 `config.json` 的 ProviderConfig 中新增可选字段：

```typescript
promptStrategy?: 'strong' | 'standard' | 'weak'  // 默认 'standard'
```

也可以自动推断：已知模型名 → 自动匹配策略（兜底 standard）。

**文件**：`src/core/bootstrap.ts`、`src/config/config-manager.ts`
**预估**：2 天

### 任务 1.3：思维链隐式引导 ✅ 已完成（2026-04-09）

**现状问题**：
- 当前行为指导只有 5 条通用规则
- 缺乏对常见工作模式的引导（先读后改、先搜后定位、修改后验证）

**改进方案**：

在行为指导中补充 **工作模式引导**（注入 system prompt）：

```markdown
## 工作模式

修改文件前：
1. 先用 read_file 阅读目标文件相关区域
2. 理解现有代码结构和上下文
3. 再用 edit_file 精确修改（而非 write_file 全量覆盖）

定位问题时：
1. 先用 grep 搜索关键词缩小范围
2. 再用 read_file 阅读具体文件
3. 避免盲目猜测文件路径

完成任务后：
1. 如有代码修改，用 bash 运行相关测试或类型检查验证
2. 如修改了配置，确认格式正确
3. 简洁报告完成状态，不重复展示已做的事
```

这类引导对弱模型效果尤其显著（GLM-5 经常跳过读文件直接改）。

**文件**：`src/core/bootstrap.ts`（behaviorGuidance 段）
**预估**：1 天

### 任务 1.4：工具结果超长截断提示 ✅ 已完成（2026-04-09）

**现状问题**：
- `RESULT_SUMMARY_MAX_LENGTH = 200` 用于 UI 展示
- `RESULT_FULL_MAX_LENGTH = 100_000` 用于持久化
- **但返回给 LLM 的 tool_result 没有长度限制**，大文件 grep/read 结果可能塞入数万 token

**改进方案**：

在 `agent-loop.ts` 工具结果回传 LLM 前，增加截断 + 提示：

```typescript
const LLM_RESULT_MAX_CHARS = 30_000  // ~7500 tokens

function truncateForLLM(output: string, toolName: string): string {
  if (output.length <= LLM_RESULT_MAX_CHARS) return output
  const truncated = output.slice(0, LLM_RESULT_MAX_CHARS)
  return truncated + `\n\n[结果已截断，共 ${output.length} 字符，仅展示前 ${LLM_RESULT_MAX_CHARS} 字符。请缩小搜索范围或指定行号范围重试。]`
}
```

按工具类型定制截断策略：
- `grep`：截断后提示"请缩小 pattern 或添加 glob 过滤"
- `read_file`：截断后提示"请使用 lines 参数指定范围"
- `bash`：截断后提示"输出过长，请用 grep/head/tail 过滤"

**文件**：`src/core/agent-loop.ts`
**预估**：0.5 天

### 任务 2.1：Rate Limit 退避 + 网络错误重试

**现状问题**：
- Provider 层无 429 (Rate Limit) 自动重试
- 网络瞬断直接报错终止
- 空 catch 吞掉异常无上下文

**改进方案**：

在 `src/providers/` 层实现通用重试包装：

```typescript
interface RetryConfig {
  maxRetries: number       // 默认 3
  baseDelayMs: number      // 默认 1000
  maxDelayMs: number       // 默认 30000
  retryableErrors: string[] // ['429', 'ECONNRESET', 'ETIMEDOUT', 'socket hang up']
}
```

重试策略：
- 429 → 指数退避（1s → 2s → 4s），读取 `Retry-After` header
- 网络错误（ECONNRESET/ETIMEDOUT）→ 固定间隔重试 2 次
- 5xx → 重试 1 次
- 4xx（非 429）→ 不重试，直接报错

实现位置：在 `anthropic.ts` 和 `openai-compat.ts` 的 `chat()` 方法外层包装，不侵入内部流式逻辑。

**文件**：新建 `src/providers/retry.ts`，修改 `anthropic.ts`、`openai-compat.ts`
**预估**：1.5 天

### 任务 2.2：工具执行错误分析增强

**现状问题**：
- 工具执行失败后 stack trace 直接传给 LLM
- LLM 需要自己理解错误并决定是否重试
- 某些常见错误（文件不存在、权限拒绝）可以给更友好的提示

**改进方案**：

在 `agent-loop.ts` 的工具结果处理中增加 **错误分类 + 建议注入**：

```typescript
function enrichErrorMessage(toolName: string, error: string): string {
  // 文件不存在
  if (error.includes('ENOENT')) {
    return `${error}\n\n[提示: 文件不存在。请先用 glob 或 grep 确认正确路径。]`
  }
  // edit_file 匹配失败
  if (toolName === 'edit_file' && error.includes('not found in file')) {
    return `${error}\n\n[提示: old_str 未匹配。请先用 read_file 查看文件当前内容，确保 old_str 完全一致（包括缩进和空白）。]`
  }
  // bash 超时
  if (error.includes('timed out')) {
    return `${error}\n\n[提示: 命令超时。可以增加 timeout 参数，或用 run_in_background 后台执行。]`
  }
  return error
}
```

**文件**：`src/core/agent-loop.ts`
**预估**：1 天

### 任务 2.3：Git 深度集成

**现状问题**：
- 仅有 commit skill，LLM 通过 bash 调用 git
- 无 diff-aware 上下文（不知道当前有哪些改动）
- 无法快速理解工作区状态

**改进方案**：

**方案 A（轻量级，推荐）**：不新增工具，通过 **SessionStart Hook + System Prompt 注入** 实现：

启动时（或 resume 时）自动执行 git 命令，将结果注入 system prompt 尾部：

```typescript
// hooks 层面，SessionStart 时自动注入
const gitStatus = execSync('git status --short').toString()
const gitBranch = execSync('git branch --show-current').toString()
const gitLog = execSync('git log --oneline -5').toString()

const gitContext = `
## 当前 Git 状态
- 分支: ${gitBranch.trim()}
- 最近提交:
${gitLog}
- 工作区变更:
${gitStatus || '(clean)'}
`
```

这样 LLM 一开始就知道当前分支、最近提交、未提交变更，无需手动 `git status`。

**方案 B（重量级，后期）**：新增 `git_diff` / `git_status` 专用工具，带结构化输出。

先做方案 A，验证效果后再决定是否需要方案 B。

**文件**：`src/core/bootstrap.ts` 或 `src/hooks/` 内置 hook
**预估**：1 天

### 任务 2.4：空 catch 治理

**现状问题**：
- 全项目大量空 catch 块（`catch { /* 静默 */ }`）
- 部分关键路径的错误被吞掉，排查困难

**改进方案**：

分类治理：

| 场景 | 策略 |
|------|------|
| 持久化写入失败（SessionLogger、TokenMeter） | 保持静默，但写入 debug.log |
| Provider API 错误 | 必须传递给 LLM + UI 展示 |
| MCP 连接失败 | 保持静默（已有 status 展示） |
| 配置读写失败 | 写入 debug.log + 降级为默认值 |
| 工具执行异常 | 已正确处理（传递 stack） |

核心改动：将关键路径的空 catch 改为 `catch (e) { dbg(...) }`，非关键路径保持静默。

**文件**：全项目 grep `catch {` 或 `catch (` 逐个审查
**预估**：1 天
