> 日期：2026-04-09
> 关联：`20260409160000_CCode能力提升实施计划.md` 任务 1.4
> 状态：✅ 已完成（2026-04-09）

---

## 一、背景

CCode 的 AgentLoop 核心循环为：

```
LLM 生成 tool_calls → 执行工具 → tool_result 塞入 history → 下一轮 LLM 调用
```

tool_result 的内容会在**后续每一轮 LLM 调用中被完整发送**（作为 history 的一部分）。如果某次工具返回了超长结果（如 `bash` 执行 `npm install` 输出 5 万字符），这段内容会：

1. **每轮重复发送**，累计消耗大量 input token（成本）
2. **挤占上下文窗口**，导致更早触发 compact 压缩
3. **稀释注意力**，模型在海量无关文本中更容易遗漏关键信息

## 二、现状分析

### 2.1 已有的截断/防膨胀机制（6 层防线）

经代码审查，项目已在多个层级建立了防御：

| 层级 | 机制 | 位置 | 阈值 | 保护对象 |
|------|------|------|------|---------|
| L1 工具内部 | read_file 内容截断 | `read-file.ts:35` | 20,000 字符 | LLM 输入 |
| L1 工具内部 | grep 结果条数限制 | `grep.ts:78-84` | 50 条 + `[结果已截断]` | LLM 输入 |
| L1 工具内部 | bash 进程输出缓冲 | `process-tracker.ts:15` | 1MB 环形缓冲（保留尾部） | 内存 |
| L2 AgentLoop | assistant args 精简 | `agent-loop.ts:199` | `summarizeArgs()` 去大段内容 | history token |
| L3 事件层 | tool_done 事件截断 | `agent-loop.ts:513-514` | 200(summary) / 100K(full) | UI / JSONL |
| L4 SubAgent | 继承 history 时截断 | `context-utils.ts:111-135` | tool_result 500字符 / text 1000字符 | 子 Agent 上下文 |

### 2.2 缺口：主 Agent history 的 tool_result

```
L1 工具内部截断 → 结果返回 → ⚠️ 直接塞入 history → 每轮 LLM 调用发送
                                ↑ 这里没有截断
```

代码路径（`agent-loop.ts`）：

**串行工具**（危险工具，逐个执行）— 第 470-479 行：
```typescript
const toolResultText = result.success
  ? result.output       // ← 无长度限制
  : [result.output, result.error].filter(Boolean).join('\n') || 'error'

history.push({
  role: 'user',
  content: [{ type: 'tool_result', toolCallId, result: toolResultText }],
})
```

**并行工具**（安全工具，批量执行）— 第 370-376 行：
```typescript
const toolResults = results.map(pr => ({
  type: 'tool_result',
  toolCallId: pr.toolCallId,
  result: pr.success
    ? pr.output          // ← 无长度限制
    : [pr.output, pr.error].filter(Boolean).join('\n') || 'error',
}))
```

### 2.3 实际风险评估

大部分工具已有 L1 内部截断，真正的风险集中在两个工具：

| 工具 | 风险场景 | 可能输出量 | L1 保护 |
|------|---------|-----------|---------|
| `bash` | `npm install`、`git log --all`、构建日志 | 数万~数十万字符 | 1MB 缓冲（远超 LLM 需要） |
| `task_output` | 长时间后台进程的累积输出 | 数万字符 | 无 |
| `read_file` | — | — | 20K 截断 ✅ |
| `grep` | — | — | 50 条限制 ✅ |
| `glob` | — | 大量文件匹配 | 无限制，但单行路径，实际不大 |
| 其他 | — | — | 输出天然短小 ✅ |

**结论**：需要一个 L2 层的兜底截断，主要防御 `bash` 和 `task_output` 的极端场景。

## 三、设计方案

### 3.1 方案选择

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| A. 各工具内部加截断 | 在 bash.ts、task-output.ts 等工具内部截断 | 可按工具定制提示 | 分散在多处，新工具容易遗漏 |
| B. AgentLoop 统一兜底 | 在 history.push 前统一截断 | 一处修改，覆盖全部工具（含 MCP/Plugin 动态工具） | 截断提示不够工具专属 |
| **C. AgentLoop 兜底 + 工具感知提示** | 统一截断函数，按工具名生成不同提示 | 兼顾覆盖面和提示质量 | 略复杂，但可控 |

**选择方案 C**：统一截断点 + 工具感知提示。

关键理由：
- MCP 和 Plugin 动态注册的工具**无法在工具内部预先截断**，只有 AgentLoop 层能兜底
- 不同工具的"缩小范围"方式不同，工具感知提示能引导模型更有效地重试

### 3.2 截断阈值

```
LLM_RESULT_MAX_CHARS = 40_000  (~10,000 tokens)
```

选择 40K 的理由：
- read_file 已限制 20K，grep 已限制 50 条（通常 < 5K）— 这些不会触发兜底截断
- bash 正常输出（测试结果、编译日志）通常 < 10K — 也不会触发
- 只有异常长输出（npm install 全量日志、git log --all 等）才会触发
- 40K ≈ 10K tokens，占 128K 上下文窗口的 ~8%，可接受

### 3.3 工具感知截断提示

截断时追加的提示按工具名差异化，引导模型用正确方式缩小范围：

| 工具 | 截断后追加的提示 |
|------|----------------|
| `bash` | `请用 grep/head/tail 过滤输出，或拆分为更小的命令` |
| `task_output` | `输出过长，请用 bash 配合 grep/tail 过滤，或等任务结束后获取关键输出` |
| `grep` | `请缩小 pattern 范围或用 glob 指定更精确的搜索路径` |
| `read_file` | `请使用行号范围参数读取特定区域` |
| 其他/MCP/Plugin | `结果过长已截断，请尝试缩小查询范围` |

### 3.4 修改点

仅修改 `src/core/agent-loop.ts` 一个文件：

1. **新增常量** `LLM_RESULT_MAX_CHARS = 40_000`
2. **新增函数** `truncateForLLM(output: string, toolName: string): string`
3. **串行入口**（~470 行）：`toolResultText` 经过 `truncateForLLM()` 后再塞入 history
4. **并行入口**（~374 行）：`pr.output` 经过 `truncateForLLM()` 后再塞入 tool_result

### 3.5 函数设计

```typescript
/** 回传 LLM 的工具结果最大字符数 */
const LLM_RESULT_MAX_CHARS = 40_000

/** 工具专属的截断后引导提示 */
const TRUNCATION_HINTS: Record<string, string> = {
  bash: '请用 grep/head/tail 过滤输出，或拆分为更小的命令',
  task_output: '输出过长，请用 bash 配合 grep/tail 过滤关键信息',
  grep: '请缩小 pattern 范围或指定更精确的搜索路径',
  read_file: '请指定行号范围读取特定区域',
}
const DEFAULT_HINT = '结果过长已截断，请尝试缩小查询范围'

/**
 * 截断工具结果后塞入 LLM history。
 * 
 * 设计要点：
 * - 只在结果超过阈值时才截断（大部分工具内部已有限制，不会触发）
 * - 截断提示按工具名差异化，引导模型用正确方式重试
 * - 保留原始长度信息，让模型知道丢了多少
 */
function truncateForLLM(output: string, toolName: string): string {
  if (output.length <= LLM_RESULT_MAX_CHARS) return output
  const hint = TRUNCATION_HINTS[toolName] ?? DEFAULT_HINT
  return output.slice(0, LLM_RESULT_MAX_CHARS) +
    `\n\n[结果已截断：共 ${output.length} 字符，仅保留前 ${LLM_RESULT_MAX_CHARS} 字符。${hint}]`
}
```

## 四、影响面评估

### 4.1 不受影响的已有机制

| 机制 | 影响 | 原因 |
|------|------|------|
| read_file 20K 截断 | 无 | 20K < 40K，兜底截断不会触发 |
| grep 50 条限制 | 无 | 50 条通常 < 5K 字符 |
| write_file 成功引导 | 无 | 输出固定一句话 |
| SubAgent context-utils 截断 | 无 | SubAgent 继承 history 时有独立截断，两层互不干扰 |
| tool_done 事件截断 | 无 | 事件截断在 history.push 之后，各自独立 |
| PostToolUse hook stdin | 无 | hook 用的是 `truncate(result.output, 1000)`，独立截断 |

### 4.2 会受影响的场景

| 场景 | 截断前 | 截断后 | 用户感知 |
|------|--------|--------|---------|
| `bash npm install` 输出 8 万字符 | 全量塞入 history | 保留前 4 万 + 截断提示 | LLM 回复更快、更准确 |
| `task_output` 后台长日志 | 全量塞入 | 保留前 4 万 + 引导提示 | LLM 不再被日志淹没 |
| 正常工具调用（< 40K） | 无变化 | 无变化 | 无感知 |

### 4.3 风险

| 风险 | 级别 | 缓解 |
|------|------|------|
| 截断丢失关键信息（如编译错误在第 40001 字符之后） | 低 | bash 进程 tracker 保留尾部（环形缓冲），错误通常在输出末尾；截断提示引导模型用 tail 读取尾部 |
| MCP 工具返回超大 JSON | 低 | 截断后 JSON 不完整，但模型会看到截断提示知道要缩小范围 |
