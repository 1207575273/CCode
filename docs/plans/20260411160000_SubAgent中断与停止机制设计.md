# SubAgent 停止机制设计

> 日期：2026-04-11（v5.0 — 一二期已实施）
> 状态：**一二期已实施**

---

## 一、背景与动机

### 1.1 现状

当前 SubAgent 一旦启动，除了以下两种隐式退出方式外，**没有任何主动控制手段**：

1. **maxTurns 轮次耗尽** — 自然退出
2. **父 Agent abort 传播** — 用户 Ctrl+C 中止主 Agent 时，通过 `onParentAbort` 级联中止所有子 Agent

缺失的能力：
- 主 Agent 无法选择性停止某一个子 Agent（只能全局中止）
- 用户无法通过 CLI 面板或 Web 界面停止某个正在运行的子 Agent
- 停止后没有结构化的汇报机制 — 主 Agent 不知道子 Agent 做到了哪里、为什么被终止

### 1.2 架构层面的根本问题

`dispatch-agent.ts` 的 `stream()` 方法中，**SubAgent 的 AbortController 和 AgentLoop 引用被埋在局部作用域里，外部完全无法触及**：

```typescript
// 当前：控制句柄全部是局部变量，外部不可达
async *stream(args, ctx) {
  const subController = new AbortController()  // ← 局部，外部拿不到
  const subLoop = new AgentLoop(...)            // ← 局部，外部拿不到
  for await (const event of subLoop.run(...)) { ... }
}
```

需要重构：将控制句柄从 `stream()` 局部作用域提升到全局可达的注册表中。

### 1.3 Claude Code 官方做法参考

Claude Code 官方的 SubAgent 模型是**原子性执行**：
- 子 Agent 启动后跑到完成，主 Agent 拿到最终结果
- 没有任何中途控制机制（没有 stop、没有 timeout）
- 唯一的"控制"是通过 `tools` 限制工具集来收窄行为范围
- 如果需要更精细的控制，官方建议用 Agent Teams（lead 可以"请求"teammate 停止）

**我们的差异化目标**：在官方原子性模型的基础上，增加"拉回来"的能力 — 弱模型会跑偏，后台 SubAgent 会浪费 token，用户需要及时止损。

## 二、核心概念：stop + 宽限期自动升级

### 2.1 设计思路

用户只需要一个概念：**stop** — "停下来，把结果告诉我"。

系统内部自动处理两个阶段：

```
stop(agentId, reason)
  │
  ▼
┌──────────────────────────────────┐
│  阶段一：优雅停止（graceful）     │
│                                   │
│  设置 AgentLoop 退出标志          │
│  等待当前 turn 完成（LLM + 工具） │
│  → 干净退出，当前轮结果完整       │
└──────────────┬───────────────────┘
               │
               │ 宽限期内退出了？
               │
          ┌────┴────┐
          │ Yes     │ No（超时）
          ▼         ▼
       干净退出   ┌───────────────────────────────┐
                  │  阶段二：强制中断（forced）     │
                  │                                │
                  │  abortController.abort()        │
                  │  → 打断当前操作，收集部分结果    │
                  └────────────────────────────────┘
               │
               ▼
         ┌───────────────┐
         │  StopReport   │
         │  汇报给主Agent │
         └───────────────┘
```

类比 Kubernetes pod 终止：先发 SIGTERM（宽限期内优雅退出），超时后 SIGKILL。

### 2.2 为什么只要一个概念

| 对比 | 两个概念（stop + break） | 一个概念（stop + 自动升级） |
|------|-------------------------|---------------------------|
| 用户心智 | 需要区分"优雅"和"立即" | 只需要"停下来"，系统处理升级 |
| LLM 工具调用 | 需要判断调哪个，可能选错 | 只有一个动作，不可能选错 |
| UI | 两个按钮，用户犹豫 | 一个按钮 |
| 实现 | 两套路径 | 一套路径 + 一个定时器 |

### 2.3 状态机

新增 `stopping` 和 `stopped` 两个状态：

```
                     ┌──────────┐
                     │ running  │
                     └────┬─────┘
           ┌──────────────┼──────────────┬───────────────┐
           │              │              │               │
           │              ▼              │               │
           │       ┌───────────┐         │               │
           │       │ stopping  │ 宽限期   │               │
           │       └─────┬─────┘         │               │
           │         ┌───┴────┐          │               │
           │         ▼        ▼          ▼               ▼
        ┌──────┐  ┌──────┐ ┌──────┐  ┌───────┐
        │ done │  │stopped│ │stopped│  │ error │
        └──────┘  │(优雅) │ │(强制) │  └───────┘
                  └──────┘ └──────┘
```

```typescript
type SubAgentStatus = 'running' | 'stopping' | 'stopped' | 'done' | 'error'
```

- `stopping` — 已收到 stop 指令，等待当前轮结束（宽限期倒计时中）
- `stopped` — 已终止（不区分优雅/强制，对外统一；`StopReport.resolution` 内部记录区分）

### 2.4 宽限期设计

```typescript
/** 默认宽限期（毫秒） */
const DEFAULT_GRACE_PERIOD_MS = 30_000  // 30 秒
```

**30 秒的考量**：

宽限期要覆盖"从设置标志位到 AgentLoop 检查到标志位"这段时间。最坏情况的时间线：

```
t=0s   stopAgent() 调用，设置 #stopRequested = true
       此时 AgentLoop 可能正处于：
       ├─ #callLLM() 中等 LLM 流      → 还需 2-8 秒
       └─ #executeToolCalls() 中执行工具 → 还需 0-20 秒（bash 命令）
t=8s   LLM 流结束
t=20s  工具执行完毕
t=20s  检查 #stopRequested → 优雅退出 ✓
```

15 秒在上述最坏情况下不够（8 + 12 = 20 秒）。30 秒更安全，对用户而言额外等待的感知不大（因为大多数情况 3-10 秒内就能优雅退出）。

可通过 `AgentDefinition.gracePeriodMs` 按类型自定义（如 explore 类型工具轻量，可设更短）。

## 三、触发入口与架构约束

### 3.1 前台 vs 后台的关键约束

**前台模式下主 Agent 无法调用 `control_agent` 工具停止子 Agent。**

原因：前台 `dispatch_agent.stream()` 通过 `yield*` 阻塞了主 AgentLoop：

```
主 AgentLoop.run()
  → for (turn) {
      llmResult = callLLM()
      executeToolCalls()
        → #executeOneTool(dispatch_agent)
           → yield* dispatch_agent.stream()   ← 主 Agent 阻塞在此
           → 子 Agent 跑完才 return
           → 主 Agent 才能进入下一轮调用其他工具
    }
```

主 Agent 的循环被 `dispatch_agent` 占据，不可能"同时"调用另一个工具。

**因此各入口的适用范围**：

| 触发入口 | 前台 SubAgent | 后台 SubAgent |
|----------|:---:|:---:|
| CLI 面板（Ctrl+B → s） | ✅ | ✅ |
| Web 界面（停止按钮） | ✅ | ✅ |
| `control_agent` 工具 | ❌ 主 Agent 被阻塞 | ✅ |
| 父 Agent Ctrl+C | ✅ 但是全局中止 | ✅ 全局中止 |

**设计推论**：
- `control_agent` 工具是**后台 SubAgent 专属**的控制手段，文档和工具 description 需明确标注
- 前台 SubAgent 的停止只能由**外部触发**（CLI/Web 用户）
- 这是当前 `yield*` 串行架构的固有约束，不是 bug

#### 3.2.3 `control_agent` 工具

适用：**仅后台 SubAgent**。

```typescript
export class ControlAgentTool implements Tool {
  readonly name = 'control_agent'
  readonly description = [
    '停止正在后台运行的子 Agent（run_in_background=true 派发的）。',
    '子 Agent 会先尝试优雅退出（完成当前轮），超时后强制中断。',
    '停止后会收到一份报告，包含子 Agent 的部分结果和执行进度。',
    '',
    '注意：此工具只对后台 SubAgent 有效。前台 SubAgent 由用户通过 CLI/Web 面板停止。',
  ].join('\n')

  get parameters() {
    return {
      type: 'object',
      properties: {
        agent_id: {
          type: 'string',
          description: 'agent ID 或 name（从 dispatch_agent 返回的 agentId，或自定义的 name）',
        },
        reason: {
          type: 'string',
          description: '停止原因（将包含在汇报中，帮助后续决策）',
        },
      },
      required: ['agent_id', 'reason'],
    }
  }

  async execute(args: Record<string, unknown>, ctx: ToolContext): Promise<ToolResult> {
    const id = String(args['agent_id'])
    const reason = String(args['reason'] ?? '主 Agent 请求停止')

    const result = stopAgent(id, 'parent_agent', reason)
    if (!result.success) {
      return { success: false, output: '', error: result.error }
    }

    // stopAgent 是异步的（设置标志后返回），此处返回"已发起"
    // 真正的 StopReport 会在后台 SubAgent 退出后通过 subagent_done 事件广播，
    // 主 Agent 可通过 task_output 查询完整报告
    return {
      success: true,
      output: JSON.stringify({
        status: 'stop_requested',
        agentId: id,
        note: '已发起停止请求，子 Agent 将在当前轮结束后退出。用 task_output 查询最终报告。',
      }),
    }
  }
}
```

加入 `ALWAYS_EXCLUDE` — 子 Agent 不能停止其他子 Agent。

## 四、StopReport — 向主 Agent 汇报

### 4.1 为什么汇报是核心设计点

当前 SubAgent 被终止后，主 Agent 拿到的信息只有最终文本。主 Agent 不知道：
- 子 Agent 做到了第几轮？完成了多少？
- 为什么被终止？是用户主动停的还是出错了？
- 谁停的？CLI 用户？Web 用户？主 Agent 自己？

缺少这些信息，主 Agent 无法做出合理的后续决策。

### 4.2 StopReport 结构

```typescript
interface StopReport {
  /** 终止的子 Agent 标识 */
  agentId: string
  name: string
  agentType: string

  /** 终止方式：优雅（当前轮完成后退出）or 强制（abort 中断） */
  resolution: 'graceful' | 'forced'

  /** 谁触发的停止 */
  source: 'user_cli' | 'user_web' | 'parent_agent' | 'timeout'

  /** 为什么终止 */
  reason: string

  /** 执行进度 */
  turn: number
  maxTurns: number

  /** 已有的结果（优雅退出时是完整的当前轮结果，强制中断时是部分结果） */
  partialResult: string
}
```

### 4.3 回传路径（前台 vs 后台）

**前台模式**：

stop 发生时 `dispatch_agent.stream()` 还在 `for await` 循环中。子 AgentLoop yield `done(reason='stopped')` 后循环退出，`stream()` 将 StopReport 编码为返回值，直接作为 `dispatch_agent` 的 ToolResult 传回主 Agent 的 history。

```
主 Agent 调 dispatch_agent(前台)
  → 子 Agent 运行中
  → 用户通过 CLI/Web 触发 stop
  → 子 Agent 优雅退出
  → dispatch_agent 返回 AgentStoppedOutput（含 StopReport）
  → 主 Agent 的 history 中看到这个 tool result
  → 主 Agent 基于 StopReport 做后续决策
```

**后台模式**：

后台 SubAgent 停止后，StopReport 需要两条路径传递：

1. **store 层**：`markSubAgentDone()` 时将 StopReport 存入 `SubAgentState.stopReport` 字段，`task_output` 工具查询时能读到
2. **EventBus**：`subagent_done` 事件的 `output` 字段携带 JSON 序列化的 StopReport

```
主 Agent 调 dispatch_agent(后台) → 立即返回 async_launched
  → 后来调 control_agent 停止 → 返回 stop_requested
  → ... 子 Agent 退出 ...
  → subagent_done 事件广播（含 StopReport）
  → 主 Agent 调 task_output 查询 → 读到完整 StopReport
```

### 4.4 AgentOutput 类型扩展

当前有三种：`completed` / `async_launched` / `error`。新增第四种：

```typescript
interface AgentStoppedOutput {
  status: 'stopped'
  agentId: string
  name: string
  agentType: string
  resolution: 'graceful' | 'forced'
  source: 'user_cli' | 'user_web' | 'parent_agent' | 'timeout'
  reason: string
  turn: number
  maxTurns: number
  partialResult: string
}

type AgentOutput =
  | AgentCompletedOutput
  | AgentAsyncLaunchedOutput
  | AgentErrorOutput
  | AgentStoppedOutput   // 🆕
```

## 五、架构重构：统一状态与控制

### 5.1 store.ts 扩展（而非新增 handle-registry）

**原方案**：新建 `handle-registry.ts`，独立于 `store.ts` 管理控制句柄。

**问题**：两个全局 Map 管同一组 SubAgent，注册/移除要操作两处，状态可能不同步。

**修正方案**：将控制句柄**合并到现有 `store.ts`** 中。store 既管展示状态，也管控制句柄。

```typescript
// store.ts 扩展

/** 控制句柄（仅 running/stopping 状态的 agent 拥有） */
export interface SubAgentControl {
  abortController: AbortController
  loop: AgentLoop
}

export interface SubAgentState {
  // ... 现有字段不变 ...
  status: 'running' | 'stopping' | 'stopped' | 'done' | 'error'

  /** 控制句柄 — running/stopping 时存在，终态后清除 */
  control?: SubAgentControl

  /** stop 请求信息 — stopping/stopped 时存在 */
  stopRequest?: {
    source: 'user_cli' | 'user_web' | 'parent_agent' | 'timeout'
    reason: string
    requestedAt: number
    gracePeriodMs: number
    timer: ReturnType<typeof setTimeout>
  }

  /** 停止报告 — stopped 时存在（供 task_output 读取） */
  stopReport?: StopReport
}
```

**好处**：
- 一个 Map，一处注册，一处查找
- 状态和控制句柄天然同步
- 现有 `listSubAgents()` / `getSubAgent()` 无需改动，UI 代码不变
- `control` 字段仅 running 期间存在，终态后清除释放引用

### 5.2 `stopAgent()` 实现（在 store.ts 中）

```typescript
/** 默认宽限期 */
const DEFAULT_GRACE_PERIOD_MS = 30_000

export function stopAgent(
  idOrName: string,
  source: StopRequest['source'],
  reason: string,
  gracePeriodMs = DEFAULT_GRACE_PERIOD_MS,
): { success: boolean; error?: string } {

  // 按 ID 或 name 查找
  const state = getSubAgent(idOrName) ?? findSubAgentByName(idOrName)
  if (!state) return { success: false, error: 'agent not found' }
  if (state.status !== 'running') {
    return { success: false, error: `agent is ${state.status}, cannot stop` }
  }
  if (!state.control) {
    return { success: false, error: 'agent has no control handle (already finishing)' }
  }

  // 幂等：已在 stopping 中则忽略
  if (state.stopRequest) return { success: true }

  // ── 阶段一：优雅停止 ──
  state.control.loop.requestStop()
  state.status = 'stopping'

  // ── 阶段二预备：宽限期定时器 ──
  const timer = setTimeout(() => {
    if (state.control && !state.control.abortController.signal.aborted) {
      state.control.abortController.abort()
    }
  }, gracePeriodMs)

  state.stopRequest = { source, reason, requestedAt: Date.now(), gracePeriodMs, timer }

  eventBus.emit({
    type: 'subagent_event',
    agentId: state.agentId,
    detail: { kind: 'text', text: `[Stopping: ${reason}]` },
  })

  return { success: true }
}
```

### 5.3 AgentLoop 新增 `requestStop()`

```typescript
export class AgentLoop {
  #stopRequested = false

  /** 外部请求优雅停止 — 当前轮结束后退出循环 */
  requestStop(): void {
    this.#stopRequested = true
  }

  async *run(messages: Message[]): AsyncIterable<AgentEvent> {
    const history = messages

    for (let turn = 0; turn < maxTurns; turn++) {
      // 🆕 检查点 1：新一轮开始前
      if (this.#stopRequested) {
        yield { type: 'done', reason: 'stopped' }
        return
      }

      const llmResult = yield* this.#callLLM(history)
      if (llmResult.aborted) return

      // ... push assistant message, check toolCalls ...

      yield* this.#executeToolCalls(llmResult.toolCalls, history)

      // 🆕 检查点 2：工具执行完毕后（不等下一轮开头）
      if (this.#stopRequested) {
        yield { type: 'done', reason: 'stopped' }
        return
      }
    }

    yield { type: 'done', reason: 'max_turns' }
  }
}
```

**检查点位置的约束**：

不能放在 `#callLLM` 和 `#executeToolCalls` 之间。原因：此时 history 末尾是 `assistant(tool_calls)` 但还没有对应的 `user(tool_results)`。如果此时退出，history 配对关系被破坏，后续回放或 resume 时 LLM API 会报 400。

两个安全的检查点：
1. 循环顶部 — history 末尾是 `tool_result`（上一轮）或初始 `user` 消息
2. `#executeToolCalls` 之后 — `tool_result` 已全部写入 history

### 5.4 dispatch-agent.ts 重构

```typescript
async *stream(args, ctx) {
  // ... 参数解析、创建 subController / subLoop（保持不变）...

  // 🆕 注册控制句柄到 store
  setSubAgentControl(agentId, {
    abortController: subController,
    loop: subLoop,
  })

  try {
    for await (const event of subLoop.run(initialMessages)) {
      // ... 现有事件处理逻辑（保持不变）...
    }

    // 循环正常退出 — 区分"正常完成" vs "优雅停止"
    const state = getSubAgent(agentId)
    if (state?.stopRequest) {
      const report = buildStopReport(state, finalText, currentTurn, 'graceful')
      state.stopReport = report
      markSubAgentDone(agentId, finalText, 'stopped')
      return { success: true, output: JSON.stringify({ status: 'stopped', ...report }) }
    }

    // ... 现有的正常完成逻辑 ...

  } catch (err) {
    const state = getSubAgent(agentId)

    if (isAbortError(err) && state?.stopRequest) {
      // abort 来自宽限期超时（而非 Ctrl+C）
      // 区分方式：检查父 signal 是否 aborted
      const fromParentAbort = ctx.signal?.aborted === true
      if (!fromParentAbort) {
        const report = buildStopReport(state, finalText, currentTurn, 'forced')
        state.stopReport = report
        markSubAgentDone(agentId, finalText, 'stopped')
        return { success: true, output: JSON.stringify({ status: 'stopped', ...report }) }
      }
    }
    // ... 现有的异常处理逻辑 ...

  } finally {
    // 清理
    const state = getSubAgent(agentId)
    if (state?.stopRequest?.timer) {
      clearTimeout(state.stopRequest.timer)
    }
    clearSubAgentControl(agentId)  // 🆕 清除控制句柄（释放引用）
    ctx.signal?.removeEventListener('abort', onParentAbort)
    sessionProvider.dispose?.()
  }
}
```

### 5.5 后台模式的改造

`runSubAgentInBackground()` 需要同样的区分逻辑：

```typescript
function runSubAgentInBackground(...) {
  // 🆕 注册控制句柄
  setSubAgentControl(agentId, { abortController: subController, loop: subLoop })

  void (async () => {
    try {
      for await (const event of subLoop.run(initialMessages)) { ... }

      // 循环正常退出
      const state = getSubAgent(agentId)
      if (state?.stopRequest) {
        const report = buildStopReport(state, finalText, currentTurn, 'graceful')
        state.stopReport = report
        markSubAgentDone(agentId, finalText, 'stopped')
        eventBus.emit({
          type: 'subagent_done', agentId, name: agentName, description,
          success: true,
          output: JSON.stringify({ status: 'stopped', ...report }),
        })
        return
      }
      // ... 正常完成逻辑 ...

    } catch (err) {
      // ... 同前台的 abort 区分逻辑 ...
    } finally {
      const state = getSubAgent(agentId)
      if (state?.stopRequest?.timer) clearTimeout(state.stopRequest.timer)
      clearSubAgentControl(agentId)
      cleanup?.()
      sessionProvider?.dispose?.()
    }
  })()
}
```

## 六、关键设计决策与权衡

### 6.1 为什么不保留 redirect / continue

方向错了应该停掉重新来，而不是"注入一条消息试图纠偏"：
- 注入消息后 LLM 是否真的会转向，取决于模型能力，不可靠
- 停掉 + 重新派发，prompt 完全可控，效果确定
- 少一个概念 = 用户心智负担更低

### 6.2 为什么合并到 store.ts 而非独立 handle-registry

- 一个 Map 管一组 SubAgent → 注册/查找/清理只需一处操作
- 状态和控制句柄天然同步，不存在"store 说 running 但 handle 说 stopping"的不一致
- 现有 `listSubAgents()` / `getSubAgent()` 接口不变，UI 层零改动
- `control` 字段用 optional 限制生命周期 — running 时存在，终态后清除释放引用

### 6.3 stopping 状态的必要性

用户点了 stop 后需要即时反馈"在处理了"：
- CLI 面板显示 `stopping  等待退出(5s)...`
- Web 面板显示倒计时进度
- 避免用户因"没反应"而反复点击

### 6.4 Ctrl+C 与 stop 的并发区分

场景：用户先通过面板 stop 了一个 SubAgent（进入 stopping），宽限期内又按了 Ctrl+C。

两者都触发 `abortController.abort()`，但语义不同：
- 宽限期超时 abort → `stopped`（该 SubAgent 的定向操作）
- Ctrl+C abort → `aborted`（全局操作，用户想退出整个对话）

**区分方式**：catch 路径中检查 `ctx.signal?.aborted`（父 signal）：
- 父 signal aborted = Ctrl+C → 走现有 error/abort 路径
- 父 signal 未 aborted 但子 abort 了 = 宽限期超时 → 走 stopped 路径

### 6.5 control_agent 是否应纳入一期

考虑到 `control_agent` 只对后台 SubAgent 有效，且后台 SubAgent 本身使用频率低于前台，**建议一期只实现 CLI/Web 触发入口**，`control_agent` 放二期。

理由：
- CLI/Web 入口覆盖前台 + 后台，已满足核心需求
- 一期少一个工具 = 少一套注册/测试/文档
- 二期加 `control_agent` 时，底层 `stopAgent()` 已就绪，只需套壳

## 八、风险与注意事项

### 8.1 history 配对安全

`#stopRequested` 检查点**严格放在** `#executeToolCalls` 之后和循环顶部。不在 `#callLLM` 与 `#executeToolCalls` 之间检查 — 此时 `tool_call` 没有对应 `tool_result`，退出会破坏 history。

### 8.2 并发安全

Node.js 单线程，`stopAgent()` 内部状态操作是原子的。幂等设计：
- 已在 stopping → 返回 success，不重复设置定时器
- 已在终态（done/error/stopped）→ 返回 error

### 8.3 control 引用的生命周期

`SubAgentState.control` 持有 AbortController 和 AgentLoop 的引用。必须在终态时调用 `clearSubAgentControl()` 置为 undefined，否则 AgentLoop（含 history 数组、provider 引用等）无法被 GC。

### 8.4 JSONL 审计

stop 操作写入子 Agent 的 JSONL：
- `{ type: 'lifecycle', action: 'stopped', resolution: 'graceful'|'forced', source, reason, turn, maxTurns, timestamp }`
- `{ type: 'session_end', status: 'stopped', totalErrors: 0, ... }`
