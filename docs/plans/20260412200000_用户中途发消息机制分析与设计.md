## 2. Claude Code CLI 的做法分析

### 2.1 核心模型：中断-重启（Interrupt-Restart）

Claude Code **并非**"Agent 继续工作 + 用户消息并行注入"的模式，而是一个更简单但高效的设计：

```
用户在 Agent streaming 时按 Enter 发送消息
    ↓
1. abort() — 立即中止当前 LLM 流（AbortSignal → fetch abort）
2. 保留已累积的 partial reply 作为 assistant 消息追加到 history
3. 将用户新消息追加到 history
4. 启动新一轮 AgentLoop（带完整上下文继续对话）
```

**本质：中止当前轮 → 保留上下文 → 开新轮。不存在"两条消息流并行"。**

### 2.2 为什么不是"消息注入"？

| 约束 | 说明 |
|------|------|
| **LLM API 限制** | 流式返回中无法追加新 user message，模型已经在生成 |
| **一致性保证** | history 是线性的 `user→assistant→user→...`，并行注入破坏结构 |
| **用户体验足够** | abort fetch 是瞬时的，新轮启动也是瞬时的，用户几乎无感 |

### 2.3 三种交互入口

| 入口 | 触发方式 | 行为 |
|------|---------|------|
| **Escape** | 键盘 | 仅中止当前响应，回到空闲 |
| **Ctrl+C** | 键盘 | 第一次：中止响应；第二次（2秒内）：退出程序 |
| **Enter（streaming 期间）** | 输入框 | 中止当前响应 + 发送新消息 + 开新轮 |

### 2.4 中断后的上下文保留

```
history 时间线：

[user]      "帮我写一个排序算法"
[assistant]  "好的，我来用快速排序... def quicksort(arr):"  ← partial reply，被中断
[user]      "[Request interrupted by user]"                 ← 中断标记（可选）
[user]      "不对，用归并排序"                                ← 新消息
[assistant]  "好的，改用归并排序..."                          ← 新一轮完整回复
```

模型能看到之前的 partial reply，理解上下文是连续的。

### 3.2 发现的 Bug：React 闭包陷阱

**位置**：`useChat.ts:225-226, 532-534`

```typescript
// submit 用 state 做守卫
const submit = useCallback((text) => {
  if (isStreaming) return  // ← 闭包捕获的 isStreaming
  // ...
  setIsStreaming(true)
  // ...async work...
  // finally { setIsStreaming(false) }
}, [isStreaming, messages, currentProvider, currentModel])

// interruptAndSubmit 依赖 submit
const interruptAndSubmit = useCallback((text) => {
  abortRef.current?.abort()         // 1. 触发 abort（同步）
  setTimeout(() => submit(text), 0) // 2. 延迟调用 submit
}, [submit])
```

**问题链**：

```
1. streaming 期间调用 interruptAndSubmit
   → 此时 isStreaming = true
   → submit 闭包中 isStreaming = true

2. abort() 同步执行
   → 触发 AbortError → async catch/finally → setIsStreaming(false)
   → 但 setIsStreaming 是 React state update，异步的！

3. setTimeout(() => submit(text), 0) 在 macrotask 中执行
   → 问：此时 submit 闭包中的 isStreaming 是 true 还是 false？

4. 答：仍然是 true！
   原因：
   - abort → AbortError → catch → finally → setIsStreaming(false)
     这条链路虽然通过 microtask 完成（Promise 链），
     但 setIsStreaming 只是调度了 React 重渲染，还没执行
   - 即使 React 重渲染了，setTimeout 中捕获的是旧版 submit
   - 旧版 submit 的闭包 isStreaming = true
   
5. 结果：submit 在 line 226 直接 return，新消息没有被发送！
```

**影响**：`interruptAndSubmit` 功能实际上是**失效的** — 用户在 streaming 时输入新消息并按 Enter，消息不会被发送。仅 abort 生效（响应停止），但新一轮对话不会启动。

### 3.3 验证方法

```
1. 启动 cCli，发送一个会触发长回复的问题
2. 在 LLM 流式输出过程中，输入新消息并按 Enter
3. 预期：中断 + 新消息发送
4. 实际：仅中断，新消息丢失，需要用户再按一次 Enter
```

## 4. 修复方案

### 4.1 方案 A：Ref 守卫（推荐）

用 `useRef` 替代 `useState` 做 submit 的重入守卫。Ref 是同步可变的，不受 React 渲染周期约束。

**变更范围**：仅 `useChat.ts`

```typescript
// === 新增 ===
const isStreamingRef = useRef(false)

// === 修改 submit ===
const submit = useCallback((text: string, _source: 'cli' | 'web' = 'cli', imageIds?: string[]) => {
  if (isStreamingRef.current) return  // ← 改用 ref
  isStreamingRef.current = true
  setIsStreaming(true)  // ← state 仍保留，驱动 UI 渲染

  // ... 原有逻辑不变 ...

  ;(async () => {
    try {
      // ... AgentLoop 执行 ...
    } finally {
      isStreamingRef.current = false  // ← 同步清除
      setIsStreaming(false)
    }
  })()
}, [/* 移除 isStreaming 依赖，改用 ref */
    messages, currentProvider, currentModel])

// === 修改 interruptAndSubmit ===
const interruptAndSubmit = useCallback((text: string) => {
  abortRef.current?.abort()
  isStreamingRef.current = false  // ← 同步清 ref，确保 submit 不被拦截
  setTimeout(() => submit(text), 0)
}, [submit])
```

**优点**：
- 改动最小，仅影响守卫逻辑
- Ref 是同步的，彻底规避 React 渲染时序问题
- `isStreaming` state 仍保留用于 UI 渲染（InputBar 的 streaming 样式等）

**风险点**：
- `submit` 的 `useCallback` 依赖从 `[isStreaming, ...]` 变为 `[messages, ...]`
- 需确认移除 `isStreaming` 依赖不会导致其他副作用

### 4.2 方案 B：Force 参数

给 `submit` 增加 `force` 参数跳过守卫检查。

```typescript
const submit = useCallback((
  text: string,
  _source: 'cli' | 'web' = 'cli',
  imageIds?: string[],
  force = false,  // ← 新增
) => {
  if (!force && isStreaming) return
  // ...
}, [isStreaming, messages, currentProvider, currentModel])

const interruptAndSubmit = useCallback((text: string) => {
  abortRef.current?.abort()
  setTimeout(() => submit(text, 'cli', undefined, true), 0)
}, [submit])
```

**优点**：改动更小
**缺点**：
- `force=true` 时可能导致两个 AgentLoop 并发（如果 abort 的 finally 还没执行完）
- 需要额外的防护逻辑确保旧 loop 真正退出

### 4.3 方案 C：Promise 等待 abort 完成

```typescript
const interruptAndSubmit = useCallback(async (text: string) => {
  abortRef.current?.abort()
  // 等待旧 loop 的 finally 执行完
  await new Promise(resolve => setTimeout(resolve, 50))
  submit(text)
}, [submit])
```

**缺点**：
- 50ms 延迟是"猜测值"，不确定是否足够
- 依赖 React 在 50ms 内完成重渲染，不可靠
- 本质上还是时序赌博

### 4.4 方案对比

| 维度 | A: Ref 守卫 | B: Force 参数 | C: Promise 等待 |
|------|------------|--------------|----------------|
| 可靠性 | ✅ 完全可靠 | ⚠️ 可能并发 | ❌ 时序不可控 |
| 改动量 | 中 | 小 | 小 |
| 副作用风险 | 低 | 中 | 高 |
| 代码可读性 | ✅ 语义清晰 | ⚠️ force 参数不直观 | ❌ magic number |

**结论：推荐方案 A（Ref 守卫）。**

## 5. 是否需要更高级的模式？

除了修复 Interrupt-Restart，还有两种更复杂的模式可以讨论：

### 5.1 消息排队注入（Queue-Inject）

```
Agent 在执行工具（比如写文件），用户发送消息
  → 消息不中断 Agent，而是放入队列
  → Agent 当前工具执行完毕后，将队列中的消息注入 history
  → Agent 在下一轮 LLM 调用时看到用户的新消息
```

**适用场景**：Agent 正在执行重要的不可中断操作（如数据库迁移），用户想补充信息但不想打断

**实现复杂度**：中等
- 需要在 AgentLoop 的 turn 间隙检查消息队列
- 需要区分"用户想中断"还是"用户想补充"
- UI 需要新的交互提示

**价值评估**：⚠️ 有限
- 大多数场景 Interrupt-Restart 已经够用
- 用户很难区分"补充"和"重新来"
- Claude Code 没有做这个

### 5.2 并行感知（Parallel-Aware）

```
Agent 收到用户消息后自行决定是否中断或调整策略
  → 需要特殊的 LLM 协议支持
  → 当前主流 LLM API 不支持
```

**价值评估**：❌ 不可行（LLM API 限制）

### 5.3 结论

**当前阶段只需要做好 Interrupt-Restart。** 这是 Claude Code 的全部做法，用户体验已经验证过了。

## 6. 完整交互流程（目标态）

### 6.1 正常对话流程

```
[idle] 用户输入 → Enter → submit()
  → isStreamingRef = true, setIsStreaming(true)
  → InputBar 切换为 streaming 样式（暗色 ❯，Enter 触发 interruptAndSubmit）
  → AgentLoop.run() 开始
  → LLM 流式输出 → 工具调用 → 继续对话
  → AgentLoop done
  → isStreamingRef = false, setIsStreaming(false)
  → InputBar 恢复正常样式
[idle]
```

### 6.2 用户中途发消息（Interrupt-Restart）

```
[streaming] 用户输入新消息 → Enter
  → InputBar.handleSubmit() 检测 streaming=true
  → onInterruptSubmit(text) → interruptAndSubmit(text)
  │
  ├── 1. abortRef.current.abort()
  │     → AbortSignal 触发
  │     → Provider fetch 中断，抛出 AbortError
  │     → AgentLoop catch → yield { type: 'done', reason: 'aborted' }
  │     → useChat catch(isAbortError)
  │       → 保存 partial reply 到 messages + session log
  │       → 记录 "[Request interrupted by user]"
  │       → finally: isStreamingRef = false, setIsStreaming(false)
  │
  ├── 2. isStreamingRef.current = false（同步，确保 submit 不被拦截）
  │
  └── 3. setTimeout(() => submit(text), 0)
        → submit 检查 isStreamingRef.current === false ✅
        → contextManager.pushUser(text)
        → 新 AgentLoop 启动
        → history 包含之前的完整对话（含 partial reply）
[streaming again]
```

### 6.3 纯中止（Escape / Ctrl+C）

```
[streaming] 用户按 Escape
  → App.tsx useInput 捕获
  → abort()
  → 保留 partial reply
  → 回到 idle
[idle]

[idle] 用户按 Ctrl+C
  → 提示 "再次 Ctrl+C 退出"

[idle] 用户再次按 Ctrl+C（2秒内）
  → process.exit(0)
```

### 6.4 工具执行中的中断

```
[streaming] Agent 正在执行工具（如 write_file）
  → 用户按 Escape 或输入新消息
  → abort() 触发
  → 但工具执行不受 AbortSignal 影响（工具不监听 signal）
  → 工具执行完毕后，下一轮 LLM 调用时 fetch abort 生效
  → AgentLoop 抛出 AbortError，走正常中断流程
  
  注意：工具可能已经产生了副作用（文件已写入等），
  中断不会回滚工具操作。这与 Claude Code 行为一致。
```

### 6.5 权限确认中的中断

```
[streaming] Agent 弹出权限确认弹窗
  → AgentLoop 在 permission_request 的 await 上暂停
  → 用户输入新消息 → interruptAndSubmit
  → abort() 触发
  → permission promise 不会 resolve（被 GC）
  → AbortError 从当前暂停点传播
  → useChat catch 保存 partial reply
  → 新一轮 submit 启动
```

## 7. Web 端同步

Web 端（通过 Bridge）的中断行为需要与 CLI 一致：

| Web 操作 | Bridge 消息 | CLI 处理 |
|---------|------------|---------|
| 点击"停止" | `{ type: 'abort' }` | `eventBus.emit({ type: 'user_input', text: '__abort__' })` → `abort()` |
| 发送新消息 | `{ type: 'chat', text }` | `eventBus.emit({ type: 'user_input', text })` → `submit()` |
| streaming 中发消息 | `{ type: 'chat', text }` | 需要识别 isStreaming → 走 `interruptAndSubmit()` |

**Web 端潜在问题**：当前 Web 端发送消息走的是 `eventBus → submit()`，没有经过 `interruptAndSubmit` 路径。需要在 eventBus 的 `user_input` handler 中判断 isStreaming 状态，自动切换到 interruptAndSubmit。

相关代码位置：`useChat.ts` 中 `eventBus.onType('user_input', ...)` 的监听逻辑。
