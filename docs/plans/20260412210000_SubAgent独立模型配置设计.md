# SubAgent 独立模型配置设计

> 日期：2026-04-12
> 状态：实施中

---

## 1. 需求

用户希望主 Agent 用强模型（如 GLM-5.1），子 Agent 用省钱/快速模型（如 GLM-5）。
当前子 Agent 默认继承父 Agent 模型，没有声明式配置入口。

## 2. 方案：全局一个字段（方案 A）

子 Agent 只有两层（不可嵌套派生），一个字段覆盖所有场景。

### 2.1 config.json 变更

```jsonc
{
  "defaultProvider": "glm",
  "defaultModel": "glm-5.1",
  "subAgentModel": "glm-5"       // ← 新增，子 Agent 默认模型，不配则继承主 Agent
}
```

### 2.2 优先级链

```
1. LLM 显式传 model 参数       → dispatch_agent({ model: "xxx" })  最高
2. config.subAgentModel         → 全局配置的子 Agent 模型
3. 继承父 Agent 当前模型        → 兜底（现有行为）
```

### 2.3 模型所属 Provider 自动解析

subAgentModel 只需配模型名，Provider 由 resolveSubAgentProvider() 自动在 config.providers 中查找。
已有逻辑完全可复用，无需额外配置 subAgentProvider。

## 3. 改动范围

| 文件 | 改动 |
|------|------|
| `src/config/config-manager.ts` | CCodeConfig 接口新增 `subAgentModel?: string` |
| `src/tools/agent/dispatch-agent.ts` | resolveSubAgentProvider 中增加 config.subAgentModel 优先级 |
| Web 配置页 | Settings 中新增子 Agent 模型选择（P1，可后做） |

## 4. 边界情况

| 边界 | 处理 |
|------|------|
| subAgentModel 配了不存在的模型 | resolveSubAgentProvider 已有兜底：用父 provider + 该 modelName，API 报错由模型侧返回 |
| subAgentModel 属于另一个 provider | 已有逻辑支持：遍历 providers 查找，自动创建对应 provider 会话 |
| 不配 subAgentModel | 走现有默认行为（继承父 Agent），完全兼容 |
| LLM 主动传了 model 参数 | 优先级最高，覆盖 subAgentModel |
