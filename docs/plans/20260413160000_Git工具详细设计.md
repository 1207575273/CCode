# Git 工具详细设计

> 日期：2026-04-13
> 对应任务：实施计划 2.3 Git 深度集成
> 目标：提供专用 git 工具，替代 LLM 通过 bash 调 git 的低效模式

---

## 一、设计目标

1. **按需调用**：LLM 需要 git 信息时主动调用，而非一次性快照
2. **输出可控**：自动截断大 diff，结构化返回，节省 token
3. **安全分级**：只读操作（status/diff/log）自由调用；写操作（commit/push）标记 dangerous
4. **启动注入**：首轮对话自动注入 git 状态摘要，LLM 无需手动调 `git status`
5. **替代 commit skill**：commit 子命令整合进 git 工具，skill 保留为快捷入口

---

## 二、工具定义

### 2.1 单一工具 + 子命令模式

采用 **一个工具 `git` + `subcommand` 参数** 的设计（而非拆成 git_status / git_diff / git_log 多个工具），理由：
- 减少工具列表长度，降低 LLM 选择困难
- 子命令之间共享 git 仓库检测、输出截断等通用逻辑
- 与 bash 工具的 command 模式一致，LLM 学习成本低

### 2.2 JSON Schema

```typescript
{
  type: 'object',
  properties: {
    subcommand: {
      type: 'string',
      enum: ['status', 'diff', 'log', 'branch', 'show', 'checkout', 'commit', 'merge', 'rebase', 'cherry_pick', 'stash', 'tag', 'reset'],
      description: '子命令类型',
    },
    // --- status 参数 ---
    // 无额外参数，返回工作区变更摘要

    // --- diff 专用 ---
    file: {
      type: 'string',
      description: '[diff/show] 指定文件路径（可选，默认全部文件）',
    },
    staged: {
      type: 'boolean',
      description: '[diff] 仅查看已暂存的变更（默认 false，查看未暂存）',
    },

    // --- log 专用 ---
    count: {
      type: 'number',
      description: '[log] 返回的提交数量（默认 10，最大 50）',
    },
    author: {
      type: 'string',
      description: '[log] 按作者过滤',
    },

    // --- show / cherry_pick 共用 ---
    ref: {
      type: 'string',
      description: '[show/cherry_pick/reset] 指定 commit hash 或引用（如 HEAD~1、abc1234）',
    },

    // --- checkout 专用 ---
    branch_name: {
      type: 'string',
      description: '[checkout] 切换到的分支名，或 -b 新分支名',
    },
    create_branch: {
      type: 'boolean',
      description: '[checkout] 是否创建新分支（默认 false，等同于 git checkout -b）',
    },

    // --- commit / tag 共用 ---
    message: {
      type: 'string',
      description: '[commit/tag] commit 提交信息（遵循 Conventional Commits）或 tag 注释信息',
    },
    files: {
      type: 'array',
      items: { type: 'string' },
      description: '[commit] 指定暂存的文件列表（必填，禁止 git add -A）',
    },

    // --- merge / rebase 共用 ---
    target: {
      type: 'string',
      description: '[merge/rebase] 目标分支名（如 main、feature/xxx）',
    },
    abort: {
      type: 'boolean',
      description: '[merge/rebase/cherry_pick] 冲突时中止操作（--abort）',
    },

    // --- reset 专用 ---
    mode: {
      type: 'string',
      enum: ['soft', 'mixed', 'hard'],
      description: '[reset] 重置模式（默认 mixed）',
    },

    // --- stash 专用 ---
    stash_action: {
      type: 'string',
      enum: ['push', 'pop', 'list', 'drop'],
      description: '[stash] 操作类型（默认 push）',
    },
    stash_message: {
      type: 'string',
      description: '[stash] push 时的描述信息',
    },

    // --- tag 专用 ---
    tag_name: {
      type: 'string',
      description: '[tag] 标签名称（创建/删除时必填，省略则列出所有 tag）',
    },
    tag_delete: {
      type: 'boolean',
      description: '[tag] 删除标签（默认 false，即创建）',
    },
  },
  required: ['subcommand'],
}
```

### 2.3 子命令详细行为

| 子命令 | dangerous | 底层命令 | 输出处理 | 
|--------|-----------|---------|---------|
| `status` | false | `git status --short --branch` | 原样返回，附加分支名和统计 |
| `diff` | false | `git diff [--staged] [file]` | 超过 2000 行截断 + 提示 |
| `log` | false | `git log --oneline --format=... -N` | 结构化：hash + date + author + subject |
| `branch` | false | `git branch -a --format=...` | 标记当前分支 `*`，远程分支列出 |
| `show` | false | `git show --stat [ref]` + `git show [ref] -- [file]` | stat 摘要 + 指定文件 diff |
| `checkout` | **动态** | `git checkout [-b] <branch>` | 切换/创建分支，返回新分支名 |
| `commit` | **true** | `git add <files> && git commit` | 返回 commit hash + 变更摘要 |
| `merge` | **true** | `git merge <target>` 或 `git merge --abort` | 成功返回合并摘要，冲突返回冲突文件列表 |
| `rebase` | **true** | `git rebase <target>` 或 `git rebase --abort` | 成功返回变基摘要，冲突返回冲突信息 |
| `cherry_pick` | **true** | `git cherry-pick <ref>` 或 `git cherry-pick --abort` | 返回拣选结果 |
| `stash` | **动态** | `git stash push/pop/list/drop` | list 只读，push/pop/drop 标记 dangerous |
| `tag` | **动态** | `git tag <name>` 或 `git tag -d <name>` | 无 tag_name 时列出（safe），有则创建/删除 |
| `reset` | **true** | `git reset [--soft/--mixed/--hard] <ref>` | 返回 reset 后状态摘要 |

### 2.4 动态 dangerous 机制

当前 Tool 接口的 `dangerous` 是静态 `readonly` 属性，但 `checkout`、`stash`、`tag` 三个子命令需要根据参数动态判断安全级别。

**方案**：GitTool 整体标记 `dangerous = false`（不弹确认），在 `execute()` 内部对写操作子命令**手动调用权限确认**：

```typescript
// 需要确认的子命令集合
const DANGEROUS_SUBCOMMANDS = new Set(['commit', 'merge', 'rebase', 'cherry_pick', 'reset'])

// 动态判断的子命令
function isDynamicDangerous(subcommand: string, args: Record<string, unknown>): boolean {
  if (subcommand === 'checkout' && args['create_branch']) return true  // 创建分支需确认
  if (subcommand === 'stash' && args['stash_action'] !== 'list') return true  // list 以外需确认
  if (subcommand === 'tag' && args['tag_name']) return true  // 有 tag_name 说明是创建/删除
  return false
}
```

**实现方式**：在 `execute()` 入口判断，如果是 dangerous 操作，返回一个带 `[需要确认]` 前缀的 output，由 agent-loop 的现有流程处理。

或者更简单：**GitTool 标记 `dangerous = true`**，所有子命令统一弹确认。只读操作确认一次后 agent-loop 会自动放行同类工具，不会反复弹窗。这样实现零成本，用户体验影响极小。

**推荐**：先用 `dangerous = true` 统一方案，后续如果用户反馈"查 status 也要确认太烦了"再改动态方案。

## 三、子命令实现细节

### 3.1 status

```
输入: { subcommand: 'status' }
执行: git status --short --branch
输出格式:
  分支: main (ahead 2, behind 0)
  
  已暂存 (2):
    M  src/foo.ts
    A  src/bar.ts
  
  未暂存 (1):
    M  src/baz.ts
  
  未跟踪 (3):
    ?? dist/
    ?? .env
    ?? tmp.log
```

### 3.2 diff

```
输入: { subcommand: 'diff', file: 'src/foo.ts', staged: true }
执行: git diff --staged -- src/foo.ts
截断: 超过 2000 行 → 截断 + "[diff 过长，共 N 行，已截断。请指定 file 参数查看单文件 diff]"
输出: 原始 diff 文本（带 +/- 标记）
```

**截断策略**：
- 全量 diff（无 file 参数）：超过 2000 行截断
- 单文件 diff：超过 3000 行截断
- 截断后附加统计摘要（`git diff --stat` 输出）

### 3.3 log

```
输入: { subcommand: 'log', count: 5 }
执行: git log --oneline --format="%h %ad %an | %s" --date=short -5
输出格式:
  abc1234 2026-04-13 codeYang | feat(git): add git tool
  def5678 2026-04-12 codeYang | fix: memory rebuild
  ...
```

- count 默认 10，最大限制 50（防止 LLM 请求 1000 条）
- 支持 author 过滤

### 3.4 branch

```
输入: { subcommand: 'branch' }
执行: git branch -a --format="%(if)%(HEAD)%(then)* %(end)%(refname:short) %(upstream:short)"
输出格式:
  * main → origin/main
    feature/git-tool
    remotes/origin/main
    remotes/origin/dev
```

### 3.5 show

```
输入: { subcommand: 'show', ref: 'HEAD~1' }
执行: git show --stat HEAD~1 (摘要) + git show HEAD~1 -- [file] (详情)
输出格式:
  commit abc1234
  Author: codeYang
  Date: 2026-04-13
  
      feat(git): add git tool
  
  3 files changed, 120 insertions(+), 15 deletions(-)
   src/tools/core/git.ts | 100 +++++++
   src/core/bootstrap.ts |  20 ++--
   ...
```

- 无 file 参数时只返回 stat 摘要
- 有 file 参数时返回该文件的完整 diff（同样受截断限制）

### 3.6 checkout

```
输入: { subcommand: 'checkout', branch_name: 'feature/xxx' }
执行: git checkout feature/xxx
输出: ✓ 已切换到分支 feature/xxx

输入: { subcommand: 'checkout', branch_name: 'feature/new', create_branch: true }
执行: git checkout -b feature/new
输出: ✓ 已创建并切换到新分支 feature/new
```

**安全分级**：
- 切换已有分支（`create_branch=false`）：safe
- 创建新分支（`create_branch=true`）：dangerous（会改变 HEAD）
- branch_name 必填，不允许空值

### 3.7 commit

```
输入: { subcommand: 'commit', message: 'feat(git): add git tool', files: ['src/tools/core/git.ts'] }
执行:
  1. git add src/tools/core/git.ts
  2. 写入临时文件 /tmp/.ccode-commit-msg，内容为 message
  3. git commit -F /tmp/.ccode-commit-msg
  4. 删除临时文件
  5. git status --short (验证)
输出格式:
  ✓ 已提交 abc1234
  feat(git): add git tool
  1 file changed, 100 insertions(+)
```

> **为什么不用 `-m`**：message 可能包含引号、换行符、特殊字符，`-m "..."` 需要复杂转义。
> 使用 `git commit -F <file>` 通过临时文件传递，避免 shell 转义问题（与 commit skill 的 HEREDOC 方案原理一致）。

**安全约束**：
- `dangerous: true`（agent-loop 会请求用户确认）
- files 必填，拒绝空数组
- 拒绝提交 .env / credentials.json 等敏感文件（硬编码黑名单）
- 不支持 `--amend`、`--no-verify`（需要用 bash 手动调）
- 不支持 push（独立操作，用 bash）

### 3.8 merge

```
输入: { subcommand: 'merge', target: 'feature/xxx' }
执行: git merge feature/xxx
输出格式（成功）:
  ✓ 已合并 feature/xxx → main
  3 files changed, 50 insertions(+), 10 deletions(-)

输出格式（冲突）:
  ✗ 合并冲突
  冲突文件:
    UU src/foo.ts
    UU src/bar.ts
  请手动解决冲突后 commit，或调用 git merge --abort 中止
```

```
输入: { subcommand: 'merge', abort: true }
执行: git merge --abort
输出: ✓ 合并已中止，恢复到合并前状态
```

**安全约束**：`dangerous: true`

### 3.9 rebase

```
输入: { subcommand: 'rebase', target: 'main' }
执行: git rebase main
输出格式（成功）:
  ✓ 已变基到 main，共 replay 3 个提交

输出格式（冲突）:
  ✗ 变基冲突（在 commit abc1234 处）
  冲突文件:
    UU src/foo.ts
  解决冲突后运行 git rebase --continue，或调用 git rebase --abort 中止
```

```
输入: { subcommand: 'rebase', abort: true }
执行: git rebase --abort
输出: ✓ 变基已中止
```

**安全约束**：`dangerous: true`
**限制**：不支持交互式 rebase（`-i`），需要交互式操作请用 bash

### 3.10 cherry_pick

```
输入: { subcommand: 'cherry_pick', ref: 'abc1234' }
执行: git cherry-pick abc1234
输出格式（成功）:
  ✓ 已拣选 abc1234 → 新提交 def5678
  feat(ui): add button component

输出格式（冲突）:
  ✗ 拣选冲突
  冲突文件:
    UU src/ui/Button.tsx
  解决冲突后运行 git cherry-pick --continue，或调用 git cherry_pick --abort 中止
```

**安全约束**：`dangerous: true`

### 3.11 stash

```
输入: { subcommand: 'stash', stash_action: 'push', stash_message: 'WIP: feature xxx' }
执行: git stash push -m "WIP: feature xxx"
输出: ✓ 已暂存工作区变更 (stash@{0})

输入: { subcommand: 'stash', stash_action: 'pop' }
执行: git stash pop
输出: ✓ 已恢复暂存 stash@{0}

输入: { subcommand: 'stash', stash_action: 'list' }
执行: git stash list
输出:
  stash@{0}: WIP: feature xxx (2 分钟前)
  stash@{1}: On main: hotfix (1 小时前)

输入: { subcommand: 'stash', stash_action: 'drop' }
执行: git stash drop
输出: ✓ 已删除 stash@{0}
```

**安全分级**：`list` 只读（safe），`push/pop/drop` 标记 `dangerous: true`

### 3.12 tag

```
输入: { subcommand: 'tag', tag_name: 'v1.0.0', message: 'Release v1.0.0' }
执行: git tag -a v1.0.0 -m "Release v1.0.0"
输出: ✓ 已创建 tag v1.0.0

输入: { subcommand: 'tag', tag_name: 'v0.9.0', tag_delete: true }
执行: git tag -d v0.9.0
输出: ✓ 已删除 tag v0.9.0

输入: { subcommand: 'tag' }  (无 tag_name)
执行: git tag -l --sort=-creatordate --format="%(refname:short) %(creatordate:short)"
输出:
  v1.0.0  2026-04-13
  v0.9.0  2026-04-12
```

**安全分级**：列出 tag 只读（safe），创建/删除标记 `dangerous: true`

### 3.13 reset

```
输入: { subcommand: 'reset', ref: 'HEAD~1', mode: 'soft' }
执行: git reset --soft HEAD~1
输出:
  ✓ 已 reset 到 HEAD~1 (abc1234)
  模式: soft（变更保留在暂存区）

输入: { subcommand: 'reset', ref: 'HEAD~3', mode: 'hard' }
执行: git reset --hard HEAD~3
输出:
  ⚠ 已 hard reset 到 HEAD~3 (abc1234)
  警告: 3 个提交的变更已永久丢弃
```

**安全约束**：`dangerous: true`
- `hard` 模式输出额外警告
- ref 必填，不允许无目标 reset

## 四、Bash 兜底说明

git 工具覆盖了最常用的操作，但不可能穷举所有 git 子命令。以下场景应引导 LLM 使用 bash 工具：

| 场景 | 说明 |
|------|------|
| `git push` / `git pull` / `git fetch` | 涉及远程操作，需要网络和认证，不纳入 git 工具 |
| `git rebase -i`（交互式变基） | 需要编辑器交互，git 工具无法支持 |
| `git bisect` | 多步交互式调试，不适合单次工具调用 |
| `git submodule` | 低频且复杂，用 bash 更灵活 |
| `git config` | 配置管理，用 bash 直接操作 |
| `git blame` | 低频查询，bash 调用即可 |
| `git reflog` | 高级恢复操作，用 bash |
| 复杂的 `git log` 格式 | 如 `--graph --all --decorate`，用 bash 自定义 |

**工具描述中需明确告知 LLM**：git 工具未覆盖的操作（如 push、pull、交互式 rebase 等），请使用 bash 工具执行原生 git 命令。

## 五、工具描述（注入给 LLM）

```
Git 版本控制工具 — 查看仓库状态、变更历史、分支管理，以及提交、合并、变基等操作。

子命令（只读）：
- status: 查看工作区变更摘要（已暂存/未暂存/未跟踪文件）
- diff: 查看文件变更详情（staged=true 查看已暂存，file 指定单文件）
- log: 查看提交历史（count 控制数量，默认 10；author 按作者过滤）
- branch: 查看所有分支及当前分支
- show: 查看指定提交详情（ref 指定 commit hash 或引用，file 查看单文件变更）

子命令（写操作，需确认）：
- checkout: 切换/创建分支（branch_name 指定分支，create_branch=true 创建新分支）
- commit: 提交代码（message 必填，files 必填且逐个暂存，遵循 Conventional Commits）
- merge: 合并分支（target 指定目标分支，abort=true 中止冲突合并）
- rebase: 变基（target 指定目标分支，abort=true 中止）。不支持交互式 -i
- cherry_pick: 拣选提交（ref 指定 commit hash，abort=true 中止）
- stash: 暂存/恢复工作区（stash_action: push/pop/list/drop；list 无需确认）
- tag: 管理标签（无 tag_name 时列出所有标签；有 tag_name 时创建，tag_delete=true 删除）
- reset: 重置 HEAD（ref 指定目标，mode: soft/mixed/hard，默认 mixed）

注意事项：
- 修改文件前建议先调 status 了解工作区状态
- diff 输出过长会自动截断，请用 file 参数缩小范围
- commit 不会自动 push，需要 push 请用 bash 工具
- push、pull、fetch、交互式 rebase、bisect、blame 等操作不在此工具范围内，请使用 bash 工具执行原生 git 命令
- 非 git 仓库会返回错误提示
```

## 六、启动注入（Bootstrap Git Context）

### 5.1 目的

让 LLM 在第一轮对话前就知道当前 git 状态，无需先调工具。

### 5.2 实现位置

`bootstrap.ts` 的 `bootstrapAll()` 中新增 Chain F'，与其他链并行执行：

```typescript
// Chain F': Git 上下文收集
const gitContextPromise = collectGitContext(cwd)
```

### 5.3 注入内容

```typescript
const GIT_CONTEXT_TIMEOUT_MS = 3000  // 3 秒超时，防止大仓库阻塞启动

async function collectGitContext(cwd: string): Promise<string> {
  try {
    // 0. 所有 git 命令共享超时，使用 AbortController
    const controller = new AbortController()
    const timer = setTimeout(() => controller.abort(), GIT_CONTEXT_TIMEOUT_MS)

    // 1. 检测是否是 git 仓库（git rev-parse --is-inside-work-tree）
    // 2a. 是 git 仓库 → 并行执行: git branch --show-current, git log --oneline -5, git status --short
    //     拼接为 markdown 段落
    // 2b. 不是 git 仓库 → 返回明确提示（非空字符串）

    clearTimeout(timer)
    return result
  } catch {
    // 超时或其他异常 → 返回降级提示，不阻塞启动
    return '## Git 状态\nGit 信息收集超时，请手动调用 git status 查看。'
  }
}
```

> **超时设计理由**：大型仓库（如 chromium、Linux kernel）的 `git status` 可能需要数秒甚至更久。
> 设置 3 秒超时，超时后返回降级提示而非阻塞整个启动流程。

**是 git 仓库时**，注入到 `buildSystemPrompt()` 的 `gitContext`：

```
## 当前 Git 状态
- 分支: main
- 最近提交:
  abc1234 feat(git): add git tool
  def5678 fix: memory rebuild
  ...
- 工作区变更:
  M src/foo.ts
  ?? tmp.log
```

**不是 git 仓库时**，同样注入明确提示：

```
## Git 状态
当前工作目录不是 Git 仓库。git 工具不可用，如需版本控制请先执行 git init 或 clone 一个仓库。
```

> **设计理由**：不静默跳过，让 LLM 明确知道当前环境无 git，避免盲目调用 git 工具后才发现报错。

### 5.4 工具层面的非 git 仓库处理

当 LLM 在非 git 仓库中调用 git 工具的任何子命令时，统一返回：

```typescript
{
  success: false,
  output: '',
  error: '当前目录不是 Git 仓库。请先执行 git init 初始化，或 cd 到一个 Git 仓库目录。'
}
```

不区分子命令，第一步统一检测 git 仓库，不通过直接返回错误。

### 5.5 Token 预算

- 是 git 仓库：分支名 + 5 条 log + status 摘要 ≈ 200-400 token
- 非 git 仓库：提示信息 ≈ 30 token
- 设上限：status 超过 50 行时截断 + "[变更较多，请调用 git status 查看完整列表]"
- 无变更时显示 `(clean)`

## 七、与 Commit Skill 的关系

### 现状
- `src/skills/builtin/commit/SKILL.md` 定义了完整的 commit 流程
- Skill 通过 bash 工具执行 `git add` + `git commit`

### 改进后
- Skill 保留为 `/commit` 快捷入口（用户输入 `/commit` 触发）
- Skill 内部改用 `git` 工具的 `commit` 子命令，而非 bash
- 好处：统一安全约束（敏感文件检查）、统一输出格式

### Skill 改动

```markdown
## 流程（更新后）

1. 调用 `git status` 子命令查看变更
2. 调用 `git log` 子命令查看提交历史
3. 分析变更，草拟 commit message
4. 调用 `git commit` 子命令提交（指定 files 和 message）
5. 调用 `git status` 确认提交成功
```

allowed-tools 从 `Bash` 改为 `git`（或两者都保留，bash 用于 push 等操作）。

## 八、权限配置（settings.local.json）

新工具注册名为 `git`，现有 `PermissionManager` 的友好名解析已支持，无需改代码。

用户在 `.ccode/settings.local.json` 的 `permissions.allow` 中添加即可：

```jsonc
{
  "permissions": {
    "allow": [
      "Bash(*)",
      "Read(*)",
      "Git(*)",    // ← 新增，放行所有 git 子命令（含 commit/merge/reset 等）
      // ...
    ]
  }
}
```

跟 `Bash(*)` 一个道理 — 加了就全部免确认，不加就每次弹确认。用户自行决定风险承受度。
