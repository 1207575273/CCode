## 一、愿景

从 **"一个人 + 一个 AI"** 进化到 **"一个需求 + N 个 AI 角色自动协作"**。

```
现在:
  开发者 ──→ CCode CLI ──→ 代码

未来:
  需求 ──→ 编排层 ──→ PM Agent ──→ 需求文档
                    ──→ 架构 Agent ──→ 技术方案
                    ──→ [PM + 架构 + 前端 + 后端] 会议讨论 ──→ 评审结论
                    ──→ PM Agent ──→ 看板任务
                    ──→ 前端 Agent ──→ 前端代码
                    ──→ 后端 Agent ──→ 后端代码
                    ──→ 架构 Agent ──→ 集成验收
```

**核心原则**：底层执行引擎可替换。CCode CLI 是执行节点之一，未来可以换成 Claude Code CLI、Cursor Agent 或任何 A2A 兼容的 Agent。

---

## 二、为什么选 A2A 协议

### 2.1 A2A 是什么

Google 提出的 Agent-to-Agent 协议，定义了 Agent 之间的发现、通信和任务协作标准：

- **Agent Card** — JSON 格式描述 Agent 的能力、技能、端点地址
- **Task Lifecycle** — `submitted → working → input-required → completed / failed` 标准状态机
- **Message Exchange** — JSON-RPC over HTTP，支持 SSE 流式推送
- **Artifact** — Agent 产出物的标准格式（文档、代码、图片等）

### 2.2 为什么不是扩展 SubAgent

| 维度 | 扩展 SubAgent | A2A 协议 |
|------|-------------|---------|
| 执行引擎 | 只能是 CCode 内部 | 任何 A2A 兼容 Agent |
| 通信边界 | 进程内 | HTTP（跨进程/跨机器/跨组织） |
| 能力发现 | 硬编码 | Agent Card 动态发现 |
| 未来扩展 | 改代码 | 注册新 Agent Card 即可 |
| 生态对接 | 无 | Claude Code / Cursor / 其他 A2A Agent |

**关键判断**：即使当前所有 Agent 都跑在本地，用 A2A 协议也不亏——协议开销极小（HTTP localhost），但架构上预留了"未来把某个角色换成远程 Agent"的能力。

### 2.3 A2A 裁剪方向（初步）

A2A 完整规范偏重跨组织场景，我们不需要全部实现：

| A2A 功能 | 是否需要 | 原因 |
|----------|---------|------|
| Agent Card | ✅ 需要 | Agent 能力发现的基础 |
| Task CRUD | ✅ 需要 | 任务生命周期核心 |
| Streaming (SSE) | ✅ 需要 | 实时进度推送到看板 |
| Artifact | ✅ 需要 | Agent 产出物传递 |
| Push Notifications | ⚠️ 后期 | 初期用轮询/SSE 够了 |
| Authentication | ⚠️ 后期 | 初期全部本地运行，不需要认证 |
| Multi-party negotiation | ❌ 不需要 | 编排层统一调度，Agent 之间不直接协商 |

## 四、核心概念

### 4.1 Coordinator Agent（主 Agent / 总指挥）

编排层不仅仅是代码逻辑，**它本身也是一个 Agent**——有 AI 大脑，能理解需求、做全局决策。

#### 为什么需要主 Agent

硬编码的 Pipeline YAML 能覆盖标准流程，但现实中充满意外：
- PM Agent 产出的需求文档模糊，要不要退回重写还是直接推进？
- 架构 Agent 说"这个需求需要先重构数据库"，原计划要不要调整？
- 并行开发中后端 Agent 阻塞了，前端要不要先用 mock 继续？
- 超出预算 80% 了，剩下的任务哪些可以砍、哪些必须做？

这些决策不能硬编码，需要一个**有全局视野的 AI 来动态判断**。

#### Coordinator 的职责

```
Coordinator Agent（永远在线，贯穿全流程）
│
├── 1. 需求理解
│   └── 接收用户需求 → 判断复杂度 → 选择合适的 Pipeline 模板
│
├── 2. 流程编排
│   └── 按 Pipeline 驱动各步骤，但可以动态调整
│   └── 例: 需求太简单 → 跳过"方案评审"直接进开发
│   └── 例: 架构设计出了 3 个备选方案 → 临时加一轮评审
│
├── 3. 任务分配
│   └── 根据 Persona 的 required_skills 匹配执行引擎
│   └── 根据 Agent 当前负载做负载均衡（未来）
│
├── 4. 进度监控
│   └── 实时汇总各 Agent 状态 → 更新 Workspace
│   └── 检测异常（超时、失败、成本超限）→ 决定重试/跳过/上报人工
│
├── 5. 冲突协调
│   └── 前后端接口定义不一致 → 拉双方讨论
│   └── 会议讨论不收敛 → 介入裁决
│
└── 6. 兜底决策
    └── 任何 Agent 上报 "我做不了" → Coordinator 判断：
        ├── 换一个更强的模型重试
        ├── 拆成更小的任务
        ├── 标记为需要人工介入
        └── 降级处理（跳过非关键步骤）
```

#### Coordinator 的 Persona 定义

```yaml
# personas/coordinator.yaml
name: "总指挥"
id: "coordinator"
system_prompt: |
  你是项目总指挥（Coordinator），负责全局统筹。你的核心原则：

  1. 你不直接写代码或做设计，你的工作是协调和决策
  2. 你能看到 Workspace 中的所有信息（需求、计划、看板、各 Agent 产出）
  3. 当流程遇到意外时，你做出判断并调整计划
  4. 你尊重各角色 Agent 的专业判断，但最终决策权在你
  5. 成本和时间是硬约束，你需要在质量和效率之间做权衡
  6. 遇到你无法判断的问题，立即上报人工，不要猜

  你的决策日志会被完整记录，供人工复盘。
preferred_model: "claude-opus-4-6"    # 需要最强推理能力
cost_limit_tokens: 300000             # 全流程的协调开销
```

#### Coordinator vs Pipeline 的关系

```
Pipeline YAML = 预设的标准路线（高速公路）
Coordinator   = 实时导航系统（遇到堵车/事故可以改道）

正常情况: Coordinator 按 Pipeline 定义的步骤推进
异常情况: Coordinator 动态调整（跳步/加步/回退/终止）
```

Coordinator 的每次"改道决策"都记录到 Workspace 的决策日志中，人工可以随时复盘。

### 4.2 Workspace（共享工作区）

所有 Agent 共享同一个 Workspace，**不是只看到自己的输入输出，而是能看到全局**。

#### 为什么不能只传 Artifact

如果只在 Agent 之间传递 Artifact（产出物），每个 Agent 只能看到"上一步给了我什么"。
但真实协作中，一个后端 Agent 需要知道：

- 原始需求是什么（不只是拆解后的任务卡片）
- 架构设计的整体方案（不只是"做一个 API"）
- 前端 Agent 的进度（接口联调需要知道对方做到哪了）
- 会议讨论的结论（有哪些约定和妥协）

**Workspace = 项目的 "战情室"，所有人都能看到全貌。**

#### Workspace 结构

```
workspace/
├── context.md              ← 项目上下文（Coordinator 维护）
│   ├── 原始需求
│   ├── 当前阶段
│   └── 关键决策记录
│
├── requirements/           ← 需求文档（PM Agent 产出）
│   └── feature-login.md
│
├── design/                 ← 设计文档（架构 Agent 产出）
│   ├── architecture.md
│   └── api-contract.yaml
│
├── meeting-notes/          ← 会议记录（自动生成）
│   └── 2026-04-13-review.md
│
├── kanban/                 ← 看板状态（结构化 JSON）
│   └── tasks.json
│
├── artifacts/              ← 各 Agent 的产出物
│   ├── frontend/
│   │   ├── LoginPage.tsx
│   │   └── useAuth.ts
│   └── backend/
│       ├── auth-controller.ts
│       └── auth-service.ts
│
├── progress/               ← 进度快照（Coordinator 定期更新）
│   └── latest.json         ← 各任务状态 + 各 Agent 当前动作 + 成本统计
│
└── decisions/              ← Coordinator 决策日志
    └── log.jsonl           ← 每条决策：时间、原因、选择、影响
```

#### 各角色的 Workspace 视图

不是所有 Agent 每次都读完整 Workspace，而是按角色注入相关上下文：

```yaml
# Coordinator — 全量视图
workspace_view:
  - context.md              # 全局上下文
  - kanban/tasks.json       # 全部任务状态
  - progress/latest.json    # 全部进度
  - decisions/log.jsonl     # 自己的决策历史

# PM Agent — 需求 + 看板视图
workspace_view:
  - context.md              # 理解全局
  - requirements/           # 自己的产出
  - design/architecture.md  # 理解技术约束
  - kanban/tasks.json       # 看板状态

# 前端 Agent — 开发视图
workspace_view:
  - context.md              # 理解全局
  - design/api-contract.yaml  # 接口契约（与后端协作的关键）
  - kanban/tasks.json       # 看自己的任务 + 后端进度
  - artifacts/backend/      # 后端已完成的代码（联调需要）
  - meeting-notes/          # 有什么约定

# 后端 Agent — 开发视图
workspace_view:
  - context.md
  - design/api-contract.yaml
  - kanban/tasks.json
  - artifacts/frontend/     # 前端已完成的代码（联调需要）
  - meeting-notes/
```

#### Workspace 注入时机

```
每次给 Agent 下发任务时:
  1. Coordinator 从 Workspace 裁剪出该角色需要的视图
  2. 序列化为 context 文本（或 Artifact 列表）
  3. 注入到 A2A Task 的 message 中

Agent 产出结果时:
  1. Artifact 写回 Workspace 对应目录
  2. Coordinator 更新 progress/latest.json
  3. 如果影响其他 Agent（如接口变更）→ 通知相关 Agent
```

#### Workspace 与 A2A 的关系

Workspace 不是 A2A 协议的一部分——A2A 只管 Agent 间的任务投递和结果返回。
Workspace 是编排层自己维护的**共享状态**，通过 A2A Task 的 message/context 注入给各 Agent。

```
A2A 管的: Agent A 给 Agent B 发任务、收结果
Workspace 管的: 所有 Agent 都能"看到"的项目全局信息
两者关系: Coordinator 从 Workspace 取信息 → 塞进 A2A Task → Agent 拿到上下文
```

### 4.3 Persona（角色模板）

角色模板定义 Agent 的行为特征，与执行引擎解耦：

```yaml
# personas/pm.yaml
name: "产品经理"
id: "pm"
system_prompt: |
  你是一位资深产品经理。你的职责是：
  1. 分析需求的可行性和优先级
  2. 编写结构化的需求文档（用户故事 + 验收标准）
  3. 将技术方案转化为可执行的看板任务
  你输出的文档需要简洁、无歧义、可验证。
required_skills:
  - "code-read"     # 需要能读代码理解现状
preferred_model: "claude-sonnet-4-6"  # 不需要最强模型
cost_limit_tokens: 100000

# personas/architect.yaml
name: "架构师"
id: "architect"
system_prompt: |
  你是一位资深系统架构师。你的职责是：
  1. 评估需求的技术复杂度
  2. 设计接口契约和模块边界
  3. 识别技术风险并提出方案
  4. 在方案评审中担任主持人
  你的方案需要考虑：可维护性、性能、安全性。
required_skills:
  - "code-read"
  - "code-write"
preferred_model: "claude-opus-4-6"    # 需要强推理
cost_limit_tokens: 200000
```

**关键设计**：Persona 只定义"做什么"和"怎么想"，不绑定"用什么引擎执行"。编排层根据 `required_skills` 从 Agent Registry 中匹配可用的执行引擎。

### 4.4 Pipeline（流水线）

```yaml
# pipelines/feature-dev.yaml
name: "功能开发流水线"
steps:
  - id: "requirement"
    name: "需求分析"
    persona: "pm"
    input: "$trigger.description"      # 来自用户输入
    output_artifact: "requirement.md"
    approval: optional                  # 人工审批可选

  - id: "design"
    name: "架构设计"
    persona: "architect"
    input:
      - "$steps.requirement.artifact"   # 上一步产出
    output_artifact: "design.md"
    approval: required                  # 必须人工审批

  - id: "review"
    name: "方案评审"
    type: "meeting"                     # 特殊步骤类型：会议
    moderator: "architect"
    participants: ["pm", "frontend", "backend"]
    input:
      - "$steps.requirement.artifact"
      - "$steps.design.artifact"
    max_rounds: 5
    convergence: "moderator_summary"    # 主持人总结即收敛
    output_artifact: "review_notes.md"

  - id: "task_breakdown"
    name: "任务拆解"
    persona: "pm"
    input:
      - "$steps.design.artifact"
      - "$steps.review.artifact"
    output_artifact: "kanban_tasks.json"

  - id: "implementation"
    name: "并行开发"
    type: "parallel"                    # 并行执行
    source: "$steps.task_breakdown.artifact"  # 从看板任务动态生成子任务
    assign_by: "task.role"              # 按任务标签分配角色
    report_interval: "per_task"         # 每完成一个任务汇报
    approval: none

  - id: "integration"
    name: "集成验收"
    persona: "architect"
    input:
      - "$steps.design.artifact"
      - "$steps.implementation.artifacts"  # 所有并行任务的产出
    output_artifact: "review_result.md"
```

### 4.5 Meeting（会议机制）

多 Agent 讨论是最复杂的部分，需要严格的收敛规则：

```
会议规则:
├── 主持人 (moderator): 控制发言顺序、判断是否收敛
├── 最大轮次: 超过则主持人强制总结
├── 发言顺序: 轮流制（Round Robin）
├── 收敛判定:
│   ├── moderator_summary — 主持人总结，其他人无异议
│   ├── vote — 投票（>50% 同意）
│   └── no_objection — 连续一轮无人提出新问题
└── 异常处理:
    ├── 死循环 — 超过 max_rounds，主持人强制裁决
    └── 单人阻塞 — 同一 Agent 连续 3 轮提同一问题，标记为 "保留意见" 继续推进
```

### 4.6 Kanban（看板）

```typescript
interface KanbanTask {
  id: string
  title: string
  description: string
  status: 'todo' | 'in_progress' | 'review' | 'done' | 'blocked'
  assignee: string        // persona id
  dependencies: string[]  // 依赖的其他 task id
  artifacts: string[]     // 产出物路径
  a2aTaskId?: string      // 关联的 A2A Task ID
  estimatedTokens?: number
  actualTokens?: number
}
```

## 五、CCode 侧需要的改动

### 5.1 暴露 A2A 端点

在现有 Bridge Server 上新增 A2A 路由：

```
现有:
  /ws              — WebSocket（CLI ↔ Web）
  /api/*           — Dashboard REST API

新增:
  /a2a/.well-known/agent.json  — Agent Card（GET）
  /a2a/tasks                   — 创建任务（POST）
  /a2a/tasks/:id               — 查询/取消任务（GET/POST）
  /a2a/tasks/:id/events        — SSE 事件流（GET）
```

### 5.3 Task Handler

// A2A Task 进来 → 转化为 AgentLoop 执行 → 事件流转化为 A2A 事件

// 输入: A2A Message（包含 persona system prompt + 用户指令 + 上游 artifact）
// 输出: A2A StatusUpdate 流 + 最终 Artifact

async function* handleA2ATask(task: A2ATask): AsyncIterable<A2AEvent> {
  // 1. 从 task.messages 提取 system prompt（persona 注入）和 user message
  // 2. 构造 AgentLoop 的 messages 数组
  // 3. 执行 AgentLoop.run()
  // 4. AgentEvent → A2AEvent 映射:
  //    - text       → a2a.StatusUpdate (state: 'working', message)
  //    - tool_start → a2a.StatusUpdate (state: 'working', message: 'using tool X')
  //    - tool_done  → a2a.StatusUpdate
  //    - done       → a2a.StatusUpdate (state: 'completed') + Artifact
  //    - error      → a2a.StatusUpdate (state: 'failed')
}

## 六、Claude Code CLI 包装为 A2A Agent（示例）

展示执行引擎可替换性：

```typescript
// adapters/claude-code-adapter.ts

// Claude Code CLI 没有 HTTP 接口，通过子进程 + stdin/stdout 通信
// 适配器将 A2A Task 转化为 Claude Code CLI 的 pipe 输入

class ClaudeCodeA2AAgent {
  async executeTask(task: A2ATask): AsyncIterable<A2AEvent> {
    const prompt = task.messages.map(m => m.content).join('\n')

    // claude -p "prompt" --json
    const process = spawn('claude', ['-p', prompt, '--json'], {
      cwd: task.metadata?.cwd,
    })

    // stdout JSON 解析 → A2A StatusUpdate
    for await (const chunk of process.stdout) {
      yield mapClaudeOutputToA2A(chunk)
    }
  }

  getAgentCard(): AgentCard {
    return {
      name: 'claude-code',
      description: 'Anthropic Claude Code CLI',
      skills: [
        { id: 'code-read', name: 'Code Reading' },
        { id: 'code-write', name: 'Code Writing' },
        { id: 'code-execute', name: 'Command Execution' },
        // Claude Code 有更多工具（WebSearch, MCP 等）
        { id: 'web-search', name: 'Web Search' },
      ],
    }
  }
}
```

## 七、风险评估

### 7.1 技术风险

| 风险 | 严重度 | 应对策略 |
|------|--------|---------|
| **会议讨论不收敛** | 高 | 严格 max_rounds + 主持人强制裁决 + "保留意见"机制 |
| **质量级联失败** | 高 | 关键步骤插入人工审批节点（至少架构设计必须人工确认） |
| **Token 成本爆炸** | 高 | 每个 Persona 有 cost_limit；Pipeline 有总预算；超限暂停通知人工 |
| **A2A 协议演进** | 中 | 只实现裁剪子集；抽象 A2A 交互为接口，协议变更只改适配层 |
| **执行引擎差异** | 中 | Agent Card 声明 skills，编排层按 skills 匹配，不假设底层能力 |

### 7.2 产品风险

| 风险 | 严重度 | 应对策略 |
|------|--------|---------|
| **用户不信任 AI 自治** | 高 | 默认每步人工审批，用户可逐步放开 |
| **产出质量不稳定** | 高 | 第一版不追求全自动，先做"AI 辅助 + 人工兜底" |
| **使用场景不清晰** | 中 | 先聚焦"CRUD 功能开发"这一个场景，跑通再泛化 |

### 7.3 成本估算（单次需求流转）

| 步骤 | 预估 Token | 模型 | 预估费用 |
|------|-----------|------|---------|
| 需求分析 | 50K | Sonnet | ~$0.15 |
| 架构设计 | 100K | Opus | ~$1.50 |
| 方案评审（5轮） | 200K | Sonnet×4 | ~$0.60 |
| 任务拆解 | 30K | Sonnet | ~$0.09 |
| 并行开发（5个任务） | 500K | Sonnet×5 | ~$1.50 |
| 集成验收 | 100K | Opus | ~$1.50 |
| **合计** | **~980K** | | **~$5.34** |

> 一个中等复杂度的功能需求，全流程自动化约 5 美元。如果用 DeepSeek/GLM 作为部分执行引擎，成本可降至 $1-2。
