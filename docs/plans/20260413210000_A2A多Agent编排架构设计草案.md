> **状态**：构思阶段（未启动）

---

## 一、愿景

从 **"一个人 + 一个 AI"** 进化到 **"一个需求 + N 个 AI 角色自动协作"**。

```
现在:
  开发者 ──→ CCode CLI ──→ 代码

未来:
  需求 ──→ 编排层 ──→ PM Agent ──→ 需求文档
                    ──→ 架构 Agent ──→ 技术方案
                    ──→ [PM + 架构 + 前端 + 后端] 会议讨论 ──→ 评审结论
                    ──→ PM Agent ──→ 看板任务
                    ──→ 前端 Agent ──→ 前端代码
                    ──→ 后端 Agent ──→ 后端代码
                    ──→ 架构 Agent ──→ 集成验收
```

**核心原则**：底层执行引擎可替换。CCode CLI 是执行节点之一，未来可以换成 Claude Code CLI、Cursor Agent 或任何 A2A 兼容的 Agent。

---

## 二、为什么选 A2A 协议

### 2.1 A2A 是什么

Google 提出的 Agent-to-Agent 协议，定义了 Agent 之间的发现、通信和任务协作标准：

- **Agent Card** — JSON 格式描述 Agent 的能力、技能、端点地址
- **Task Lifecycle** — `submitted → working → input-required → completed / failed` 标准状态机
- **Message Exchange** — JSON-RPC over HTTP，支持 SSE 流式推送
- **Artifact** — Agent 产出物的标准格式（文档、代码、图片等）

### 2.2 为什么不是扩展 SubAgent

| 维度 | 扩展 SubAgent | A2A 协议 |
|------|-------------|---------|
| 执行引擎 | 只能是 CCode 内部 | 任何 A2A 兼容 Agent |
| 通信边界 | 进程内 | HTTP（跨进程/跨机器/跨组织） |
| 能力发现 | 硬编码 | Agent Card 动态发现 |
| 未来扩展 | 改代码 | 注册新 Agent Card 即可 |
| 生态对接 | 无 | Claude Code / Cursor / 其他 A2A Agent |

**关键判断**：即使当前所有 Agent 都跑在本地，用 A2A 协议也不亏——协议开销极小（HTTP localhost），但架构上预留了"未来把某个角色换成远程 Agent"的能力。

### 2.3 A2A 裁剪方向（初步）

A2A 完整规范偏重跨组织场景，我们不需要全部实现：

| A2A 功能 | 是否需要 | 原因 |
|----------|---------|------|
| Agent Card | ✅ 需要 | Agent 能力发现的基础 |
| Task CRUD | ✅ 需要 | 任务生命周期核心 |
| Streaming (SSE) | ✅ 需要 | 实时进度推送到看板 |
| Artifact | ✅ 需要 | Agent 产出物传递 |
| Push Notifications | ⚠️ 后期 | 初期用轮询/SSE 够了 |
| Authentication | ⚠️ 后期 | 初期全部本地运行，不需要认证 |
| Multi-party negotiation | ❌ 不需要 | 编排层统一调度，Agent 之间不直接协商 |

## 四、核心概念

### 4.1 Persona（角色模板）

角色模板定义 Agent 的行为特征，与执行引擎解耦：

```yaml
# personas/pm.yaml
name: "产品经理"
id: "pm"
system_prompt: |
  你是一位资深产品经理。你的职责是：
  1. 分析需求的可行性和优先级
  2. 编写结构化的需求文档（用户故事 + 验收标准）
  3. 将技术方案转化为可执行的看板任务
  你输出的文档需要简洁、无歧义、可验证。
required_skills:
  - "code-read"     # 需要能读代码理解现状
preferred_model: "claude-sonnet-4-6"  # 不需要最强模型
cost_limit_tokens: 100000

# personas/architect.yaml
name: "架构师"
id: "architect"
system_prompt: |
  你是一位资深系统架构师。你的职责是：
  1. 评估需求的技术复杂度
  2. 设计接口契约和模块边界
  3. 识别技术风险并提出方案
  4. 在方案评审中担任主持人
  你的方案需要考虑：可维护性、性能、安全性。
required_skills:
  - "code-read"
  - "code-write"
preferred_model: "claude-opus-4-6"    # 需要强推理
cost_limit_tokens: 200000
```

**关键设计**：Persona 只定义"做什么"和"怎么想"，不绑定"用什么引擎执行"。编排层根据 `required_skills` 从 Agent Registry 中匹配可用的执行引擎。

### 4.2 Pipeline（流水线）

```yaml
# pipelines/feature-dev.yaml
name: "功能开发流水线"
steps:
  - id: "requirement"
    name: "需求分析"
    persona: "pm"
    input: "$trigger.description"      # 来自用户输入
    output_artifact: "requirement.md"
    approval: optional                  # 人工审批可选

  - id: "design"
    name: "架构设计"
    persona: "architect"
    input:
      - "$steps.requirement.artifact"   # 上一步产出
    output_artifact: "design.md"
    approval: required                  # 必须人工审批

  - id: "review"
    name: "方案评审"
    type: "meeting"                     # 特殊步骤类型：会议
    moderator: "architect"
    participants: ["pm", "frontend", "backend"]
    input:
      - "$steps.requirement.artifact"
      - "$steps.design.artifact"
    max_rounds: 5
    convergence: "moderator_summary"    # 主持人总结即收敛
    output_artifact: "review_notes.md"

  - id: "task_breakdown"
    name: "任务拆解"
    persona: "pm"
    input:
      - "$steps.design.artifact"
      - "$steps.review.artifact"
    output_artifact: "kanban_tasks.json"

  - id: "implementation"
    name: "并行开发"
    type: "parallel"                    # 并行执行
    source: "$steps.task_breakdown.artifact"  # 从看板任务动态生成子任务
    assign_by: "task.role"              # 按任务标签分配角色
    report_interval: "per_task"         # 每完成一个任务汇报
    approval: none

  - id: "integration"
    name: "集成验收"
    persona: "architect"
    input:
      - "$steps.design.artifact"
      - "$steps.implementation.artifacts"  # 所有并行任务的产出
    output_artifact: "review_result.md"
```

### 4.3 Meeting（会议机制）

多 Agent 讨论是最复杂的部分，需要严格的收敛规则：

```
会议规则:
├── 主持人 (moderator): 控制发言顺序、判断是否收敛
├── 最大轮次: 超过则主持人强制总结
├── 发言顺序: 轮流制（Round Robin）
├── 收敛判定:
│   ├── moderator_summary — 主持人总结，其他人无异议
│   ├── vote — 投票（>50% 同意）
│   └── no_objection — 连续一轮无人提出新问题
└── 异常处理:
    ├── 死循环 — 超过 max_rounds，主持人强制裁决
    └── 单人阻塞 — 同一 Agent 连续 3 轮提同一问题，标记为 "保留意见" 继续推进
```

### 4.4 Kanban（看板）

### 5.3 Task Handler

```typescript
// A2A Task 进来 → 转化为 AgentLoop 执行 → 事件流转化为 A2A 事件

// 输入: A2A Message（包含 persona system prompt + 用户指令 + 上游 artifact）
// 输出: A2A StatusUpdate 流 + 最终 Artifact

async function* handleA2ATask(task: A2ATask): AsyncIterable<A2AEvent> {
  // 1. 从 task.messages 提取 system prompt（persona 注入）和 user message
  // 2. 构造 AgentLoop 的 messages 数组
  // 3. 执行 AgentLoop.run()
  // 4. AgentEvent → A2AEvent 映射:
  //    - text       → a2a.StatusUpdate (state: 'working', message)
  //    - tool_start → a2a.StatusUpdate (state: 'working', message: 'using tool X')
  //    - tool_done  → a2a.StatusUpdate
  //    - done       → a2a.StatusUpdate (state: 'completed') + Artifact
  //    - error      → a2a.StatusUpdate (state: 'failed')
}
```

## 六、Claude Code CLI 包装为 A2A Agent（示例）

展示执行引擎可替换性：

```typescript
// adapters/claude-code-adapter.ts

// Claude Code CLI 没有 HTTP 接口，通过子进程 + stdin/stdout 通信
// 适配器将 A2A Task 转化为 Claude Code CLI 的 pipe 输入

class ClaudeCodeA2AAgent {
  async executeTask(task: A2ATask): AsyncIterable<A2AEvent> {
    const prompt = task.messages.map(m => m.content).join('\n')

    // claude -p "prompt" --json
    const process = spawn('claude', ['-p', prompt, '--json'], {
      cwd: task.metadata?.cwd,
    })

    // stdout JSON 解析 → A2A StatusUpdate
    for await (const chunk of process.stdout) {
      yield mapClaudeOutputToA2A(chunk)
    }
  }

  getAgentCard(): AgentCard {
    return {
      name: 'claude-code',
      description: 'Anthropic Claude Code CLI',
      skills: [
        { id: 'code-read', name: 'Code Reading' },
        { id: 'code-write', name: 'Code Writing' },
        { id: 'code-execute', name: 'Command Execution' },
        // Claude Code 有更多工具（WebSearch, MCP 等）
        { id: 'web-search', name: 'Web Search' },
      ],
    }
  }
}
```

## 七、风险评估

### 7.1 技术风险

| 风险 | 严重度 | 应对策略 |
|------|--------|---------|
| **会议讨论不收敛** | 高 | 严格 max_rounds + 主持人强制裁决 + "保留意见"机制 |
| **质量级联失败** | 高 | 关键步骤插入人工审批节点（至少架构设计必须人工确认） |
| **Token 成本爆炸** | 高 | 每个 Persona 有 cost_limit；Pipeline 有总预算；超限暂停通知人工 |
| **A2A 协议演进** | 中 | 只实现裁剪子集；抽象 A2A 交互为接口，协议变更只改适配层 |
| **执行引擎差异** | 中 | Agent Card 声明 skills，编排层按 skills 匹配，不假设底层能力 |

### 7.2 产品风险

| 风险 | 严重度 | 应对策略 |
|------|--------|---------|
| **用户不信任 AI 自治** | 高 | 默认每步人工审批，用户可逐步放开 |
| **产出质量不稳定** | 高 | 第一版不追求全自动，先做"AI 辅助 + 人工兜底" |
| **使用场景不清晰** | 中 | 先聚焦"CRUD 功能开发"这一个场景，跑通再泛化 |

### 7.3 成本估算（单次需求流转）

| 步骤 | 预估 Token | 模型 | 预估费用 |
|------|-----------|------|---------|
| 需求分析 | 50K | Sonnet | ~$0.15 |
| 架构设计 | 100K | Opus | ~$1.50 |
| 方案评审（5轮） | 200K | Sonnet×4 | ~$0.60 |
| 任务拆解 | 30K | Sonnet | ~$0.09 |
| 并行开发（5个任务） | 500K | Sonnet×5 | ~$1.50 |
| 集成验收 | 100K | Opus | ~$1.50 |
| **合计** | **~980K** | | **~$5.34** |

> 一个中等复杂度的功能需求，全流程自动化约 5 美元。如果用 DeepSeek/GLM 作为部分执行引擎，成本可降至 $1-2。
