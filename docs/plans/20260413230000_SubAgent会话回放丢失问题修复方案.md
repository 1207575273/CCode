# SubAgent 会话回放丢失问题修复方案

## 一、问题描述

### 现象

在 Web Dashboard 的会话回放（ConversationsPage）和实时对话（ChatPage）中，SubAgent 的执行历史无法正确关联到主对话流的 `dispatch_agent` 工具调用位置。SubAgent 的 JSONL 数据已完整记录，API 也能正确返回，但前端渲染时"找不到"对应的 SubAgent 卡片。

### 用户体验影响

- 回放历史会话时，看到 `dispatch_agent` 工具条目但看不到子 Agent 做了什么
- 子 Agent 数据只能通过 SubAgentDrawer（悬浮按钮 + 抽屉面板）查看，与主对话流脱节
- 无法直观理解"主 Agent 在哪一步派发了子 Agent、子 Agent 做了什么、结果如何"

## 二、根因分析

### 问题 1：agentId 正则匹配失败（直接 bug）

**位置**：`web/src/components/ToolStatus.tsx` `extractAgentId()` 函数

```typescript
function extractAgentId(resultSummary?: string): string | null {
  if (!resultSummary) return null
  const match = resultSummary.match(/agentId:\s*([a-f0-9]+)/)  // ← 错误正则
  return match?.[1] ?? null
}
```

`dispatch_agent` 的 `resultSummary` 是 JSON 字符串，实际格式为：
```json
{"status":"completed","agentId":"abc123def456789","name":"test-agent",...}
```

正则 `/agentId:\s*([a-f0-9]+)/` 期望的是 `agentId: abc123` 格式，**永远匹配不到 JSON 中的 `"agentId":"abc123"`**。

这意味着 `ToolStatus` 组件中每个 `dispatch_agent` 工具条目旁边的 `SubAgentCard` **从未被渲染过**。

### 问题 2：subAgents 数据仅在侧面板展示，未嵌入对话流（架构缺陷）

当前数据流：

```
API /conversations/:id
  → { messages: [...], subagents: [...] }  // 两个独立数组

ConversationsPage
  → messages 按顺序渲染 MessageBubble
  → subAgents 作为 Map 传给 MessageBubble → ToolStatus
  → ToolStatus 通过 extractAgentId(resultSummary) 尝试关联 → 失败（问题 1）
  → SubAgentDrawer 接收 subAgents Map → 可展示但与消息流脱节
```

即使修复了正则（问题 1），关联链路仍然脆弱：
- 依赖从 `resultSummary`（截断到 200 字符的文本）中正则提取 agentId
- `resultSummary` 是 JSON 字符串，正则提取不如结构化字段可靠
- 后台模式子 Agent 的 `dispatch_agent` tool_done 事件的 `resultSummary` 是 `async_launched` 的 JSON

### 问题 3：后台子 Agent 完成后无法追溯（缺失场景）

后台子 Agent（`run_in_background=true`）的结果通过 `task_output` 工具读取，不再经过 `dispatch_agent` 的 `tool_done` 事件。这意味着：
- `dispatch_agent` 的 `resultSummary` 只包含 `async_launched` 信息
- 后台子 Agent 完成后，其详细执行过程无法在主对话流中找到锚点

## 三、修复方案

### 方案概述

分两层修复：**L1 修复关联断裂（快速止血）** + **L2 增强结构化关联（根本解决）**

---

### L1：修复 extractAgentId 正则 + 补充 resultFull 提取

**改动文件**：`web/src/components/ToolStatus.tsx`

**改动 1**：修复正则，支持 JSON 格式

```typescript
// 修复前
function extractAgentId(resultSummary?: string): string | null {
  if (!resultSummary) return null
  const match = resultSummary.match(/agentId:\s*([a-f0-9]+)/)
  return match?.[1] ?? null
}

// 修复后：支持 JSON 格式 "agentId":"xxx" 和旧格式 agentId: xxx
function extractAgentId(text?: string): string | null {
  if (!text) return null
  // JSON 格式："agentId":"abc123def456789"
  const jsonMatch = text.match(/"agentId"\s*:\s*"([a-f0-9]+)"/)
  if (jsonMatch?.[1]) return jsonMatch[1]
  // 旧格式兼容：agentId: abc123def456789
  const plainMatch = text.match(/agentId:\s*([a-f0-9]+)/)
  return plainMatch?.[1] ?? null
}
```

**改动 2**：优先从 `resultFull` 提取（不受 200 字符截断影响）

```typescript
// ToolStatus 组件中
const agentId = e.toolName === 'dispatch_agent'
  ? extractAgentId(e.resultFull ?? e.resultSummary)
  : null
```

**效果**：修复后 SubAgentCard 能正确渲染在 dispatch_agent 工具条目旁边。

---

### L2：tool_done 事件增加结构化 agentId 字段

L1 的关联仍然依赖正则从文本中提取 agentId，不够健壮。L2 通过在数据源头添加结构化字段，彻底消除正则依赖。

#### L2-a：AgentLoop tool_done 事件的 meta 字段

`dispatch_agent` 已经支持通过 `ToolResult.meta` 返回结构化元数据（`ToolResultMeta` 联合类型）。

**改动文件**：`src/tools/core/types.ts`

在 `ToolResultMeta` 联合类型中新增：

```typescript
| { type: 'dispatch-agent'; agentId: string; agentName: string; agentType: string; status: string }
```

**改动文件**：`src/tools/agent/dispatch-agent.ts`

在所有 `return { success: true, output: ... }` 处补充 `meta` 字段：

```typescript
return {
  success: true,
  output: JSON.stringify(output),
  meta: { type: 'dispatch-agent', agentId, agentName: agentName, agentType, status: 'completed' },
}
```

#### L2-b：JSONL tool_call_end 事件持久化 agentId

**改动文件**：`src/observability/session-logger.ts`

`tool_call_end` 事件已经写入 `resultSummary` 和 `resultFull`，额外写入 `meta` 中的 `agentId`：

```typescript
case 'tool_done':
  this.#appendEvent('tool_call_end', {
    toolName: event.toolName,
    toolCallId: event.toolCallId,
    durationMs: event.durationMs,
    success: event.success,
    resultSummary: event.resultSummary,
    resultFull: event.resultFull,
    // L2：结构化 agentId（dispatch_agent 专用）
    ...(event.meta?.type === 'dispatch-agent' ? { agentId: event.meta.agentId } : {}),
  })
  break
```

#### L2-c：SessionStore loadMessages 在 SnapshotToolEvent 中携带 agentId

**改动文件**：`src/persistence/session-store.ts`

`loadMessages()` 构建 `SnapshotToolEvent` 时提取 `agentId`：

```typescript
if (event.type === 'tool_call_end' && event.toolCallId && event.toolName) {
  const toolEvt: SnapshotToolEvent = {
    toolCallId: event.toolCallId,
    toolName: event.toolName,
    args: toolStartArgs.get(event.toolCallId) ?? {},
    // L2：关联 SubAgent
    ...(event.agentId ? { agentId: event.agentId } : {}),
  }
  // ...
}
```

**改动文件**：`src/persistence/session-types.ts`

```typescript
export interface SnapshotToolEvent {
  // ... 已有字段
  /** dispatch_agent 关联的子 Agent ID */
  agentId?: string
}
```

#### L2-d：前端使用结构化 agentId

**改动文件**：`web/src/types.ts`

```typescript
export interface ToolEvent {
  // ... 已有字段
  agentId?: string
}
```

**改动文件**：`web/src/components/ToolStatus.tsx`

```typescript
// 优先使用结构化 agentId，降级到正则提取
const agentId = e.toolName === 'dispatch_agent'
  ? (e.agentId ?? extractAgentId(e.resultFull ?? e.resultSummary))
  : null
```

---

### 方案对比

| 维度 | L1（正则修复） | L2（结构化字段） |
|------|---------------|-----------------|
| 改动范围 | 1 个文件 | 6 个文件 |
| 可靠性 | 中（依赖文本正则） | 高（结构化字段直传） |
| 向后兼容 | 完全兼容 | 完全兼容（L2-d 有降级逻辑） |
| 历史数据 | 能修复所有已有 JSONL | 只对新产生的 JSONL 有效，旧数据降级到 L1 正则 |

**建议**：L1 和 L2 一起做。L1 保证旧数据也能正确关联，L2 保证新数据结构化可靠。两者互为补充。
