# 流式工具结果（Streaming Tool Results）分析与轻量方案

## 一、需求背景

bash 工具执行长命令时（如 `npm install`、`docker build`），用户只能看到 spinner 转圈，不知道命令是否在正常运行，产生"卡死感"。

期望效果：执行期间能看到进度信息，用户知道"还在跑"。

## 二、当前链路

```
tool_start → UI 显示 spinner "Running command..." → 等待进程退出 → tool_done → 显示结果
```

- bash 工具用 `spawn()` 执行命令，stdout/stderr **缓存在内存中**
- 进程退出后一次性返回 `ToolResult`
- 执行期间 UI 只有一个旋转 spinner，无任何进度信息

## 三、完整方案评估（StreamableTool 改造）

### 改动范围（5-7 层）

| 层 | 改动 | 复杂度 |
|---|------|--------|
| BashTool | 从 Tool 改为 StreamableTool，stream() yield 中间输出 | 中 |
| AgentEvent | 新增 `tool_output` 事件类型 | 低 |
| AgentLoop | StreamableTool 串行执行策略适配 | **高风险** |
| CLI UI | ToolStatusLine 滚动展示中间输出 | 中 |
| Web UI | ChatPage/ToolStatus 处理 tool_output 事件 | 中 |
| SessionLogger | tool_output 事件持久化决策 | 设计决策 |
| Bridge | tool_output 事件广播 | 低 |

### 核心矛盾：并行执行能力受损

当前设计：
- **Safe 工具**（read_file、glob、grep、bash）→ `Promise.allSettled` **并行**执行
- **Dangerous / StreamableTool**（dispatch_agent）→ **串行**执行

bash 一旦变成 StreamableTool：
- LLM 同时返回 3 个 bash tool_call → 只能串行 → **整体变慢**
- 改造 parallel-executor 支持多 StreamableTool 并行 yield → 多 generator 合并极其复杂

### 实际收益

| 场景 | 频率 | 耗时 | 流式收益 |
|------|------|------|---------|
| `git status` / `ls` | 高 | <1s | 无 |
| `tsc --noEmit` | 中 | 5-15s | 改善不大 |
| `npm install` | 中 | 30-60s | 有价值 |
| `docker build` | 低 | 1-5min | 有价值 |

**结论：80% 的 bash 调用在 5 秒内完成，完整流式改造投入产出比不高。**

### 评估结论

| 维度 | 评估 |
|------|------|
| 价值 | 中等 — 只改善少数长命令场景 |
| 成本 | 高 — 5-7 层改动，预计 2-3 天 |
| 风险 | bash 并行能力受损，可能引入回归 |
| **决策** | **不做，采用轻量替代方案** |

## 四、轻量方案：ToolStatusLine 显示已执行时长

### 方案描述

在工具 running 状态时显示实时计时：

```
修改前：⟳ Bash(npm install)
修改后：⟳ Bash(npm install)  12s
```

### 改动范围

仅修改 `src/ui/ToolStatusLine.tsx`，零后端改动。

### 优势

- 解决 80% 的"卡死感"— 用户看到时间在走，知道没卡死
- 零风险 — 不改执行链路、不影响并行策略
- 工时 < 0.5 小时

### 未来扩展空间

如果后续确实需要完整流式，可以分阶段推进：
1. 先做本方案（计时器）→ 已完成
2. 观察用户反馈，确认长命令场景是否高频
3. 如有必要，再做 StreamableTool 改造（仅对 bash 工具，不影响并行分组逻辑——可以用"达到时间阈值后自动升级为后台模式"的策略替代）
