## 2. 性能层指标详解

### 2.1 TTFT（Time To First Token）— 首 Token 延迟

**定义**：从发送 API 请求到收到第一个 streaming token 的时间。

**为什么重要**：
- 直接影响用户感知的"响应速度"——用户按 Enter 后多久看到 AI 开始"打字"
- TTFT > 3s 用户会觉得"卡死了"，> 5s 会尝试中断重发
- 不同模型差异很大：小模型 ~200ms，大模型 ~1-3s

**行业基准（2025）**：

| 模型 | 典型 TTFT | 说明 |
|------|----------|------|
| Claude Sonnet 4 | 0.8 - 1.5s | 受 Prompt Caching 影响大，命中时可降至 ~300ms |
| Claude Opus 4 | 1.5 - 3.0s | 更大模型，推理更深 |
| GPT-4o | 0.5 - 1.2s | |
| Gemini 1.5 Pro | 0.3 - 1.0s | Google 基础设施优势 |
| DeepSeek V3 | 0.5 - 2.0s | 受部署区域影响 |
| GLM-4 | 0.3 - 1.0s | 国内部署延迟低 |

**影响 TTFT 的因素**：
1. **输入 token 数**——越长越慢（prefill 阶段处理整个 prompt）
2. **Prompt Caching 命中**——命中时跳过 prefill，TTFT 大幅下降
3. **网络延迟**——跨洋请求 +200-400ms
4. **模型大小**——Opus 比 Sonnet 慢
5. **并发负载**——高峰期排队等待

### 2.2 TPS（Tokens Per Second）— 输出吞吐率

**定义**：每秒生成的输出 token 数。反映模型的"打字速度"。

**为什么重要**：
- 决定长回复的等待时间——1000 token 的回复，80 TPS 需要 12.5s，30 TPS 需要 33s
- 直接影响 Agent 多轮交互的总耗时——Agent 可能连续调用 10 次 LLM
- 不同模型差异显著，小模型通常更快

**行业基准（2025）**：

| 模型 | 典型 TPS | 说明 |
|------|---------|------|
| Claude Sonnet 4 | 80 - 120 | |
| Claude Opus 4 | 40 - 70 | 大模型生成更慢 |
| Claude Haiku | 150 - 200 | 小模型极快 |
| GPT-4o | 60 - 100 | |
| GPT-4o-mini | 120 - 180 | |
| Gemini 1.5 Pro | 100 - 150 | |
| DeepSeek V3 | 40 - 80 | |

**计算方式**：`TPS = outputTokens / (e2eLatency - ttft)`

### 2.3 TPOT（Time Per Output Token）— 单 Token 生成时间

**定义**：平均每个输出 token 的生成耗时。是 TPS 的倒数。

`TPOT = 1000 / TPS (ms)`

| TPS | TPOT |
|-----|------|
| 30 | 33ms |
| 80 | 12.5ms |
| 120 | 8.3ms |
| 200 | 5ms |

**用户感知阈值**：TPOT < 15ms 时（TPS > 67），streaming 输出在视觉上接近"即时"。

### 2.4 ITL（Inter-Token Latency）— Token 间延迟

**定义**：相邻两个 token 之间的时间间隔。与 TPOT 的区别：

- TPOT 是**平均值**
- ITL 是**逐 token 的实际间隔**，可以画分布图

ITL 波动大说明模型推理不稳定（可能遇到了难的 token、或触发了 KV-cache 重计算）。

**在 CCode 中的价值**：ITL 方差可用于检测"卡顿"——如果某个 token 的 ITL 突然 >500ms，可能是服务端出了问题。

### 2.5 E2E Latency（端到端延迟）

**定义**：从请求发出到最后一个 token 收到的总时间。

`E2E = TTFT + (outputTokens × TPOT)`

**在 Agent 场景中的特殊性**：
- 一个用户请求可能触发 5-15 次 LLM 调用（多轮工具 use → observe → act）
- **用户感知的 E2E** ≠ **单次 LLM 调用的 E2E**
- 需要同时记录 per-call E2E 和 per-turn E2E

## 3. 资源层指标

### 3.1 Cache Hit Rate — 缓存命中率

**定义**：`cacheHitRate = cacheReadTokens / (cacheReadTokens + inputTokens)`

**为什么重要**：
- Anthropic Prompt Caching 命中时：TTFT 降低 ~80%，input 成本降低 ~90%
- 命中率直接反映 system prompt 和 context 的复用效率
- 目标值：多轮对话中应 > 60%（system prompt + 历史消息被缓存）

### 3.2 Context Window 使用率

**定义**：`contextUsage = totalInputTokens / maxContextTokens`

**阈值**：
- < 50%：安全区
- 50-80%：注意区，考虑 compact
- 80%：预警，触发自动 compact
- > 90%：危险，可能截断丢失上下文

### 3.3 Token 四维分解

CCode 记录 4 个 token 维度，全部独立计价：

| 维度 | 含义 | 计价特点 |
|------|------|---------|
| inputTokens | 未命中缓存的输入 token | 标准输入价格 |
| outputTokens | 模型生成的输出 token | 最贵，通常是 input 的 3-5 倍 |
| cacheReadTokens | 命中缓存的 token | 最便宜，通常是 input 的 10% |
| cacheWriteTokens | 写入缓存的 token | 比 input 贵 ~25%，但只交一次 |

## 4. 可靠性层指标

### 4.1 错误率

**定义**：`errorRate = failedCalls / totalCalls`

**分类**：
- 400 Bad Request — 通常是 prompt 构造错误
- 401 Unauthorized — API Key 问题
- 429 Rate Limited — 限流，需要退避重试
- 500 Server Error — 模型服务端问题
- 503 Overloaded — 高峰期过载
- Timeout — 超时未响应

**目标**：< 1%（排除 429 限流）

### 4.2 重试率

**定义**：`retryRate = retriedCalls / totalCalls`

**CCode 现状**：有重试机制（providers/retry.ts），但尚未统计重试次数（Phase 3 待实施）。

### 4.3 超时率

**定义**：请求超过设定时间未完成的比例。

不同场景的合理超时阈值：
- 短回复（< 500 token）：30s
- 长回复（> 2000 token）：120s
- 工具调用响应：60s

## 5. 质量层指标

### 5.1 工具调用准确率

**定义**：模型正确选择并调用工具的比例。

**评估维度**：
- **选择准确率**：应该调用工具时是否调用了？不应该调用时是否误调？
- **参数准确率**：调用参数是否正确？路径、命令、搜索词是否合理？
- **调用效率**：完成任务所需的工具调用次数是否合理？（过多说明模型在"瞎试"）

**CCode 中可间接衡量**：
- 工具调用失败率（tool_result 中 isError 的比例）
- 平均每个 turn 的工具调用次数
- 重复调用同一工具的频率（RepetitionDetector 已有部分能力）

### 5.2 指令遵循率

**定义**：模型是否按照 system prompt 和用户指令的要求行事。

**CCode 场景中的表现**：
- 是否遵循 CCODE.md / CLAUDE.md 中的编码规范？
- 是否在需要时使用中文回复？
- 是否在修改文件前先读取文件？

**难以自动化衡量**，通常靠人工评估或 LLM-as-Judge。

### 5.3 上下文保持能力

**定义**：长对话中模型是否还记得早期的信息。

**衡量方式**：
- 对话第 N 轮时，模型是否还能正确引用第 1 轮提到的文件/变量？
- compact 后信息损失率——compact 前后的回答质量差异

### 5.4 拒绝率

**定义**：模型拒绝执行合理请求的比例（安全过滤误报）。

**CCode 场景**：模型拒绝写代码、拒绝执行 bash 命令、拒绝修改文件。
**目标**：< 2%（对合理的编程请求）。

## 6. 行业参考

### 6.1 OpenTelemetry GenAI Semantic Conventions

OpenTelemetry 在 2024 年发布了 GenAI 语义约定，定义了标准的 span 属性：

```
gen_ai.system              — 供应商（anthropic / openai / ...）
gen_ai.request.model       — 模型名
gen_ai.request.max_tokens  — 最大输出 token
gen_ai.request.temperature — 温度
gen_ai.usage.input_tokens  — 输入 token 数
gen_ai.usage.output_tokens — 输出 token 数
gen_ai.response.model      — 实际使用的模型（可能与请求不同）
gen_ai.response.finish_reason — 结束原因（stop / length / tool_use）
```

**目前缺失但值得跟进的标准属性**：
- `gen_ai.usage.cache_read_tokens` — 尚未纳入标准，但 Anthropic 已有
- `gen_ai.latency.ttft` — 社区讨论中
- `gen_ai.latency.tps` — 社区讨论中

### 6.2 主流可观测性平台对比

| 平台 | 模式 | 核心能力 | 适合场景 |
|------|------|---------|---------|
| **Helicone** | 代理层（改 base_url） | 无侵入采集，按 request 粒度 | 快速接入 |
| **Langsmith** | SDK 集成 | trace 级追踪，支持 chain/agent 分步 | 复杂 Agent |
| **Portkey** | 网关 | 内置 fallback/retry/cache，统一多模型 | 多模型管理 |
| **Braintrust** | SDK + eval | 自动评估质量（LLM-as-Judge） | 质量优化 |

### 6.3 各平台的可视化设计模式

#### Helicone — 代理层可观测

核心指标卡片 + 时间序列图的组合：
- **顶部**：4 个数字卡片（请求数、延迟 P50/P99、成本、错误率）
- **中间**：Latency 时间序列图（散点+趋势线）、请求量折线图
- **底部**：按模型分组的对比表

特点：**P50/P99 双分位数**是核心指标，比平均值更有意义。

#### Langsmith — Trace 追踪平台

- **Trace 级延迟瀑布图**：每次 LLM 调用展示为一个 span（TTFT + generation 两段）
- **延迟分布直方图**：X 轴=延迟区间，Y 轴=请求数，看分布形态
- **P50/P90/P99 趋势线**：三条分位线叠加在同一图上

特点：**延迟分布**比点值更有价值，能发现长尾问题。

#### Portkey — AI 网关

- **实时仪表盘**：TPS gauge 表盘 + Latency gauge 表盘
- **模型对比雷达图**：多维度同时对比（TTFT、TPS、成本、错误率）
- **成本效率散点图**：X=延迟，Y=成本，每个点是一个模型

特点：**模型选型导向**——所有图表都在回答"该选哪个模型"。

#### Grafana + OpenTelemetry — 通用方案

RED 方法论（Rate、Errors、Duration）：
- **Rate**：请求量折线图（按 provider 分色）
- **Errors**：错误率折线图 + 错误类型饼图
- **Duration**：延迟热力图（时间 × 延迟区间 × 颜色深浅=请求密度）

特点：**热力图**是延迟分析的终极可视化——同时看趋势、分布、异常。

### 6.4 CCode 的定位

**自建观测，不依赖第三方平台**。数据存本地 SQLite，Web Dashboard 展示。

- 优点：数据完全私有，无外部依赖
- 取舍：不做 LLM-as-Judge（质量层），不做分布式 trace（单机场景）

## 📅 2026-04-16 · 方案设计

> 初次梳理 LLM 观测指标体系，识别 CCode 覆盖缺口，决定补齐优先级。

### 现状判断

CCode 当时的观测能力**仅覆盖资源层**（token 四维 + 费用 + context 使用率 + 调用计数）。

五层观测模型中：
- **资源层** ✅ 已覆盖
- **性能层** ❌ 空白 — TTFT/TPS/E2E 全无
- **可靠性层** ❌ 空白
- **质量层** ❌ 空白
- **业务层** ❌ 空白

### 阶段性结论

**现阶段优先补齐性能层。理由：**

1. **客观可测量** — TTFT、TPS、E2E 都是纯数值，采集即记录，不需要定义"什么算好"
2. **采集点明确** — 全部集中在 agent-loop.ts 的 LLM 调用处，改动面极小
3. **用户最直接的痛点** — "为什么这么慢"是用户最常问的问题，有数据才能回答
4. **为模型选型提供依据** — 多模型场景下（Claude/GPT/GLM/DeepSeek），性能对比是选型的硬指标
5. **Cache 优化的前提** — 不知道 TTFT 和 Cache Hit Rate，就无法衡量 Prompt Caching 是否生效

**质量层暂不做。理由：**

1. **定义不清** — "工具调用准确率"怎么算？调了一个不该调的工具算不准确，还是调了但参数错了算不准确？边界模糊
2. **评估成本高** — 指令遵循率需要 LLM-as-Judge，本身就要额外 API 调用和成本
3. **样本不足** — 质量评估需要大量标注数据做基线，当前用户量不够
4. **主观性强** — 同一个回答，不同用户可能有不同评价，难以自动化

**可靠性层部分可顺带做**（错误分类统计成本极低，在 provider 层 catch 里加计数）。

## 📅 2026-04-17 · Phase 1 实施：资源层补齐 + 性能层采集

### 现状审计

#### usage_logs 表字段 vs 实际写入

```
建表定义                              TokenMeter INSERT 实际写入
─────────                             ──────────────────────
input_tokens / output_tokens          ✅
```

**结论**：`duration_ms` 是一个"预留但空置"的字段，可以直接利用，无需加新列。

#### agent-loop.ts `#callLLM` 计时情况

```
现有计时点                                缺失
──────────                               ──────
工具执行：Date.now() 前后 ✅               LLM 调用：无任何计时 ❌

                                          输出吞吐率：无 ❌
```

#### AgentEvent `llm_done` 当前字段

原字段：`inputTokens / outputTokens / cacheReadTokens / cacheWriteTokens / stopReason`——纯 token 统计，无时间维度。

### 数据库版本化迁移框架

#### 问题背景

当前 db.ts 没有任何版本管理：

```
现状                                     问题
────                                    ────
CREATE TABLE IF NOT EXISTS              只管"有没有表"，不管"表结构对不对"
seedDefaultPricing（INSERT 去重）        只管"数据有没有"，不管"数据是否最新"
无版本号                                 没法知道用户本地 DB 是哪个版本的 schema
新增字段只能 try/catch ALTER TABLE       脆弱、不可追踪、无法回溯
```

#### 方案：PRAGMA user_version + 迁移函数数组

SQLite 内置 `PRAGMA user_version`——一个存在 DB 文件头的整数，无需额外建表：

```typescript
db.pragma('user_version')         // 读 → 返回 [{ user_version: 0 }]
db.pragma('user_version = 3')     // 写 → 原子更新
```

**核心设计**：

```
                        ┌──────────────────────────────────────────┐
                        │     SCHEMA_VERSION = migrations.length   │
                        │     （代码中声明的目标版本号）              │
                        └──────────────┬───────────────────────────┘
                                       │
          ┌────────────────────────────┼────────────────────────────┐
          │                            │                            │
    user_version = 0             user_version = 1            user_version = N
    （全新 / 旧库）              （v1 schema）                （已是最新）
          │                            │                            │
          ▼                            ▼                            ▼
    执行 migrations[0]           执行 migrations[1..]         跳过，直接返回
    （v0 → v1: 删表重建）        （增量迁移）
          │                            │
          └────────┬───────────────────┘
                   ▼
         设置 user_version = SCHEMA_VERSION（事务外）
```

#### 迁移函数数组

每个迁移函数负责**从版本 N 升级到 N+1**，按索引顺序排列：

```typescript
type MigrationFn = (db: DatabaseType) => void

const migrations: MigrationFn[] = [
  // ── v0 → v1：基线 schema ──
  // 当前阶段项目处于早期迭代，历史数据价值有限，直接 DROP + 重建
  (db) => {
    db.exec('DROP TABLE IF EXISTS usage_logs')
    db.exec('DROP TABLE IF EXISTS pricing_rules')
    db.exec('DROP TABLE IF EXISTS schema_comments')
    db.exec('DROP TABLE IF EXISTS memory_meta')
    db.exec('DROP TABLE IF EXISTS memory_vectors')
    db.exec('DROP INDEX IF EXISTS idx_memory_vec')

    // 建 usage_logs / pricing_rules / schema_comments / memory_meta
    // memory_vectors 延迟建表（维度是运行时参数）

    seedDefaultPricing(db)
    seedSchemaComments(db)
  },
  // ── v1 → v2：预留（未来扩展）──
]
```

#### 迁移执行器

```typescript
const SCHEMA_VERSION = migrations.length

function runMigrations(db: DatabaseType): void {
  const current = (db.pragma('user_version') as any[])[0]?.user_version ?? 0
  if (current >= SCHEMA_VERSION) return

  // 逐个执行（不套外层事务——migrations 内部的 seed 函数自带事务，
  // SQLite 不支持嵌套事务）
  for (let i = current; i < SCHEMA_VERSION; i++) {
    const migration = migrations[i]
    if (migration) migration(db)
  }

  // PRAGMA user_version 不受事务保护，必须迁移成功后才设置
  db.pragma(`user_version = ${SCHEMA_VERSION}`)
}
```

#### memory_vectors：延迟建表 + 独立版本追踪

`memory_vectors` 的 schema 依赖运行时参数 `dimension`（向量维度），无法在静态迁移数组中硬编码。

**方案**：migrations 只管 `memory_meta`。`memory_vectors` 由 `ensureMemoryVectors()` 独立管理，版本号存在 `memory_meta.vectors_schema_version` 键下。

```typescript
const VECTORS_SCHEMA_VERSION = 1

export function ensureMemoryVectors(dimension: number): void {
  const db = getDb()
  if (dimension <= 0) return

  // 维度变化 → DROP + 重建（F32_BLOB 维度写死在列定义里）
  const existingDim = getMemoryMeta(db, 'embedding_dimension')
  if (existingDim !== null && parseInt(existingDim, 10) !== dimension) {
    db.exec('DROP TABLE IF EXISTS memory_vectors')
    db.exec('DROP INDEX IF EXISTS idx_memory_vec')
    setMemoryMeta(db, 'vectors_schema_version', '0')
  }

  // 按版本号增量迁移
  const currentVer = parseInt(getMemoryMeta(db, 'vectors_schema_version') ?? '0', 10)
  if (currentVer < VECTORS_SCHEMA_VERSION) {
    // 建表 + 索引
    setMemoryMeta(db, 'vectors_schema_version', String(VECTORS_SCHEMA_VERSION))
  }
  setMemoryMeta(db, 'embedding_dimension', String(dimension))
}
```

```
┌─────────────────────────────────────────────────────────┐
│                    PRAGMA user_version                    │

│                                                          │
```

#### 踩坑记录：SQLite 嵌套事务

初版的 `runMigrations` 把 DDL 执行包在 `db.transaction()` 里，但 migration 内部的 `seedDefaultPricing` 也用了 `db.transaction()` 做批量插入。SQLite 不支持嵌套事务，报 `cannot start a transaction within a transaction`。

### TPS 计算特殊处理（实施后发现）

**问题**：Anthropic provider 的 `tool_call` chunk 是在流式循环结束后才 yield 的。对于纯工具调用响应（无文本输出），TTFT 触发时已接近 E2E，导致 `generationMs ≈ 0`，TPS 被除成 0。

**现象**：用户反馈 TTFT 有数据但 TPS 总是 0。

**修复**：agent-loop.ts `#callLLM` 的 TPS 计算引入退化分支：

```typescript
const e2eMs = Date.now() - requestStart
const generationMs = e2eMs - ttftMs
// 50ms 阈值避免极小值放大噪声
const tpsBase = generationMs > 50 ? generationMs : e2eMs
const tps = tpsBase > 0 && outputTokens > 0
  ? Math.round(outputTokens / (tpsBase / 1000) * 10) / 10
  : 0
```

**语义含义**：
- 有流式文本生成 → TPS = 纯 generation 阶段吞吐率（精确）
- 纯工具调用 / 极短响应 → TPS = 整体吞吐率（退化但有意义）

debug 日志输出 `base=Xms` 标记区分两种情况。历史数据（此前提交前的 TPS=0）无法回填，新调用会被正确计算。

### 潜在改动点

1. **usage_logs 表**：新增 `error_code` / `retry_count` 字段（走 migrations[1] 增量迁移，不动历史数据）
2. **AgentEvent**：`llm_error` 扩展 HTTP status 信息
3. **Web OverviewPage**：新增"可靠性层"Tab，展示错误率饼图、重试率趋势

### 实施前的前置工作

- 在 providers 层定义统一的错误分类常量
- 确认 retry.ts 的重试机制能正确区分"主动重试"vs"被动失败"
- 评估高流量场景下 error_code 字段对写入性能的影响

### 何时启动

- 当用户反馈"调用失败率异常"类问题超过 3 次
- 或用户量上升到百调用/天量级，错误统计有意义时
- 或发起一次完整的可靠性专项优化时
