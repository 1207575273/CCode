## 二、性能层指标详解

### 2.1 TTFT（Time To First Token）— 首 Token 延迟

**定义**：从发送 API 请求到收到第一个 streaming token 的时间。

**为什么重要**：
- 直接影响用户感知的"响应速度"——用户按 Enter 后多久看到 AI 开始"打字"
- TTFT > 3s 用户会觉得"卡死了"，> 5s 会尝试中断重发
- 不同模型差异很大：小模型 ~200ms，大模型 ~1-3s

**行业基准（2025）**：

| 模型 | 典型 TTFT | 说明 |
|------|----------|------|
| Claude Sonnet 4 | 0.8 - 1.5s | 受 Prompt Caching 影响大，命中时可降至 ~300ms |
| Claude Opus 4 | 1.5 - 3.0s | 更大模型，推理更深 |
| GPT-4o | 0.5 - 1.2s | |
| Gemini 1.5 Pro | 0.3 - 1.0s | Google 基础设施优势 |
| DeepSeek V3 | 0.5 - 2.0s | 受部署区域影响 |
| GLM-4 | 0.3 - 1.0s | 国内部署延迟低 |

**影响 TTFT 的因素**：
1. **输入 token 数**——越长越慢（prefill 阶段处理整个 prompt）
2. **Prompt Caching 命中**——命中时跳过 prefill，TTFT 大幅下降
3. **网络延迟**——跨洋请求 +200-400ms
4. **模型大小**——Opus 比 Sonnet 慢
5. **并发负载**——高峰期排队等待

**在 CCode 中的采集点**：

```typescript
// agent-loop.ts 中 LLM 调用
const requestStart = Date.now()
const stream = await provider.chat(messages, { stream: true })

for await (const chunk of stream) {
  if (firstChunk) {
    const ttft = Date.now() - requestStart   // ← 这就是 TTFT
    firstChunk = false
  }
  // ... 处理 chunk
}
const e2eLatency = Date.now() - requestStart  // ← 这是 E2E
```

### 2.2 TPS（Tokens Per Second）— 输出吞吐率

**定义**：每秒生成的输出 token 数。反映模型的"打字速度"。

**为什么重要**：
- 决定长回复的等待时间——1000 token 的回复，80 TPS 需要 12.5s，30 TPS 需要 33s
- 直接影响 Agent 多轮交互的总耗时——Agent 可能连续调用 10 次 LLM
- 不同模型差异显著，小模型通常更快

**行业基准（2025）**：

| 模型 | 典型 TPS | 说明 |
|------|---------|------|
| Claude Sonnet 4 | 80 - 120 | |
| Claude Opus 4 | 40 - 70 | 大模型生成更慢 |
| Claude Haiku | 150 - 200 | 小模型极快 |
| GPT-4o | 60 - 100 | |
| GPT-4o-mini | 120 - 180 | |
| Gemini 1.5 Pro | 100 - 150 | |
| DeepSeek V3 | 40 - 80 | |

**计算方式**：`TPS = outputTokens / (e2eLatency - ttft)`

### 2.3 TPOT（Time Per Output Token）— 单 Token 生成时间

**定义**：平均每个输出 token 的生成耗时。是 TPS 的倒数。

`TPOT = 1000 / TPS (ms)`

| TPS | TPOT |
|-----|------|
| 30 | 33ms |
| 80 | 12.5ms |
| 120 | 8.3ms |
| 200 | 5ms |

**用户感知阈值**：TPOT < 15ms 时（TPS > 67），streaming 输出在视觉上接近"即时"。

### 2.4 ITL（Inter-Token Latency）— Token 间延迟

**定义**：相邻两个 token 之间的时间间隔。与 TPOT 的区别：

- TPOT 是**平均值**
- ITL 是**逐 token 的实际间隔**，可以画分布图

ITL 波动大说明模型推理不稳定（可能遇到了难的 token、或触发了 KV-cache 重计算）。

**在 CCode 中的价值**：ITL 方差可用于检测"卡顿"——如果某个 token 的 ITL 突然 >500ms，可能是服务端出了问题。

### 2.5 E2E Latency（端到端延迟）

**定义**：从请求发出到最后一个 token 收到的总时间。

`E2E = TTFT + (outputTokens × TPOT)`

**在 Agent 场景中的特殊性**：
- 一个用户请求可能触发 5-15 次 LLM 调用（多轮工具 use → observe → act）
- **用户感知的 E2E** ≠ **单次 LLM 调用的 E2E**
- 需要同时记录 per-call E2E 和 per-turn E2E

## 三、资源层指标补充

### 3.1 Cache Hit Rate — 缓存命中率

**定义**：`cacheHitRate = cacheReadTokens / (cacheReadTokens + inputTokens)`

**为什么重要**：
- Anthropic Prompt Caching 命中时：TTFT 降低 ~80%，input 成本降低 ~90%
- 命中率直接反映 system prompt 和 context 的复用效率
- 目标值：多轮对话中应 > 60%（system prompt + 历史消息被缓存）

**CCode 现状**：已有 cacheReadTokens 和 cacheWriteTokens 原始数据，但没算比率。只需一行计算即可补齐。

### 3.2 Context Window 使用率

**定义**：`contextUsage = totalInputTokens / maxContextTokens`

**阈值**：
- < 50%：安全区
- 50-80%：注意区，考虑 compact
- 80%：预警，触发自动 compact
- > 90%：危险，可能截断丢失上下文

**CCode 现状**：已实现，StatusBar 展示 Ctx 百分比 + 颜色分级。

## 四、可靠性层指标

### 4.1 错误率

**定义**：`errorRate = failedCalls / totalCalls`

**分类**：
- 400 Bad Request — 通常是 prompt 构造错误
- 401 Unauthorized — API Key 问题
- 429 Rate Limited — 限流，需要退避重试
- 500 Server Error — 模型服务端问题
- 503 Overloaded — 高峰期过载
- Timeout — 超时未响应

**目标**：< 1%（排除 429 限流）

### 4.2 重试率

**定义**：`retryRate = retriedCalls / totalCalls`

**CCode 现状**：有重试机制（providers/retry.ts），但未统计重试次数。

### 4.3 超时率

**定义**：请求超过设定时间未完成的比例。

不同场景的合理超时阈值：
- 短回复（< 500 token）：30s
- 长回复（> 2000 token）：120s
- 工具调用响应：60s

## 五、质量层指标

### 5.1 工具调用准确率

**定义**：模型正确选择并调用工具的比例。

**评估维度**：
- **选择准确率**：应该调用工具时是否调用了？不应该调用时是否误调？
- **参数准确率**：调用参数是否正确？路径、命令、搜索词是否合理？
- **调用效率**：完成任务所需的工具调用次数是否合理？（过多说明模型在"瞎试"）

**CCode 中可间接衡量**：
- 工具调用失败率（tool_result 中 isError 的比例）
- 平均每个 turn 的工具调用次数
- 重复调用同一工具的频率（RepetitionDetector 已有部分能力）

### 5.2 指令遵循率

**定义**：模型是否按照 system prompt 和用户指令的要求行事。

**CCode 场景中的表现**：
- 是否遵循 CCODE.md / CLAUDE.md 中的编码规范？
- 是否在需要时使用中文回复？
- 是否在修改文件前先读取文件？

**难以自动化衡量**，通常靠人工评估或 LLM-as-Judge。

### 5.3 上下文保持能力

**定义**：长对话中模型是否还记得早期的信息。

**衡量方式**：
- 对话第 N 轮时，模型是否还能正确引用第 1 轮提到的文件/变量？
- compact 后信息损失率——compact 前后的回答质量差异

### 5.4 拒绝率

**定义**：模型拒绝执行合理请求的比例（安全过滤误报）。

**CCode 场景**：模型拒绝写代码、拒绝执行 bash 命令、拒绝修改文件。
**目标**：< 2%（对合理的编程请求）。

## 六、行业参考：OpenTelemetry GenAI Semantic Conventions

OpenTelemetry 在 2024 年发布了 GenAI 语义约定，定义了标准的 span 属性：

```
gen_ai.system              — 供应商（anthropic / openai / ...）
gen_ai.request.model       — 模型名
gen_ai.request.max_tokens  — 最大输出 token
gen_ai.request.temperature — 温度
gen_ai.usage.input_tokens  — 输入 token 数
gen_ai.usage.output_tokens — 输出 token 数
gen_ai.response.model      — 实际使用的模型（可能与请求不同）
gen_ai.response.finish_reason — 结束原因（stop / length / tool_use）
```

**目前缺失但值得跟进的标准属性**：
- `gen_ai.usage.cache_read_tokens` — 尚未纳入标准，但 Anthropic 已有
- `gen_ai.latency.ttft` — 社区讨论中
- `gen_ai.latency.tps` — 社区讨论中

## 八、阶段性决策与补齐路径

### 8.1 现状判断

CCode 当前的观测能力**仅覆盖资源层**（token 四维 + 费用 + context 使用率 + 调用计数）。

五层观测模型中：
- **资源层** ✅ 已覆盖
- **性能层** ❌ 空白 — TTFT/TPS/E2E/错误率全无
- **可靠性层** ❌ 空白
- **质量层** ❌ 空白
- **业务层** ❌ 空白

### 8.2 阶段性结论

**现阶段优先补齐性能层。理由：**

1. **客观可测量** — TTFT、TPS、E2E 都是纯数值，采集即记录，不需要定义"什么算好"
2. **采集点明确** — 全部集中在 agent-loop.ts 的 LLM 调用处，改动面极小
3. **用户最直接的痛点** — "为什么这么慢"是用户最常问的问题，有数据才能回答
4. **为模型选型提供依据** — 多模型场景下（Claude/GPT/GLM/DeepSeek），性能对比是选型的硬指标
5. **Cache 优化的前提** — 不知道 TTFT 和 Cache Hit Rate，就无法衡量 Prompt Caching 是否生效

**质量层暂不做。理由：**

1. **定义不清** — "工具调用准确率"怎么算？调了一个不该调的工具算不准确，还是调了但参数错了算不准确？边界模糊
2. **评估成本高** — 指令遵循率需要 LLM-as-Judge，本身就要额外 API 调用和成本
3. **样本不足** — 质量评估需要大量标注数据做基线，当前用户量不够
4. **主观性强** — 同一个回答，不同用户可能有不同评价，难以自动化

**可靠性层部分可顺带做**（错误分类统计成本极低，在 provider 层 catch 里加计数）。

### 8.3 性能层补齐方案

#### Phase 1：采集与存储（~20 行代码）

| 指标 | 采集点 | 代码改动 |
|------|--------|---------|
| **TTFT** | agent-loop.ts，首个 streaming chunk 到达时 `Date.now() - requestStart` | 5 行 |
| **E2E Latency** | agent-loop.ts，最后一个 chunk 到达时 `Date.now() - requestStart` | 3 行 |
| **TPS** | 计算值：`outputTokens / ((e2e - ttft) / 1000)` | 1 行 |
| **Cache Hit Rate** | 计算值：`cacheReadTokens / (cacheReadTokens + inputTokens)` | 1 行 |
| **错误分类** | providers 层 catch 中按 HTTP status 分类计数 | 10 行 |

存储：扩展 usage_logs 表，新增 `ttft_ms`、`tps` 字段，`duration_ms` 已有字段填值。详见第九章最终方案（以 9.2.8 字段表为准）。

#### Phase 2：展示（Web Dashboard）

- OverviewPage 新增"性能"Tab
- TTFT 分布直方图（P50 / P90 / P99）
- TPS 趋势折线图（按模型分组）
- 错误率饼图（按 HTTP status 分类）
- Cache Hit Rate 仪表盘

#### Phase 3：可顺带做的可靠性指标

- 重试率：retry.ts 中记录重试次数
- 超时率：provider 调用超时计数
- 模型间对比：按 provider+model 分组的所有指标

### 8.4 质量层留白

质量层指标暂不实施，但记录已识别的方向供将来参考：

| 指标 | 可行的衡量方式 | 前置条件 |
|------|--------------|---------|
| 工具调用失败率 | tool_result 中 isError 比例 | 需要定义"失败"的边界 |
| 工具调用效率 | turn 平均调用次数 | 需要基线数据 |
| 指令遵循率 | LLM-as-Judge 自动评估 | 需要评估 prompt + 额外 API 成本 |
| 上下文保持 | compact 前后回答质量对比 | 需要设计对比实验 |
| 拒绝率 | 安全过滤导致的无效响应比例 | 需要区分合理拒绝和误报 |

**等用户量和使用场景积累到一定程度，再回头定义质量层的评估标准。**

### 9.2 数据库版本化迁移框架

#### 9.2.1 问题：当前 db.ts 没有任何版本管理

```
现状                                    问题
────                                    ────
CREATE TABLE IF NOT EXISTS              只管"有没有表"，不管"表结构对不对"
seedDefaultPricing（INSERT 去重）        只管"数据有没有"，不管"数据是否最新"
无版本号                                 没法知道用户本地 DB 是哪个版本的 schema
新增字段只能 try/catch ALTER TABLE       脆弱、不可追踪、无法回溯
```

#### 9.2.2 方案：PRAGMA user_version + 迁移函数数组

SQLite 内置 `PRAGMA user_version`——一个存在 DB 文件头的整数，无需额外建表：

```typescript
db.pragma('user_version')         // 读 → 返回 [{ user_version: 0 }]
db.pragma('user_version = 3')     // 写 → 原子更新
```

**核心设计**：

```
                        ┌──────────────────────────────────────────┐
                        │           SCHEMA_VERSION = 2             │
                        │     （代码中声明的目标版本号）              │
                        └──────────────┬───────────────────────────┘
                                       │
          ┌────────────────────────────┼────────────────────────────┐
          │                            │                            │
    user_version = 0             user_version = 1            user_version = 2
    （全新 / 旧库）              （v1 schema）                （已是最新）
          │                            │                            │
          ▼                            ▼                            ▼
    执行 migrations[0]           执行 migrations[1]            跳过，直接返回
    （v0 → v1: 删表重建）        （v1 → v2: ALTER TABLE）
          │                            │
          ▼                            ▼
    执行 migrations[1]            设置 user_version = 2
    （v1 → v2: ALTER TABLE）
          │
          ▼
    设置 user_version = 2
```

#### 9.2.3 迁移函数数组设计

每个迁移函数负责**从版本 N 升级到 N+1**，按索引顺序排列：

```typescript
/**
 * 迁移函数数组 — 按索引顺序执行。
 *
 * migrations[0] = v0 → v1（基线建表，包含全部表）
 * migrations[N] = vN → v(N+1)（未来扩展...）
 *
 * 约定：
 * - 每个函数接收 db 实例，内部自行执行 SQL
 * - 破坏性迁移（DROP TABLE）只允许出现在早期版本（用户量小时）
 * - 数据迁移（INSERT/UPDATE）放在同一个函数内，和 DDL 一起执行
 * - 函数内部不设置 user_version（由 runMigrations 统一设置）
 */
type MigrationFn = (db: DatabaseType) => void

const migrations: MigrationFn[] = [
  // ══════════════════════════════════════════════
  // v0 → v1：基线 schema — 全量建表
  // ══════════════════════════════════════════════
  // 全新安装或旧库（无版本号）统一走这里。
  //
  // ⚠️ 本次迁移会清空全部历史数据（usage_logs / pricing_rules / memory_vectors）。
  // 当前阶段项目处于早期迭代，历史数据价值有限，可以接受。
  (db) => {
```

#### 9.2.4 迁移执行器

```typescript
/** 目标版本 = migrations 数组长度 */
const SCHEMA_VERSION = migrations.length  // 当前 = 1

/**
 * 执行数据库迁移。
 *
 * 读取 user_version → 事务内逐个执行待跑的迁移 → 成功后更新 user_version。
 *
 * ⚠️ PRAGMA user_version 不参与事务回滚（它写在 DB 文件头，不受 rollback 影响）。
 * 因此必须放在事务外面，迁移事务成功后再设置。
 * 如果放在事务里，迁移失败回滚了但 user_version 已经写成新值，
 * 下次启动会跳过这个迁移 → schema 不完整。
 */
function runMigrations(db: DatabaseType): void {
  const current = (db.pragma('user_version') as Array<{ user_version: number }>)[0].user_version

  if (current >= SCHEMA_VERSION) return  // 已是最新

  // 事务内执行 DDL + DML
  const tx = db.transaction(() => {
    for (let i = current; i < SCHEMA_VERSION; i++) {
      migrations[i](db)
    }
  })
  tx()

  // 迁移成功后才更新版本号（事务外！）
  db.pragma(`user_version = ${SCHEMA_VERSION}`)
}
```

#### 9.2.6 memory_vectors：延迟建表 + 维度感知重建

`memory_vectors` 表的特殊性在于：**schema 依赖运行时参数 `dimension`**（向量维度，由 EmbeddingProvider 决定）。
无法在静态迁移数组中硬编码维度。

**方案**：migration 只管 `memory_meta` 表。`memory_vectors` 由独立函数按需创建，但**版本追踪统一用 `memory_meta` 表的 `vectors_schema_version` 键**。

```typescript
/**
 * 确保 memory_vectors 表存在且 schema 正确。
 *
 * 调用时机：LibsqlVectorStore.initialize()
 * 调用者传入 dimension，本函数负责：
 *   1. 维度变化 → DROP + 重建（向量索引不支持动态修改维度）
 *   2. vectors_schema_version 落后 → 执行增量迁移
 *   3. 全部到位 → 跳过
 */
const VECTORS_SCHEMA_VERSION = 1

export function ensureMemoryVectors(dimension: number): void {
  const db = getDb()

  // 检查维度是否变化
  const existingDim = getMemoryMeta(db, 'embedding_dimension')
  if (existingDim !== null && parseInt(existingDim, 10) !== dimension) {
    // 维度变化：必须 DROP 重建（F32_BLOB 维度写死在列定义里）
    db.exec('DROP TABLE IF EXISTS memory_vectors')
    db.exec('DROP INDEX IF EXISTS idx_memory_vec')
    setMemoryMeta(db, 'vectors_schema_version', '0')
  }

  // 检查 schema 版本
  const currentVer = parseInt(getMemoryMeta(db, 'vectors_schema_version') ?? '0', 10)

  if (currentVer < VECTORS_SCHEMA_VERSION) {
    // v0 → v1：初始建表
    db.exec(`
      CREATE TABLE IF NOT EXISTS memory_vectors (
        id          TEXT PRIMARY KEY,
        entry_id    TEXT NOT NULL,
        scope       TEXT NOT NULL,
        project_slug TEXT,
        embedding   F32_BLOB(${dimension}),
        chunk_text  TEXT NOT NULL,
        chunk_index INTEGER NOT NULL,
        tags        TEXT,
        type        TEXT NOT NULL,
        source      TEXT NOT NULL,
        created     TEXT NOT NULL,
        updated     TEXT NOT NULL
      );
      CREATE INDEX IF NOT EXISTS idx_memory_vec
        ON memory_vectors(libsql_vector_idx(embedding));
      CREATE INDEX IF NOT EXISTS idx_memory_entry_id
        ON memory_vectors(entry_id);
      CREATE INDEX IF NOT EXISTS idx_memory_scope
        ON memory_vectors(scope);
    `)

    setMemoryMeta(db, 'vectors_schema_version', String(VECTORS_SCHEMA_VERSION))
  }

  // 记录当前维度
  setMemoryMeta(db, 'embedding_dimension', String(dimension))
}

// ── 辅助函数 ──

function getMemoryMeta(db: DatabaseType, key: string): string | null {
  try {
    const row = db.prepare('SELECT value FROM memory_meta WHERE key = ?')
      .get(key) as { value: string } | undefined
    return row?.value ?? null
  } catch { return null }
}

function setMemoryMeta(db: DatabaseType, key: string, value: string): void {
  db.prepare('INSERT OR REPLACE INTO memory_meta (key, value) VALUES (?, ?)')
    .run(key, value)
}
```

#### 9.2.7 版本管理全景

改造后，整个 DB 层的版本管理分为两条线：

```
┌─────────────────────────────────────────────────────────┐
│                    PRAGMA user_version                    │
│                  （主 schema 版本号）                      │
│                                                          │
│  管辖范围：                                               │
│    usage_logs · pricing_rules · schema_comments          │
│    memory_meta                                           │
│                                                          │
│  迁移方式：migrations[] 数组 + runMigrations()            │
│  版本存储：SQLite 文件头（PRAGMA user_version）            │
├─────────────────────────────────────────────────────────┤
│              memory_meta.vectors_schema_version           │
│                （向量表 schema 版本号）                     │
│                                                          │
│  管辖范围：                                               │
│    memory_vectors                                        │
│                                                          │
│  迁移方式：ensureMemoryVectors() 内部版本判断              │
│  版本存储：memory_meta 表的 vectors_schema_version 键      │
│  特殊性：维度变化时无条件 DROP + 重建（向量列定义不可变）    │
└─────────────────────────────────────────────────────────┘
```

**为什么分两条线**：
- 主 schema 在 `createDb()` 时一次性完成，生命周期 = 进程
- `memory_vectors` 在 `LibsqlVectorStore.initialize()` 时延迟创建，维度是运行时参数
- 两者独立演进，互不阻塞

### 9.4 agent-loop.ts `#callLLM` 计时改造

改动集中在 `#callLLM` 方法内，不影响任何外部接口：

```typescript
async *#callLLM(history: Message[]):
  AsyncGenerator<AgentEvent, { toolCalls: ToolCallContent[]; text: string; aborted: boolean }> {

  // ... chatRequest 构建、yield llm_start 不变 ...

  let inputTokens = 0, outputTokens = 0
  let cacheReadTokens = 0, cacheWriteTokens = 0
  let doneStopReason = 'end_turn'

  // ── 性能层：计时变量 ──
  const requestStart = Date.now()         // ← 新增
  let firstChunkReceived = false          // ← 新增
  let ttftMs = 0                          // ← 新增

  try {
    for await (const chunk of this.#provider.chat(chatRequest)) {

      // ── 性能层：首个有内容的 chunk 才算 TTFT ──
      // 注意：部分 Provider（如 Anthropic）第一个 chunk 可能是 message_start 或空的
      // content_block_start，不含实际内容。如果把这些算作首 token，TTFT 会偏小。
      // 只在 text / thinking / tool_call 类型时才记录 TTFT。
      if (!firstChunkReceived && (chunk.type === 'text' || chunk.type === 'thinking' || chunk.type === 'tool_call')) {
        ttftMs = Date.now() - requestStart           // ← 新增
        firstChunkReceived = true                    // ← 新增
      }

      // ... 原有 mapChunk / usage / done 处理不变 ...
    }

    // ── 性能层：E2E + TPS 计算 ──
    const e2eMs = Date.now() - requestStart                         // ← 新增
    const generationMs = e2eMs - ttftMs                             // ← 新增
    const tps = generationMs > 0                                    // ← 新增
      ? Math.round(outputTokens / (generationMs / 1000) * 10) / 10
      : 0

    // ... Prompt Cache 破裂检测不变 ...

    yield {
      type: 'llm_done', inputTokens, outputTokens,
      cacheReadTokens, cacheWriteTokens, stopReason: doneStopReason,
      ttftMs, e2eMs, tps,   // ← 新增三个字段
    }

    // ... contextTracker.update、return 不变 ...

  } catch (err) {
    if (isAbortError(err)) {
      const e2eMs = Date.now() - requestStart                      // ← 新增
      yield {
        type: 'llm_done', inputTokens, outputTokens,
        cacheReadTokens, cacheWriteTokens, stopReason: 'abort',
        ttftMs, e2eMs, tps: 0,   // ← abort 时 tps 无意义，填 0
      }
    } else {
      yield makeLlmError(...)
    }
    throw err
  }
}
```

**新增行数**：~12 行（变量声明 3 + 首 chunk 判断 3 + E2E/TPS 计算 3 + yield 字段 3）。

### 9.6 Cache Hit Rate（资源层补齐）

不需要新字段、不需要改存储。两个获取方式：

#### 方式一：会话级实时计算（TokenMeter getter）

```typescript
/** 当前会话缓存命中率（0~1），用于 StatusBar 实时展示 */
getCacheHitRate(): number {
  const total = this.#stats.totalCacheReadTokens + this.#stats.totalInputTokens
  return total > 0 ? this.#stats.totalCacheReadTokens / total : 0
}
```

#### 方式二：SQL 聚合查询（Dashboard / 报表）

```sql
-- 按会话统计 Cache Hit Rate
SELECT
  session_id,
  ROUND(SUM(cache_read) * 100.0 / (SUM(cache_read) + SUM(input_tokens)), 1)
    AS cache_hit_rate_pct
FROM usage_logs
GROUP BY session_id;

-- 按模型统计（模型选型参考）
SELECT
  provider, model,
  ROUND(SUM(cache_read) * 100.0 / (SUM(cache_read) + SUM(input_tokens)), 1)
    AS cache_hit_rate_pct,
  COUNT(*) AS call_count
FROM usage_logs
GROUP BY provider, model;
```

### 10.2 行业参考：LLM 可观测性平台怎么做

#### Helicone（代理层可观测）

核心指标卡片 + 时间序列图的组合：
- **顶部**：4 个数字卡片（请求数、延迟 P50/P99、成本、错误率）
- **中间**：Latency 时间序列图（散点+趋势线）、请求量折线图
- **底部**：按模型分组的对比表

特点：**P50/P99 双分位数**是核心指标，比平均值更有意义。

#### Langsmith（Trace 追踪平台）

- **Trace 级延迟瀑布图**：每次 LLM 调用展示为一个 span（TTFT + generation 两段）
- **延迟分布直方图**：X 轴=延迟区间，Y 轴=请求数，看分布形态
- **P50/P90/P99 趋势线**：三条分位线叠加在同一图上

特点：**延迟分布**比点值更有价值，能发现长尾问题。

#### Portkey（AI 网关）

- **实时仪表盘**：TPS gauge 表盘 + Latency gauge 表盘
- **模型对比雷达图**：多维度同时对比（TTFT、TPS、成本、错误率）
- **成本效率散点图**：X=延迟，Y=成本，每个点是一个模型

特点：**模型选型导向**——所有图表都在回答"该选哪个模型"。

#### Grafana + OpenTelemetry（通用方案）

RED 方法论（Rate、Errors、Duration）：
- **Rate**：请求量折线图（按 provider 分色）
- **Errors**：错误率折线图 + 错误类型饼图
- **Duration**：延迟热力图（时间 × 延迟区间 × 颜色深浅=请求密度）

特点：**热力图**是延迟分析的终极可视化——同时看趋势、分布、异常。

### 10.3 设计决策

#### 定位：在 OverviewPage 中新增"性能"区块

不单独建页面。原因：
1. 性能和资源是同一维度（一次 LLM 调用的属性），放一起看更自然
2. 共享 Tab 切换（当日/本周/本月）避免重复 UI
3. OverviewPage 已有 Recharts 引用，零额外依赖

#### 放在哪里

在现有"折线图"和"四维饼图"之间插入，逻辑顺序变为：

```
汇总卡片 → 资源趋势图 → 【性能指标区块】→ 供应商分布 → 模型明细表 → 会话列表
```

#### 图表选型

| 指标 | 图表类型 | 理由 |
|------|---------|------|
| TTFT / E2E 概览 | **数字卡片**（P50 + P99） | 一眼看到关键分位数，Helicone 模式 |
| TPS 概览 | **数字卡片**（均值） | 吞吐率看均值即可 |
| Cache Hit Rate | **数字卡片**（百分比） | 资源层补齐 |
| TTFT 趋势 | **折线图**（P50 + P99 双线） | 看延迟随时间的变化 |
| TPS 趋势 | **折线图**（按模型分组） | 不同模型吞吐率差异大 |
| TTFT 分布 | **柱状图**（直方图） | 看分布形态、发现长尾 |
| 模型性能对比 | **扩展现有表格** | 在模型明细表加 3 列（Avg TTFT / Avg TPS / Cache Hit%） |

> 不做雷达图、热力图——当前数据量不够，这些图在百/千次调用级别才有意义。先把基础图做好。

### 10.4 API 扩展：`/api/overview` 新增性能数据

#### 新增查询函数

```typescript
// api.ts 中新增

/** 性能指标按时间范围聚合 */
const perfByRange = (since: string, until?: string) => {
  const { where, params } = buildWhere(since, until)
  return db.prepare(`
    SELECT provider, model,
      COUNT(*) as callCount,
      -- TTFT 分位数（用 AVG 近似 P50，用 MAX 近似 P99 在小数据量下足够）
      ROUND(AVG(ttft_ms)) as avgTtft,
      MIN(ttft_ms) as minTtft,
      MAX(ttft_ms) as maxTtft,
      -- E2E
      ROUND(AVG(duration_ms)) as avgE2e,
      MAX(duration_ms) as maxE2e,
      -- TPS
      ROUND(AVG(tps), 1) as avgTps,
      -- Cache Hit Rate
      ROUND(
        SUM(cache_read) * 100.0 / NULLIF(SUM(cache_read) + SUM(input_tokens), 0),
        1
      ) as cacheHitPct
    FROM usage_logs ${where}
    AND ttft_ms IS NOT NULL
    GROUP BY provider, model
  `).all(...params)
}

/** TTFT 趋势（按时间分组的分位数） */
const perfTrend = (since: string, until?: string, byHour = false) => {
  const { where, params } = buildWhere(since, until)
  const dateExpr = byHour
    ? "strftime('%Y-%m-%d %H:00', timestamp)"
    : "DATE(timestamp)"
  return db.prepare(`
    SELECT ${dateExpr} as date,
      ROUND(AVG(ttft_ms)) as avgTtft,
      MAX(ttft_ms) as maxTtft,
      ROUND(AVG(tps), 1) as avgTps,
      ROUND(AVG(duration_ms)) as avgE2e,
      COUNT(*) as callCount,
      ROUND(
        SUM(cache_read) * 100.0 / NULLIF(SUM(cache_read) + SUM(input_tokens), 0),
        1
      ) as cacheHitPct
    FROM usage_logs ${where}
    AND ttft_ms IS NOT NULL
    GROUP BY ${dateExpr}
    ORDER BY date
  `).all(...params)
}

/** TTFT 分布直方图（区间统计） */
const ttftDistribution = (since: string, until?: string) => {
  const { where, params } = buildWhere(since, until)
  return db.prepare(`
    SELECT
      CASE
        WHEN ttft_ms < 500 THEN '<500ms'
        WHEN ttft_ms < 1000 THEN '500ms-1s'
        WHEN ttft_ms < 2000 THEN '1s-2s'
        WHEN ttft_ms < 3000 THEN '2s-3s'
        WHEN ttft_ms < 5000 THEN '3s-5s'
        ELSE '>5s'
      END as bucket,
      COUNT(*) as count,
      -- 保持排序用
      CASE
        WHEN ttft_ms < 500 THEN 1
        WHEN ttft_ms < 1000 THEN 2
        WHEN ttft_ms < 2000 THEN 3
        WHEN ttft_ms < 3000 THEN 4
        WHEN ttft_ms < 5000 THEN 5
        ELSE 6
      END as sortOrder
    FROM usage_logs ${where}
    AND ttft_ms IS NOT NULL
    GROUP BY bucket, sortOrder
    ORDER BY sortOrder
  `).all(...params)
}
```
