# Web 主界面 SubAgent 卡片挂载修复

> 背景：Web 端复杂任务派发子 Agent 时，**主对话界面**（ChatView 里嵌在 dispatch_agent
> 工具调用后的 `<SubAgentCard>`）在子 Agent 运行期间**经常不显示**，直到子 Agent
> 结束才短暂出现；但右侧的 `<SubAgentDrawer>` 抽屉始终正常。同时卡片上的 "⟳" 图标
> 不会旋转（抽屉里的同一图标会）。用户反馈后定位为两个独立问题，都是架构层面的 bug。

---

## 一、问题 1：SubAgent 卡片 running 期间不挂载

### 1.1 现象

- Web 主聊天流里对应 `dispatch_agent` 工具调用后面**不显示** `<SubAgentCard>`
- 右侧 `<SubAgentDrawer>` 抽屉**完全正常**，running 卡片和进度都在
- 子 Agent 跑完之后，主界面卡片才短暂一闪 / 或者根本看不到

### 1.2 根因链路

经代码走查确认，**不是 race condition**，而是**后端 → 前端的事件协议缺少
"dispatch_agent 工具调用 ↔ 子 Agent" 的关联键**。

#### 前端挂载条件（`ToolStatus.tsx:100-119`）

```tsx
{events.map(e => {
  const agentId = e.toolName === 'dispatch_agent'
    ? (e.agentId ?? extractAgentId(e.resultFull ?? e.resultSummary))
    : null
  const agentInfo = agentId && subAgents ? subAgents.get(agentId) : undefined
  return (
    <div key={e.toolCallId}>
      <ToolStatusItem event={e} />
      {agentInfo && <SubAgentCard agent={agentInfo} />}
    </div>
  )
})}
```

挂载需要三个条件同时满足：

1. `e.toolName === 'dispatch_agent'` ✓
2. `agentId` 能从 `e.agentId` 或 `extractAgentId(resultFull/resultSummary)` 取到
3. `subAgents.get(agentId)` 返回非 undefined

#### 前端 ToolEvent 的 agentId 生命周期（`ChatPage.tsx:191-214`）

```tsx
case 'tool_start':
  setToolEvents(prev => [...prev, {
    toolCallId, toolName, args, status: 'running', startedAt: Date.now(),
    // ❌ 这里没有 agentId
  }])
  break
case 'tool_done': {
  const toolAgentId = event.meta?.type === 'dispatch-agent' ? event.meta.agentId : undefined
  // ✓ 只有 tool_done 时才能从 meta 补 agentId
  setToolEvents(prev => prev.map(e => e.toolCallId === event.toolCallId
    ? { ...e, status: 'done', ..., ...(toolAgentId ? { agentId: toolAgentId } : {}) }
    : e))
  break
}
```

**ToolEvent 的 agentId 只能等到 tool_done 才能填上**。在 dispatch_agent 运行的
几十秒到几分钟里，`e.agentId === undefined`，`resultFull === undefined`，前端
三个挂载条件中第 2 条不成立，**卡片死活不挂**。

#### 抽屉为什么正常？

抽屉显示条件（`SubAgentDrawer.tsx:47`）：

```tsx
if (agentList.length === 0) return null
```

只要 `subAgents` Map 有任意一个条目就显示。抽屉**不经过 ToolEvent**，直接消费
`subagent_progress` / `subagent_done` 事件更新的 Map，所以不受影响。

### 1.3 方案选型

| 方案 | 描述 | 缺陷 | 判断 |
|---|---|---|---|
| A. 前端启发式回补 | 收到 subagent_progress 时，把最近一个 running 的 dispatch_agent ToolEvent.agentId 补上 | 并行派发 2+ dispatch_agent 时会误配 | 快但不彻底 |
| B. subagent_progress 加 parentToolCallId | 后端 + 前端改动 | 事件语义上 progress 本不需要 parentId，侵入性 | 不推荐 |
| C. 新事件 subagent_spawn（最终选用） | dispatch_agent 生成 agentId 的瞬间 yield 专用事件，带 parentToolCallId + agentId | 需要 ToolContext 加 toolCallId | **采用** |

**为什么选 C**：
- 语义清晰：spawn 就是"派生宣告"，专职做关联，不污染其他事件
- 消除歧义：toolCallId 是天然唯一键，并行派发也零错配
- 未来扩展点：spawn 事件本身可以携带更多元数据（比如初始化参数摘要）

### 1.4 实施

#### 后端改动（3 文件）

**`cCli/src/core/event-bus.ts`** — 新增事件类型：

```ts
| {
    type: 'subagent_spawn';
    parentToolCallId: string;
    agentId: string;
    name: string;
    agentType: string;
    description: string;
    maxTurns: number;
  }
```

**`cCli/src/core/agent-loop.ts`**：

1. `AgentEvent` 联合类型加上同样的 `subagent_spawn`
2. `buildToolContext()` 新增 `toolCallId?: string` 形参，按 exactOptionalPropertyTypes 规则条件写入
3. `#executeOneTool()` 调用 `buildToolContext` 时传入 `tc.toolCallId`

```ts
// 从
const ctx = buildToolContext(this.#provider, this.#registry, this.#config, history)
// 改为
const ctx = buildToolContext(this.#provider, this.#registry, this.#config, history, tc.toolCallId)
```

**`cCli/src/tools/core/types.ts`** — `ToolContext` 加字段：

```ts
/**
 * 当前工具调用的 toolCallId — StreamableTool 需要它建立
 * "tool_start 事件 ↔ 自己 yield 的子事件" 的关联。
 */
toolCallId?: string
```

**`cCli/src/tools/agent/dispatch-agent.ts`** — 在 `registerSubAgent` 之后立即 yield：

```ts
registerSubAgent({ agentId, name: agentName, description, agentType, modelName, maxTurns })
if (subLogger.sessionId) setSubAgentSessionId(agentId, subLogger.sessionId)

// 派生宣告 — UI 据此在 running 期间就挂载卡片
if (ctx.toolCallId) {
  yield {
    type: 'subagent_spawn',
    parentToolCallId: ctx.toolCallId,
    agentId,
    name: agentName,
    agentType,
    description,
    maxTurns,
  }
}
```

#### 前端改动（2 文件）

**`cCli/web/src/types.ts`** — ServerEvent 加类型。

**`cCli/web/src/pages/ChatPage.tsx`** — 新 case：

```tsx
case 'subagent_spawn':
  // 1. 立即建 Map 占位条目（turn=0, status=running）— 抽屉更早显示
  setSubAgents(prev => {
    if (prev.has(event.agentId)) return prev
    const next = new Map(prev)
    next.set(event.agentId, {
      agentId: event.agentId,
      name: event.name,
      agentType: event.agentType,
      description: event.description,
      status: 'running',
      turn: 0,
      maxTurns: event.maxTurns,
      events: [],
    })
    return next
  })
  // 2. 用 parentToolCallId 精确回补对应 ToolEvent 的 agentId — 主界面立即挂载
  setToolEvents(prev => prev.map(e =>
    e.toolCallId === event.parentToolCallId
      ? { ...e, agentId: event.agentId }
      : e,
  ))
  break
```

### 1.5 时序对比

**改前**：

```
T0    主 Agent: tool_start(dispatch_agent)              → ToolEvent.agentId = undefined ❌
T1-N  子 Agent: subagent_progress × N                   → Map 有条目，但 ToolEvent 无关联
T∞    主 Agent: tool_done(meta.agentId)                 → 补上，但子 Agent 已结束
```

**改后**：

```
T0    主 Agent: tool_start(dispatch_agent)              → ToolEvent 创建
T0+ε  dispatch: subagent_spawn(parentToolCallId, id)    → ✨ ToolEvent.agentId 立即回补
                                                          Map 同帧建占位条目
T1-N  子 Agent: subagent_progress × N                   → Map 正常更新 turn
T∞    主 Agent: tool_done                               → 收尾
```

核心是让关联信息**在 agentId 生成的同一事件循环帧**随事件下发，消除"ToolEvent
和 subagent 信息各走各路"的脱节。

## 二、问题 2：主界面 SubAgentCard 图标不旋转

### 2.1 现象

- `SubAgentDrawer` 里每行的 ⟳ 图标 running 时**会旋转**
- `SubAgentCard`（主对话流内嵌）的 ⟳ 图标 running 时**不旋转**

### 2.2 根因

对比代码：

**抽屉里 `SubAgentDrawer.tsx:179 + 295`**：

```tsx
{(agent.status === 'running' || agent.status === 'stopping') ? <Spinner /> : statusIcon}

function Spinner() {
  return (
    <span className="inline-block animate-spin text-warning" style={{ animationDuration: '1s' }}>
      ⟳
    </span>
  )
}
```

走专用 `<Spinner>` 组件，带 `animate-spin` class，1 秒转一圈。

**主界面 `SubAgentCard.tsx:68`**（原始）：

```tsx
<span className={`${statusColor} font-mono`}>{statusIcon}</span>
```

只是静态渲染字符 `⟳`，没有动画 class。

**为什么会这样**：两个组件独立演进，`SubAgentDrawer` 后加的 `<Spinner>` 没同步到
`SubAgentCard`。

### 2.3 修复

`SubAgentCard.tsx` 提取 `isSpinning` 标志，给 icon span 条件性加 `animate-spin`：

```tsx
const isSpinning = agent.status === 'running' || agent.status === 'stopping'
const statusIcon = isSpinning
  ? '⟳'
  : agent.status === 'done' ? '✓'
  : agent.status === 'stopped' ? '◼'
  : '✗'

// 渲染：
<span
  className={`${statusColor} font-mono ${isSpinning ? 'inline-block animate-spin' : ''}`}
  style={isSpinning ? { animationDuration: '1s' } : undefined}
>
  {statusIcon}
</span>
```

保留原 statusColor 逻辑，只在 running/stopping 时额外加 animate-spin。不动
Spinner 组件的位置（它在 SubAgentDrawer 内部，暂不抽出独立文件，避免为这一个
class 改动扩大模块影响）。

### 2.4 后续优化建议（未做）

长远看应该把 `Spinner` 抽成 `cCli/web/src/components/common/Spinner.tsx`，
让两处（以及未来可能第三处）共用。但本次改动聚焦在修 bug，不做范围扩张。

## 五、可复用设计沉淀

### 5.1 "派生宣告"作为独立事件类型的价值

工具调用外壳（AgentLoop 发的 `tool_start` / `tool_done`）是**语义不变式**，不该
因为某一个工具的特殊需求而膨胀字段。新增独立的 `subagent_spawn` 事件比"给
tool_start 加可选字段"或"给 subagent_progress 加回溯字段"更干净：

- **职责单一**：spawn 只做一件事 —— 宣告关联
- **反向可追**：任何未来想扩展子 Agent 元数据的地方（派生原因、父任务 ID、
  资源预算等），都有天然归口
- **事件可独立禁用**：如果未来某个通道不关心关联（比如纯文本日志），可以直接
  skip 这个事件类型而不影响其他流程

### 5.2 ToolContext 承载调用身份信息的合理性

工具执行期间它的 `toolCallId` 是天然的执行身份标识，把它放到 `ToolContext`
里让 StreamableTool 能显式引用自己的调用 ID，是消除"工具 yield 事件与外层
tool_start 事件关联"类问题的通用解法。后续其他需要建立此类关联的 StreamableTool
都能受益。

### 5.3 UI 组件同类能力下沉的必要性

`Spinner` 在两处独立存在（一处是完整组件，一处是静态字符）已经出 bug。同类 UI
能力应该抽到 `components/common/` 让两侧共用。本次修不做抽取是为了聚焦；后续
优化时应一起处理。
