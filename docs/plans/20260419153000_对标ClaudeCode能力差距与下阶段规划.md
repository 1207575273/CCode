## 一、背景与目标

v0.13.0 发版后,需要明确 v0.14+ 的能力演进方向。本文档不做功能清单式罗列,而是以**"对标 Claude Code 官方 CLI"**为镜子,逐项判断:

- 哪些能力 **已经做了**(避免重复造)
- 哪些能力 **差临门一脚**(顺势补齐,性价比高)
- 哪些能力 **改造成本大、但战略价值高**(需要规划)
- 哪些能力 **不值得自己做**(外部生态 / 长尾场景)

目标是产出一份**可执行、可审计、有优先级**的改造路线,避免"看见什么就做什么"。

## 三、差距候选清单(按"是否值得做")

### 3.1 值得立刻做(临门一脚,低成本高收益)

#### A. Prompt Caching 写入

**问题**:Anthropic SDK 支持 `cache_control: {"type": "ephemeral"}`,长 system prompt + 常驻 skills 可被缓存 5 分钟复用。我们现在只读了 `cache_read_input_tokens` 统计,**但请求里没设 cache_control**,等于每轮都重算。

**改动位置**:`cCli/src/providers/anthropic.ts` 构造请求处。

**方案要点**:
1. `system` 字段拆成结构化数组 `[{type: 'text', text: systemPrompt, cache_control: {type: 'ephemeral'}}]`
2. 常驻工具定义(tools 数组最后一个)加 `cache_control`
3. 多轮对话中,上一轮助手消息也可选择性加 cache_control(每 5~10 轮加一次)

**收益**:长会话 input token 立省 50~90%,延迟也下降(缓存命中路径更快)。实测 Claude Code 在 200+ 轮对话里重复计费的 system token 可以降到几乎为 0。

**风险**:
- cache_control 一旦设置后,被缓存的内容不能在后续请求里改(改了就失效)。所以 system prompt 里不能掺时间戳这类每次都变的东西——**需要先审查 system prompt 里有没有动态字段**。
- cache 有最小 token 数要求(1024 tokens),短 system prompt 触发不了缓存。

**难度**:S(2~4 小时,含测试)

##### 实施记录(2026-04-19 完成)

**状态**:✅ 已实施并实测通过

**最终改动**:

1. `cCli/src/providers/anthropic.ts:184~222` — `#chatOnce()` 改造:
   - `system` 字段从 `string` 改为 `[{type:'text', text, cache_control:{type:'ephemeral'}}]`
   - `tools` 数组末尾元素加 `cache_control: {type:'ephemeral'}`(缓存断点向前累积,一处即覆盖整个 tools 数组)
   - 长度保护:`systemPrompt.length < 4000` 字符(约 1000 tokens)时回退字符串形式,避免请求体复杂化却拿不到缓存
   - 环境开关:`CCODE_DISABLE_PROMPT_CACHE=1` 一键回退到原路径

2. `cCli/src/ui/App.tsx:478~` — `/usage` 增加缓存命中观测行:
   - 有 read 或 write → `X read / Y write | hit rate N%`
   - 会话无请求 → `--（本次会话尚无请求）`
   - 有请求但全零 → `--（未命中:provider 可能不支持 Prompt Caching 或 system 过短）`

**方案调整**:原方案第 3 点(消息级 cache_control,每 5~10 轮标一次助手消息)**本次未做**——先验证基础 system + tools 缓存的效果,等第一期稳定后再评估是否需要。

**过程中遇到的问题**:

| # | 问题 | 定位过程 | 解决方案 |
|---|---|---|---|
| 1 | glm-5.1 测试 `cache_read_input_tokens` 恒为 0,响应里连 `cache_creation_input_tokens` 字段都没有 | 翻 `debug.log` 看到 `message_delta.usage` 缺字段,判断服务端压根没实现 | 不是 bug:第三方 Anthropic 兼容代理普遍不实现 Prompt Caching(Anthropic 官方功能,非协议必需部分)。改动无副作用(请求多发几十字节,服务端忽略)。 |
| 2 | `/usage` 在 glm 场景下显示"首轮对话仅写入缓存",暗示"等一下会写入",误导 | 用户反馈后检查文案逻辑 | 文案三分支化:有命中 / 无请求 / 有请求但全零(中立提示,不暗示"马上会写入") |
| 3 | MiniMax-M2.5 观察到 `cache_read_input_tokens > 0` 但 `cache_creation_input_tokens` 始终 = 0 | 对比 debug.log 响应字段,排除代码问题 | MiniMax 走**隐式缓存**(服务端自动识别重复前缀,不依赖 cache_control)。我们发 cache_control 字段它忽略,但它自己的缓存路径在工作 → 无需特殊处理。 |
| 4 | 切 provider 后原 `/usage` 数据还是旧 provider 的 | 用户第一次贴数据是 glm 会话的,切 MiniMax 后没刷新 | 非 bug,会话独立计费是预期行为。明确提示用户在新会话里 /usage 看 |

**实测数据**(MiniMax-M2.5,本项目 system prompt 约 11K tokens):

| 轮次 | input | cache_read | cache_write | hit rate 累计 |
|---|---|---|---|---|
| 1 | 9316 | 5 | 0 | ~0%(首轮) |
| 2 | 1551 | 9240 | 0 | **46.0%** |

**第二轮 input 成本估算**:
- 无缓存:10791 tokens × 1.0 = 10791 等效
- 有缓存:1551 × 1.0 + 9240 × 0.1 = 2475 等效
- **省 77%**(按 Anthropic 官方 cache_read 价 ≈ input 10% 折算)

**结论**:
- ✅ 代码改动正确,Anthropic 原生协议 + MiniMax 均生效
- ❌ glm-5.1 不支持(服务端限制,非代码问题)
- ⏳ 官方 Anthropic + Claude 模型未直接测试,但协议对齐,预期正常返回 cache_creation + cache_read 双字段

**附带发现(非本次范围)**:TokenMeter 缺 `MiniMax-M2.5` 定价规则,会话 cost 显示 `--`。独立问题,后续单独修。

#### B. Checkpoint / Rewind

**对标澄清(2026-04-19 补)**:
Claude Code 的 `/rewind` 核心不是"切分支",是**按用户消息粒度对被工具改过的文件做磁盘快照**,支持"只回代码 / 只回对话 / 两个都回"三种模式。快照独立于 git、本地存 30 天,只覆盖 `Write/Edit/NotebookEdit` 动过的文件——bash 改的、外部副作用(HTTP/DB/部署)一律不管。
这块是我们**当前完全没有**的能力轴:我们有 `resume`(从 JSONL 恢复会话)和 session 分支树(uuid+parentUuid fork),都只处理对话维度,不处理磁盘文件。

**当前项目已有(对话维度)**:
- `commands/resume.ts` — 从 JSONL 加载历史 session
- `persistence/session-store.ts` — 消息树 + parentUuid,已支持任意点 fork

**缺口(代码维度)**:
- 没有文件快照机制
- 没有按消息粒度选回滚点的 UI

**决策:不做完整版,仅保留轻量兜底(延后或不做)**
理由:
1. 文件快照 + 三种恢复模式的落地量不小(拦截工具、快照存储、过期清理、UI 挑点),和"解决用户实际痛点"的比例不划算——我们项目当前用户基本有 git 习惯,`git stash` / `git diff` 已经够用。
2. 用户已经通过 `resume` + session 分支树覆盖了"回到某个对话点再往下试"的场景。
3. Claude Code 自己的 /rewind 也明确不管 bash 改的东西和外部副作用,价值边界很窄。

**如果未来真要做,做到哪一步为止**:
- 只回退**对话历史**,不做文件快照。
- 回退时在新分支的第一条 user message 后注入一条系统提示:"已从 checkpoint X 回滚,磁盘文件未恢复,如需确认当前文件状态请先读取相关文件。" 让 LLM 自己去 Read 一遍来对齐认知。
- 入口可以复用现有的 resume UI 做"选消息而不是选 session"的变体,不新增独立的 `/checkpoint` 指令。

**难度**(仅作备忘,本阶段不排期):S(0.5 天)— 只回退对话 + 注入提示;完整版(含文件快照)不考虑。

#### C. Plan Mode

**问题**:现在 agent 拿到任务直接动手,没有"先出方案再执行"的开关。对不熟悉的代码库 / 高风险任务这是刚需。Claude Code 的 `EnterPlanMode` / `ExitPlanMode` 是个很好的参考。

**改动位置**:
- `cCli/src/core/agent-loop.ts` 增加 `mode: 'plan' | 'exec'` 字段到 AgentLoop 上下文
- `cCli/src/commands/` 新增 `/plan` 指令切 plan mode,`/approve` 退出 plan mode 并执行
- `cCli/src/core/permission.ts`(或类似)在 plan mode 下拦截所有 mutating 工具

**方案要点**:
1. Plan mode 下,所有"写类工具"(Write / Edit / Bash 可变命令 / dispatch_agent)被拒绝,返回一条标准结果"plan mode 下不能执行,请先输出方案"
2. 系统提示词追加一段"当前是 plan mode,请先用 Markdown 输出完整方案,等用户批准"
3. 只读工具(Read / Grep / Glob / WebFetch)仍可用,方便 agent 做调研
4. 用户 `/approve` 后,把 plan mode 下产生的方案作为 system context 注入下一轮,切回 exec

**取舍**:
- 是不是用现有 Hook 系统实现(PreToolUse 钩子 reject)?**否**——Hook 目的是用户可插拔,这是核心产品能力,应该是一等公民。
- 哪些工具算 mutating?建议在 Tool 接口上加个 `mutating: boolean` 字段,工具自己声明,比在 agent-loop 里硬编码白名单好维护。

**难度**:M(1~2 天,含 UI 提示)

---

#### D. 图片输入(仅 Web 端)

**问题**:视觉多模态是 Opus 4.x 的基本盘,截图发 bug / 设计稿转代码是高频需求。Ink(终端)端图片协议太碎(iTerm2 / kitty / sixel 各不相同),先只做 Web。

**改动位置**:
- `cCli/src/providers/anthropic.ts` 消息结构支持 `content: [{type: 'image', source: ...}, {type: 'text', text: ...}]`
- `cCli/web/src/` 增加拖拽 / 粘贴上传 UI + base64 编码
- `cCli/src/core/*` 消息类型扩展(从 string 到 MessageContent[])

**方案要点**:
1. 消息类型先扩:用户消息和助手消息的 `content` 从 `string` 变 `string | MessageContent[]`,保持 string 路径兼容
2. Web 端:输入框支持拖拽 / 粘贴 / 点击上传,图片以 base64 塞进 message
3. Provider 层:Anthropic 原生支持 multimodal,OpenAI 兼容端按需处理(大部分兼容端也支持 vision)
4. 持久化:JSONL 存 base64 可能过大,可以单独存到 `.ccode/attachments/`,event 里只存路径

**取舍**:
- Ink 端不做 — 不值得为长尾场景加大复杂度
- 持久化:内联 base64 简单但文件会爆炸,外挂文件正确但多一套管理

**难度**:M(2~3 天,改消息类型是最大的面)

### 3.2 需要设计决策先行(中等成本,先对齐再开工)

#### E. `/compact` 策略升级

**现状**:`src/commands/compact.ts` 已实现,但是手动触发且策略单一。

**可改进方向**:
1. **自动触发**:监测 token 数达到模型上下文 75% 时自动 compact(可配置)
2. **Pinned 保留**:用户能标记"这条消息不能被压缩"(比如关键的需求说明 / TODO 列表)
3. **分层压缩**:工具调用结果 > 助手文本 > 用户文本,优先压缩 token 最高的工具调用结果(grep 大输出之类)
4. **增量压缩**:不每次全量压缩,只压缩最老的 N 条,保留近期 K 条原文

**为什么先对齐**:每一种策略背后都有一堆边界(什么时机触发?压缩到什么程度?压完丢不丢工具调用链?),开工前需要先出一份策略文档定型。

**难度**:M(策略设计 1 天 + 实现 2~3 天)

---

#### F. 钩子脚本化(Hook 用户可配置)

**现状**:`hook-manager.ts` 支持在代码里注册 Pre/Post 钩子,但**没有读取用户的 `.ccode/hooks/*.sh`**(待确认)。这一层加上后,用户就能自己挂 lint / 格式化 / 审计 / 通知。

**改动位置**:`cCli/src/hooks/hook-manager.ts` 增加 `loadUserHooks()`,按约定路径加载 shell 脚本或 JS 文件。

**方案要点**:
1. 约定 `.ccode/hooks/{pre-tool-use,post-tool-use,user-prompt-submit}.sh` 文件名
2. 钩子通过环境变量传入工具参数(JSON 序列化)和上下文
3. 脚本输出 JSON 到 stdout 表示"修改后的参数"或"拒绝原因"
4. 支持超时(默认 5s 强制杀掉)

**风险**:
- Shell 脚本安全边界?默认只执行 `.ccode/hooks/` 目录下的,跟 `settings.local.json` 一样的信任模型
- Windows 兼容?`.ps1` / `.bat` 作为 fallback,或用 Node 脚本 `.cjs`

**难度**:M(1.5~2 天,Windows 兼容是主要坑)

### 3.3 改造大、战略价值有争议(建议慎重,或外包给生态)

| 能力 | 建议 | 理由 |
|---|---|---|
| IDE 集成(VS Code / JetBrains) | **不做** | 每个 IDE 都是独立项目。让开发者用 CLI + Web,或者出 SDK 让社区做。 |
| Agent SDK(程序化调用) | **暂缓** | 需要先稳定内部 API 边界,v0.13 还在快速迭代,现在出 SDK 会成为枷锁。等 v0.15~0.16 再考虑。 |
| 持久化重构(SQLite 索引) | **暂缓** | JSONL 当前规模足够,只有当单会话突破 10 万 events 时才有明显瓶颈。不是主矛盾。 |
| Remote trigger / 通知 | **不做** | 除非有明确的"远程下发任务"场景。Claude Code 的 PushNotification 是为桌面通知,CLI 场景不急。 |
