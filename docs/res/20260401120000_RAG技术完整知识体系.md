# RAG（Retrieval-Augmented Generation）技术完整知识体系

> 调研日期：2026-04-01  
> 覆盖范围：RAG 核心原理、关键组件、最佳实践、Agent 结合范式、工程实践

## 一、背景与动机

### 1.1 为什么需要 RAG

大语言模型（LLM）虽然在文本生成方面表现出色，但存在几个根本性缺陷：

| 问题 | 描述 |
|------|------|
| **幻觉（Hallucination）** | LLM 会"一本正经地胡说八道"，生成看似合理但事实错误的内容 |
| **知识过时（Knowledge Cutoff）** | 训练数据有截止日期，无法获取最新信息 |
| **私有数据盲区** | LLM 无法访问企业内部文档、数据库等私有知识 |
| **缺乏溯源** | 生成结果无法追溯信息来源，难以验证准确性 |
| **上下文窗口限制** | 即使上下文窗口不断扩大，将所有知识塞入 prompt 仍不现实且成本高昂 |

RAG 的核心思路是：**先检索，再生成**——让 LLM 在回答问题前，先从外部知识库中检索到相关信息，再基于这些信息生成回答。据行业数据，使用 RAG 的系统可实现 **60-80% 的幻觉率降低**和 **3 倍的领域问答准确率提升**。

### 1.2 发展历程

| 时间 | 里程碑 |
|------|--------|
| 2020 | Meta（Facebook AI）发表 RAG 论文，首次提出将检索与生成结合的端到端框架 |
| 2022 | 随着 ChatGPT 爆发，RAG 成为解决 LLM 幻觉问题的主流方案 |
| 2023 | LangChain、LlamaIndex 等框架成熟，RAG 进入工程化阶段；RAGAS 评估框架发布 |
| 2024 | Advanced RAG 技术涌现：Graph RAG、Corrective RAG、Self-RAG；向量数据库市场爆发 |
| 2025 | Agentic RAG 成为主流范式；RAG 从实验性技术演变为生产级架构；85% 企业 AI 应用采用 RAG |
| 2026 | RAG 架构进一步成熟，Multi-Agent RAG、Late Chunking 等新模式出现 |

## 二、核心原理

### 2.1 基本架构

RAG 的核心架构由两部分组成：**Retrieval（检索）** 和 **Generation（生成）**。

```
用户查询 (Query)
    │
    ▼
┌─────────────┐     ┌──────────────────┐
│  Query      │     │  知识库           │
│  Embedding  │     │  (文档 → 切分     │
│  (向量化)    │     │   → Embedding    │
└──────┬──────┘     │   → 向量存储)     │
       │            └────────┬─────────┘
       ▼                     │
┌──────────────────────────────────────┐
│        向量相似度检索                  │
│   (Vector Similarity Search)         │
└──────────────────┬───────────────────┘
                   │
                   ▼  Top-K 相关文档
┌──────────────────────────────────────┐
│       上下文注入 (Context Injection)   │
│  System Prompt + Retrieved Context   │
│  + User Query → LLM                 │
└──────────────────┬───────────────────┘
                   │
                   ▼
            生成回答 (Response)
```

**完整流程分为两个阶段：**

**离线索引阶段（Indexing）：**
1. **文档加载**：从各类数据源（PDF、网页、数据库等）加载原始文档
2. **文档切分（Chunking）**：将长文档切分为适合检索的小块
3. **向量嵌入（Embedding）**：使用 Embedding 模型将文本块转为高维向量
4. **向量存储**：将向量存入向量数据库，建立索引

**在线查询阶段（Query）：**
1. **查询向量化**：将用户问题转为向量
2. **相似度检索**：在向量数据库中检索最相关的文本块
3. **上下文注入**：将检索结果拼接到 LLM 的 prompt 中
4. **生成回答**：LLM 基于检索到的上下文生成回答

### 2.2 向量嵌入（Embedding）原理

Embedding 是将文本映射到高维向量空间的过程，核心思想是 **语义相近的文本在向量空间中距离也相近**。

```
"猫喜欢吃鱼"  →  [0.12, -0.34, 0.56, ..., 0.78]  (1024维)
"猫爱吃鱼"    →  [0.13, -0.33, 0.55, ..., 0.77]  (距离很近)
"今天天气真好"  →  [-0.45, 0.67, -0.12, ..., 0.34]  (距离较远)
```

**关键特性：**
- **维度**：常见 384~4096 维，维度越高表达能力越强，但计算和存储成本也越高
- **Matryoshka 表示**：现代 Embedding 模型（如 text-embedding-3-large）支持维度裁剪，前 N 维即可保留大部分语义信息
- **相似度度量**：余弦相似度（Cosine Similarity）是最常用的度量方式

### 2.3 向量数据库（Vector Store）的作用

向量数据库是 RAG 系统的"记忆中枢"，核心职责：

1. **高效存储**：存储海量高维向量及其关联的元数据
2. **近似最近邻搜索（ANN）**：使用 HNSW、IVF 等索引算法实现毫秒级检索
3. **元数据过滤**：支持在向量检索的同时进行结构化过滤（如按时间、类别、来源过滤）
4. **可扩展性**：支持从百万到十亿级向量的水平扩展

### 2.4 检索策略

| 策略 | 原理 | 优势 | 劣势 |
|------|------|------|------|
| **向量相似度搜索** | 基于 Embedding 向量的余弦/欧氏距离 | 语义理解强，能捕获同义词 | 对精确关键词匹配弱 |
| **关键词搜索（BM25）** | 基于词频-逆文档频率的经典信息检索算法 | 精确关键词匹配强 | 无法理解语义 |
| **混合检索（Hybrid Search）** | 向量搜索 + BM25 结果融合（通常用 RRF） | 兼顾语义与关键词 | 需要调参融合权重 |
| **三路混合检索** | Dense + Sparse + Full-text 同时检索 | 最佳召回率 | 架构复杂度高 |

### 2.5 上下文注入

检索到的文档块通过 prompt 模板注入 LLM：

```
你是一个问答助手。请仅根据以下参考资料回答问题，如果资料中没有相关信息，请明确说明。

参考资料：
---
{retrieved_chunk_1}
---
{retrieved_chunk_2}
---
{retrieved_chunk_3}

用户问题：{user_query}
```

关键注意点：
- 检索结果的**排列顺序**会影响 LLM 的关注度（研究发现存在 "Lost in the Middle" 现象）
- **上下文窗口有上限**：注入过多内容会导致"context rot"——Chroma 2025 年研究发现约 2500 tokens 处存在"context cliff"，超过后回答质量显著下降

### 3.3 文档切分策略（Chunking）

Chunking 是 RAG 系统中**影响最大的环节之一**——Vectara 在 NAACL 2025 发表的研究指出："Chunking 配置对检索质量的影响与 Embedding 模型的选择同等重要甚至更大。"

#### 七种主要切分策略

| 策略 | 原理 | 适用场景 | 不适用场景 |
|------|------|---------|-----------|
| **固定长度切分** | 按固定字符/token 数切割 | 日志、同质化文档 | 有表格或结构的文档 |
| **递归字符切分** | 按层级分隔符（段落→句子→字符）递归切割 | 通用 RAG（推荐默认） | 短文档 |
| **句子级切分** | 按句子分组直到达到目标大小 | 技术文档、结构化文章 | 多跳分析查询 |
| **文档感知切分** | 利用文档原生结构（标题、页码、函数） | PDF、Markdown、代码 | 页面分割无意义的文档 |
| **语义切分** | Embedding 句子后，在相似度骤降处切割 | 多主题文档 | 同质化语料、预算紧张 |
| **LLM 辅助切分** | 用 LLM 识别语义单元 | 高价值文档（合同） | 大规模批量处理 |
| **Late Chunking** | 先对完整文档 Embedding，再切分 | 大量交叉引用的文档 | 超出 Embedding 上下文的文档 |

#### Benchmark 数据（2026）

FloTorch 2026 基准测试（50 篇学术论文，905K tokens，端到端准确率）：

| 策略 | 准确率 |
|------|--------|
| 递归 512 tokens | **69%** |
| 固定 512 tokens | 67% |
| 语义切分 | 54%（平均仅 43 token 碎片） |

**核心发现**：语义切分虽然在**检索召回率**上表现最好（91.9%），但在端到端准确率上反而最差，因为产生的碎片过小（平均 43 tokens），导致 LLM 缺乏足够上下文来生成正确答案。

#### 推荐参数

| 文档类型 | 策略 | Chunk 大小 | Overlap | 依据 |
|---------|------|-----------|---------|------|
| 通用文本/文章 | 递归字符切分 | 512 tokens | 50-100（10-20%） | FloTorch 2026 |
| 学术论文 | 递归字符切分 | 512 tokens | 50-100（10-20%） | FloTorch 2026 |
| 财务报告 | 页面级 / 固定 | 1024 tokens | 15% | NVIDIA 2024 |
| 技术文档 | 标题切分 + 递归 | 512/section | 10-20% | Microsoft Azure |
| 代码 | 语言感知切分 | 函数级 | 无 | LangChain |
| 短 FAQ | 不切分 | 完整文档 | N/A | Firecrawl |

**默认推荐**：`RecursiveCharacterTextSplitter`，512 tokens，50-100 tokens overlap。这是 2026 年最大规模真实文档测试中得分最高的方案，零模型调用成本，且优于所有更昂贵的替代方案。

### 3.4 检索优化技术

#### Re-ranking（重排序）

检索后使用 Cross-Encoder 或 ColBERT 对候选文档重新排序，提升最终精度。

```
Query → 检索 Top-50 → Re-ranker → 取 Top-5 → 注入 LLM
```

常用 Re-ranker：Cohere Rerank、BGE-Reranker、ColBERT v2。三路混合检索（Dense + Sparse + Full-text）+ ColBERT Re-ranking 在 MRR 指标上表现最优。

#### HyDE（Hypothetical Document Embeddings）

用 LLM 先生成一个"假想答案"，再用该答案的 Embedding 去检索真实文档。适用于查询措辞与文档措辞差异大的场景。

```
用户查询："如何优化数据库性能"
    ↓ LLM 生成假想答案
"数据库性能优化通常包括索引优化、查询计划分析、连接池配置..."
    ↓ Embedding 假想答案
检索与假想答案语义相近的真实文档
```

#### MultiQuery（多查询扩展）

用 LLM 将原始查询改写为多个不同角度的查询，分别检索后合并结果：

```
原始查询："RAG 的缺点是什么？"
    ↓ LLM 改写
  - "RAG 系统有哪些局限性？"
  - "检索增强生成的常见问题？"
  - "RAG 在哪些场景下效果不好？"
    ↓ 分别检索 → 合并去重 → 重排序
```

#### Contextual Compression（上下文压缩）

从检索到的长文档块中提取与查询最相关的部分，丢弃不相关内容，减少噪声注入。

## 四、最佳实践

### 4.1 Chunk 大小与重叠经验值

- **默认起点**：512 tokens，10-20% overlap（50-100 tokens）
- **保守方案**：512 tokens，25% overlap（Microsoft Azure 推荐）
- **上限警告**：超过 2500 tokens 会触发"context cliff"，回答质量骤降
- **下限警告**：低于 200 tokens 的碎片会导致 LLM 缺乏上下文

**调优流程**：
1. 从递归切分 512 tokens + 50-100 overlap 开始
2. 在真实查询上测量检索质量
3. 如果指标低且文档有结构，尝试文档感知切分
4. 如果仍不满意且文档存在明显主题切换，测试语义切分（设置 200-400 token 下限）
5. 仅在特殊场景尝试 LLM 切分或 Late Chunking

### 4.2 Embedding 维度与性能权衡

| 维度范围 | 适用场景 | 存储成本（百万向量） | 检索延迟 |
|---------|---------|-------------------|---------|
| 384 | 轻量级/边缘设备 | ~1.5 GB | 最快 |
| 768-1024 | 大多数生产场景 | ~3-4 GB | 快 |
| 2048-3072 | 高精度需求 | ~8-12 GB | 中等 |
| 4096+ | 研究 / 极致精度 | ~16+ GB | 较慢 |

**实用建议**：支持 Matryoshka 表示的模型（如 text-embedding-3-large）可以先用高维度训练，然后在部署时截断到较低维度以平衡精度与成本。

### 4.3 混合检索实践

生产级 RAG 系统推荐的检索流水线：

```
用户查询
    ├── BM25 关键词检索 → 候选集 A
    ├── Dense 向量检索 → 候选集 B
    └── (可选) Sparse 向量检索 → 候选集 C
         ↓
    RRF (Reciprocal Rank Fusion) 融合
         ↓
    Cross-Encoder / ColBERT Re-ranking
         ↓
    Top-K 结果 → LLM
```

**RRF（Reciprocal Rank Fusion）** 公式：`RRF_score(d) = Σ 1/(k + rank_i(d))`，其中 k 通常取 60。

### 4.4 元数据过滤

在向量检索中结合元数据过滤可显著提升精度：

- **时间过滤**：只检索最近 N 天的文档（适合新闻、法规等时效性强的内容）
- **来源过滤**：按文档来源/类别缩小搜索范围
- **权限过滤**：结合用户权限，只返回有权访问的文档
- **标签过滤**：预先给文档打标签，检索时按标签过滤

### 4.5 评估指标

RAG 系统的评估分为 **检索质量** 和 **生成质量** 两个维度：

#### 检索质量指标

| 指标 | 含义 |
|------|------|
| **Precision@K** | Top-K 检索结果中相关文档的比例 |
| **Recall@K** | 所有相关文档中被检索到的比例 |
| **MRR（Mean Reciprocal Rank）** | 第一个相关文档的排名位置的倒数 |
| **NDCG** | 考虑排名位置的加权相关性评分 |

#### 生成质量指标

| 指标 | 含义 | 推荐阈值 |
|------|------|---------|
| **Faithfulness（忠实度）** | 回答中的每个论断是否有检索上下文支持 | 通用 0.8+，监管行业 0.9+ |
| **Answer Relevance（回答相关性）** | 回答是否切题 | 0.75+ |
| **Context Precision（上下文精度）** | 检索到的内容是否真的被用于回答 | 0.7+ |
| **Context Recall（上下文召回率）** | 回答所需信息是否都在检索到的上下文中 | 0.75+ |
| **Hallucination Rate（幻觉率）** | 回答中无依据论断的比例 | 越低越好 |

#### 评估框架对比

| 框架 | 最佳场景 | 核心优势 |
|------|---------|---------|
| **RAGAS** | 快速实验、研究 | 内置合成数据生成，最广泛采用的开源框架 |
| **DeepEval** | CI/CD 集成 | 原生 pytest 集成，支持 pass/fail 阈值 |
| **TruLens** | 开发阶段监控 | 实时 instrumentation，可视化仪表盘 |

**评估数据集规模建议**：50 条精心构造的 golden questions 即可获得稳定指标；合成回归测试集推荐 200-500 条。

### 4.6 常见陷阱与避坑

| 陷阱 | 说明 | 解决方案 |
|------|------|---------|
| **盲目选择语义切分** | 语义切分听起来更高级，但 benchmark 显示递归切分在端到端准确率上更优 | 默认递归切分，有明确数据支持时再换 |
| **Chunk 过小** | 碎片化导致 LLM 上下文不足 | 设置 200 token 最小下限 |
| **Chunk 过大** | 稀释了相关性 | 控制在 512-1024 tokens |
| **忽略元数据** | 纯向量检索在大规模知识库中精度不够 | 结合元数据过滤缩小搜索范围 |
| **只用向量搜索** | 精确关键词匹配场景（如型号、编号）表现差 | 混合检索（向量 + BM25） |
| **不做评估就上线** | 无法量化系统表现 | 上线前跑 RAGAS 评估，设定基线 |
| **忽视 Embedding 漂移** | 数据更新后 Embedding 质量退化 | 增量更新 + 定期重建索引 |
| **低估成本** | 只关注 Embedding API 费用，忽略存储和重建成本 | 全面预算（Embedding + 存储 + 计算 + 重建） |

## 五、RAG 与 Agent 的结合范式

### 5.1 演进路线

```
Naive RAG        → Advanced RAG      → Agentic RAG
(静态检索+生成)    (优化检索+后处理)    (智能体驱动的动态检索)
```

传统 RAG 是线性的"检索→生成"流水线，**Agentic RAG 引入了自主决策和迭代优化能力**，让 Agent 能够：
- 自主判断是否需要检索
- 动态选择检索策略
- 评估检索结果质量
- 在结果不佳时自动调整查询并重试

### 5.2 Agentic RAG 核心架构

Agentic RAG 集成四个核心要素：

1. **LLM（推理引擎）**：负责理解、推理、生成
2. **Memory（记忆系统）**：短期记忆（当前对话上下文）+ 长期记忆（累积知识）
3. **Planning（规划机制）**：任务分解、自反思、策略调整
4. **Tools（工具集）**：向量搜索、Web API、数据库查询等

四种驱动 Agentic RAG 的核心模式：

| 模式 | 描述 |
|------|------|
| **Reflection（反思）** | Agent 迭代评估自身输出，通过自反馈循环提升准确率 |
| **Planning（规划）** | 将复杂任务自主分解为可管理的子任务 |
| **Tool Use（工具使用）** | 集成外部 API、数据库等超越预训练知识的资源 |
| **Multi-Agent Collaboration（多 Agent 协作）** | 跨 Agent 的任务专业化与中间结果共享 |

### 5.3 Self-RAG（自反思检索）

Self-RAG 在生成过程中引入**自检索和自评估**机制：

```
用户查询
    ↓
LLM 判断：是否需要检索？
    ├── 不需要 → 直接生成回答
    └── 需要 → 生成检索查询 → 检索
                    ↓
              LLM 评估检索结果：是否充分支持回答？
                    ├── 充分 → 生成回答 → 自评估回答质量
                    └── 不充分 → 重新构造查询 → 再次检索
                                        ↓
                                  迭代直到满意
```

**关键特征**：
- 模型自主决定何时检索（不是每次都检索）
- 检索后会自评估证据是否充分
- 生成后会自评估回答质量
- 形成自纠正的闭环，无需外部验证器

### 5.4 Corrective RAG（CRAG）

CRAG 在检索后增加了**纠错机制**，由五个专业化 Agent 协作：

```
用户查询
    ↓
① Context Retrieval Agent — 从向量数据库检索文档
    ↓
② Relevance Evaluation Agent — 将文档拆为"知识条带"逐条评估相关性
    ├── 相关性达标 → ⑤ Response Synthesis Agent → 生成回答
    └── 相关性不达标 ↓
③ Query Refinement Agent — 利用语义理解重写查询
    ↓
④ External Knowledge Retrieval Agent — Web 搜索补充外部知识
    ↓
⑤ Response Synthesis Agent — 整合验证后的信息生成回答
```

**核心优势**：通过"迭代纠正"最大限度减少幻觉，对检索质量进行动态验证。

### 5.5 多步推理 RAG（Iterative RAG）

适用于需要多跳推理的复杂问题：

```
问题："A 公司去年营收最高的产品线负责人是谁？"

第一轮检索：查找 A 公司各产品线营收数据
  → 得知 X 产品线营收最高

第二轮检索：查找 X 产品线负责人信息
  → 得知负责人是张三

生成最终回答
```

### 5.6 RAG 作为 Agent 工具（Tool Use 模式）

在 Agent 框架中，RAG 被封装为一个工具（Tool），Agent 根据需要决定是否调用：

```typescript
// 概念示例：RAG 作为 Agent 的一个工具
const ragTool = {
  name: "knowledge_search",
  description: "在企业知识库中搜索相关信息",
  parameters: {
    query: "string — 搜索查询",
    filters: "object — 可选的元数据过滤条件"
  },
  execute: async (params) => {
    // 1. Embedding query
    // 2. Vector search + optional BM25
    // 3. Re-ranking
    // 4. 返回 Top-K 结果
  }
};
```

Agent 在思考链中决定何时调用 RAG，何时使用其他工具（代码执行、Web 搜索等），何时直接回答。

### 5.7 Multi-Agent RAG

在多 Agent 架构中，RAG 可以作为**知识专家 Agent** 角色：

```
Coordinator Agent（协调者）
    ├── Knowledge Agent — 负责内部知识库检索（RAG）
    ├── SQL Agent — 负责结构化数据查询
    ├── Web Agent — 负责实时网络信息检索
    └── Analysis Agent — 负责数据分析和推理

协调者根据查询类型分派给对应的专家 Agent，
多个 Agent 可并行处理，最终由协调者整合结果。
```

每个文档甚至可以有专属的 Document Agent，由 Meta-Agent 编排交互，实现更精细的检索决策。

## 六、工程实践考量

### 6.1 本地 vs 云端部署

| 维度 | 本地部署 | 云端部署 |
|------|---------|---------|
| **数据安全** | 数据不出内网，合规性强 | 需评估云服务商的数据处理策略 |
| **成本模型** | 一次性硬件投入 + 运维人力 | 按量付费，初期成本低 |
| **扩展性** | 需自行规划容量 | 弹性扩缩容 |
| **运维复杂度** | 高（需要 GPU、向量数据库运维） | 低（托管服务） |
| **延迟** | 内网延迟低 | 受网络影响 |
| **适合场景** | 金融/医疗/政务等监管行业 | 创业公司、快速迭代的产品 |

**混合方案**：Embedding 和向量数据库自托管（数据安全），LLM 调用云端 API（降低 GPU 成本）。

### 6.2 增量索引 vs 全量重建

| 策略 | 适用场景 | 注意事项 |
|------|---------|---------|
| **增量索引** | 日常文档更新、新增内容 | 需要处理文档版本控制和去重 |
| **全量重建** | 更换 Embedding 模型、切分策略调整 | 耗时且成本高，建议非高峰期执行 |
| **混合策略** | 生产环境推荐 | 日常增量更新 + 季度全量重建 |

**生产实践建议**：
- 增量 Embedding 处理日常更新
- 监控 Embedding 漂移（新旧文档的向量分布差异）
- 冷数据每季度重新 Embedding
- 更换 Embedding 模型时必须全量重建

### 6.3 实时性要求

| 实时等级 | 方案 | 延迟 |
|---------|------|------|
| **准实时（分钟级）** | 文档变更触发增量索引 pipeline | 1-5 分钟 |
| **近实时（小时级）** | 定时批量索引任务 | 1-6 小时 |
| **离线（天级）** | 每日全量/增量构建 | 24 小时 |

### 6.4 成本控制

成本构成与优化策略：

| 成本项 | 占比估算 | 优化手段 |
|--------|---------|---------|
| **Embedding API** | 15-25% | 批量调用、使用开源模型自托管、缓存重复文档 |
| **向量存储** | 20-35% | 降低向量维度、压缩索引、分层存储（热/冷数据） |
| **LLM 推理** | 30-40% | 控制注入上下文长度、使用更小的模型处理简单查询 |
| **计算资源** | 10-20% | 按需扩缩容、预留实例 |

**关键发现**：大多数团队在预算上的误差达 2-3 倍，因为他们只关注 Embedding API 费用，忽略了存储超支和重建索引的周期性成本。更小的 Chunking 策略可能产生 3-5 倍的向量数量，直接影响 Embedding 调用、存储和计算成本。

**中型系统月度成本参考**：$14,000-$27,000/月（含平台费 $8,000-$15,000 + 工程人力 $6,000-$12,000）。监管行业的合规治理成本额外增加 20-30%。

### 6.5 与现有 Agent 框架的集成方式

| 框架 | 集成方式 | 说明 |
|------|---------|------|
| **LangChain** | `Retriever` 接口 + `RetrievalQA` Chain | 最成熟的 RAG 生态，支持所有主流向量数据库 |
| **LlamaIndex** | 原生 RAG pipeline，`VectorStoreIndex` | 专为 RAG 设计，数据连接器丰富 |
| **Semantic Kernel** | `Memory` + `TextMemoryPlugin` | 微软生态，与 Azure 深度集成 |
| **自定义 Agent** | RAG 封装为 Tool | 最灵活，适合有特殊需求的团队 |

**集成最佳实践**：
- 将 RAG 封装为标准的 Tool/Function，遵循框架的工具调用协议
- 在工具描述中清晰说明 RAG 的能力边界，帮助 Agent 做出正确的工具选择
- 返回结果附带元数据（来源、相关性分数），方便 Agent 判断可信度
- 支持流式返回，避免大量检索结果阻塞 Agent 主循环

## 七、总结与趋势

### 2026 年 RAG 技术关键趋势

1. **Agentic RAG 成为标配**：从静态检索向 Agent 驱动的动态检索演进，Agent 自主决策何时、如何检索
2. **混合检索是生产标准**：Dense + Sparse + Full-text 三路混合 + Re-ranking 成为高质量 RAG 的标配
3. **评估驱动开发**：RAGAS 等评估框架成为 RAG 开发的必备工具，无评估不上线
4. **Graph RAG 兴起**：结合知识图谱的 RAG 在跨文档推理和主题发现方面表现突出
5. **成本意识增强**：企业开始精细化管理 RAG 系统的 TCO，分层存储和模型分级成为常见优化手段
6. **开源 Embedding 追赶**：Qwen3-Embedding、BGE-M3 等开源模型在 MTEB 上已接近甚至超越商业模型

### 技术选型速查

| 需求 | 推荐方案 |
|------|---------|
| 快速原型 | Chroma + text-embedding-3-large + 递归切分 512 tokens |
| 生产部署（云端） | Pinecone/Qdrant Cloud + Cohere embed-v4 + 混合检索 + Re-ranking |
| 生产部署（本地） | Milvus/Qdrant + BGE-M3 + BM25 混合检索 |
| 多语言场景 | BGE-M3 或 Cohere embed-v4 |
| 预算敏感 | pgvector + Nomic embed-text-v1.5 |
| 复杂推理 | Agentic RAG + Multi-Agent 架构 |
